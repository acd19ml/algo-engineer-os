# Runbook Entry Template

`WORK/runbooks/<domain>/runbook.md` 里单条 runbook 条目的形态。

症状导向：读者带着"我看到了什么问题"来查，每条给出可立即操作的**下一步**。**不是教科书也不是设计文档**——因果的"为什么"住在 KNOWLEDGE 节点，本条目只放**可操作面 + 回链**。

---

## 两件先决判断（写条目前先回答，决定整条形态）

### 1. 动作是哪种形态？ → `type`

- **`remedy-menu`**（候选修法菜单）：N 个**相互独立**的修法，**按期望解决成本排序**（P(有效)×成本，最先该试的在前）。读者**可跳着试**。多数性能 / 质量类条目是这种。
- **`diagnostic-procedure`**（诊断流程）：一串**定位关卡**，每步是一个测试，**测试结果决定下一步**；"修法"是定位到的故障所要求的动作。顺序是**拓扑的（上游→下游），不是成本的**，且**不允许跳步**——跳步会把上游故障伪装成下游故障。

> **判据**：各步相互独立、谁先谁后只看成本 → `menu`。某步的结果决定走向、跳步会误导 → `procedure`。
> **反例信号**：如果你发现要在正文写"必须按顺序、跳步会怎样"——说明你在写 procedure，却套了 menu 的壳。把它声明成 `type: diagnostic-procedure`，那句散文补丁就不用写了。

### 2. blast radius 是什么？ → `severity`

- **`reversible-cheap`**：失败可重试、代价小（绝大多数）。动作照"最便宜先试"走。
- **`irreversible`**：动真实系统 / 有副作用 / 安全相关。动作的**第一步永远是先止血**（断权限 / 回滚 / 硬中断），**再**诊断。不许"先试最便宜的修法看看"。

## 字段

```markdown
## P-XXX：<一句话症状概述>

`type: remedy-menu | diagnostic-procedure`   `severity: reversible-cheap | irreversible`

**症状**
- 我能观察到的现象（按"三个月后我会搜什么关键词"反向写，描述现象、不解释机制）
- 在哪种环境 / 前提下更频繁
- （可选）我此刻最可能的错误第一反应——它本身是个搜索钩子

**根因**
为什么会发生，一段话。定量数据（百分比 / token / 延迟）放这里。
**只写定位所需的因果，不复述机制**——完整的"为什么"在 `源` 指向的节点里。超过两三句、或开始讲背景理论，就是密度漏出来了：删掉，回链。
（procedure 例外：若某句因果是"为什么不许跳这一步"的依据，它该留——它在为 type 的 no-skip 语义辩护。）

**前置条件**（可选；procedure 常需要）
执行下面的流程需要先有什么（如：能拿到失败样本、能读到实际 chunk、自动化 eval harness）。没有前置条件，流程跑不动——别让读者走到第三步才发现缺工具。

**当前方案**
> menu：按期望解决成本排，每条独立可操作。
> procedure：按上游→下游排，每关 = **测什么 + 结果怎么读 + 结果指向的修法**。不写"这一关为什么存在"（那在节点里）。

*方案 1 / 关卡 1：<一行总结>*
展开（具体操作 / 测试 / 代码片段）。

*方案 2 / 关卡 2：<一行总结>*
…

**演进日志**（可选；append-only，倒序）
- `YYYY-MM`：把 X 提为当前首选 ／ 用 Y 取代 X，因为 Z
- 历史方案放这里，**不要混进"当前方案"破坏 best-first 排序**。
- 当某个新方案在同一时段成为**多条**条目的新首选 → 先判定它是运行时恢复过程，还是让症状 / 有害后果不可能发生的结构性不变量；只有后者才沉淀成 design commitment pattern。

**适用与失效**
- 适用：什么情况这条有用（模型规模 / 推理后端 / 工作流长度 / 单轮 vs 多轮）
- 失效：什么情况不适用 / 会误导 → 链到正确条目或节点
（注意：这一栏只放"适用/失效"。前置条件去上面那栏，前瞻预防去 design checklist。）

**源**（单一出处，本条的**动作**从哪蒸馏来）
`[[node-name]]`，或跨节点综合时显式写 `synthesized across [[a]] [[b]]`。
验证状态：未验证 / 部分验证 / 已验证。**部分验证必须写清没验证的是哪部分。**

**See also**（导航，可空）
`[[related-node]]` · `P-YYY 简述`。**不重复列 `源` 已有的节点。**
```

## 字段写作要点

| 字段 | 关键 | 反例 |
|---|---|---|
| `type` | 选错就得用散文打补丁 | menu 里写"必须按顺序、不许跳步"（这是 procedure）|
| `severity` | 决定第一步是"先试便宜的"还是"先止血" | 给 irreversible 条目排"最便宜先试" |
| 症状 | 查找入口，描述现象不解释机制 | "reflection 触发条件错"（这是根因）|
| 根因 | 只写定位所需的因果，≤2–3 句 | 把节点的机制叙述整段搬过来 |
| 前置条件 | 缺它流程跑不动，提前告知 | 把它埋进"适用与失效"|
| 当前方案 | menu 按成本 / procedure 按拓扑；都只放可操作面 | 在方案里解释"为什么该有这一关"|
| 演进日志 | 抓"运行时修法是否暴露出结构性不变量"那一刻 | 把历史方案混进当前菜单 |
| 适用与失效 | 防腐关键，少了它半年后照搬不适用的规则 | 留空 / "按情况判断" |
| 源 | 单一出处，让条目来源可追踪 | 源和 see also 重复列同一节点 |

## 薄 vs 厚

runbook 条目必须**薄**：动作 + 回链。密度（"为什么"的因果叙述）只活在节点里一份。

- procedure 合法地比 menu 厚一点，因为每关需要"怎么测 + 结果怎么读"。但**不包括**"这一关为什么存在"的因果理论——那回链到节点。
- **薄判据**：删掉条目里所有"为什么"句子后，每一步还能照着做吗？能 → 厚度合适；不能 → 你把节点内容塞进 runbook 了。

## 长度

floor 很低：简单 menu **5 行就够**（症状 + 一个方案 + 源）。`前置条件 / 演进日志 / see also` 都是可选的，复杂度按条目实际需要才长。不要为格式对称硬撑。

## 编号

`P-001` 全局递增，**不按 domain 重置**——将来拆子目录不冲突。

## 一个内容信号：前瞻预防内容要"毕业"

条目里出现"上线后持续监控 X ／ 为预防这类、设计时该 Y"这种**前瞻预防**内容——它不属于 runbook（反应式）。先判定它是否是结构性不变量；如果只是恢复过程或诊断步骤，留在 runbook。只有能让症状 / 有害后果不可能发生的不变量，才沉淀到 `WORK/design-commitment-patterns/`。在条目里留一句指过去，正文别展开。
