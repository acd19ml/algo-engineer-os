# Runbook Entry Template

WORK/runbooks/`<domain>`/runbook.md 里单条 runbook 条目的形态。

症状导向：读者带着"我看到了什么问题"来查，每条给出可立即操作的候选方案。**不是教科书也不是设计文档**。

---

## 字段（按顺序）

```markdown
## P-XXX：<一句话症状概述>

**症状**
- 我能观察到的现象 1
- 我能观察到的现象 2
- 在哪种环境/前提下更频繁

**根因**
为什么会发生。如果有定量数据（百分比、token 数、延迟）放在这里。一段话，不要 bullet。

**候选方案**

*方案 1：<一行总结>（默认先上 / 最便宜）*

展开说明。给出具体操作步骤或代码片段（可选）。

*方案 2：<一行总结>*

展开说明。

*方案 3：<一行总结>*

展开说明。

**适用与失效**
- 适用：什么情况这条有用（模型规模、推理后端、工作流长度等）
- 失效：什么情况这条不适用 / 会误导，链接到正确条目

**来源**
外部材料名 / 内部踩坑。验证状态：未验证 / 部分验证 / 已验证。

**关联**
P-XXX 简述；P-YYY 简述。链接到 KNOWLEDGE 节点用相对路径或 `[[node-name]]`。
```

---

## 字段写作要点

| 字段 | 关键 | 反例 |
|---|---|---|
| **症状** | 这是查找入口，按"三个月后我会用什么关键词搜"反向校准。描述现象，不解释机制 | "reflection step 触发条件错误"（这是根因不是症状）|
| **根因** | 一段话讲清机制 + 定量数据 | 重复症状描述 |
| **候选方案** | 按"便宜→贵"排序，最先该试的放最前。每条独立可操作 | 把多个方案揉成一段 |
| **适用与失效** | 这一栏是防腐烂的关键。少了它，半年后会照搬不适用的规则 | 留空 / 写"按情况判断" |
| **来源 + 验证状态** | 区分外部 claim 和自己验证过的 | 只写"来自某论文" |
| **关联** | 让查找形成网。即使目标条目还没写，写 `[[name]]` 也行——marks a TODO | 留空 |

## 长度

不是所有条目都需要 5 个方案。**简单问题 5 行就够**（症状 + 一个方案 + 来源）。不要为了格式对称把短条目硬撑长。

## 编号

P-001、P-002、... 全局递增。**不按 domain 重置**——这样将来拆子目录时编号不冲突。
