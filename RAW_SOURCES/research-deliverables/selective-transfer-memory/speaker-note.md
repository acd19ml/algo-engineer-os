10分钟英文汇报讲稿中文版

第1页：标题页

大家下午好。我们是 Li Mengxiao、Ye Hengrui、Jiang Yuchen 和 Cheng Zhaoyang。我们的项目题目是：记忆增强型大语言模型智能体中的选择性迁移。我们提出的核心问题很简单，但要求很严格：在固定经验预算下，记忆能否帮助结构匹配的目标任务，同时不伤害结构不匹配的任务？

第2页：目录

今天我们会先介绍问题背景，然后说明试点实验设置和工作流程，接着讲解一个诊断案例和两个修复方法，最后总结选择性迁移信号、结论和下一步工作。

第3页：团队分工与贡献

这一页展示了我们的分工。Li Mengxiao 主要负责研究框架和试点实验设置。Cheng Zhaoyang 负责工作流程和可观测性框架设计。Jiang Yuchen 负责诊断分析和匹配修复。Ye Hengrui 负责算子抽象、选择性分析和最终总结。

第4页：研究框架

我们的研究动机是：只看平均 benchmark 提升，并不足以评价记忆是否真正有效。模型可能只是因为多了一段上下文而表现更好，但这段记忆并不一定匹配目标任务的推理结构。同时，粗粒度的相关性标签也可能造成误判，因为从大类别上看相关的记忆，在具体算子层面可能仍然是不匹配的。

因此，我们设计了一个更严格的测试。我们在相同经验预算和相同 prompt 框架下，比较匹配条件和不匹配条件。在这种设置下，真正的选择性迁移信号应该表现为：匹配路径提升，而不匹配控制组保持不变。如果两者都提升，那可能只是一般性的上下文效果。如果一个原本被标为匹配的样本在细查后失败，那更可能说明这是评估标签的问题，而不是真正的记忆失败。

所以，我们的试点实验重点不是单纯追求平均分提升，而是构建可解释的、案例级别的证据。

第5页：实验设置

我们的试点实验使用从 HotpotQA 源任务到 2WikiMultiHopQA 目标任务的近迁移问答流程。源任务用于构建记忆材料，然后这些记忆会被提供给模型，用来回答目标问题。

我们比较三种记忆条件：无记忆、情节式轨迹记忆和跨情节整合记忆。无记忆条件下，模型只看到当前上下文和问题。情节式轨迹记忆会分别提供五个源任务经验。跨情节整合记忆则会把同样的五个源任务压缩成一个抽象经验。

为了保证比较公平，我们固定了关键条件：模型使用本地 Qwen3.5-9B，解码方式使用 greedy decoding，每个条件都获得相同的五个已解决源任务作为经验预算。我们还并列报告匹配和不匹配源任务划分。这样，当结果出现差异时，我们可以把差异解释为记忆形式或结构匹配带来的影响，而不是模型能力或数据预算不同造成的影响。

接下来进入工作流程和可观测性框架。

第6页：工作流程与可观测性

我们的工作流程是为了让整个修复链条都可以被审计。

流程从采样、标注和重新审计开始。然后我们生成固定的记忆材料，使用共享结构化 prompt 框架运行试点实验，构建角色感知的聚合视图，挖掘 relation-chain bridge 案例，最后进行重路由、修复，并把发现更新回解释层。

一个关键设计是右侧的共享 prompt 框架。在所有条件下，模型都会看到相同的部分：Context、Question、Past Experience、Instructions、Reasoning 和 Final Answer。这样做可以让输出更容易解析和评分，也能让模型如何使用记忆变得可观察。

基于这个框架，我们追踪两类指标。结果指标包括 Exact Match、token-level F1，以及匹配和不匹配条件的比较。过程指标包括推理过程是否可见、最终答案是否可解析，以及过去经验是被明确使用、拒绝，还是忽略。

这很重要，因为它让我们能够区分迁移效果和工具性因素。如果结果发生变化，我们可以进一步判断原因是真正的结构迁移、格式问题、案例选择问题，还是模型与记忆交互的问题。

第7页：诊断案例：wiki_dev_2639

这个目标问题问的是：Harriet Pelham-Holles, Duchess of Newcastle-upon-Tyne 的 sibling-in-law 是谁？标准答案是 Henry Pelham。为了正确回答，模型需要走出一条关系路径，比如 wife，然后 husband，再到 brother。

在最初的试点结果中，无记忆 baseline 是正确的，但两个相关记忆条件都失败了。表面上看，这似乎说明记忆造成了负面影响：baseline 答对了，而相关记忆路径反而降到了 0。

但是，真正隐藏的问题在右侧。这个目标实际上是一个 relation-chain bridge 案例，而原始源记忆只是按照粗粒度 bridge 标签被标为相关。也就是说，源任务从大类别上看相关，但它没有保留目标所需要的算子结构。原来的 matched 标签其实比我们想象中更弱。

因此，这个案例成为我们的核心诊断案例。它说明，一个看起来像负迁移的结果，实际上可能来自目标推理路径和源任务匹配粒度之间的不一致。

第8页：修复一：匹配粒度

我们的第一个修复只改变 matched source set，其他条件都不变。在重路由之前，匹配源任务来自 attribute-bridge 集合。重路由之后，匹配源任务来自 relation-chain 集合，并且具有和目标任务相同的亲属关系算子模式。

重要的是，我们固定了目标问题、模型、prompt 框架、评分方式和不匹配控制组。所以这是一个受控修复。

右侧展示了结果。重路由之后，matched episodic 结果从 0 恢复到 1，而 mismatched control 仍然保持为 0。这是修复一的关键量化结果。

所以，这里的结论不只是“记忆有帮助”。更强的结论是：匹配粒度本身就是结果的一部分。粗粒度 bridge 标签对这个案例来说太宽泛了；只有当源任务保留了正确的算子模式时，情节式记忆路径才恢复正确。

第9页：修复二：可执行算子抽象

在第一个修复之后，matched episodic 路径恢复了，但 matched consolidation 路径仍然是错误的。所以第二个问题是：为什么抽象记忆仍然失败？

原因是之前的抽象太偏描述性。它提到了如何思考 spouse branches，但没有以模型可以一步步执行的形式保留算子结构。

因此，在修复二中，我们改变了 Past Experience block 中的 heuristic。我们不再只给描述性指导，而是把 sibling-in-law 规范化为有顺序的算子路径，例如 spouse-of then sibling，或者 sibling then spouse-of。

量化结果也很清楚。matched consolidation 结果从 0 变成 1，而 mismatched control 仍然保持为 0。这说明抽象记忆本身并不够，抽象必须以可执行形式保留算子结构。

第10页：选择性信号与过程信号

这一页总结了我们的证据。

左侧展示了 anchor case 上的最小选择性检查。在最初的粗粒度匹配设置中，两个 matched memory paths 都是 0。经过源任务重路由之后，matched episodic path 变成 1，而两个 mismatched controls 仍然是 0。经过算子修复之后，matched consolidation path 也变成 1，并且 mismatched controls 依旧保持为 0。

这给出了一个受控的选择性模式：匹配路径提升，不匹配控制组保持不变。

右侧展示了另一个目标案例 wiki_dev_8896 的过程层面例子。在这个案例中，最终答案可能保持不变，但推理轨迹不同。相关记忆被明确使用，而不相关记忆被明确拒绝。这一点很重要，因为它说明我们的框架不仅能捕捉最终任务效果，也能捕捉模型的记忆行为。

所以，我们并不是声称已经证明了大规模平均提升。我们的主张更窄，也更严谨：在受控比较下，我们可以暴露选择性迁移，并把它和一般上下文效果区分开。

第11页：结论与下一步

我们最后总结三点。

第一，只有当匹配和不匹配条件共享相同预算和相同 prompt 框架时，选择性迁移才会变得可见。第二，匹配粒度很重要：粗粒度相关标签可能掩盖目标案例真正的推理结构。第三，整合记忆必须以可执行形式保留算子结构，尤其是在亲属关系这类推理路径中。

同时，我们也保持结论边界。当前试点中最强的修复证据仍然来自一个核心诊断案例，所有实验也都使用同一个 Qwen3.5-9B 设置。因此，下一步应该在更多 relation-chain targets 上进行更大规模的修复后评估，并结合适用性判断和记忆形式设计。

我们的核心信息是：在问“记忆平均上是否有帮助”之前，我们必须先让相关性、匹配关系和抽象形式，都能服务于目标推理路径。谢谢大家，欢迎提问。

5分钟问答准备

1. 为什么你们选择自己的 proposal，而不是老师提供的 proposal？

我们选择这个 proposal，是因为它给了我们一个更清晰的因果问题，并且有可测量的输出。我们不是简单问“记忆平均上有没有帮助”，而是问“记忆是否能帮助结构匹配的任务，同时不伤害结构不匹配的任务”。这让我们可以构建匹配与不匹配的受控比较，观察推理轨迹，并形成一条可审计的修复链。因此，这个项目产出的不是单纯的现象描述，而是一个可以被测试的机制。

2. 你们最核心的量化结果是什么？

最清楚的量化结果来自 anchor diagnostic case。经过 matched source set 重路由之后，matched episodic 结果从 0 提升到 1，而 mismatched control 仍然是 0。经过 operator repair 之后，matched consolidation 结果也从 0 提升到 1，而 mismatched control 仍然是 0。所以我们的主要信号不是整体提升，而是只有 matched path 出现受控恢复。

3. 为什么平均提升不够？

因为平均提升不能说明记忆是否因为正确原因发挥作用。额外上下文可能提升表现，即使这段记忆并不匹配目标结构。同时，粗粒度相关标签也可能隐藏真正的不匹配。因此，平均提升会混合真实迁移、一般上下文收益和评估标签误差。我们的设计通过相同预算和相同框架下的匹配/不匹配比较，把这些因素区分开。

4. 你们如何定义 matched 和 mismatched？

Matched 指源记忆保留了目标任务所需的结构模式。Mismatched 指源记忆没有保留该模式，即使它看起来在大类别上相关。我们的诊断案例说明，这种区分必须到算子层面，而不能只停留在类别层面。一个粗粒度 bridge 标签是不够的，源任务还必须保留 relation-chain 结构。

5. 你们比较中固定了哪些变量？

我们固定了模型、解码策略、经验预算、prompt 框架、目标问题和评分规则。在修复一中，我们只改变 matched source routing。在修复二中，我们只改变 Past Experience block 中的抽象形式。所以我们把提升解释为这些具体修复带来的结果。

6. 你们怎么知道修复一真的造成了提升？

因为修复一只改变 matched source set。目标问题、模型、prompt 框架、评分方式和 mismatched control 都保持不变。这个改变之后，matched episodic path 从 0 变成 1，而 mismatched path 仍然是 0。这是一个受控的前后对比。

7. 你们怎么知道修复二不只是一次偶然有效的 prompt tweak？

我们的解释比“prompt tweak 有效”更窄。真正重要的变化是从描述性抽象变成可执行算子抽象。matched consolidation path 从 0 变成 1，而 mismatched control 仍然是 0。所以证据支持的是一个具体说法：抽象必须以模型能够执行的形式保留算子结构。

8. 为什么使用 Qwen3.5-9B 和 greedy decoding？

我们想要一个稳定、可控的试点设置。使用单一本地模型和 greedy decoding 可以减少 sampling 带来的随机性，让比较更干净。当然，这也限制了泛化性，我们也明确说明了这一点。当前结论只针对一个模型上的受控试点，而不是跨模型的普遍结论。

9. 为什么只用五个源任务和六个目标案例？是不是太少了？

是的，如果要做大规模性能结论，这个规模确实太小。但我们并不是把它作为大规模 benchmark 结果，而是作为诊断性 pilot。小而固定的经验预算可以迫使记忆设计体现选择性，案例集的作用是检验评估逻辑是否有效。下一步应当把修复后的设置扩展到更多 relation-chain targets。

10. 一个 anchor diagnostic case 足够支撑你们的结论吗？

足够支撑一个有边界的结论，但不足以支撑广泛结论。我们最强的证据来自一个 anchor case，所以我们不声称在整个 benchmark 上有普遍性能提升。我们声称的是：这个 pilot 暴露了一个真实的评估问题，并且两个受控修复让 matched performance 恢复，同时 mismatched controls 保持不变。因此下一步是扩展实验，而不是过度声称。

11. 你们的结论为什么是严谨的，而不是描述性的？

严谨性来自受控比较和明确边界。我们固定关键变量，在每次修复中只改变一个因素，并且并列追踪 matched 和 mismatched controls，同时使用结果指标和过程指标。我们也清楚说明这个 pilot 不能证明什么。因此结论是具体的、受控的、有边界的。

12. 如果最终准确率才是主要目标，为什么过程指标重要？

因为最终准确率不能告诉我们记忆是否被正确使用。过程指标可以显示相关记忆是否被明确使用、不相关记忆是否被拒绝，以及输出是否可解析和可评分。这有助于区分真实迁移、格式问题和偶然答对。

13. 你们项目的新意是什么？

新意在于三个方面的结合：选择性迁移的研究框架、固定经验预算下的 matched-versus-mismatched 评估设计，以及一条从诊断分析到具体记忆修复的可审计链条。换句话说，我们不只是问“记忆有没有帮助”，而是问评估能否区分结构迁移和一般上下文效果。

14. 当前研究最大的局限是什么？

最大的局限是规模。最强的修复证据集中在一个 anchor diagnostic case 上，所有实验也都使用同一个模型配置。因此，我们把它定位为诊断性试点，而不是最终 benchmark 结论。

15. 如果有更多时间，你们下一步实验会做什么？

最明确的下一步是在更多 relation-chain targets 上进行更大规模的修复后评估，使用修正后的匹配逻辑和可执行抽象。之后，我们会测试这种模式是否能跨模型成立，并探索是否应该把 applicability judgment 和 memory-form design 联合学习。

16. 提升会不会只是因为给模型更多文本？

我们的证据说明，这不是完整解释。所有记忆条件都使用相同的 prompt 框架，而关键信号是 matched paths 提升，mismatched controls 保持不变。如果只是“更多文本有帮助”，我们应该会看到更广泛的提升，包括 mismatched controls 也提升。

17. 为什么选择 HotpotQA 到 2WikiMultiHopQA？

我们选择这组任务，是因为它们都涉及 multi-hop QA，所以足够接近，可以进行受控迁移；但它们又有差异，使结构匹配并不是自动成立的。这让它们适合作为 near-transfer pilot，用来测试记忆是否因为正确的结构原因发挥作用。

18. 如果老师要求更多细节，应该怎么回答？

我们可以打开 appendix 或 report，展示详细的 audit trail，包括案例标注、source rerouting 逻辑，以及具体的 repair comparison。
