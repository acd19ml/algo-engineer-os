今天,聊一个每个agent的开发者都要做,但大部分人做得很粗糙的东西,系统提示词。你可能觉得系统提示词就是开头写一段你是一个什么什么助手,然后加几条规则就完事了。Cloud Code的系统提示词有900多行。打开圆码你会发现,这不是一段文本,而是一个由十几个模块化段落拼接而成的工程系统。每个段落解决一个不同的问题,有些段落是所有用户共享的,有些是每个绘画动态生成的,中间还有一个明确的分界标记来控制缓存策略。 这是CloudCode圆码系列的第八期。今天我挑五个最有启发性的设计来讲。第一个设计,静态段和动态段的分离。CloudCode系统提示词被一个边界标记分成了两半。前半段是静态内容,身份介绍,安全指令,代码风格规则,工具使用指南,行动风险评估框架。这些内容对所有用户,所有绘画都是一样的。后半段是动态内容。 当前绘画启用了哪些工具记忆系统的索引内容工作目录和操作系统信息用户的语言偏好连接的外部插件的使用说明这些内容每个绘画都不同为什么要这样分因为缓存大模型的API有一个提示词缓存机制如果两次请求的提示词前缀一样服务端可以复用之前的计算结果省钱省时间如果你把动态内容和静态内容混在一起每个用户的提示词都不一样缓存命中率就很低Cloud Code的做法是把所有用户通用的内容放在前半段 所有绘画特有的内容放在后半段,这样前半段可以跨用户共享缓存。你和另一个用户用的是同一个cloud code,你们的系统提示词前半段完全一样,服务端只需要计算一次。这个设计对初学者的启发是,如果你的agent有多个用户,把系统提示词里所有用户通用的规则和当前用户特有的信息分开放,通用规则放前面,特有信息放后面。这一个简单的调整就能显著提高缓存命中率,降低成本。第二个设计,代码风格指令极其具体。 大部分人写系统提示词里的代码规则,写的是请写高质量的代码或者遵循最佳实践。这种指令等于没写,因为模型不知道你说的高质量具体是什么意思。Cloud Code的做法完全不同,它的代码风格指令是一条一条非常具体的规则。比如,不要给你没改过的代码,加注式,文档,字符串,或类型标注,只在逻辑不自明的地方加注式。比如,不要为假设的未来需求做设计,不要给一次性操作创建工具函数,三行相似的代码比一个过早的抽象好。 比如,不要加你没被要求加的功能。一个bug修复不需要顺便清理周围的代码。一个简单功能不需要额外的可配置性。比如,不要加错误处理,降级方案或校验逻辑来应对不可能发生的场景。信任内部代码和框架的保证只在系统边界做校验。每一条都是具体的,可执行的,有明确边界的。模型看到这些规则,知道该做什么,不该做什么。这个设计的启发是,你的系统提示词里的规则越具体越好。不要写,请简洁。 写工具调用之间的文本不超过25个词,最终回复不超过100个词。不要写写好代码,写不要给你没改的代码加注释。模型不怕规则多,怕规则模糊。第三个设计,行动风险评估框架。Cloud Code的系统提示词里,有一整段专门讲什么操作需要确认,什么操作可以直接做。它不是简单地列一个危险操作清单,而是给了模型一个判断框架。核心思路是两个维度,可逆性和影响范围。 本地的可逆的操作,比如编辑文件,跑测试,可以自由执行,不可逆的影响共享系统的操作,比如强制推送代码,删除分支,发送消息,修改基础设施权限,必须先确认。然后他列了一系列具体的例子,删除文件和分支,删除数据库表,强制推送,修改已发布的提交,发送消息到外部服务,上传内容到第三方工具。更有意思的是,他还教模型怎么处理意外状态。 如果发现了不认识的文件、分支或配置,先调查再行动。不要直接删除或覆盖,因为那可能是用户正在进行的工作。如果遇到锁文件,先查是什么进程持有的,不要直接删掉。这个框架的价值在于,它不是一个死板的规则列表,而是教模型一种思维方式。模型遇到新的,没见过的操作时,可以用这个框架自己判断该不该确认。第四个设计,安全指令由专门团队维护。原码里有一段叫Cyber Risk Instruction的安全指令,它定义了模型在安全相关请求上的行为边界。 可以协助授权的安全测试、防御性安全、CTF挑战和教育场景,但拒绝破坏性技术、拒绝服务攻击、大规模目标攻击、供应链攻击和恶意目的的检测规避。这段指令本身不长,但圆马注释非常引人注目。注释里写了这段指令的负责人名字,明确要求任何修改必须经过安全团队审批和评估,因为对这段文本的修改会对模型处理安全请求的方式产生重大影响。这个设计的启发不是技术层面的,而是工程管理层面 系统提示词不是一个随便谁都能改的文本文件。它是你的agent的行为规范。某些段落的修改可能带来严重的安全后果。在团队协作中,关键的提示词段落应该有明确的负责人和审批流程,就像关键代码需要code review一样。第五个设计,防止模型说谎。这是我在原码里看到的最务实的一段提示词。它直接告诉模型,如实报告结果。具体怎么写的呢?如果测试失败了,就说失败了,附上相关输出。如果你没有执行验证步骤, 就说你没执行,不要暗示它成功了。永远不要在输出显示失败的时候声称所有测试通过。永远不要压缩或简化失败的检查来制造一个绿色的结果。永远不要把未完成或有问题的工作说成已完成。但它不只是防止说谎,还防止过度谨慎。后面紧跟着一句,同样的,当检查确实通过了,或者任务确实完成了,就直接说。不要对已确认的结果加不必要的免责声明。不要把已完成的工作降级成部分完成。不要重新验证你已经检查过的东西。 目标是准确的报告,不是防御性的报告。圆马柱是说,这段是针对特定模型版本接近30%的虚假声明率加的。也就是说,没有这段提示词的时候,模型大约每三次就有一次会在报告结果时说谎或夸大。这个设计的启发是,如果你的agent需要报告执行结果,你必须在系统提示词里明确告诉他如实报告。而且,要从两个方向约束,既不能把失败说成成功,也不能把成功说得像失败。模型的默认倾向是讨好用户,所以它会倾向于报告好消息。 你需要用明确的指令来对抗这个倾向。五个设计讲完了,回过头来看整个系统提示词的架构。900多行,十几个模块,静态段和动态段分离,每个段落解决一个具体问题。身份介绍,告诉模型它是谁。安全指令,画出不可逾越的底线。代码风格规则,用具体的条目替代模糊的写好代码。行动风险框架,教模型怎么判断该不该确认。防说谎指令,对抗模型,讨好用户的默认倾向。这些段落不是一次写好的。 远马注释里到处都是针对某个模型版本加的,某个PR引入的,某次评估发现的问题。系统提示词是一个持续演进的工程产物,每一条规则背后都有一个真实的生产问题。如果你在给自己的Agent写系统提示词,从Cloud Code的设计里可以带走三个原则。第一,模块化。不要写一大段文本,把不同职责的内容拆成独立的段落。身份是一段,规则是一段,工具指南是一段,环境信息是一段。这样你可以独立修改每个段落,也可以根据场景动态组合。第二,具体化。 每一条规则都要具体到模型能直接执行。写好代码不是规则,不要给你没改的代码加注释才是规则。第三,分离缓存。把所有用户通用的内容放前面,绘画特有的内容放后面,这一个调整就能让你的API成本降下来。一句话总结,系统提示词不是一段介绍文字,是你的agent的操作系统。它定义了agent的身份、能力、边界、行为规范和风险判断框架。写得越精确,agent的行为就越可行。
