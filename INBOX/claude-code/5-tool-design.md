今天聊一个大部分人忽略的话题,agent的工具设计。你可能觉得工具设计没什么好讲的,不就是定义一个函数,写个描述,让模型调用吗?如果你这么想,说明你还没踩过坑。我先描述一个场景,看你有没有遇到过。你给agent接了一个命令行工具,让他可以执行任意命令。然后,你又接了一个文件读取工具,一个搜索工具。你觉得分工很清楚,读文件用文件工具,搜索用搜索工具,跑命令用命令行工具。结果呢,模型拿到命令行工具之后,什么都用它来做。 用命令行跑cat搜索用命令行跑grab猎目录用命令行跑ls你精心设计的专用工具模型根本不用这不是模型笨恰恰相反这是模型太聪明了他发现命令行工具什么都能做那为什么还要学其他工具的参数格式呢用一个万能工具解决所有问题对模型来说是最省力的策略但对你来说这是灾难因为命令行工具的输出是无结构的纯文本你没法做格式化展示命令行工具的权限是全开的你没法做细利度的安全控制 命令行工具的输出大小是不可预测的。 这是我在原码里看到的最反直觉的设计。Cloud Code的命令行工具,也就是Bash工具。它的使用说明里有一段非常醒目的文字。大意是,不要用这个工具来执行读文件,搜索,列目录这些操作。除非你已经确认专用工具无法完成任务。然后列了一个工具偏号表。文件搜索用glob工具,内容搜索用grab工具,读文件用read工具,编辑文件用edit工具,写文件用write工具。 一个工具的使用说明里花了大量篇幅告诉模型别用我去用别的工具。为什么要这样设计?因为万能工具是安全和可控性的天敌。你给模型一个能做一切的工具,它就会用这个工具做一切。你的权限系统被绕过了,因为所有操作都走同一个工具入口。你的输出管理被绕过了,因为命令行输出是无结构的。你的UI展示被绕过了,因为你没法区分这是一个读操作还是一个写操作。专用工具的价值不在于它能做什么,而在于它把操作限制在一个可控的范围内。 read工具只能读文件,所以你可以精确控制它的输出格式,做文件缓存去重,做token预算检查。edit工具只能编辑文件,所以你可以做差异展示,做路径保护,做权限细分,但模型不会自动理解这些。模型的默认策略是用最少的工具完成最多的事。所以,你必须在万能工具的说明里显示地引导它。这些事情有更好的工具去用那些工具。这个原则对初学者的启发是什么?如果你的agent有一个功能很强大的通用工具, 你一定要在它的描述里告诉模型哪些场景应该用其他工具。不是靠模型自己判断,是你显示的写出来。否则模型会把所有操作都塞进这一个工具里。你后面做的所有精细化管理都白费了。第二个原则,工具不要一股脑全给。按需发现,Cloud Code有40多个工具。如果把它们全部塞进系统提示词,光是工具定义就要占掉大量的上下文空间。而且模型面对太多工具的时候,选择准确率会明显下降。 之前我在讲Multi-Agent那期提到过,Manus团队发现给模型100个以上的工具会导致幻觉参数或调用错误。Cloud Code的解法不是拆成多个Agent,而是把工具分成两集。第一集是核心工具,启动时就加载,模型立刻可用。这些是最高频的工具,命令行,文件读取,文件编辑,文件写入,搜索,Agent工具,大概七八个。第二集是延迟工具,模型只知道它们的名字,不知道参数格式,所以没法直接调用。 网页搜索,网页抓取,笔记本编辑,任务管理,团队协作,这些都是延迟工具。当模型需要用一个延迟工具的时候,它先调用一个叫to search的工具。to search就像是工具的搜索引擎。模型传入关键词,to search返回匹配的工具的完整定义,包括参数格式。拿到定义之后,模型就能正常调用了。to search的搜索算法也很讲究。每个工具除了名字和完整的使用说明之外,还有一个叫search hint的字段。 3-10个词的能力短语,比如文件读取工具的Search Hint是读取文件、图片、PDF、笔记本。搜索工具的是用正则搜索文件内容。这些短语专门为关键词匹配优化,用的词尽量不和工具名重复,覆盖模型可能搜索的各种表述。搜索的时候,工具名精确匹配权重最高,Search Hint匹配次质,完整使用说明里的关键词匹配权重最低。这样,模型用简短的关键词就能快速找到需要的工具。 这个两极加载还有一个很实际的好处缓存稳定性。原码里的工具排序逻辑把内置工具排在前面作为连续前缀,外部插件工具排在后面。这样即使插件工具变化了,内置工具的前缀不变,服务端的提示词缓存不会失效。如果把所有工具混在一起排序,每次插件变化都会打乱前缀,缓存就废了。这个原则对初学者的启发是什么?如果你的agent工具超过10个,就应该考虑分级加载了。核心工具直接给模型,其余的按需发现。 你不需要实现一个完整的to search。最简单的做法是在系统提示词里列出所有工具的名字和一句话描述。然后告诉模型,如果你需要用某个工具,先告诉我,我会给你它的完整定义。这比把所有工具定义一股脑塞进去,效果好得多。第三个原则,工具输出要有预算。这个原则直接呼应上期讲的上下文工程。上期讲的是对话历史怎么压缩,这期讲的是工具输出怎么控制。Cloud Code里每个工具都有一个输出大小上限,大部分工具是十万字符。 搜索工具更严格,只有2万字符。文件读取工具比较特殊,它的上限是无穷大。原因我一会儿说,当工具输出超过上限的时候,系统不会直接截断。截断意味着信息丢失,模型可能因为看不到关键信息而做出错误判断。它的做法是把完整输出存到磁盘文件,然后给模型一个预览,加一个文件路径。预览是输出的前几千字节,让模型大致了解内容。如果模型需要完整内容,可以用文件读取工具去读那个文件。 为什么文件读取工具的上限是无穷大?因为如果文件读取工具的输出也被存到磁盘,模型读回来的时候又会触发存储,形成一个死循环。所以文件读取工具用另一种方式控制输出。它有自己的token预算检查,超过预算直接报错,告诉模型用偏移和行数参数来分段读取。搜索工具还有一个额外的保护。它的搜索结果默认限制在250条以内。圆马注释说得很直白,无界搜索可以轻松填满上下文。 250条对于探索性搜索已经足够了。如果模型确实需要更多,可以显示传入一个参数来解除限制。这个原则对初学者的启发是什么?你的每一个工具都应该有输出预算,不要假设工具输出是可控的。一个数据库查询可能返回几万行,一个日志搜索可能返回几十万字符。如果你不控制输出大小,一次工具调用就能把整个上下文窗口称爆,后面的所有推理都会受影响。最简单的做法是设一个字符数上线,超过就阶段并附上提示输出以阶段,共多少条结果。 更好的做法是像Cloud Code一样,存到外部,给摘要加纸针。三个原则讲完了,回过头来看,你会发现这三个原则指向同一个核心认知,工具设计本质上也是上下文工程。第一个原则,在万能工具里写别用我,是在管理模型的注意力引导模型走受控路径,而不是让它自由发挥。第二个原则,工具按需发现是在管理上下文空间,不把不需要的工具定义塞进上下文,把空间留给真正有用的信息。第三个原则, 工具输出有预算是在管理上下文质量防止一次工具调用的输出淹没其他所有信息上期我说过上下文窗口是稀缺资源关键不是装不装得下而是该不该装进去工具设计是这个原则的另一个战场你的工具定义占多少上下文你的工具输出占多少上下文你的工具引导模型走什么路径这些问题和对话历史怎么压缩记忆怎么提取是同一类问题一句话总结 好的工具设计不是给模型更多能力是给模型更清晰的边界边界越清晰模型在边界内的表现就越好
