今天聊Agent记忆系统里一个大部分人没想过的问题记忆由谁来写?前两期我们拆了Cloud Code记忆系统的存什么和怎么取但有一个问题一直没回答这些记忆文件是怎么产生的?你可能以为是用户手动告诉Agent记住这个Agent就写一条记忆确实有这种方式但它不是主要的如果记忆全靠用户手动触发大部分值得记住的东西都会被漏掉因为用户在对话的时候注意力在当前任务上不会时刻想着这条信息值得长期保存Cloud Code的做法是 让一个后台的agent自动帮你写记忆。原码里这个系统叫Extract Memories。每次对话结束的时候,也就是主模型给出最终回答。不再调用任何工具的时候,系统会在后台悄悄启动一个独立的子agent。这个子agent的唯一任务就是回顾刚才的对话,判断有没有值得长期保存的信息。如果有,就写入记忆文件。整个过程用户完全无感。不谈框,不确认,不打断。你继续用你的agent,后台的记忆提取,默默完成。 这个设计听起来简单,但圆马里的实现细节非常讲究。我挑四个最有启发性的设计来讲。第一个设计,权限锁死。这个后台的agent不是一个拥有全部能力的agent,它的权限被严格限制。它能做什么?读文件,搜索文件,用只读的命令行,命令查看目录和文件内容,以及在记忆目录里写入和编辑文件。它不能做什么?不能执行任何会产生副作用的命令。不能修改你的代码,不能启动子agent,不能调用任何外部服务。 原码里对命令行工具的限制特别细,不是简单的禁止使用命令行,而是只允许只读命令。查看目录,搜索文件内容,查看文件状态这些可以。任何涉及写入,重定项,修改状态的命令都会被拒绝。为什么要这样设计?因为记忆提取是一个高价值但高风险的操作。它在后台运行,用户看不到它在做什么。如果它有权限修改代码或执行命令,一个错误的判断就可能在用户不知情的情况下改坏项目。 把权限锁死在只读加记忆目录写入,确保了即使子agent犯错,最坏的结果也只是写了一条不太准确的记忆,不会影响你的代码。这个原则对初学者的启发是,如果你要做后台自动化的agent任务,一定要给他最小权限,不是他大概率不会做坏事,而是即使他想做坏事也做不了。第二个设计,两轮对话策略。这个子agent被限制在最多五轮对话内完成任务,但实际上,设计意图是两轮就搞定。第一轮,并行读取所有需要的文件。 它会读取现有的记忆文件列表,了解已经存了什么,避免写重复的内容。同时,读取最近的对话内容,找出值得保存的信息。第二轮,并行写入所有记忆文件,把提炼出来的记忆一次性写完。为什么要强调并行?因为这个子Agent共享主模型的提示词缓存。它和主模型看到的系统提示词,工具定义是完全一样的。如果它的对话轮次越少,缓存命中率就越高,成本就越低。两轮对话,第一轮全读,第二轮全写,是成本最优的策略。 原码注释里记录了实际的缓存命中数据。一次典型的记忆提取,缓存命中率可以达到很高的水平。意味着大部分输入Token都是从缓存读取的,几乎不产生额外的计算成本。五轮的硬上线是一个安全阀,正常情况下两轮就够了。但如果子Agent因为某种原因需要多读几个文件或者多改几条记忆,还有余量,超过五轮直接中止,防止它陷入无意义的循环。第三个设计,互斥机制。 原码里有一个很精妙的判断。如果主模型在对话过程中已经自己写了记忆文件,后台的记忆提取就直接跳过。什么意思?Cloud Code的主模型本身也有写记忆的能力。如果用户说记住我偏好用Tabs,主模型会直接写一条记忆,不需要等后台子Agent。这时候,如果后台子Agent再跑一遍,他会看到同样的对话内容,可能会写出一条重复的记忆。所以原码里做了护斥,检查最近的对话里有没有主模型写入记忆目录的操作。如果有,说明主模型已经处理了记忆写入。 后台子Agent就不需要再跑了。直接把游标推进到最新位置,下次只看新的对话内容。这个设计避免了重复劳动,也避免了主模型和子Agent对同一条信息产生两条不同措辞的记忆。第四个设计,游标和合并。后台子Agent不是每次都回顾整段对话历史,它维护一个游标,记录上次处理到了哪条消息,每次运行只看游标之后的新消息。这意味着如果你和Agent对话了二十轮,前十轮已经被提取过了,下次提取只看后十轮。 这大幅降低了每次提取的输入量和成本。但如果提取失败了呢?游标不会推进。下次提取会重新处理这些消息,确保不会因为一次失败就永久丢失值得保存的信息。只有成功完成提取之后,游标才会推进。还有一个合并机制,如果上一次提取还在运行,新的一轮对话又结束了,系统不会启动第二个并行的提取。它会把新的上下文暂存起来,等当前提取完成后,再用最新的上下文跑一次。 这样避免了多个提取子agent同时运行互相冲突的问题。而且圆马里还有一个节流机制,不是每轮对话结束都触发提取,而是可以配置间隔,比如每两轮或每三轮才触发一次。对于高频对话的场景,这可以显著降低后台的计算开销。回过头来看整个Extract Memories的设计,它解决的核心问题是让记忆的生产自动化,但不失控。自动化体现在,用户不需要手动告诉agent的什么该记,后台在agent每轮对话结束后自动分析和提取,不失控体现在四个层面。 这个设计对初学者的启发是什么?如果你想给自己的agent加自动记忆提取,最简单的实现方式是,每次对话结束后,把最近的对话内容发给一个便宜的模型,附上一段提示词,让它判断有没有值得长期保存的信息。如果有,写入一个记忆文件。 但你一定要做三件事。第一,限制这个提取过程的权限,只允许它读对话和写记忆文件,不允许它做任何其他事。第二,记录上次处理到哪里了,下次只看新的内容,不要每次都重新处理整段历史。第三,如果主模型已经自己写了记忆,就跳过自动提取,避免重复。这三件事做到了,你就有了一个基本可用的自动记忆生产系统。一句话总结,好的记忆生产不靠用户手动出发,靠后台自动提取。 但自动不等于不受控。权限要锁死。轮次要封顶。重复要护斥。进度要有游标。自动化和可控性缺一不可。
