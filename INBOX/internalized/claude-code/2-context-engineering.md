Cloud Code的上下文管理是整个Harness工程含量最高的部分。今天打开圆码,你会发现这句话还说清了。光Compact目录下就有15个文件,加起来超过15000行。上下文管理不是一个功能,它是一个完整的子系统。先说一个数据。圆码注释里,有一条来自2026年3月10日的生产数据。1279个绘画出现了50次以上的连续自动压缩失败。最多的一个绘画失败了3272次。全局每天浪费大约25万次API调用。修复方案是三行代码。 连续失败超过三次就停止重视。这个细节说明了什么?上下文管理不是一个设计好就完事的系统,它是一个在生产环境中持续踩坑,持续修补的战场。今天我们看到的每一个设计决策,背后都有真实的生产事故。好,进入正题,Cloud Code的上下文管理分成四层。我按照从轻到重的顺序来讲。第一层,微压缩。微压缩是成本最低的一层,不调用模型,纯规则驱动。它的核心逻辑是,按工具类型白名单。 保留最近N个工具结果,把更早的结果清掉。白名单包括File Read, Bash, Grab, Glob, Web Search, Web Fetch, File Edit, File Write。这些工具的输出通常很大,但时效性短。你10分钟前读的文件内容,大概率已经不需要了,但怎么清,有讲究。Anthropic的API有一个Prompt Cache机制,你发过去的Prompt前缀会被缓存在服务端。下次请求如果前缀一样,就不用重新处理,省钱省时间。 所以,微压缩在清理旧结果的时候,要考虑一个问题。我这一刀下去,会不会把服务端的缓存搞失效?原码里有两条路径来处理这个问题。第一条路径叫Cached Micro Compact。如果用户一直在连续对话,服务端缓存还在,微压缩就走这条精细路径。它不修改本地消息内容,而是通过Cache Editing API告诉服务端把这几个工具结果删了,但不动Prompt前缀,缓存继续有效。 它维护一个全局状态,追踪所有已注册的工具结果,按技术预值决定哪些该删。第二条路径叫Time-Based Micro Compact。如果用户离开了一段时间回来,服务端缓存早就过期了,那就没必要小心翼翼了。系统计算距离上一条助手消息的时间间隔,如果超过配置预值,就直接改消息内容,把旧的工具输出替换成一个站位符。这里有一个很细的工程细节。 CachedMicroCompact只对主线程生效。为什么?因为CloudCode会Fork出TheAgent来做绘画、记忆提取、Prompt建议等后台任务。如果这些TheAgent把自己的工具结果注册到全局状态里,主线程下次尝试删除的时候会去删一些在自己对话历史里根本不存在的工具。所以圆马里用QuerySource做了主线程隔离。第二层,绘画记忆压缩。这一层和微压缩有本质区别。微压缩是删东西,绘画记忆压缩是提炼东西。 它的核心思路是不要把对话做摘要,而是从对话中提取学到了什么。项目结构,用户偏好,任务进度,这些结构化的事实被持久化到本地的记忆目录里。原码里的实现先等待后台的记忆提取完成,然后读取Memory.md的内容,用这些结构化的记忆来替代传统的对话摘要。关键设计是保留多少最近的消息。原码里有三个配置参数,最小tokens数量默认1万,最小消息数量默认5条,最大token是数量默认4万。 系统从最后一条以总结的消息开始,往前扩展。直到同时满足最小Token数和最小消息数,或者触及最大Token上限。这里有一个非常精细的边界处理。扩展的时候,不能把Tool Use和Tool Result拆开。如果你保留了一条包含Tool Result的用户消息,就必须同时保留包含对应Tool Use的助手消息,否则API会报错。它还要处理Thinking Block的情况。流逝传输时,同一个Message ID可能被拆成多条消息。Thinking在一条,Tool Use在另一条,不能只保留后者。 绘画记忆压缩的优势是什么?它不需要调用模型来做摘要,直接用已经提取好的结构化记忆。成本比完整压缩低得多,而且保留了最近的原始消息。模型可以看到最近几轮对话的完整细节。第三层,完整压缩。当绘画记忆压缩不可用或者不够用的时候,系统回退到完整压缩。这一层需要调用模型。原码里的Prompt设计非常值得研究。它要求模型按九个维度来总结对话,用户的请求和意图,关键技术概念,文件和代码片段,错误和修复。 问题解决过程,所有用户消息,代办任务,当前工作,可选的下一步。但最精妙的是Analysis机制。Prompt要求模型先在Analysis标签里做详细的分析,然后再在Summary标签里输出最终摘要。Analysis部分在格式化的时候会被完全剥离,不会进入压缩后的上下文。它的作用纯粹是让模型在输出摘要之前先想清楚,提高摘要质量。 这本质上就是一个内嵌的chain of thought。还有一个很实际的问题,压缩请求本身可能触发prompt to long错误。对,你没听错,用来缩短上下文的压缩操作,本身的输入可能就超长了。原码里的解法是,按API轮次分组,从最早的组开始丢弃,最多重视三次。压缩完成后,系统还要做一系列善后工作。清空文件读取缓存,重新注入最近访问的文件内容。最多五个文件,总预算5万token。重新注入plan文件,skill内容,mcp工具说明,agent列表。 这些都是压缩过程中被丢掉的上下纹,需要在压缩后恢复。第四层,自动触发。前面三层都是压缩的,怎么做?这一层解决什么时候做?原码里的AutoCompact系统会在每次模型调用之后检查当前Token用量。如果超过了预值,上下纹窗口减去13000Token的缓冲,就自动触发压缩。触发顺序是,先尝试绘画机压缩。如果不可用或者压缩后仍然超标,再回退到完整压缩。这里有一个熔断器设计。 连续失败超过三次就停止重视,避免在上下文不可恢复的情况下,反复浪费API调用。就是我开头提到的那个25万次API调用的修复。还有一个很有意思的排除逻辑。Session Memory和Compact这两个Query Source的请求不会触发自动压缩。为什么?因为它们本身就是Fork出来的子Agent。如果它们也触发压缩,就会死锁。好,四层讲完了。但上下文工程不止压缩,工具系统也是上下文工程的一部分。Cloud Code内置了80多个工具。 但不是一开始就全部塞进上下文。核心工具启动时加载。扩展工具通过Tool Search按需发现和加载。每个工具用Zod Schema定义输入参数。模型输出的JSON必须通过验证才能执行。如果工具输出太大,系统不会直接截断,而是存到外部的Tool Result Storage,给模型一个摘要加一个指针。模型按需取用,而不是被迫消化一整块巨大的输出。还有一个叫Snip Compact的机制。 它可以在不做完整压缩的情况下从对话历史中减掉特定的消息段释放token空间这是一种比完整压缩更轻量的中间方案回过头来看整个上下文管理系统四层压缩从微压缩的零成本规则清理到绘画记忆的结构化事实提取到完整压缩的模型调用摘要到自动触发的熔断保护每一层都假设上一层可能不够用每一层都有自己的成本和适用场景这套系统最让我印象深刻的不是某一个单独的设计 而是它对上下文是稀缺资源这个认知的彻底贯彻。从微压缩区分缓存是否有效来决定清理策略,到绘画记忆压缩精确计算保留多少条消息,到完整压缩用analysis标签提高摘要质量,到自动触发的熔断器防止失控。每一个决策都在问同一个问题,这个token值不值得留在上下文里?这就是我之前讲的上下文工程的核心原则在圆满级别的体现。关键不是装不装得下,而是该不该装进去。一句话总结。 Cloud Code的上下文管理证明了一件事。Agent的上下文窗口不是一个被动的容器,是一个需要主动管理的稀缺资源。管理的好,Agent可以在长达数小时的绘画中保持连贯。管理的不好,25万次API调用就这么烧掉了。下一期我们拆权限与安全系统,看看YOLO Classifier的两阶段LLMS Judge架构在圆码里到底是怎么实现的。
