面试官问你,multi-agent的架构到底什么时候该用,什么时候不该用?你如果回答,任务复杂就用多agent?面试官一定会追问,怎么定义复杂,代价是什么?有没有不该用的情况?你说不清楚,这个问题就丢分了。今天我们从Anthropic,Clawed Code,Manus,OpenCode,OpenClaw这些前沿产品的实战经验里提炼出一套清晰的判断框架。先说最关键的一点,Anthropic的立场非常明确,大多数团队不需要multi-agent系统。 他们发现在单个agent上反复改进prompt,效果可以匹配那些花了几个月搭建的复杂多agent架构,而且代价很大。在anthropic的数据中,普通agent消耗大约4倍于聊天的token,而multiagent系统消耗大约15倍。这些额外开销来自跨agent的上下文复制、协调消息和结果汇总。所以multiagent需要任务价值足够高,高到能为这些开销买单。那什么时候该用三个场景?第一个,上下文污染。 不同子任务积累的信息互相干扰,导致推理质量下降。这时候,用独立的子Agent做上下文隔离。Manus的核心原则是通过通信来共享上下文,而不是通过共享上下文来通信。对于有明确输入输出的独立任务,启动全新的子Agent只传入具体指令。只有当子Agent必须理解完整轨迹时,比如需要看到之前所有错误尝试的调试Agent才共享完整上下文。 第二个,并行探索。任务可以拆成多个独立方向同时推进时,Multi-agent的效果是碾压性的。Anthropic的研究系统用多agent架构比单agent高出90%。为什么?因为多agent帮助投入了足够多的token来解决问题。在他们的分析中,token使用量本身就解释了80%的性能方差。Cloud的研究功能就是这么做的。主agent同时生成3到5个子agent,每个子agent又并行执行多个工具调用,对于复杂查询可以减少多达90%的研究时间。 第三个,工具过多。当Agent需要管理20个以上的工具时,选择准确率会明显下降。Manus发现给模型100个以上的工具会导致幻觉参数或调用错误。解决方案是按领域拆分给拥有聚焦工具级的专业化Agent。不过Cloud Code提供了一个替代思路,用一个工具去搜索工具,让模型按需动态发现工具,而不是一次性加载所有定义,也就是目前比较火的Skill渐进式加载的思路,可以减少多达85%的Token使用量。 说完该用的,再说三种不该用的情况。第一,大多数编码任务。编码的子任务之间偶合度高,真正可并行化的部分比研究类任务少得多。涉及一到三个文件的修改,单agent更快也更便宜。第二,简单查询和低价值任务。Anthropic在研究系统的prompt里直接写了缩放规则,简单事实核查只用一个agent三到十次工具调用。 直接对比用2到4个子agent,只有复杂研究问题才用10个以上子agent。早期版本甚至出现过对简单查询生成50个子agent的失控情况。第三,拟人化的角色分工。不要按组织架构图给agent排角色,什么经理、设计师、程序员互相聊天。Manus团队说得很直接,把子agent当作工具来调用,不要当作同事来对话。主agent调用一个函数,底层启动临时子agent循环并返回结构化结果。Anthropic也指出,按角色分工会产生持续的协调开销。 最后补两个关键的实战经验。第一个,子agent的任务指令必须极其具体,要包含目标、输出格式、工具范围和任务边界。Anthropic一开始允许主agent给出简短指令,比如研究半导体短缺,结果发现多个子agent在做完全相同的搜索,完全没有有效分工。第二个,并行不是万能要。OpenCloud在同一绘画内一次只处理一条消息,按绘画序列化执行。原因很明确。 当多个agent的共享状态时,并发会导致工具冲突和状态损坏。这是刻意的设计选择,不是限制。面试的时候建议你这样组织答案,先说原则,从最简单方案开始,只在有明确证据时才引入多agent。然后说三个该用的场景,上下文污染用隔离,并行探索用分发,工具过多,用专业化。再说三个不该用的情况,编码任务偶合度高,低价值查询不划算,拟人化角色分工是反模式。 最后补两个加分点。任务委派指令要极其具体,共享状态时序列化比并行更安全。把这些讲清楚,这个问题就稳了。
