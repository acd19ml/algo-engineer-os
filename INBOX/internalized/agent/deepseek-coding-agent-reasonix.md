DeepSeek最近放了个大招,继续在性价比路线上突飞猛进。今天要介绍的就是他们最新发布的Coding Agent,名字叫 Reasonix。它的核心优势是专门围绕前缀缓存稳定性来设计,实测缓存命中率能做到99%以上,单日4.35亿输入Token只花了1.38美元。同样的工作量如果没有缓存优化,要花61美元。 这个数字不是实验室跑出来的,是真实用户在2026年5月1日的DeepSeek后台截图。那问题来了,DeepSeek的API本身就有自动前缀缓存,为什么不同客户端的命中率差这么多?答案是,缓存是DeepSeek提供的,但命中率是客户端决定的。DeepSeek的前缀缓存机制是这样的,如果你这次请求的前缀字节跟上一次完全一样,缓存命中,输入Token的价格降到原来的五十分之一。 注意,是完全一样,差一个字节就miss。大多数agent的框架怎么做的?每次请求都重新序列化工具定义,重新排列历史消息,注入时间戳,甚至随机打乱fue shot的顺序。结果就是,每次请求的前缀都不一样,缓存命中率只有2-30%,甚至更低。Resenix的做法完全不同,它把整个上下文分成三个区域,每个区域有严格的不变性约束。第一个区域叫Immutable Prefix,不可变前缀。 这个区域包含系统提示词、工具定义列表和少数样本的示例。它在绘画开始时计算一次,之后整个绘画期间不变。代码里做了指纹追踪,如果有任何代码路径意外修改了前缀,指纹会飘移,系统会立刻报错。这个设计的关键约束是,工具定义的顺序不能变,系统提示词不能动态注入变量,FueShot不能随机化。任何一个变化都会导致前缀字节不同,缓存就miss了。第二个区域叫Append-only Log,只追加日志。 对话历史只能追加,不能修改,不能重排,不能删除。每一轮的assistant回复,tool调用结果都按时间顺序追加到日之末尾。为什么这很重要?因为Deepseek的前缀缓存是按字节前缀匹配的。如果你在第十轮修改了第三轮的消息,那从第三轮开始的所有字节都变了,缓存全部失效。只追加的设计保证了,每次新请求的前缀就是上一次请求的完整内容加上新的消息。 前坠永远是上一次的超级,所以缓存永远命中。第三个区域叫volatile scratch,一时性暂存区。模型的思维链,临时的计划状态,每轮的中间笔记都放在这个区域。关键是,这个区域的内容不会发送给API。它只在本地存在,每轮开始实清空。为什么要单独隔离?因为思维链每次都不一样。如果你把上一轮的思维链塞进下一轮的请求里,前坠就变了。 ReasonX的做法是,思维链产生的有用信息通过工具调用修复管道提取出来,以结构化的方式追加到Append-only Log里。思维链本身永远不进入请求体。这三个区域的组合效果是,每次API请求的前缀都是上一次请求的完整前缀,加上新追加的消息。Deepseek的缓存机制天然命中。但光有这三个区域还不够,常会话会称报。上下文窗口,你总得压缩,压缩的时候怎么保持缓存命中。ReasonX的做法叫, Cash Line Fold缓存对齐的折叠。当上下文使用率超过70%时,系统会触发自动折叠。折叠的方式是,把旧的消息总结成一条摘要消息,追加到前缀后面。注意,是追加,不是替换前缀。前缀本身不动,所以缓存还是命中的。具体流程是这样的,系统计算一个tail budget,比如上下文窗口的20%,然后从日之末尾往前数,保留最近的消息直到填满tail budget。剩下的旧消息, 调用一次V4 Flash模型来生成摘要。摘要生成后,用摘要消息加上保留的尾部消息,替换整个日志。这里有一个精妙的设计,摘要调用本身也附用了主绘化的前缀。因为摘要调用的Messages数组以相同的System Prompt和Tool Specs开头,所以它也能命中Deep Seek的前缀缓存。一次折叠操作不会额外浪费太多钱。如果75%的折叠还不够,到了78%会触发激进折叠。Tail Budget缩小到10%。 到了80%会直接强制退出当前turn,生成一个总结性恢复。到了90%会在turn开始前就预防性折叠。这是一个多级防御体系。接下来讲第二个支柱,工具调用修复管道。这不是直接的缓存优化,但它保证了agent循环的稳定性,间接保护了缓存命中率。因为如果工具调用失败导致重视、回滚或者消息重排,缓存就会被破坏。Deep Seek模型有几个已知的工具调用问题。第一, 模型有时候会把工具调用的JSON写在思维链里,但忘了放到正式的2COS字段。 ReasonX会扫描思维链的文本,用正则和JSON解析器把遗漏的调用捞回来。它能识别三种格式,DSML标记格式,标准OpenAI格式和RE自由格式。第二,当工具的参数schema超过10个叶子节点或者嵌套超过二层时,模型容易丢参数。 Resenix的Flatten模块会在注册工具时自动检测这种情况,把深层嵌套的schema斩平成点号分隔的路径。模型看到的是扁平的参数列表,Dispatch的时候再自动还原成嵌套结构。第三,模型有时候会因为最大Token长度限制,输出到一半就截断了。JSON不完整,Trunkation模块会检测不平衡的括号,尝试补权。如果补不了,会标记为不可恢复,让工具层返回无效JSON错误给模型。 而不是静默用空对象执行。第四,模型有时候会陷入循环,反复用相同的参数调用同一个工具。Storm模块维护一个滑动窗口,如果同一个工具名加参数组合在窗口内出现超过三次,就抑制这次调用,注入一个反思,Turn。但有一个巧妙的设计,写操作会清空窗口里的读操作记录,因为编辑文件,读文件验证是正常的工作流,不应该被当成循环。 这四个修复步骤按顺序执行。每轮用户输入开始时,Storm的窗口会重置,因为新的用户意图不应该继承旧的重复检测状态。第三个支柱是成本控制。Resenix默认用V4 Flash,不用V4 Pro。V4 Pro贵12倍。但大多数编码任务不需要那么强的推理能力。它有一个自我升级机制,如果模型判断当前任务超出了Flash的能力,它会在回复的第一行输出一个特殊标记,叫Need Pro。 系统检测到这个标记后会终止当前的Flash调用,用Pro重视这一轮。这是模型自己决定的,不是基于错误技术或者启发式规则。另一个重要的成本机制是Turn End Auto Compaction。每轮结束后,超过3000Token的工具结果会被自动压缩。逻辑是,模型在产生回复的那一轮已经看过完整的工具输出了,后续的轮次只需要一个摘要。如果后面真的需要完整内容,模型可以再调一次读文件工具。 一次额外的读取,比每轮都拖着一堆工具输出,便宜得多。还有一个40%的主动压缩预值,在涨的多步骤绘画内部,如果上下文使用率超过40%,就会提前触发压缩,不等到75%的紧急预值。所有辅助调用,折叠摘要,子Agent,截断修复重视,都强制用V4 Flash并且不起用思维链,因为这些任务本质上是把工具结果改写成散文,不需要推理能力,用Pro是浪费。 最后讲讲这个项目对开发者构建agent的启示。第一个启示,如果你用Deepseek的API,缓存命中率是你最重要的成本杠杆。缓存命中的输入token价格是miss的五十分之一。这意味着,如果你能把命中率从30%提到99%,你的输入成本直接降97%,这比任何模型选择任何prompt压缩技巧都有效。具体怎么做?三条规则。系统提示词和工具定义在绘画内不要变,对话历史只追加,不修改。 思维链和临时状态不要混进请求体。第二个启示,上下纹压缩必须是缓存感知的。大多数agent框架的压缩策略是删掉旧消息或者用摘要替换旧消息。但如果你的压缩操作改变了前缀的字节,缓存就全废了。正确的做法是,压缩产生的摘要追加到前缀后面,前缀本身不动。或者说,压缩操作只能修改日志的尾部,不能碰头部。第三个启示。 工具调用的鲁棒性直接影响缓存命中率。如果工具调用失败了,你怎么处理?如果你的策略是回滚到上一个状态重视,那你就破坏了Append Only Log的不变性,缓存就miss了。 ReasonX的策略是永远不回滚,只向前修复。Scavenge捞回遗漏的调用,Trunkation补权阶段的Jason,Storm抑制重复调用。所有修复都是在当前状态上追加新信息,不修改历史。第四个启示。 成本控制不是事后优化,是架构决策。 ReasonX的三个支柱,缓存优先循环,工具调用修复,成本控制,不是独立的功能,它们是一个整体。缓存优先循环,降低输入成本,工具调用修复保护缓存稳定性,成本控制选择最便宜的模型和最小的上下文,三者互相依赖。如果你在设计agent的架构,从第一天就要把缓存策略作为核心约束,不是先做功能,再优化成本。 而是成本约束决定架构形状。第五个启示,Deep Seek Only,不是限制,是特性。 ReasonX明确声明只支持Deep Seek,不做多模型兼容。为什么?因为每个模型提供商的缓存机制不同。Anthropic The Prompt Caching需要显示标记缓存断点。OpenAI的缓存是自动的,但力度不同。如果你要兼容所有后端,你就不能针对任何一个做深度优化。 这给我们的启示是,如果你的agent主要用一个模型提供商,不要用通用框架的最大公约数,要针对那个提供商的具体机制做深度适配。缓存机制、定价结构、模型的已知缺陷都是你可以利用的信息。总结一下,DeepSeek ReasonX的核心创新不是什么新算法,而是一个工程洞察。对于DeepSeek的API,前缀缓存命中率是成本的决定性因素,而命中率完全取决于客户端怎么组织请求。 它的四个关键机制,不可变前缀,只追加日志,一时性暂存,区隔离,缓存对齐的折叠压缩。加上工具调用修复管道保护循环稳定性,加上Flash First的成本分层。这些组合在一起,让一个重度使用的Coding Agent每天只花一两美元,而不是60美元。对于正在构建Agent的开发者来说,核心take away是不要把LLM API当成黑盒,了解你用的模型提供商的缓存机制,定价结构和已知缺陷,然后围绕这些约束来设计你的Agent循环。 这比换一个更强的模型或者写一个更好的prompt,对成本的影响大得多。
