今天分享一个弱模型通过Harness工程能逼近前沿B原模型性能的经验。这个实践中,Deep Seek V4 Pro在一个AI编程工具的内部评测,10次里有6次胜过Cloud Opus 4.7。不是在某个特定Benchmark上刷分,是在真实的编码任务里读文件,执行命令,写代码,跑测试,端到端底赢了。而且两天前,同一个Deep Seek模型连最简单的文件读取都跑不通,每一个工具调用都被校验器打回来,模型收到报错之后完全无法恢复。 两天之间发生了什么?不是模型升级了。是有人写了不到300行代码的工具输入。修复层。这个案例来自Command Code AI,一个开源的AI命令行工具,类似Cloud Code的开源替代品。他们的CEO分享了完整的技术细节。他的核心结论是,当你听到这个开源模型不会调工具的时候,90%的情况下那不是模型的问题,是Harness的问题。今天我们来拆它具体做了什么?为什么开源模型工具调用总失败? 核心论点是,这不是能力问题,是工具接口的定义问题。什么意思?前沿的B元模型为什么工具调用看起来很稳?因为Cloud和GPT参数量巨大,并且在预训练及后训练阶段见过海量的工具调用格式,各种工具格式定义它都见过,所以它能猜对你随便定义的任何格式。它不是真的理解了你的合约定义,它是见过足够多类似的合约,靠统计分布蒙对了。开元模型没有这个优势,它的训练数据里没见过你定义的那个特定格式。 所以它会犯一些很固定的格式错误。但这些错误不是模型不会推理,而是模型没记住你的格式。区别在哪?如果是推理能力不够,你没办法修。如果只是格式记忆不够,你在Harness层加一个修复就行了。他们发现跨Deep Seek,千万,质朴这些模型,工具调用的失败模式惊人的一致,不是随机的,是一个很小的,有限的,可没举的集合。总共就四种错误。第一种,可选字段传了No,而不是省略它。 字典里定义了一个可选字段,模型不想填的时候,它传了一个显示的no进去,而不是直接不传这个字段。教验器一看,这个字段的类型不包含no,报错。第二种,把数组写成了长得像数组的字符串。调用格式期望的是一个真正的数组,模型输出的是一个字符串,内容恰好长得像数组,类型不对,报错。第三种,调用格式期望数组,模型传了一个裸字符串,比如文件路径,字段期望,字符串数组。 模型直接传了一个单独的字符串,而不是把它包在数组里。第四种,期望输入数组。模型传了一个空对象作为站位服务。四种错误,每种修复大概30到100行代码,就这些。当他听到这个开源模型不会掉工具的时候,他现在默认假设是这四种之一。到目前为止,90%的情况下,这个假设是对的。但这里有一个很关键的工程细节,修复的顺序很重要。字符串化数组的解析必须在裸字符串包装之前执行。为什么? 如果你先做裸字符串包装,遇到一个内容是数组格式的字符串。你会把它整个包成一个只有一个元素的数组,那个元素就是原始字符串本身,而不是把它解析成一个多元素数组。顺序错了,修复就变成了破坏。除了这四种通用错误之外,他们还发现了一个特别有意思的失败模式。Deep Seek Flash在写文件的时候,有时候会把文件路径输出成Markdown自动链接格式,本来应该是一个普通的文件路径。它输出的时候,把文件名包在了Markdown链接语法里。 他们的写文件工具老老实实地去创建了一个文件名里带着方括号和圆括号的文件,直到有人发现为止。这不是幻觉,这是后训练的聊天分布泄露到了工具调用的边界。什么意思?DeepSeek在后训练阶段被奖励,在对话输出中自动给文件名加超链接。这在聊天场景里是合理的,用户看到一个可点击的链接体验更好。但模型没有区分这个输出是给人看的聊天气泡,和这个输出是要传给文件系统的路径。 它把聊天场景的鲜艳带到了工具调用场景里。修复是两行正则表达式,只处理退化情况。链接文本等于链接地址,去掉协议头的部分。真正的Markdown链接不受影响。这个案例揭示了一个更深层的认知。与其说模型工具调用能力差,不如说模型在工具边界上发生了分布混淆。模型知道怎么格式化一个路径,它只是没被足够清楚地告知,这个路径是要传给系统调用的,不是要渲染到聊天界面的。 所以,他们在合约层面做了一个改动,把通用的字符串类型换成了自定义的路径字符串类型。这个类型在描述里明确告诉模型,这是一个文件系统路径,不要加任何格式化。一次改动,所有路径字段的泄露问题同时解决。接下来是整个实践中最重要的架构决策。他一开始的做法是,先预处理,再校验。在校验器之前,跑一遍预处理,去掉No,解析字符串化的数组,标准化格式,然后再交给校验器。这个方案立刻就崩了。 写文件工具的内容字段里,用户要写入的内容恰好是JSON格式的文本。预处理器看到一个长得像竖组的字符串,把它解析成了真正的竖组。文件内容被静默篡改了,写入词盘的不是用户想要的内容。这种问题在冒烟测试里很难发现,正确的做法是反过来,先笑咽,再修复。具体流程是这样的,第一步,拿到模型的原始输入,直接交给校验器。如果校验通过,直接用,不做任何修改。 合法输入永远不被触碰。第二步,如果校验失败,拿到校验器返回的问题列表,每个问题里有精确的路径信息,告诉你是哪个字段,在什么位置,期望什么类型,实际收到什么类型。第三步,沿着issue列表,对每个出错的字段按顺序尝试四种修复,哪个能让这个字段通过校验就用哪个。第四步,修复完再校验一次,通过了,寄一条工具输入已修复的日志。 然后正常执行。没通过记一条工具输入无效的日志,给模型返回一条可读的重试提示。这个设计的结构性洞察是什么?当你做预处理的时候,你在编码一个关于什么可能坏了的鲜艳假设。你必须猜哪些字段需要修复,而你的猜测可能误伤正常数据。当你让校验器先跑的时候,schema本身就是鲜艳。你只在校验器明确指出这里不对的路径上花修复预算,校验器帮你做了bug定位的工作。你不需要猜,你有证据。 这跟Agent的开发里到处出现的先走快路径,失败了再走慢路径是同一个模式。正常情况零开销,异常情况精确修复。而且,这个设计附送了免费的摇测数据,你可以按模型和工具两个维度统计修复率。哪个模型在哪个工具上修复率突然升高了,说明模型在这个合约上退化了。你能在用户发现之前就补货回归。最后还有一类问题,四种修复搞不定。 读文件工具有一个关系不变量,如果你传了偏移量,就必须同时传形数限制。反过来也一样,Deepseek经常只传一个形数限制,不传偏移量。这个问题没法用输入修复解决,因为每个字段单独看都是合法的,行数限制是一个合法的数字,偏移量缺失也不违反类型约束。问题在字段之间的关系上,不在单个字段的形状上。他们的解法是不报错,而是推断模型的意图,只传了行数限制,没传偏移量。 默认偏移量等于0,只传了偏移量没传行数限制,默认行数限制等于2000行。然后在返回结果里告诉模型你做了什么选择。注意,行数限制未提供以默认为2000行,如需读取更多或更少,请同时提供偏移量和行数限制重视。注意,这条提示没有错误前缀,终端不会把它标红。模型看到的是一个正常的执行结果加一条说明,而不是一个需要恢复的错误。如果默认值不对,模型下一轮自然会带上正确的参数重视。 这里的原则是能修复的修复,不能修复的扩展语义。无论哪种,都把你的选择透明地告诉模型。回过头来看整个实践。四种输入修复,每种几十行代码。两行正则处理markdown链接泄漏。一个关系默认值,一个错误前缀的改动,总共加起来可能不到300行代码。模型没变,合约变得更宽容了,恰好在需要宽容的地方宽容。因为这些改动。 Deep Seek V4 Pro能够在内部评测中10次里6次胜过OPAS 4.7。这个结果的深层含义是什么?严格的数据格式定义是一个有成本的选择。它过滤噪声,但它也过滤掉了那些可恢复的噪声。来自任何没有记住你特定数据格式的模型,前沿的B原模型因为见过足够多的格式,能隐式地吃掉这个成本。开源模型没有这个统计优势,所以它们的错误被放大了,然后被贴上能力不足的标签。 但如果你在Harness层做了恰当的中介,在模型的输出分布和你的工具数据格式定义之间做翻译,开源模型的实际推理能力就释放出来了。但这里有一个更大的问题值得想清楚。你可能已经注意到了,这整套修复是针对Deep Seek的特定失败模式设计的。Markdown链接泄漏是Deep Seek后训练的产物。换一个模型可能不会犯这个错,但会犯别的错。千万可能在数组处理上更稳,但在嵌套对象上出问题。 质朴可能路径没问题,但在波尔值处理上不稳定。这意味着Harness Engineering在当前阶段本质上是高度模型定制化的。你不是在写一套通用的中间件,你是在为每个模型的特定训练分布画一张错误地图,然后在地图上标注的每个坑上加一块电板。这跟我之前讲的快速探测模型边界是同一件事。每个模型的边界不一样,边界的形状取决于它的预训练数据分布,后训练的RL奖励信号,以及它见过的工具合约的多样性。 你没法写一套harness适配所有模型,就像你没法写一套测试用力覆盖所有bug。但这不意味着每次换模型都要从零开始,这个案例里的架构决策先校验再修复,而不是先预处理再校验,是通用的。四种形状修复的分类框架是通用的,关系不变量,用语意扩展,而不是输入修复,这个原则是通用的,变的是具体的修复规则,不变的是发现和组织修复规则的方法论。而且,先校验再修复的架构天然支持这种定制化。 因为修复逻辑是按校验器指出的具体路径精确触发的。你可以按模型维度做条件分支。Deep Seek触发链接泄漏修复。千万触发嵌套对象修复。其他模型走通用的四种形状修复。遥测数据会告诉你每个模型在哪些工具上需要什么修复。你的Harness随着使用数据的积累自动变得更精确。这也解释了为什么快速探测模型边界是Agent的开发里最不会过时的能力。模型会换。每个模型的错误地图不一样。 一句话总结,当你听到这个模型工具调用不行的时候,先问一个问题。 是模型不会推理,还是模型没记住你的格式?如果是后者,那不是模型的问题,是Harness的问题。四种修复,两行正责,一个默认值,可能就是开源模型和商业模型之间的全部差距。但记住,这些修复是模型特定的。Harness Engineering的核心能力不是写出一套通用修复,而是能快速为任何新模型画出错误地图,然后在地图上做工程。
