今天来讲一个Agent记忆系统中最难解决的问题。你告诉AI助手,我住在北京旺京。过了一个月,你说我搬去上海了。大部分AI记忆系统能正确更新你的地址。但问题来了,这个AI同时还记得,你每天骑车15分钟到公司,你交的是北京社保,你的健身房在旺京。当你搬走之后,这些事实哪些还成立?哪些需要被更新?哪些应该变成,我不确定了而不是,这个肯定是假的。 最近有一篇论文叫Meme,专门测试了这个场景,结果是所有主流的agent的记忆系统在这种集联更新任务上的准确率只有3%。肖王推理就是一个相关实体被移除后,依赖它的事实还算不算数准确率只有1%,不是某个系统差,是所有系统都差。研究者试了更强的模型,更好的提示词,更深的检索,更少的噪音,全部失败。 唯一部分有效的配置是把所有记忆放进一个文件,让最强的模型一次性读完,但代价是成本增加70倍。这意味着什么?意味着当前所有agent记忆系统的底层架构有一个根本性的缺陷。它们是孤立的点存储,不是图结构。每个事实被独立记住,但系统不知道这个事实依赖于谁,它的成立需要什么前提条件。今天聊一下这个问题为什么这么难,以及目前有哪些可能的解决方向。先说清楚问题的本质。 记忆系统需要处理的更新有三个层次。第一层,直接替换。住在北京变成住在上海。这是最简单的,大部分系统都能做到。本质上就是检测到同一个属性的新值,覆盖旧值。第二层,急连失效。住在北京变了之后,每天骑车15分钟到公司,这个事实还成立吗?它不是直接矛盾,你搬去上海之后,理论上也可能骑车15分钟到公司,但它的成立前提变了,需要被重新确认。 这是当前系统做不到的。第三层,肖王推理。你之前说我和张三是同事,后来张三离职了。那我和张三是同事这个事实现在是假的?还是应该变成我和张三曾经是同事?系统需要区分事实被否定和事实变成历史。第二层和第三层的共同难点是,系统需要理解事实之间的依赖关系。而当前的记忆系统,无论是向量数据库,文件存储,还是结构化数据库,都不显示存储这种依赖。 为什么现有系统做不到?以目前最常见的agent的记忆架构为例,写入的时候,系统把用户说的话提炼成一条条独立的事实,存进向量数据库或者文件,读取的时候根据当前对话的语意去检索相关的事实,注入上下文。这个架构的问题在于写入和读取之间没有维护环节,事实被存进去之后就是静态的,直到被直接覆盖或者被检索出来,没有任何机制会在一个事实变化的时候主动去检查其他事实是否受影响。 这就像一个数据库,没有外建约束。你可以删除一个用户,但所有引用这个用户的订单不会自动失效。在传统数据库里,我们用外建和集联删除来解决这个问题。但在自然语言记忆系统里,依赖关系不是显示定义的,需要靠推理来发现。那有哪些可能的解决方向?第一个方向,写入时触发影响分析。思路是,每次写入新记忆的时候,不只是追加,而是触发一轮检查,用模型去判断这条新信息是否与已有记忆矛盾。 如果矛盾,哪些已有记忆需要被标记为待确认?具体做法可能是新记忆进来后检索所有语意相关的已有记忆,把它们和新记忆一起交给模型,让模型判断每条旧记忆的状态仍然有效,可能失效需要确认,还是明确矛盾已失效,限制。这个方案的成本跟已有记忆的数量成正比。如果你有一千条记忆,每次写入都要检索和判断一大批,延迟和成本都会很高。 而且模型的判断本身不一定准确。论文已经证明了,即使是最强的模型,在集联推理上也只有部分准确。第二个方向,显示依赖图。思路是,存记忆的时候不只存事实本身,还存它依赖的前提条件。比如存每天骑车15分钟到公司的时候,同时标注前提,住在望京附近。当住在望京被更新时,系统自动把所有依赖它的事实标记为待确认。这本质上是把记忆从平面列表变成有像图。 结点是事实,边是依赖关系。限制,依赖关系本身需要模型来推断,而这个推断不一定准确。每天骑车15分钟到公司,依赖住在旺京。这个依赖关系对人来说很明显,但模型不一定能可靠地识别出来。而且依赖关系可能是隐士的多层的交北京社保,依赖在北京工作,在北京工作依赖住在北京。这种多挑依赖更难自动发现。第三个方向,置信度衰减加主动确认。 思路是,不试图自动解决即连更新,而是承认系统做不到完美的依赖推理。每条记忆带一个最后确认时间和自信度。当一个上游事实被更新时,所有可能相关的下游事实的自信度自动降低。回答用户问题时,如果涉及低自信度的记忆,系统主动追问确认而不是直接使用。比如,用户搬家之后问,我的健身房在哪儿?系统不是直接说忘经邮寄。 而是说,你之前提到过忘经邮寄,但你最近搬家了,这个信息可能已经过时。要确认一下吗?限制。这个方案把问题推给了用户。如果系统频繁追问确认,用户体验会很差。而且,可能相关的范围很难界定。搬家之后,你的所有记忆,理论上都可能受影响。你不能每条都问。第四个方向,全量上下文方案。这就是秘密论文里唯一部分有效的方案。把所有记忆放在一个文件里。 每次查询时,让模型读完整个文件再回答。模型在一次推理中能同时看到所有相关事实,从而自然地发现依赖关系并做出正确判断。限制,成本增加70倍,而且不scale。记忆量大了之后文件太长,超出上下文窗口。这不是一个工程上可持续的方案,但它证明了一件事,问题不在于模型的推理能力不够,而在于当前的记忆架构没有给模型足够的信息来做推理。 回过头来看,这四个方向其实指向同一个根本矛盾。吉连更新需要模型在正确的时机看到所有相关的事实,然后做一次推理。但正确的时机和所有相关的事实这两个条件很难同时满足。写入时触发分析时机。对了,但不确定哪些事实相关。显示依赖图知道哪些相关,但依赖关系本身可能不准确。 我个人觉得最务实的方案可能是一个组合。写入时做一轮粗筛,把可能受影响的记忆标记为代验证。读取时,如果碰到代验证的记忆,触发一轮精确推理,确认它是否仍然成立,后台定期批量扫描清理。这是写入时粗筛加读取时精确验证的两阶段方案。粗筛的成本低但可能有漏网之余,精确验证的成本高但只在需要的时候触发。但据我所知,目前没有哪个开源项目或商业产品完整实现了这个流程。 这仍然是一个开放问题。这篇论文的价值不在于它给出了解法,而在于它用数据证明了一件事。我们以为AI记忆系统已经够用了,但实际上在稍微复杂一点的场景下,它们的表现接近于0。3%和1%的准确率意味着这不是偶尔出错,而是几乎从来不对。对做Agent产品的人来说,这意味着你不能假设记忆系统会自动处理依赖关系。 如果你的产品场景涉及用户生活状态的变化,搬家,换工作,换城市,换伴侣,你必须在产品层面设计降级策略,而不是指望底层记忆系统能自动搞定。对用AI的人来说,当你发现AI在跨绘化后记错了或者给出过时的建议,问题可能不是他记不住,而是他的记忆缺乏结构。他记住了每一条事实,但不知道这些事实之间的关系。 一句话总结,当前所有agent的记忆系统的根本缺陷是,它们存的是孤立的事实,不是事实之间的关系。一个事实变了,依赖它的其他事实不会自动更新。这个问题目前没有完善的解法,但它正在成为agent从能用到可信赖的关键平台。
