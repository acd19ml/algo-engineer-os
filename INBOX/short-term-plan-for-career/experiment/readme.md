这个目录放了我硕士第一年期间自主做的研究关于帮助我自己理解agent memory的工作，有实验报告，也有我和Claude的聊天关于理解人类记忆帮助我来设计agent记忆模块。

从学习中我学到的核心思想是，

AWM是一个局部的、依赖于条件的记忆机制。

它确实抓住了程序性记忆的一个重要部分，因为它将工作流外部化并在不同的任务中复用。这是一个实质性的贡献。

但它主要在三个条件同时满足时才起帮助：工作流族具有可复用性、检索到的程序与当前任务匹配、且基线模型仍有改进空间。

目前仍然缺失的是一个强大的修正闭环（revision loop）。系统还没有一种稳健的方法来检测有害的工作流、在失败后修改它们，或者在任务匹配度较弱时抑制它们。

所以我的结论不是简单地说 AWM 有效或无效。而是 AWM 展示了程序性记忆在实践中可能的样子，但它也表明了为什么单靠工作流复用是不够的。如果没有由失败驱动的修正机制，记忆虽然有帮助，但并非完全可靠。

在完成报告后，我一直在思考：接下来的问题到底仅仅是“应该改进哪个 AWM 模块”，还是更深层次的问题——“记忆对象本身的形态（shape of the memory object）”。

有两个因素促使我朝这个方向思考：

在一项小型的记忆迁移研究中，我发现单靠抽象化是不够的。一个记忆即使在主题上高度相关，仍然可能会失效，除非它保留了可执行的结构（executable structure）。

最近关于“经验对象（experience-object）”设计的研究表明，运行时形态（runtime shape）至关重要。问题不仅在于存储了什么经验，还在于在测试阶段返回给模型的是什么样的对象。

因此，这部分内容是为了开启一个更上游的讨论。
这里我并不是在对 Skill、AWM 和 Gene 进行排名。我是将它们视为经验对象空间中不同的设计点。

Skill 是外部的程序性对象，但它仍然更偏向于人类可读且包含大量散文式文本。

AWM 将 Web 环境中的经验表示为带有动作槽位、以上下文为条件的工作流模板。它已经比普通的总结更具执行性，但其选择、边界和修正机制依然薄弱。

Gene 则强调了一个更加紧凑的运行时控制对象，其中触发器（trigger）、策略（strategy）、避免约束（avoid constraints）和验证（validation）都被做得更加明确。

所以更深层的问题是：在不同的环境下，什么形态的程序性记忆对象才是最合适的。

然后，我试图将这个问题落地到一个更具工业背景的场景中。

想象一个运维诊断场景，模型本身已经很强，工具也非常成熟。模型或许能够从零开始完整地解决一次故障。问题不在于它不能解决，而在于每次都从头解决是一种浪费。

真正有用的诊断路径并不是通用常识。它依赖于当前的拓扑结构、服务依赖、告警模式和监控埋点。这些是外部结构，而不仅仅是内部知识。因此，在这种场景下，程序性记忆的主要目的不是为了补充缺失的知识，而是保留可复用的搜索路径、验证顺序和边界条件，以便更高效地排查后续的故障。

构建完整的工业系统成本太高。所以我目前的想法是模拟诊断的结构：构建一个包含隐藏拓扑、基于工具的查询、症状相同但根本原因不同、并且带有轻微环境漂移的轻量级图结构环境。

这将使我能够测试：外部程序性记忆能否降低重复搜索成本，并在不断变化的环境下依然支持选择性的复用。

您认为程序性记忆的哪一部分会被更强的模型内部化吸收，哪一部分会长期保持外部化？
导师给我的答复是：当我们从生产中得到了高质量的数据，我们可以有很多种做法，你可以拿来微调、也可以拿来做memory，在不断变化的外部环境中，如果经常需要变动、每次变动都做一次微调未免成本太高，此时memory弥补了这一点。

我还思考，从一条成功/失败轨迹就来改动memory是一个不严谨的决定。假设是web agent的操作，网站不稳定或者，记忆设计分层不清晰，比如我在携程的订机票操作，复用到去哪儿网，这都是一个旅行订票的机制/网站，属于同一个范围，但如果携程的操作成功经验到去哪儿网操作失败了，是不是要改这个经验？如果改了可能带着这个经验到携程又不好用了，如果在同一个经验内增了就会出现注意力稀释同样可能影响模型判断，如果拆开新增一个经验有回粒度太细、导致无限累积的问题。而且程序性记忆是需要同模型一起构建的，base model就能解决的问题不需要召回一条经验会增加开销。所以在运维下钻分析场景中，设计一套好的工具可以让模型了解系统情况，并且记忆管理好，开再多新session也不需要模型重新了解系统，只在系统变更时重新跑一遍了解新系统的情况，就是我能想到最好的解决方案，节约开销，我还希望能用故障域来聚合记忆，假如一个故障域发生故障，我可以把曾经排查成功的轨迹同时开多个subagent并行验证，那么都没排查出来的情况下，不要走老路，直接查没走过的新方向，就很好。失败的轨迹还可以根据失败原因来进行分类。由此我理解，记忆系统可能需要根据业务场景来定制，而且需要分层设计。如果从一个成功/失败的轨迹就做出经验层的改动可能复用性很差、适用范围小，一个适用范围广的经验也未必能真正影响成功率。由此，我们需要保留好原始轨迹，并且不断迭代设计解决方案，系统没有成熟时，可能需要工程师来对经验进行标注，从而迭代。我想学怎么做到这种数据飞轮，我没见过什么样的数据是好的训练数据。为达到这个目标我需要更多实践。并且我可能需要专门训练一个对轨迹分析生产记忆的模型。所以在训练方面也需要累积更多经验。
