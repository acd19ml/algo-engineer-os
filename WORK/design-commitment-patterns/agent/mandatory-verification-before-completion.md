# Mandatory Verification Before Completion

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计的 agent 任务有可验证的完成标准（代码可运行、文件内容符合格式、测试通过等），并且任务完成后需要 agent 报告结果时，先打开这条。

不适用：纯生成类任务（创意写作、摘要等）没有客观可验证标准；验证步骤的成本远高于任务本身时。

## 结构性不变量

Agent 任务的完成报告必须经过一个独立的验证步骤（不依赖 agent 自我评估"看起来对了"）；验证步骤必须由系统在 agent 退出节点自动触发，不依赖 agent 自愿执行；验证失败时不允许 agent 声称任务完成。

## 它预防的症状簇

- P-033：Agent 在任务完成前跳过验证直接输出 → 系统在退出节点拦截，强制执行验证，agent 无法在不经验证的情况下报告完成。
- P-028：评审者宽容/LLM 评估偏向表扬生成结果 → 独立验证步骤（而非 agent 自评）提供客观信号，天然的自我表扬偏差无法影响验证结果。
- P-031：Agent 报告结果时夸大成功或掩盖失败 → 验证步骤产生客观状态，报告基于验证结果而非 agent 自我判断。

## 留在 runbook 的部分

- P-028 评估器调优过程（人工迭代循环，读日志→改 prompt→再跑）：这是让评估器变严格的运行时过程，不是结构性不变量。
- P-033 推理三明治策略（规划/验证阶段用高推理强度）：这是验证质量优化的实现选型，留在 runbook。
- P-031 system prompt 里的具体诚实性规则措辞：这是软约束层，是本 pattern 的补充而非替代。

## 代价 / 范围

- 代价：每个任务都需要一个验证步骤，增加延迟；需要为每类任务定义"可验证标准"（what counts as passing）。
- 适用：任何产出有客观可验证标准的 agent 任务（代码执行、格式校验、测试通过）。
- 失效：纯主观输出任务（摘要、创意写作）没有客观验证标准，这层不适用；对这类任务，参考 P-028 的评分维度设计。

## 实例化成项目 DC 时要补的校验

- 架构：agent loop 退出节点存在一个中间件钩子，在 agent 声明"完成"时拦截并触发验证器；验证器不是 agent 本身的调用，是独立的确定性检查或独立 LLM 调用。
- 静态检查：扫描所有"agent 任务完成路径"，确认每条路径都经过验证器（不存在绕过验证直接退出的路径）。
- eval：构造"agent 报告完成但实际有错误"的用例（如 agent 说测试通过但 mock 测试实际没有运行），验证系统拦截并继续工作。
- 人工 review TODO：每新增一类 agent 任务类型时，必须定义该类型的验证标准；没有验证标准的任务类型不得进入自动完成路径。

## 前沿问题

- 验证器本身出错（误判通过 / 误判失败）时，系统应该 fail-closed（保守拦截）还是 fail-open（放行）？
- 对长任务的多阶段验证：每个阶段都要验证还是只在最终完成时验证？中间阶段验证失败的代价和最终验证失败的代价不同。
- LLM-as-judge 型验证器（而非确定性验证）如何保证不重蹈 P-028"自我表扬"的问题？需要结合 [[reviewer-isolation-fail-closed]]。
