# Prerequisite Gate Before Terminal Tool

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计多步工具调用工作流，且部分工具（如 `submit` / `finalize` / `apply`）依赖前置工具的输出（如 `search_list` 返回的 ID）才能合法执行时，先打开这条。

不适用：工具间无显式数据依赖；工作流是单步的；依赖关系全部是运行时动态数据依赖（无法静态编码时）。

## 结构性不变量

Dispatcher 在前置工具尚未被调用的情况下，不得执行声明了依赖关系的终端工具；前置条件检查发生在工具真正执行之前，不依赖模型自觉性。

## 它预防的症状簇

- P-003：模型跳过必要的中间步骤直接调终端工具 → 前置门控让"跳步调用"在架构层被拦截，终端工具永远无法在依赖未满足的情况下真正执行。

## 留在 runbook 的部分

- P-003 方案 2 升级式纠正消息（礼貌 → 直接 → 强硬）：这是跳步被拦截后向模型注入的纠正信号，属于运行时恢复流程，不是不变量本体。
- P-007 纠正计数器硬上限：和前置门控配合使用（门控拦截后触发纠正，纠正次数受 [[retry-budget-hard-cap]] 约束）。

## 代价 / 范围

- 代价：需要在工具注册时显式声明依赖图；Dispatcher 每次调用前要做依赖检查（确定性 O(1)，代价极低）。
- 适用：工作流依赖关系静态已知且可在工具注册时编码的场景（如 search → get_detail → submit 的线性或 DAG 结构）。
- 失效：依赖关系是纯运行时数据依赖（"用第 2 步返回的 ID"，但哪个 ID 未知），无法静态声明时退化为 system prompt 约束 + 纠正消息方案。
- 失效：无状态单步工具集，不存在跨步依赖。

## 实例化成项目 DC 时要补的校验

- 架构：工具注册接口支持 `requires: [tool_id_list]` 字段；Dispatcher 维护已调用工具的有序集合。
- 静态检查：扫描工具定义，确认所有声明了 `requires` 的工具，其前置工具已在同一工作流中被注册（防止前置工具被删导致永远满足不了条件）。
- eval：构造「跳步攻击」用例：模型上下文中没有前置工具返回值，直接触发终端工具调用 → 期望 Dispatcher 拦截并注入纠正消息，终端工具不执行。
- 人工 review TODO：若暂不实现自动依赖图，代码评审必须检查所有终端工具的 docstring 中是否注明前置条件，以及 Dispatcher 是否有对应的前置检查逻辑。

## 前沿问题

- 运行时动态依赖（"工具 B 需要工具 A 某次特定调用的输出"）能否通过类型化返回值 + 依赖注入自动推导？
- 当工作流是 DAG（并行分支）而非线性时，依赖检查怎么表达：路径敏感的满足条件 vs 全局满足条件？
- 如果 Dispatcher 拦截本身失败（如依赖图被错误声明），失败模式应该 fail-closed 还是 fail-open？
