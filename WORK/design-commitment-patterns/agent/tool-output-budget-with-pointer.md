# Tool Output Budget with Pointer

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你的 agent 使用任何可能返回不定长输出的工具时（搜索 / 数据库查询 / 日志检索 / 文件读取 / 目录列举），先打开这条。

不适用：所有工具输出都有已知上界且很小（如 `get_status` 返回单个状态码）；纯文本生成 agent，工具不返回大块数据。

## 结构性不变量

所有可能返回不定长输出的工具必须声明明确的 token / 字节预算；超出预算时，系统必须将完整输出存入外部存储（磁盘 / 数据库），向模型返回 preview（前 N 字节 / 摘要行）和文件指针（外部路径）；工具层不得直接截断输出后静默返回（让模型误以为已获得完整数据）。

`read` 类工具（负责从外部存储读回数据的工具）必须使用独立的 offset + line_count 分段机制，不套用"超长存外部"逻辑，防止自指死循环。

## 它预防的症状簇

- P-021：单次工具调用输出一次撑爆上下文窗口 → 工具输出预算保证单次工具调用对上下文的影响有上界；overflow 存外部 + 返回指针，让模型拿到"获取信息的能力"而非一次性塞入全部信息（just-in-time context 思路）。

## 留在 runbook 的部分

- P-021 各工具类型的具体预算数字（搜索工具 2 万字符 / 大部分工具 10 万字符）：这些数值是经验性取值，不是不变量本体，且会随业务规模和模型窗口大小变化，留在 runbook。
- P-018 fail-loud-on-bad-json：截断检测是另一个独立的 fail-loud 机制，见 [[fail-loud-on-bad-json]]。本 pattern 处理"输出太长"，P-018 处理"输出被截断"，两者互补。

## 代价 / 范围

- 代价：需要为每个工具设计两条路径（正常返回 / 超预算外存 + 指针）；需要引入外部存储（磁盘 / 临时目录）；preview 内容的选择需要按工具类型定制（结构化数据取摘要 / 头尾几条，而非前 N 字节）。
- 适用：任何会返回不定长输出的工具，普遍适用。尤其是：搜索工具（"无界搜索可以轻松填满上下文"）、数据库查询、日志检索、大文件读取。
- 失效：工具输出本来就有已知上界且小（如枚举类型工具、状态查询）。

## 实例化成项目 DC 时要补的校验

- 架构：每个不定长工具有明确的 `max_output_tokens` / `max_output_chars` 配置；工具实现中存在 overflow 检测 + 写外存 + 返回 preview + 指针的路径；read 类工具使用 offset + line_count 接口。
- 静态检查：枚举所有工具，确认每个工具要么有明确预算，要么有文档说明它为什么不需要（输出有上界）。
- eval：对搜索类工具喂入"会返回大量结果"的 query，检查：（1）模型上下文中工具返回值 ≤ 预算；（2）完整输出确实已存外部；（3）指针可用（模型能通过指针拿到后续内容）。
- 人工 review TODO：新增工具时，代码评审必须检查：（1）是否设置了 max_output；（2）overflow 时的行为是外存 + 指针还是静默截断（不允许静默截断）。

## 前沿问题

- Preview 内容的最优策略因工具类型差异很大（结构化数据取前 N 行 vs 摘要行 vs 统计信息），是否应该作为工具元数据的一部分可配置？
- 外部存储文件的生命周期管理：会话结束后如何清理？文件路径是否需要包含 session_id 防止跨会话污染？
- 当模型需要多次用指针读取大文件时，多次 read 工具调用会不会本身占满上下文？（递归 overflow 问题）
