# Three-Tier Deny Overrides Allow

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你为操作真实系统（文件 / 命令 / API / 数据库）的 agent 设计权限模型时，先打开这条。

不适用：纯只读 agent（如查询助手），不存在越界操作的可能；agent 操作完全在沙箱内，超出沙箱的操作被基础设施层拦截（不需要应用层权限模型）。

## 结构性不变量

权限模型必须使用三档结构：**禁止** / **需要确认** / **允许**；禁止档永远优先于允许档（可以"允许所有文件编辑"同时"禁止修改 .gitconfig"）；权限粒度到**操作内容**而非工具整体（同一个 bash 工具，`git log` 允许、`git push --force` 禁止）；"需要确认"档应尽量压小（理想状态：用户每天只见到 1-3 次确认框）。

## 它预防的症状簇

- P-022：Agent 做了超出指令范围的事 → 三档模型让"允许大部分操作"和"堵住关键危险点"不再互斥，无需在"全放行"和"逐次确认"之间二选一；禁止档的存在让越界操作从架构上不可能发生（不只是被纠正）。
- P-022：确认框疲劳无脑点击 → "需要确认"档压小后，用户只在真正需要思考的决策时被打扰，而不是在每次安全操作上也弹确认框。

## 留在 runbook 的部分

- P-022 方案 0 止血（暂停写权限 / 回滚）：这是已经发生越界后的紧急响应，是运行时恢复流程，不是权限架构本身。
- P-022 各档位的具体配置示例（git status → 允许，git commit → 确认，rm -rf → 禁止）：这些是示例而非不变量，项目依据实际操作集自行配置。
- P-025 不可绕过底线清单（.zshrc / .ssh / .gitconfig 等）：底线清单是三档中"禁止"档的最高优先级子集，但底线清单的"不可配置"特性是额外不变量，见 [[non-bypassable-bottom-line]]。

## 代价 / 范围

- 代价：需要提前梳理所有工具的操作集并分档（不能事后补补丁）；禁止档和允许档如果配置冲突，需要明确优先级解析规则；维护权限配置本身有成本。
- 适用：任何让 agent 操作真实系统的场景，无论规模大小。
- 失效：沙箱完全隔离（越界不可能传出沙箱）——应用层权限模型变成锦上添花而非必要；但即使如此，三档模型也有助于控制确认框频率。

## 实例化成项目 DC 时要补的校验

- 架构：权限配置结构包含三个 key（deny / confirm / allow），deny 档规则在匹配时直接拦截，不再检查 allow 档；工具 Dispatcher 在执行前查询权限配置，返回 deny / confirm / allow 三种结果，不是 bool。
- 静态检查：权限配置文件扫描，确认 deny 档条目中不存在和 allow 档冲突且 deny 未赢的情况（"禁止优先"是系统约束，配置错误应被检测而非静默忽略）。
- eval：构造"越界操作"测试集（agent 被指示做的操作落在 deny 档），期望系统返回明确拒绝而非执行；构造"安全操作"集，期望不弹确认框直接执行（验证"允许"档工作正常）。
- 人工 review TODO：上线前必须梳理操作列表 → 分档 → review 分档结果，不能用默认配置；每次新增工具时重新 review 新工具的操作集应归入哪档。

## 前沿问题

- 当 agent 被一个恶意 prompt 操控去尝试高频"需要确认"操作消耗用户注意力时（确认框 DDoS），三档模型如何防御？
- 动态权限（某次任务临时升权）和静态三档模型如何共存？升权操作本身是否需要在"需要确认"档？
- 如何让非技术用户在不理解操作语义的情况下完成合理的权限分档？
