# Agent Config Binary Rules Bounded

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你为 agent 编写 system prompt、CLAUDE.md、AGENTS.md 或任何行为配置文件，且需要 agent 可靠地遵守这些规则时，先打开这条。

不适用：高代价安全规则——这类规则不属于"软约束层"，必须用代码层确定性实现，见 [[high-cost-rules-in-code]]。

## 结构性不变量

Agent 行为配置（system prompt / CLAUDE.md / AGENTS.md 等）中的每条规则必须是 binary 可检查的（"做了"或"没做"，而非"做得好不好"）；规则总数必须控制在正交覆盖失败模式的最小集合内（实测天花板：12 条 / 200 行）；超出天花板的配置等同于无配置，必须精简。

## 它预防的症状簇

- P-034：CLAUDE.md/AGENTS.md 过长，agent 合规率急剧下降 → 200 行 / 12 条的硬上限让配置保持在有效范围内；每次新增规则前先检查数量是否达到天花板。
- P-030：System prompt 里的规则太模糊，agent 执行时不遵守 → Binary 可检查要求过滤掉"请小心"这类无法测试的指令；只有模型能自我验证的规则才能进入配置。

## 留在 runbook 的部分

- P-030 方案 3 避免无法测试的修饰词的具体例子（"小心"→"明确陈述你的假设"）：这是规则改写的具体操作，留在 runbook。
- P-034 方案 4 不在配置文件里放例子的原因：这是一个具体操作指南，不是结构性不变量。

## 代价 / 范围

- 代价：每条规则都需要能回答"这条规则预防的是什么具体失败"；规则数量的压力可能导致遗漏某些重要约束——此时应该考虑代码层实现而非在 prompt 里堆规则。
- 适用：所有使用 CLAUDE.md / AGENTS.md / system prompt 约束 agent 行为的场景。
- 失效：高代价安全规则——不能只靠 prompt 约束，必须走代码层，见 [[high-cost-rules-in-code]]。

## 实例化成项目 DC 时要补的校验

- 架构：项目中有 CLAUDE.md / system prompt 的，存在一个"规则质量检查清单"；新增规则前过清单（是 binary 可检查的？覆盖的失败模式是否正交于已有规则？总数是否超过天花板？）。
- 静态检查：扫描 CLAUDE.md / system prompt，统计规则数量；扫描规则内容，标记包含"小心""多想""努力做好"等无法测试词汇的规则为待改写。
- eval：从配置文件里随机取一条规则，构造"模型遵守"和"模型违反"两类样本，验证规则可被 binary 判断（能区分两类）；如果无法区分，该规则是噪音。
- 人工 review TODO：每次修改配置文件时，review 新增规则是否 binary 可检查，是否与已有规则正交，总数是否在天花板内。

## 前沿问题

- 12 条 / 200 行的天花板是否随模型能力提升而变化？更强的模型是否能处理更多规则？
- binary 规则和抽象原则各有优势（前者稳定，后者灵活）——是否存在两者的最优组合比例？
- 当规则之间存在隐式冲突（A 说不要加注释，B 说要解释复杂逻辑）时，如何在配置层显式定义优先级？
