# Schema Validate Before Repair

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计工具调用层对 LLM 输出的格式修复逻辑（rescue parse、字段类型修复、格式规范化等），先打开这条。

不适用：工具输入来自确定性代码路径（非 LLM），不存在格式漂移问题；没有任何格式修复逻辑，直接拒绝不合规输入。

## 结构性不变量

工具层对 LLM 输出的所有格式修复逻辑，必须在 schema 校验之后触发（校验失败 → 读取失败路径 → 精确修复）；不得在 schema 校验之前对输入做预处理变换（猜测性修复）；正常路径（校验通过的输入）不得被修复逻辑触碰。

## 它预防的症状簇

- P-037：校验前对工具输入预处理，静默损坏合法输入 → 先校验再修复的顺序保证合法输入永远直接执行，修复逻辑只在校验失败后按失败路径精确触发，无法误伤合法输入。

## 留在 runbook 的部分

- P-037 修复顺序（字符串化数组解析必须在裸字符串包装之前执行）：这是修复逻辑内部的执行顺序，是实现细节，留在 runbook。
- P-009 方案 3 验证 + 重试（语法层兜底）：这是修复失败后的下游运行时恢复过程，留在 runbook；本 pattern 的不变量是修复的触发时机（校验后），不是修复失败后怎么办。

## 代价 / 范围

- 代价：需要在工具层实现双阶段处理（校验 → 条件修复）而非单阶段（直接修复后校验）；校验失败时需要读取并解析 issue 列表，增加轻微开销。
- 适用：所有对 LLM 输出做格式修复的场景，无例外。
- 失效：没有格式修复逻辑（工具直接拒绝不合规输入）时，不需要这层——但此时应确认 [[fail-loud-on-bad-json]] 不变量是否满足。

## 实例化成项目 DC 时要补的校验

- 架构：工具层处理 LLM 输出的函数结构为：`validate(input)` → 通过则 `execute(input)`，失败则 `repair(input, issues)` → `validate(repaired)` → 通过则 `execute(repaired)`，失败则 `fail()`；不存在 `repair(input)` → `validate(repaired)` 的顺序。
- 静态检查：扫描工具输入处理代码，确认所有"类型转换 / 格式规范化 / 字段补全"操作都发生在 schema 校验失败分支内（而非在 try 块外或校验前）。
- eval：构造一个本来合法的字符串输入（内容恰好看起来像数组），验证工具不会把它转换成数组（合法输入不被触碰）；构造一个真正格式错误的输入，验证工具能正确修复后执行。
- 人工 review TODO：任何新增"智能修复"逻辑（判断类型 / 猜测格式）时，代码评审必须检查它是否在校验失败分支内触发。

## 前沿问题

- Schema 本身如何设计才能提供足够精确的 issue 信息（字段路径 + 期望类型 + 实际值），从而让修复逻辑精确定位而不是猜测？
- 修复逻辑的粒度：是每个字段独立修复（精确但复杂），还是整体重试（简单但可能丢修复信息）？
- 如果修复后再次校验仍然失败（修复不够），是继续尝试多次修复还是直接 fail？和 [[retry-budget-hard-cap]] 的关系是什么？
