# Retry Budget Hard Cap

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计任何含有"纠正 → 重试"环路的 agent 系统时，先打开这条。包括：格式错误后的救援重试、工具名幻觉后的纠正重试、跳步被拦截后的升级式纠正、语义验证失败后的重试。

不适用：纯单次调用（没有重试环路）；重试逻辑由外部框架完全托管且已有硬限制。

## 结构性不变量

所有纠正 / 重试环路必须有全局硬上限（默认 N 次）；超限时强制终止当前 turn 并向上层报错，不得继续注入纠正消息；成功的工具调用重置该计数器。

## 它预防的症状簇

- P-007：纠正循环无限重试 → 硬上限让"一次格式失败"无法演化为无界循环，计数器到顶立即中止并上报。
- P-003 第三次纠正强硬终止：升级式纠正三步后必须 break，这是 P-007 硬上限的一种具体触发条件。
- P-011 验证重试：业务规则验证失败后的重试也必须受此约束，防止语义修复循环无界。

## 留在 runbook 的部分

- P-001 救援解析（rescue parse）：三种格式尝试是在判定失败之前的逐步降级，不是循环重试，不占用纠正计数。
- P-002 注入工具列表：是纠正消息的内容策略（"工具不存在，从 [list] 选一个"），属于运行时恢复措辞，不是计数机制。
- P-003 方案 2 升级式纠正语气：礼貌 → 直接 → 强硬的语气升级是消息内容，不是计数门控。
- P-004 工具错误分类（HardError vs ToolResolutionError）：分类规则决定什么算一次"失败纠正"，见 [[tool-error-split-counting]]。

## 代价 / 范围

- 代价：需要在 agent 框架层维护一个 per-turn 的纠正计数器，并在循环入口检查；调用成功时主动重置。实现成本极低（单整数变量 + 条件检查）。
- 适用：任何 agent 框架，包含格式纠正、工具名纠正、跳步纠正、语义验证重试等任意类型的重试环。
- 失效：外部系统（如 LLM 服务商 SDK）自带的自动重试不受此计数器控制——需明确在框架层区分"业务重试"和"基础设施重试"，后者走 SDK 层的 backoff，不纳入此计数。

## 实例化成项目 DC 时要补的校验

- 架构：agent loop 中存在唯一的纠正计数器（或按类型分计数），在 turn 开始时初始化，工具成功调用时重置，纠正消息注入时自增，超阈值时 raise 或 return error。
- 静态检查：grep 代码，确认不存在未连接到计数器的"直接 continue / 直接 inject_correction"路径。
- eval：构造「永久失败」用例（工具被 mock 成永远返回格式错误）→ 期望系统在 N 次内退出并上报，不无限循环。测试计数器重置：穿插一次成功调用，确认重置后重新从 0 开始计。
- 人工 review TODO：代码评审检查每个"向模型注入纠正消息"的调用点是否都连接到同一计数器。

## 前沿问题

- N（默认 3）在不同任务复杂度下是否需要动态调整？（探索型任务可能需要更多重试，写入型任务需要更严格的上限）
- 超限后"向上报错"的语义：应该 escalate 给人类审批，还是 abort-and-report，还是切换到更保守的 fallback 策略？
- 多层嵌套 agent 的情况下，内层 agent 的计数器和外层 orchestrator 的计数器应该独立还是联动？
