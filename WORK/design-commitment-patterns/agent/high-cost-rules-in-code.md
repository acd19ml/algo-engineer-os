# High-Cost Rules in Code

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计长生命周期 / 多模型迁移 / 多人协作的 agent 系统，且系统中存在安全或合规性规则时，先打开这条。特别是在你打算用 system prompt 表达"不要做 X"时，先问：如果模型换版本后 X 发生了，代价有多高？

不适用：一次性 / 临时性 agent（只跑一次，跑完就丢），不值得为它写代码级规则；规则被违反的代价低（只是"不太美观"）。

## 结构性不变量

被违反代价高的安全规则（触发频率高 / 副作用持久 / 检测成本高，三者满足其中两项即可）必须实现为确定性代码检查，不得仅依赖 system prompt 中的自然语言约束；低代价偏好规则（被违反后影响可撤销且即时可见）可以放 prompt。

## 它预防的症状簇

- P-024：安全规则写在 prompt 里，模型版本一变就失效 → 高代价规则编码为确定性检查后，模型版本迁移不影响规则执行：代码层的正则 / 路径匹配 / 命令黑名单不随模型权重变化。

## 留在 runbook 的部分

- P-024 方案 3 升级判断框架（破坏触发频率 / 副作用持久性 / 检测成本三个维度）：这是帮助判断哪些规则需要升级的决策工具，属于评估流程，不是不变量本体。
- P-024 方案 2 低代价规则放 prompt 的具体示例（风格偏好 / 流程建议）：这是配套的分类参考，留在 runbook。

## 代价 / 范围

- 代价：高代价规则需要在代码层维护（而非只改 prompt），版本迭代时规则本身也需要更新；规则太多会导致维护负担过重——判断框架是用来"少加好加"，不是"全加上"。
- 适用：任何长生命周期 / 可能更换模型版本的 agent 系统；任何操作有副作用的 agent（写文件、执行命令、调 API）。
- 失效：一次性实验性 agent，规则维护成本高于收益；所有操作都在沙箱内、副作用不出沙箱的系统。

## 实例化成项目 DC 时要补的校验

- 架构：存在独立的"规则分级清单"文档（列出哪些规则是代码层，哪些是 prompt 层）；代码层规则有对应的可测试实现（不只是注释说"这里应该有规则"）。
- 静态检查：扫描 system prompt，找出所有"不要 / 禁止 / 必须"条目，对照规则分级清单，确认高代价规则没有遗漏在 prompt 里。
- eval：构造"模型版本切换测试"：在不同模型（或不同温度）下尝试触发高代价规则，验证代码层检查拦截了违规（而非依赖模型听话）。
- 人工 review TODO：每次模型版本升级时，必须 review 规则分级清单，确认没有高代价规则因为"新模型更听话了"而被降级回 prompt（没有量化数据证明安全时不降级）。

## 前沿问题

- 判断"副作用持久性"的标准如何量化？（一次 git push 可以被 revert，.zshrc 修改持续存在直到被手动修复——边界在哪里？）
- 代码层规则和 prompt 层规则之间是否需要显式的"优先级解析"（二者冲突时谁赢）？
- 随模型能力提升，未来是否会有"可证明安全的 prompt 规则"（高代价规则可以放 prompt 且有形式化保证）？这对 pattern 的有效性有什么影响？
