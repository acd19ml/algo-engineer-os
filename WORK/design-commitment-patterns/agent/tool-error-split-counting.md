# Tool Error Split Counting

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计的工作流中存在"工具失败后决定是否继续"的逻辑，且工具集中包含查询 / 搜索 / 探索型工具（失败原因可能是"资源不存在"而非"系统出错"）时，先打开这条。

不适用：所有工具都是写入型 / 副作用型且"资源不存在"本身就是前提错误的场景（如"删除用户 X"，X 不存在不是 404 而是请求错了）；工具失败必然意味着系统问题的场景。

## 结构性不变量

系统必须对工具失败显式分类：HardError（工具本身损坏 / 系统异常）累加进全局错误计数器触发终止；ToolResolutionError（格式合法但资源不存在，类 HTTP 404）不累加全局计数，只将"资源不存在"语义返回模型，让它换参数重试。两类错误使用独立计数器，不得混用同一计数器。

## 它预防的症状簇

- P-004：工具错误计数混淆，系统过早放弃 → 分类计数让"探索型参数猜错几次"不会耗尽全局错误预算，模型在没出现系统故障的情况下不会被过早中止。

## 留在 runbook 的部分

- P-004 具体的实现方案（异常类型 vs 返回值 `error_kind` 字段）：这是实现机制的选择（抛异常 vs 返回结构体），属于工程细节，不是结构性不变量本体。
- P-007 纠正循环上限：ToolResolutionError 后的换参数重试同样需要受 [[retry-budget-hard-cap]] 约束（合法探索也不能无界）。

## 代价 / 范围

- 代价：工具层需要返回带分类标签的错误结构（而非统一的 Exception 或 bool）；调用层需要分支处理两类错误。工程量少，但对工具接口定义有约束（不能只返回 success / failure 二元结论）。
- 适用：搜索 / 查询 / 探索类工具为主导的工作流（探索型任务中，参数猜错是正常行为而非系统异常）。
- 失效：写入 / 副作用型工具，"目标不存在"本身就意味着请求前提错误，不应被当作可重试的探索行为。此类工具上应把"不存在"也归为 HardError 或直接向用户报错。
- 失效：工具是黑盒 / 第三方 API，无法区分 4xx 和 5xx 语义时，退化为统一计数。

## 实例化成项目 DC 时要补的校验

- 架构：工具返回类型定义中存在区分 HardError 和 ToolResolutionError 的字段（枚举 / 异常类型层级）；Dispatcher 中存在对应的分支路由逻辑。
- 静态检查：扫描所有工具实现，确认没有工具把"资源不存在"（404 语义）和"系统异常"（500 语义）混在同一个 raise Exception 里抛出。
- eval：构造两组用例：（1）查询不存在的资源 N 次后成功找到 → 期望全局计数器不被耗尽；（2）工具服务挂掉 2 次 → 期望全局计数器耗尽后中止。
- 人工 review TODO：代码评审检查每个工具的错误处理逻辑，确认是否正确区分了两类错误语义。

## 前沿问题

- 对于外部第三方 API（如搜索引擎、数据库），如何可靠区分"资源不存在"和"服务降级返回空结果"？
- ToolResolutionError 后的换参数重试次数应该独立设上限吗？还是统一受 [[retry-budget-hard-cap]] 的全局计数约束？
- 多工具并发调用时，独立计数器怎么聚合？（某工具频繁 ToolResolutionError 是否需要触发单工具级别的早停？）
