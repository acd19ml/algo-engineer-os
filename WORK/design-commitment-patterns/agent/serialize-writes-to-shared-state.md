# Serialize Writes to Shared State

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计多 agent 系统，且多个 agent 需要写入共享的状态（文件系统 / 数据库 / 外部 API 状态等）时，先打开这条。

不适用：多 agent 只读共享状态（不写），不存在并发写入问题；每个 agent 有独立工作空间，共享只有最终结果合并（不共享中间状态）。

## 结构性不变量

多个 agent 对同一共享可变状态的写入操作必须序列化执行（任意时刻最多一个 agent 在写入共享状态），不得并发写入；或者每个 agent 分配完全独立的工作空间，消除共享可变状态（以隔离代替序列化）。

## 它预防的症状簇

- P-036：多个 Agent 并行写入共享状态，导致工具冲突和状态损坏 → 序列化执行让任意时刻只有一个 agent 修改共享状态，竞态条件从架构层不可能发生；独立工作空间方案让并发写入不再是"共享"写入。

## 留在 runbook 的部分

- P-036 方案 0 止血（检查并恢复损坏状态）：这是已经发生竞态损坏后的紧急处理过程，留在 runbook。
- P-036 方案 3 只读并行分析（读操作可以并行）：这是一个优化建议，明确了序列化的精确范围（只写操作），留在 runbook 作为实施细节。

## 代价 / 范围

- 代价：序列化执行丧失并行带来的速度收益；独立工作空间方案需要最终合并步骤，增加协调复杂度。
- 适用：任何多 agent 协作且写入共享文件 / 数据库 / 外部状态的场景。
- 失效：agent 任务完全独立（不共享任何中间状态）时不需要这层；沙箱基础设施已在文件系统层保证隔离时，应用层不需要重复实现。

## 实例化成项目 DC 时要补的校验

- 架构：存在明确的共享状态写入协调机制（序列化队列 / 锁 / 独立工作空间分配策略）；多 agent 架构文档中明确识别了哪些状态是共享的、哪些是独立的。
- 静态检查：枚举所有可能被多个 agent 写入的资源（文件路径 / 数据库表 / API 端点），确认每个资源有对应的序列化保护或独立副本。
- eval：构造两个 agent 并行写入同一文件的测试场景，验证最终文件内容完整（不存在部分覆盖 / 交织写入）。
- 人工 review TODO：代码评审时，对每个"启动并行 agent"的调用点，检查这些 agent 是否会写入共同资源；如果会，检查是否有序列化保护。

## 前沿问题

- 序列化执行降低了并行带来的速度优势——如何在"需要协调的写入"和"可以真正并行的读写"之间精确画边界？
- 独立工作空间方案的最终合并步骤本身也是写入共享状态——合并步骤是否也需要序列化保护？
- 基础设施层（如 Git 的 merge / rebase 机制）是否可以代替应用层的序列化保护？什么时候委托给基础设施层更合适？
