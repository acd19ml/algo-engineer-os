# Business Rule Validation Layer

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你的系统用 LLM 生成结构化输出，且输出中存在字段取值范围约束或字段间逻辑关系（如 `status: "completed"` 时 `completion_time` 不能为 null）时，先打开这条。约束解码 / strict mode 保不了这类语义约束。

不适用：输出字段无业务规则（只有类型约束）；纯单次调用无需重试。

## 结构性不变量

LLM 结构化输出在进入工具执行或下游流程之前必须经过独立的业务规则验证层；验证层与约束解码层相互独立、分工明确：约束解码保语法合法性，业务规则验证层保语义合理性；验证失败时必须向模型注入包含具体失败字段和期望值的错误信息，不得只注入"验证失败"等模糊提示。

## 它预防的症状簇

- P-011：JSON 语法对但值不合理（语义错误）→ 业务规则验证层在语法检查之后再做语义检查，字段越界 / 字段间矛盾在执行前被拦截；具体错误反馈让模型能精准修复（"rating 必须在 1-5，你输出了 7"），而不是盲目重试。

## 留在 runbook 的部分

- P-011 具体实现库选择（Pydantic / Zod）：框架选型是项目级决定，不是结构性不变量。
- P-011 验证失败重试的次数控制：重试次数上限属于 [[retry-budget-hard-cap]] 的职责范围，本 pattern 只保证"验证必须存在"和"错误信息必须具体"。
- P-009 方案 3 语法层 rescue parse / 重试：语法层恢复过程独立于业务规则验证，留在 runbook。

## 代价 / 范围

- 代价：需要在项目中维护业务规则 schema（Pydantic model / Zod schema）；每次 LLM 输出都要经过一次验证器；规则改变时需同步维护 schema（避免验证和业务逻辑脱节）。
- 适用：有实际业务规则的结构化输出（取值范围、字段间逻辑关系、枚举外的复杂约束）；几乎所有真实业务场景都包含这类约束。
- 失效：输出字段只有类型约束（全是 int / str / bool），约束解码已经足够，业务规则层是冗余；但即使如此加一层防御也无害。

## 实例化成项目 DC 时要补的校验

- 架构：存在独立的 `validate_output(output, schema)` 函数（或等价中间件），位于 LLM 调用结果处理和工具/下游执行之间；返回结构化的失败信息（失败字段名 + 期望值 + 实际值），不只是 bool。
- 静态检查：扫描所有"LLM 输出 → 工具执行"路径，确认每条路径都经过验证器（不存在绕过验证直接执行的代码路径）。
- eval：构造已知违反业务规则的 LLM 响应样本（rating=7 / status+time 矛盾），喂入系统，期望：（1）验证器拦截；（2）注入的错误信息包含具体字段和期望值；（3）模型重试后输出合法值。
- 人工 review TODO：代码评审检查错误注入模板，确认不存在"验证失败，请重试"这类模糊消息；审查业务规则 schema 是否与实际业务逻辑一致（最容易脱节的地方）。

## 前沿问题

- 业务规则 schema 和产品业务逻辑往往双写，如何保证同步（schema 是代码，业务逻辑是 spec / DB）？
- 当验证器本身发生错误（验证库 bug / schema 定义错误）时，应该 fail-closed（拦截输出）还是 fail-open（跳过验证放行）？
- 对于字段间复杂逻辑（如"当 type=A 时字段 X 必须满足条件 Y"），Pydantic validator 和 LLM 两种方式的维护成本如何权衡？
