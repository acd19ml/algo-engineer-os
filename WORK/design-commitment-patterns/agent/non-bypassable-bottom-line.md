# Non-Bypassable Bottom Line

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你为 agent 设计"最高信任 / 跳过确认 / YOLO 模式"时，先打开这条。凡是存在"用户可以选择跳过所有确认"的功能，都需要同时设计不可绕过的底线。

不适用：没有信任分级的 agent（每次都确认），底线概念不适用（但这类 agent 可用性差）；agent 完全在沙箱中运行，越界操作被基础设施层拦截。

## 结构性不变量

底线保护清单（shell 启动文件 / SSH 配置 / agent 自身配置 / 版本控制配置 / 系统级路径 / 项目敏感文件等）必须在代码中硬编码；即使最高信任模式也不例外；底线清单不得设计为用户可配置项（配置接口本身是攻击面）；底线被触发时必须向用户展示具体影响说明（不只是普通确认框）。

## 它预防的症状簇

- P-025：最高信任模式下安全底线（如 .zshrc）被破 → 硬编码底线清单让"完全信任"和"无限制"脱钩，副作用持久且检测成本高的操作即使在 YOLO 模式下也不会静默执行。

## 留在 runbook 的部分

- P-025 方案 3 底线被触发时的具体警告措辞（"这个操作会修改终端启动文件，每次开终端都会执行其中内容，建议先查看 diff 再确认"）：这是运行时 UX 设计，不是不变量本体。
- P-025 方案 0 止血（停止 agent、切断写权限、检查 diff）：这是已经发生破线后的紧急响应，属于运行时恢复，留在 runbook。

## 代价 / 范围

- 代价：底线清单需要在代码中硬编码（不能用配置文件），版本迭代时必须在代码 PR 里显式更新；不可配置性让高级用户无法自定义底线（如果用户真的需要修改 .zshrc，需要退出 agent 手动操作）。
- 适用：任何有"最高信任 / 跳过确认"模式的 agent，无例外。
- 失效：没有信任分级的 agent（每次都确认），底线概念不适用。
- 反向陷阱：不要为了用户体验把底线清单变成可配置项——配置的入口本身就是攻击面（恶意 agent 可以通过修改配置文件来删除自身的底线保护）。

## 实例化成项目 DC 时要补的校验

- 架构：底线清单是代码中的 `const`（不可在运行时修改）；Dispatcher 在执行任何写操作之前，对目标路径 / 命令做底线匹配，命中则不论信任级别一律拦截并警告；警告消息包含具体影响说明（不只是"被禁止"）。
- 静态检查：确认底线清单没有被挪到配置文件、环境变量或数据库（常见的"让用户自定义"重构陷阱）；扫描 Dispatcher 中的最高信任模式分支，确认不存在"YOLO 模式跳过底线检查"的代码路径。
- eval：在最高信任 / YOLO 模式下发送"修改 .zshrc"指令，期望系统拦截并展示具体警告；构造"agent 自我修改其配置文件"场景，验证同样被拦截。
- 人工 review TODO：每次修改 Dispatcher 的"最高信任模式"相关代码时，必须 review 底线清单检查逻辑是否完整覆盖（最高信任模式的改动最容易误伤底线保护）；原码注释必须写明"这些检查是不可绕过的"，对抗未来"为了便利而放宽"的 PR。

## 前沿问题

- 底线清单的边界如何定义？（.zshrc 明确应该在里面；某个项目特定的 `config.json` 是否应该在里面？）
- 当 agent 在容器 / 虚拟机环境中运行时，宿主机的 .zshrc 是否还需要保护（agent 可能无法触及），还是只需要保护容器内的 shell 配置？
- 是否存在"经用户完整信息告知后可绕过底线"的场景？如果有，如何设计这条绕过路径使其本身不成为攻击面？
