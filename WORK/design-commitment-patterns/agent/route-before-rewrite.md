# Route Before Rewrite

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你为 RAG 系统添加 query rewriting 模块时，先打开这条。

不适用：系统只有一类 query，不存在"简单直接查询"和"复杂需要改写的查询"的区分；改写组件已经内置了分类逻辑。

## 结构性不变量

RAG 系统中 query rewriting 必须由路由分类前置控制：系统先判断 query 是否需要改写（简单查询 / 含明确业务词 / 直接命中等情况），只有判定为需要改写的 query 才进入 rewriting 流程；不得对所有 query 无条件触发改写。

## 它预防的症状簇

- P-039：Query 改写后检索效果反而下降 → 路由分类把不需要改写的查询（简单查询 / 含业务词 / 直接表达）直接送检索，绕过可能产生负效果的改写；改写带来的召回率下降从架构层被限制在真正需要改写的子集上。

## 留在 runbook 的部分

- P-039 方案 2 业务术语注入（在改写 prompt 里动态注入术语解释）：这是改写流程内部的质量保证措施，是改写逻辑的细节，留在 runbook。
- P-039 方案 3 实体识别 + 保护：这是改写过程中对特定词汇的保护策略，也是实现细节，留在 runbook。

## 代价 / 范围

- 代价：需要实现路由分类器（规则 / 小模型 / 主模型自判三选一，各有成本权衡）；路由器本身的误判会把需要改写的查询漏掉（路由不改写 → 召回不足）或把不需要改写的查询错误送改写（改写 → 降质）。
- 适用：任何部署了 query rewriting 的 RAG 系统，且系统的 query 混合了简单查询和复杂查询。
- 失效：系统的所有 query 本身就需要改写（如全是高度模糊的开放问题，没有简单直接查询），路由分类无意义。

## 实例化成项目 DC 时要补的校验

- 架构：query 处理 pipeline 中，query rewriting 模块的入口被路由分类器包裹；路由分类器输出"需要改写"/"直接检索"/"专用术语查询"等分类，rewriting 只在"需要改写"分支触发。
- 静态检查：扫描 query 处理流，确认 rewriting 函数不在主流程的必经路径上（所有 query 都经过），而是在路由分叉后的子分支中。
- eval：构造一批"简单直接查询"样本（单实体 / 直问直答），验证这批 query 的召回率在加路由之后不低于无 rewriting 时；构造一批"复杂模糊查询"样本，验证加 rewriting 后召回率提升。
- 人工 review TODO：上线前必须有简单查询 vs 复杂查询的 recall 对比数据；不能靠"感觉 rewriting 有帮助"就全量上线。

## 前沿问题

- 路由分类器本身应该用规则 / 小模型 / 主模型？不同方案在成本、可解释性、误路由风险上差异很大，尚无共识最优解。
- 在业务场景下，"简单查询"和"复杂查询"的边界如何自动学习而非人工规则定义？
- Planning cache（见 P-042）和 route-before-rewrite 的交互：路由分类结果是否可以作为 planning cache 的意图标签？
