# Memory Temporal Annotation

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计跨会话记忆系统，且注入的记忆内容可能包含关于代码 / 文件位置 / 项目状态等会随时间变化的信息时，先打开这条。

不适用：记忆内容本就是"历史档案"语义（如"用户在 2024 年说过…"），历史属性是预期语义；记忆只包含稳定不变的元数据（用户偏好、联系方式等极低衰减率的信息）。

## 结构性不变量

所有注入到 agent 上下文的记忆条目，必须附带时效标注；超过时效阈值的记忆，注入时必须附加明确的过期声明（"这条记忆是 N 天前写的，其中代码位置 / 状态等信息可能已过时，在做决策前请先验证当前实际状态"）；不得将记忆中的描述以当前事实身份呈现给模型，无论记忆多么具体和权威。

## 它预防的症状簇

- P-040：过时记忆被当作当前事实权威引用 → 时效标注打破记忆的"隐式权威感"——模型看到过期声明后，会将记忆视为"历史观察"而非"当前事实"，在行动前验证而非直接引用。具体性越高的记忆（文件路径、行号）被错误引用的危害越大，时效标注尤其重要。
- P-041：Skills/SOP 未维护导致过时流程被执行 → 给 Skill 文件加 `last_validated_at` 字段，超时后在索引中附加过期警告，阻止 agent 盲信过时 SOP。

## 留在 runbook 的部分

- P-040 方案 3 在访问记忆后加验证步骤（用工具验证 claim）：这是记忆访问后的运行时主动核查过程，留在 runbook；本 pattern 的不变量是注入时必须有时效声明，不是说访问记忆后一定要验证（验证由时效声明触发的 agent 判断决定）。
- P-040 方案 2 按记忆类型设不同时效阈值（代码 48h / 项目状态 2 周 / 用户偏好 30 天）：这是时效阈值的具体配置，是实现参数，留在 runbook。
- P-041 方案 2 Skill 执行失败时触发审查：这是 Skill 维护的运行时过程，留在 runbook。

## 代价 / 范围

- 代价：需要为每条记忆维护 `written_at` / `last_validated_at` 时间戳；注入逻辑需要按时效计算并生成过期声明；时效阈值需要按记忆类型维护（增加配置复杂度）。
- 适用：任何使用跨会话记忆的 agent 系统，且记忆中包含可能随时间失效的技术描述（代码位置 / 项目状态 / 接口名称等）。
- 失效：记忆内容本就是历史档案性质（用户说了什么、决策了什么），这类记忆本身就是历史，时效标注反而造成混乱——需要区分"过去事实"（本来就是历史）和"状态描述"（曾经是当前的，可能已变化）两类记忆。

## 实例化成项目 DC 时要补的校验

- 架构：记忆存储 schema 中有 `written_at` 字段；记忆注入函数在写入上下文前检查时效，超阈值时附加过期声明模板；Skill 文件 frontmatter 中有 `last_validated_at` 字段。
- 静态检查：扫描记忆注入代码，确认所有注入路径都经过时效检查（不存在直接注入原始记忆内容的路径）。
- eval：构造一条 7 天前写的、内容关于"函数在文件 X 的第 Y 行"的记忆，注入后验证：（1）上下文中存在过期声明；（2）模型在引用该记忆时使用了验证语气（"根据之前记录，但可能已过时"）而非断言语气（"函数在 X 文件第 Y 行"）。
- 人工 review TODO：新增记忆写入路径时，确认同时设置了 `written_at` 时间戳；新增 Skill 文件时，确认有 `last_validated_at` 字段和初始值。

## 前沿问题

- "过去事实"（历史记录）vs"状态描述"（曾经的当前状态）的区分——如何在写入记忆时自动识别这两类，以便只对状态描述类记忆应用时效标注？
- 时效阈值的最优设置：代码相关记忆 48h、项目状态 2 周这些数字是否有数据支撑？不同团队的代码变化速率不同，阈值是否应该可配置？
- 级联更新问题（见 [[agent-memory-cascading-update]] 节点）：时效标注只解决"知道这条记忆可能过时"，但不解决"搬家后通勤时间的记忆也应该标记为过时"——这是更深层的记忆依赖图问题，尚无完善解。
