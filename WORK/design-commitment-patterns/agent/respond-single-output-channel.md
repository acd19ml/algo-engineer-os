# Respond Single Output Channel

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你决定用小模型 / 本地模型 / 量化模型做**多步工具调用 agent**，并且模型需要在"继续调工具"和"输出最终文本"之间来回切换时，先打开这条。

不适用：前沿模型在目标工具集上模式选择已经稳定；纯单步工具调用；没有工具通道的纯文本助手。

## 结构性不变量

Agent 只有一个输出通道：工具调用。文本回复也通过合成的 `respond` 工具发出，模型不在"输出文本"和"调工具"之间做模式选择。

## 它预防的症状簇

- P-008：模型在"调工具"和"输出文本"之间频繁选错 → 模式选择这个分支从架构里消失。
- P-001 方案 3：该调工具时输出了自由文本 → 最终文本回复也成为工具调用后，自由文本直出路径被关闭。

## 留在 runbook 的部分

- P-001 方案 1 rescue parse：这是格式错误后的运行时恢复，不是不变量。
- P-001 方案 2 后端切换：这是诊断 / 环境选择，不是不变量。
- P-002 注入可用工具列表：这是工具名错误后的恢复过程，不是不变量。
- P-005 后端 ablation：这是 diagnostic-procedure，帮助判断模型 / 后端边界，不是设计承诺本体。

## 代价 / 范围

- 代价：所有文本输出包一层工具调用，可读性 / token 略增，调试多一层。
- 适用：模型模式选择不稳定时，尤其是 ≤30B、本地、量化模型，或 eval 显示模式选择错误显著高于格式恢复能解释的范围。
- 失效：前沿云模型在目标工具集上已经稳定时，这条可能是冗余开销。项目可以不采纳，或采纳成"软单通道"。

## 实例化成项目 DC 时要补的校验

- 架构：agent loop 只有一个出口 = 工具分发；不存在直接 return 文本给用户的代码路径。
- 静态检查：定位任何绕过 `respond` 的输出点。
- eval：构造混合用例，包含"应该继续调工具"和"应该输出最终答复"两类；模型走非工具通道即 fail。
- 人工 review TODO：如果暂时没有静态检查，代码评审必须检查所有用户可见输出是否经过 `respond`。

## 前沿问题

- 模型能力到什么程度后，单通道收益低于 token / 调试成本？
- 是否存在"软单通道"：默认工具模式，特定低风险文本允许直出，同时不重新引入 P-008？
- `respond` 工具会不会污染工具选择分布，让模型在应该调用业务工具时过早 respond？
