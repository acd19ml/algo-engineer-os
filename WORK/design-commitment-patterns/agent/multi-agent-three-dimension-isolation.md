# Multi-Agent Three-Dimension Isolation

`status: candidate`   `instantiates-to: project-design-commitment`

## 何时打开

当你设计多 agent 系统，且各 agent 需要承担不同职责（探索 / 规划 / 实现 / 验证等阶段角色），先打开这条。

不适用：单 agent 系统；所有 agent 承担完全相同的职责（如并行研究任务的同构 agent）。

## 结构性不变量

多 agent 系统中按职责阶段划分的每个角色，必须同时从三个维度做硬隔离：（1）工具集隔离——角色只拥有执行其职责所需的工具，物理上无法调用越界工具；（2）Prompt 隔离——角色有独立的 system prompt，明确定义能做什么和不能做什么；（3）权限隔离——角色在 runtime 层被锁定相应权限。三维缺任一维，仅剩软约束（prompt）会退化为无效。

## 它预防的症状簇

- P-029：Agent 做任务时"顺手"做了范围外的事（角色越界）→ 工具集物理隔离让探索阶段 agent 想写文件也写不了，因为工具列表里没有写文件的工具；不依赖模型自觉性。

## 留在 runbook 的部分

- P-029 方案 3 Prompt 里的具体约束规则列表（"仅改动绝对必要的部分"等）：这是三维中 Prompt 维的具体内容，属于实现细节，留在 runbook；本 pattern 的不变量是"三维都必须有"而非"Prompt 该写什么"。
- P-022 三档权限规则（禁止/确认/允许分层）：这是权限维的具体配置策略，见 [[three-tier-deny-overrides-allow]]；本 pattern 的不变量是"权限隔离是三维之一，缺一不可"。

## 代价 / 范围

- 代价：需要为每个角色分别维护三套配置（工具集、system prompt、权限）；角色越多，维护成本越高；角色间通信需要结构化（不能靠共享上下文，而是靠结构化输出传递）。
- 适用：任何需要按阶段分工的多 agent 系统（探索→规划→实现→验证的典型模式）。
- 失效：同构并行 agent（多个 agent 做完全相同的工作，只是并行提速），不需要角色隔离。
- 额外收益：工具集隔离还带来模型成本优化——探索阶段 agent（只读工具）可以用更小、更便宜的模型（如 Haiku），验证阶段 agent 才用更强模型。

## 实例化成项目 DC 时要补的校验

- 架构：每个角色对应独立的（工具集配置 + system prompt + 权限配置）三元组；不存在"所有角色共用一套配置但靠 prompt 切换"的实现。
- 静态检查：扫描角色定义，确认每个角色的工具列表只包含其职责所需的工具；探索类角色（只读）的工具列表中不存在写 / 执行类工具。
- eval：构造"探索阶段 agent 尝试写文件"测试用例，期望系统在工具层面直接拒绝（而非靠 agent 自己决定不写）。
- 人工 review TODO：新增角色时，代码评审必须同时检查三个维度的配置，缺任一维不接受。

## 前沿问题

- 按阶段拆 vs 按功能拆之外，是否存在混合拆法（先按阶段、再按功能细分）？二级拆分会不会再次引入角色冲突？
- 当任务确实需要同一个 agent 在阶段间切换时（如探索中发现需要立即修复），三维硬隔离如何优雅降级（临时升权 vs 重新委派给实现阶段 agent）？
- 角色间通信必须通过文件 / 结构化输出，这套通信协议的设计有没有标准化方案？
