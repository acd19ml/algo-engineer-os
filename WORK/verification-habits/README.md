# Verification Habits: 不知道自己错了就不知道自己对了

> 和 playbook 配对使用。playbook 告诉你「怎么做」，本文告诉你「怎么知道自己做错了」。
> 起源：`PROJECTS/work/neo-official-support-agent/` 全周期设计复盘——14 个 Milestone 的计划是显式的，但验证习惯是隐式的。

## 它要解决什么问题

做完一个 Milestone、写完一份设计文档、跑完一轮 eval——产出看起来是对的。但「看起来对」可能是：

- **真的对**——设计和实现一致，假设成立
- **构造性为真**——你验证的方式只能确认它对了，不能证伪它错了
- **还没撞到边界**——当前输入没触发漏洞，换一批输入就暴露

三种状态的共同点是：**你无法区分它们**。如果你只做确认性验证（跑一遍预期路径、看结果对不对），后两种状态和第一种看起来完全一样。

本文给出 6 个验证习惯——不是为了「检查质量」，是为了**主动暴露自己可能错在哪里**。

## 核心区分：确认性验证 vs 证伪性验证

| | 确认性验证 | 证伪性验证 |
|---|---|---|
| 问什么 | 这个产出对不对？ | 如果这个产出是错的，我怎么发现？ |
| 方式 | 按预期路径跑一遍 | 故意破坏前提，看不变量是否仍成立 |
| 能发现 | 实现是否忠实于设计 | 设计本身是否鲁棒 |
| 风险 | 给你虚假的安全感——「跑通了」≠「对的」 | 需要额外设计验证场景 |
| 例子 | 确认性验证：按预期路径跑一遍，结果对 | 证伪性验证：故意破坏前提，看不变量是否仍成立 |

**最重要的判断**：如果你只能做一种验证，做证伪性验证。确认性验证只能证明「没有明显的 bug」，证伪性验证能证明「这个不变量在极端条件下仍然成立」。

## 6 个验证习惯

### V1 · 承诺审计

**做什么**：定期回溯源 pattern / 源文档，逐条检查项目 DC 的措辞是否忠实映射了源的设计边界。

**什么时候做**：写完 DC 后、项目中期复盘时、面试准备前（因为你会在面试中被追问）。

**案例**（完整记录见 `[[PROJECTS/work/neo-official-support-agent]]` 复盘）：一条 DC 的措辞在实例化时漂出了源 pattern 的设计边界——源 pattern 管的是工具级 deny-overrides-allow，DC 写成了 intent 验证。两者都「听起来合理」，回溯源 pattern 才发现。

**为什么容易漏**：DC 写完后你不会主动怀疑它——它是你自己写的。漂移是渐进的——每次微调措辞都「听起来合理」，累积起来就过了边界。

### V2 · 结构 vs 文字区分

**做什么**：对每条安全规则，追问「这个安全是代码保证的还是文字约定的？」代码保证的（如动作空间=只读）比文字约定的（如 deny 关键词列表）可靠一个数量级。把安全预算从文字侧挪到结构侧。

**什么时候做**：安全架构设计完成后、每发现一条新的 deny 规则时。

**案例**（完整记录见 `[[PROJECTS/work/neo-official-support-agent]]` 复盘）：一个 agent 的动作空间是全部只读的——没有写链工具。但安全规则仍然花了很多精力在关键词 deny 上。回溯发现：写链的真正控制是「能力不存在」（结构性），不是关键词（文字性）。重新分层后把鲁棒性预算从文字侧挪到结构侧。

**为什么容易漏**：加一条关键词 deny 比设计一个结构性安全机制快得多——短期效率掩盖长期脆弱性。

### V3 · 原则门控

**做什么**：在设计方案进入执行前，拿 KNOWLEDGE 节点的原则逐条对照——「这个方案有没有违反已知原则？」不是 review 方案本身，是拿外部原则当筛子。

**什么时候做**：任何非 trivial 的设计决策被提出后、编码前。

**案例**（完整记录见 `[[PROJECTS/work/neo-official-support-agent]]` 复盘）：一个检测层硬化计划在编码前被 KNOWLEDGE 节点拦住——计划在关键词失效证据出现前就建 LLM overlay（违反「只在犯过的错误上投入」），且用同一模型做路由和验证（违反「生成和评估必须分离」）。修正为 heldout 先行、硬化靠证据驱动。

**为什么容易漏**：设计方案是自己想的，没有人会在提交前主动拿别人的原则来挑自己的刺。

### V4 · 泛化怀疑

**做什么**：在主 eval 集上拿到 100% 后，建一个**没参与任何调参**的 heldout 集，用完全相同的规则跑。heldout 上的表现才是真正的泛化能力。

**什么时候做**：任何时候在主 eval 集上做了优化后。

**案例**（完整记录见 `[[PROJECTS/work/neo-official-support-agent]]` 复盘）：主 eval 集上某检测规则修复后达到 100%。建了一个完全独立的 heldout 集（deny/路由规则从未见过这批输入），验证修复不是对主集过拟合。

**为什么容易漏**：100% 的满足感是真实的——你会倾向于认为「做完了」而不是「还没证明它不是过拟合」。

### V5 · 最坏情况注入

**做什么**：对每个「不依赖 X 也能成立」的声称，构造一个 X 的最坏情况，看不变量是否仍成立。

**什么时候做**：任何声称「与 Y 无关」「不依赖 Z」的设计被提出后。

**案例**（完整记录见 `[[PROJECTS/work/neo-official-support-agent]]` 复盘）：某不变量声称「与 router 能力无关」。注入了一个故意全错的最坏 router——整体正确率崩了，但该不变量仍然成立。证明了不变量真正独立于被破坏的组件。

**为什么容易漏**：正常测试路径下 router 不会错得这么离谱，你不会想到构造一个「故意全错」的 router。

### V6 · 模型能力消融

**做什么**：对任何声称「与模型能力无关」的安全不变量，用不同规模/能力的模型逐档跑同一个 harness。如果最弱的模型上不变量也成立，声称才成立。

**什么时候做**：任何基于模型行为的假设被提出后。

**案例**（完整记录见 `[[PROJECTS/work/neo-official-support-agent]]` 复盘）：四种不同规模的模型逐档跑同一个 harness。安全不变量在最弱的模型上仍然成立——即使该模型的整体正确率远低于强模型。证明了不变量与模型能力无关。

**为什么容易漏**：你会倾向于用最强的模型验证，因为「最强的都过了，弱的肯定也过」——但这个逻辑在概率系统里不成立。弱模型可能用完全不同的方式失败。

## 和 Milestone 计划的关系

14 个 Milestone 的每个 M 末尾都有自检 checklist——那些是**确认性验证**（「每个组件的边界清楚」「安全组件在语义组件之前」）。本文的 6 个习惯是**证伪性验证**——在完成每个 M 后，额外问：

```
M 完成后 → V1-V6 里哪些适用于这个 M 的产出？
         → 用什么方式主动暴露这个产出可能错在哪？
```

| Milestone | 优先适用的验证习惯 |
|---|---|
| M2 安全架构 | V2（结构 vs 文字）、V1（DC 措辞审计） |
| M3 系统架构 | V3（原则门控——context-engineering / agent-tool-design 的原则） |
| M4 实现契约 | V1（DC 校验是否忠实于源 pattern） |
| M6 Eval 体系 | V4（heldout 设计） |
| M7 运行验证 | V4（泛化怀疑）、V5（最坏情况注入）、V6（模型能力消融） |
| M8 方法论提取 | V3（提取的原则是否只在这个项目场景下成立？在其他场景下呢？） |
| M9 生产部署 | V2（上线门槛是否全部有结构性保证？） |
| M11 事故响应 | V5（Kill switch 在最坏情况下生效吗？） |
| M13 持续改进 | V6（新模型→横扫→和旧 baseline 对比，不变量仍成立吗？） |

---

## 使用方式

不是 checklist——**是习惯**。不需要每条都做、每次都做。但每完成一个重要产出，停一下，问自己：

> 我现在的验证方式，是确认它对了，还是试图证明它错了？如果是前者，我有没有漏掉一个可以证伪它的验证？
