## P-013：Agent 跑几十步后开始机械重复之前的动作

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 在长任务（几十步以上）跑着跑着进入"复读机"状态
- 不断重复之前的动作模式：比如批量处理简历时，对每份简历都重复完全相同的操作序列，即使该简历有特殊情况
- 输出 schema 没错，但**动作选择被上下文中的历史 pattern "锁定"**

**根因**

Agent 执行几十步之后，上下文里积累大量类似动作和观测，模型会被历史 pattern 强化，进入"模式锁定"状态。它看起来还在按 schema 输出，但动作选择已经不再充分响应当前样本。

**当前方案**

注意：本条方案尚不成熟。

*方案 1：上下文压缩（实际可行）*

减少重复 pattern 在上下文中的堆积。参考 P-006 的三阶段 Tiered Compact——旧的相似动作 / 观测被截断或丢弃，减少模型可以"模仿"的 pattern 数量。

*方案 2：阶段性 Context Reset（实际可行）*

在长任务的关键节点显式重置上下文——只保留结构化的总结，把详细历史丢掉。模型从一个干净的 context 重新开始决策。参见 P-027 Context Reset 机制。

*方案 3：输出微变异（思路占位，未广泛验证）*

Manus 思路：输出 schema 不变，但同一个 schema 下的**序列化方式有微小变化**（字段顺序 / 空白 / 等价表示），打破模型的自回归惯性。**节点明确标注：具体实现细节在公开资料中披露较少，不算被广泛验证的标准做法**。

**适用与失效**
- 适用：长任务 agent（步数 ≥ 数十步）+ 任务本身有重复结构（批处理、多对象循环）
- 失效：短任务 / 任务步骤差异大、本来就不重复的场景不会触发
- 注意：方案 3 在生产前先在自己场景做 A/B，不要照抄

**源**

`[[structured-output]]`（Layer 6 + Manus 实践分享）。验证状态：未验证（方案 3 在公开资料中本身也未充分验证）。

**See also**
- P-006 多步工作流上下文膨胀（共享方案 1 上下文压缩）
- P-027 上下文焦虑提前收工（共享方案 2 Context Reset）
- `[[agent-context-compaction]]`

## P-029：Agent 做任务时"顺手"做了范围外的事（角色越界）

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 让 agent 改一个 bug，他改完顺手把半个文件重构了
- 让他加一个功能，他给你加了三层抽象和五个你没要求的错误处理
- 让他加一行注释，他把整个文件的注释都重写了
- 和 P-022（权限越界）不同：P-022 是操作了权限范围外的系统；P-029 是在被允许的范围内做了过多的事

**根因**

单个 agent 同时承担多个角色（探索 + 规划 + 实现 + 验证），各角色的行为边界相互污染。"探索的好奇心污染了实现的克制"——不是能力问题，是架构问题。光用 prompt 切换角色解决不了这个——当上下文里同时塞着探索和实现两个任务的痕迹时，模型自己都分不清现在该是什么角色。

**当前方案**

*方案 1：按职责阶段拆分 agent（首选）*

按阶段分（探索 / 规划 / 实现 / 验证），不按功能分（前端 / 后端 / 测试）——前者解决角色冲突，后者不解决。探索 agent、规划 agent、实现 agent、验证 agent 各有各的 system prompt、工具集、权限边界。

*方案 2：工具集硬隔离*

探索阶段 agent 的工具列表里没有写文件工具；实现 agent 只有编辑工具，没有大范围代码库搜索工具。物理上切断"探索冲动"的触发路径——不依赖模型自觉性。

*方案 3：Prompt 里加具体可执行的约束规则*

具体规则比抽象原则有效：
- "仅改动绝对必要的部分"
- "切勿顺手优化相邻代码、注释或格式"
- "未出问题的代码绝不重构"
- "不要为假设的未来需求做设计"

在 prompt 里写这些是因为有些行为约束无法用工具集硬切（语言 / 注释风格类）——**硬约束兜底，软约束补缝**。

**适用与失效**
- 适用：多步编码 agent；任何发现 agent "超纲"行为的场景
- 失效：任务本身确实需要 agent 在多个阶段间切换时，这条不适用

**源**

`[[agent-role-isolation]]`（Cloud Code 六 agent 按阶段拆分；`getSimpleDoingTasksSection()` 具体规则列表）。验证状态：未验证（Cloud Code 外部 claim）。

**See also**
- P-022 Agent 做了超出指令范围的事（权限层越界，互补症状）
- `[[multi-agent-three-dimension-isolation]]`（DC pattern）
- `[[agent-permission-system]]` · `[[agent-role-isolation]]`

## P-035：给子 Agent 下达模糊任务指令，多个子 agent 做了相同的事

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 多 agent 并行研究任务，完成后发现几个子 agent 搜索了完全相同的内容
- 主 agent 输出"请各子 agent 研究某主题"但没有分工
- 子 agent 的结果几乎重复，并行带来的覆盖面提升几乎为零
- 成本是单 agent 的 N 倍，效果和单 agent 差不多

**根因**

子 agent 委派指令太简短 / 模糊，没有包含"目标、输出格式、工具范围、任务边界"——子 agent 只能基于自己的判断去理解任务，而判断逻辑几乎相同，所以做了一样的事。

**当前方案**

*方案 1：委派指令包含四要素（默认先上）*

目标（期望输出是什么）+ 输出格式（结构化结果怎么组织）+ 工具范围（只用哪些工具）+ 任务边界（明确说"不要搜 X，只搜 Y"）。每个子 agent 的指令应该显式包含其"专属角度"——"子 agent A 专注于 X 维度，子 agent B 专注于 Y 维度"。

*方案 2：主 agent 先生成任务分解计划*

主 agent 先输出一份包含明确分工的计划（各子 agent 负责什么 / 不负责什么），由 orchestrator 或用户确认后再委派——避免"感觉分工了但实际没有"。

*方案 3：按规模缩放使用多 agent*

参考 Anthropic Research 系统的缩放规则：
- 简单事实核查 → 只用一个 agent，3-10 次工具调用
- 直接对比 → 2-4 个子 agent
- 只有复杂研究问题 → 才用 10 个以上子 agent

不是任务"感觉复杂"就应该用多 agent，是任务确实可以并行化分工才用。

**适用与失效**
- 适用：多 agent 并行工作场景（研究、分析、数据收集类任务）
- 失效：子 agent 任务本来就相同（并行独立验证而非并行分工）

**源**

`[[multi-agent]]`（Anthropic 研究系统早期案例：允许主 agent 简短委派时，多个子 agent 做了完全相同的搜索）。验证状态：未验证。

**See also**
- P-013 模式锁定（长任务中另一种"重复"问题）
- `[[multi-agent]]`

## P-042：Planning cache 命中了错误模板，执行了错误的检索路径

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agentic RAG 系统使用了 Planning cache，某次命中的模板对当前 query 不适用
- 比如"苹果公司 2024 营收"的查询命中了"苹果手机产品介绍"的模板，执行了完全不相关的检索步骤
- 系统没有发现执行结果和任务不一致，直接输出了基于错误路径的答案
- 或者模板命中率极低（<30%），Planning cache 带来的延迟收益微乎其微

**根因**

高层意图标签太粗或相似度计算不准确，导致意图接近但任务结构不同的 query 命中了错误模板。Planning cache 节省了规划开销，但也移除了每次规划时的"适应性检查"。

**当前方案**

*方案 1：模板命中后先验证再执行*

命中模板后不直接执行，让小模型用一句话确认"这个模板的步骤框架是否适用于当前 query"；只有确认后才进入执行。验证成本远低于完整规划，但能过滤掉明显的误命中。

*方案 2：记录误命中并细化意图标签*

记录模板误命中案例（执行完发现结果和任务不匹配），把误命中的 query 对反馈给意图标签系统，细化分类粒度——把"苹果"这类歧义词在标签系统中分成不同子类。

*方案 3：给模板加典型 query 示例*

命中时把模板附带的"典型 query 示例"和当前 query 对比；差异过大时降级为完整 Agentic RAG 流程而非使用缓存模板。

**适用与失效**
- 适用：使用 Planning cache 的 Agentic RAG 系统
- 失效：任务类型高度多样化、重复率低的系统，Planning cache 价值有限，也不值得花精力解决误命中问题

**源**

`[[agentic-rag-planning-cache]]`（高层意图标签粒度的开放问题；模板命中率 vs 误命中率的 trade-off）。验证状态：未验证。

**See also**
- P-019 RAG 全链路诊断
- P-039 Query 改写降低检索效果
