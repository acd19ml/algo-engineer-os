# Agent Runbook

**症状导向的 agent 故障排查手册**。读者带着"我看到了什么问题"来查，每条给出可立即操作的候选方案。结构上接近 SRE 故障排查手册，**不是教科书也不是设计文档**。

和 KNOWLEDGE 节点的区别：KNOWLEDGE 节点是"概念为什么是这样"（因果叙述 + 反事实），runbook 条目是"我看到了 X，下一步做什么"（症状 → 候选方案）。形态不同，用途不同。

---

## 怎么用

1. 遇到问题，ctrl-F 关键词搜 `runbook.md`（或下面的索引）
2. 命中条目后，**按方案 1 → 方案 N 的顺序试**——越靠前越便宜，最先该试
3. 看"适用与失效"判断是否真适用你的场景
4. 试完后，如果"验证状态"还是"未验证"，回来更新

## 文件组织规则

**当前阶段（≤30 条）：单文件**

- 所有正文在 `runbook.md`
- 索引在本 README

**超过 30 条后：按 agent 生命周期拆子目录**

```
WORK/runbooks/agent/
├── perception/    # 感知/输入解析
├── planning/      # 规划/推理
├── tool-use/      # 工具调用（P-001 / P-007 大概率在这里）
├── memory/        # 记忆/上下文
├── execution/     # 执行/副作用
└── reliability/   # 重试/降级/可观测
```

**不要按"问题难度"或"出现频率"分**——这两个会随时间变化，归档结构会失效，之后要么不维护要么大规模迁移。

## 写新条目

模板在 [`META/templates/runbook_entry.template.md`](../../../META/templates/runbook_entry.template.md)。

7 字段：**症状 / 根因 / 候选方案 / 适用与失效 / 来源 + 验证状态 / 关联**。

关键校准：**症状字段按"我会用什么关键词搜"反向写**——人遇到问题时第一反应是描述现象（"模型不动了"、"循环停不下来"），不是描述根因。症状字段写不到位，三个月后翻不出来。

## 维护节奏

跑两周看你**是否真的回来查**。
- 不回来查 → 几乎一定是症状字段没写到位，回来改
- 真的查到 → 结构成立，再扩

不强求覆盖率。一条没用过的条目 = 一条会过时的条目。

## 引用方向

- runbook 条目可反向链 KNOWLEDGE 节点（在"根因"或"关联"段）
- KNOWLEDGE 节点**不感知 runbook**（保持仓库硬约束：knowledge 不依赖不稳定层）
