## P-022：Agent 做了超出指令范围的事 / 确认框疲劳无脑点击

`type: remedy-menu`   `severity: irreversible`

**症状**
- 你让 agent 改一个配置文件，他改完顺手改了环境变量文件
- 你让他跑测试，他自作主张地跑了部署
- 你加了"敏感操作弹确认框"兜底，但**确认框太多用户开始无脑点允许**，等于没有

**根因**

Agent 越界本质是信任边界没被结构化表达：用户给的是具体任务，agent 可能扩张成更大授权。二元确认开关又会在"确认太多导致疲劳"和"确认太少拦不住"之间摇摆。

**当前方案**

*方案 0：先止血（暂停写权限 / 回滚最近动作）*

如果已经发生越界操作，先把 agent 切到只读或暂停执行队列，收回高风险工具权限，保存当前 trace / diff / 命令日志。能回滚的先回滚，不能回滚的先冻结现场，再做下面的规则重构。

*方案 1：三档分层规则 + 禁止优先级*

不要做开关，做三档：禁止（最高优先级）/ 需要确认 / 允许。**禁止永远覆盖允许**——你可以配"允许所有文件编辑"同时配"禁止修改版本管理的配置文件"。

*方案 2：规则支持内容级别匹配*

不是"允许 bash"或"禁止 bash"——而是同一工具按内容分三档：`git status` / `git log` → 允许；`git commit` → 需要确认；`rm -rf` / `git push --force` → 禁止。

*方案 3：把"需要确认"压到最少*

确认框疲劳的根因是"需要确认"档放太多。理想状态：用户一天只看到 1-3 次确认框，每次都是真的需要思考的决定。

**适用与失效**
- 适用：任何让 agent 操作真实系统（文件 / 命令 / API）的场景
- 失效：纯只读 agent（如查询助手）不需要这层

**源**

`[[agent-permission-system]]`（分层规则段，Cloud Code 设计）。验证状态：未验证（外部 case）。

**See also**
- P-023 agent 自审被绕过
- P-025 硬底线不可绕过
- `[[three-tier-deny-overrides-allow]]`（DC pattern）

## P-023：Agent 自己审查自己，被自我辩护绕过

`type: remedy-menu`   `severity: irreversible`

**症状**
- 在 system prompt 里写"做敏感操作前要解释为什么"
- Agent 真的解释了——但解释里**写"用户已经授权了所有操作"**之类的话
- 你的审查机制（如果也是 LLM 做）看到这段解释就放行了

**根因**

Agent 自己审查自己有结构性漏洞：生成者和评估者没有隔离，被评审者可以通过自我辩护影响评审。如果第二个 LLM 审查者也读取 agent 的解释文本，它仍可能被"用户已授权"这类话术误导。

**当前方案**

*方案 0：先止血（评审失败时默认拦截）*

如果发现审查被 agent 自我辩护绕过，先把敏感操作改成 fail closed：评审异常、解析失败、超时、审查输入不完整时一律拦截。暂停依赖该评审链路的自动放行动作，再改架构。

*方案 1：生成和评估分离，且评审者不看被评审者的自我辩护*

用一个**独立的大模型调用**专门审查主 agent 的每个操作。审查者**只看事实**：用户输入 + 工具调用（"agent 要干什么"）。审查者**不看 agent 的文字回复**（"agent 怎么解释自己"）。**只看 agent 做了什么，不看 agent 说了什么**。

*方案 2：两阶段架构，把算力花在刀刃上*

- **第一阶段**：输出空间极小（只 yes / no），快速判断，**大部分安全操作直接放行**
- **第二阶段**：第一阶段觉得可疑的才进入，让模型做完整推理

*方案 3：错误处理用"宁可误杀不可放过"*

解析失败、API 错误、超时——**全部默认拦截**而非放行。安全机制的失败模式应该 fail closed 不是 fail open。

**适用与失效**
- 适用：用 LLM 做审查的所有场景（multi-agent 系统的 reviewer / critic 角色都适用）
- 失效：审查者用的是确定性规则而非 LLM——不存在被自然语言操纵的问题

**源**

`[[agent-permission-system]]`（YOLO Classifier 段，Cloud Code 设计）。验证状态：未验证。

**See also**
- P-022 权限规则系统
- P-024 软约束失效
- `[[reviewer-isolation-fail-closed]]`（DC pattern）

## P-024：安全规则写在 prompt 里，模型版本一变就失效

`type: remedy-menu`   `severity: irreversible`

**症状**
- 在 system prompt 里写过"不要修改 .zshrc / 不要 force push / 不要 rm -rf"
- 旧模型版本下表现良好
- 换新模型版本后，**同样的指令模型不再遵守**
- 你重新调 prompt 又能压住，但下次版本更新又失效——周而复始

**根因**

Prompt 里的规则是建议，代码里的规则才是法律。安全规则如果只靠自然语言约束，就会随模型版本、采样参数和上下文注意力变化；有些命令注入类风险模型也未必真正理解，必须用确定性检查兜住。

**当前方案**

*方案 0：先止血（停止把 prompt 当安全边界）*

如果换模型后安全规则失效，先冻结模型升级或把高风险工具切到确认 / 禁止档。不要继续靠补 prompt 试运气；先把已经暴露的高代价规则搬到确定性检查里。

*方案 1：高代价规则编码成确定性检查*

- 命令注入防御（零宽空格 / 等号展开 / 空字节）→ 必须代码层
- "不可逆操作必须确认"（rm -rf、force push、drop table）→ 必须代码层
- "不能修改特定文件"（.zshrc / .ssh / 项目根 .env）→ 必须代码层

*方案 2：低代价偏好规则可以放 prompt*

- 风格偏好（用 4 空格缩进 / 用 snake_case）→ prompt 即可
- 流程建议（写代码前先看测试）→ prompt 即可

*方案 3：升级判断框架*

三个维度决定规则是否需要升级为代码：破坏触发频率 / 副作用持久性 / 检测成本。三个维度都高 → **必须编码成不可绕过规则**。

**适用与失效**
- 适用：任何长生命周期 / 多模型迁移 / 多人协作的 agent 系统
- 失效：一次性 / 临时性 agent（只跑一次，跑完就丢）—— 不值得为它写代码级规则

**源**

`[[agent-permission-system]]`（硬约束 vs 软约束段）。验证状态：未验证。

**See also**
- P-022 分层规则（这条讲规则该放哪一层）
- P-023 评审分离（也是"不依赖模型自觉性"）
- P-025 不可绕过底线（方案 1 的具体应用）
- `[[high-cost-rules-in-code]]`（DC pattern）

## P-025：最高信任模式下，安全底线（如 .zshrc）被破

`type: remedy-menu`   `severity: irreversible`

**症状**
- Agent 跑在"完全信任 / 最高权限 / YOLO 模式"下，跳过所有确认框
- 某次 agent 修改了 `.zshrc` / `.bashrc` / `.ssh/config` / agent 自身配置
- 用户长时间没发现——直到某天注意到行为异常

**根因**

"完全信任"不能等于"完全没限制"。像 shell 启动文件、SSH 配置、agent 自身配置这类持久化入口，副作用长期存在且检测成本高；系统不能假设用户选择最高信任时已经 fully informed。

**当前方案**

*方案 0：先止血（隔离 agent 并检查持久化入口）*

先停止 agent，切断继续写入权限，检查 diff 和启动文件，恢复到可信版本后再重新放行任何自动化动作。

*方案 1：明确不可绕过清单*

至少包括：终端配置文件（.zshrc / .bashrc / .profile）/ Shell 启动文件 / Cron tab / SSH 配置（~/.ssh/config / authorized_keys）/ 版本控制配置（.gitconfig / .git/hooks/）/ Agent 自身配置（防止 agent 自我修改绕过规则）/ 系统级路径（/etc/ / /usr/local/etc/）/ 项目敏感文件（.env / credentials.json / secrets/）。

清单要**在代码里硬编码**，不靠用户配置。

*方案 2：原码注释明确"不可绕过"*

不只是给当下开发者看，是**给未来想"为了用户便利"放宽这些的人看**——任何"我能不能让这些规则可配置"的 PR 都应该被拒。

*方案 3：底线被触发时不只确认，要明确警告*

不是普通的"是否允许？"弹窗，要明确写操作的具体影响，让用户在完整信息下决策，而不是把"完全信任"当全自动通行证。

**适用与失效**
- 适用：任何有"最高信任 / 跳过确认"模式的 agent
- 反向陷阱：**不要为了"用户体验"把底线清单变成可配置**。配置的入口本身就是攻击面

**源**

`[[agent-permission-system]]`（不可绕过底线段，Cloud Code 设计）。验证状态：未验证。

**See also**
- P-022 分层规则（这条是"禁止"档最高优先级的延伸）
- P-024 软约束失效（不可绕过底线**必须**硬编码，是 P-024 方案 1 的最高级别应用）
- `[[non-bypassable-bottom-line]]`（DC pattern）

## P-036：多个 Agent 并行写入共享状态，导致工具冲突和状态损坏

`type: remedy-menu`   `severity: irreversible`

**症状**
- 多个 agent 并发运行，完成后发现文件内容被部分覆盖
- 数据库记录不一致，工具结果被竞态破坏
- 单个任务的工具调用结果莫名其妙（因为另一个并行 agent 改了共享文件）
- 并行并没有带来速度提升，反而引入了数据损坏

**根因**

并行 agent 共享可变状态时，并发写入不是"更快"，是"更危险"——工具冲突（多 agent 同时写同一文件）和状态损坏（一个 agent 读到另一个 agent 写了一半的结果）是结构性风险，不是偶发 bug。

**当前方案**

*方案 0：先止血（停止并行，检查并恢复）*

如果已发生状态损坏，先停止所有并行 agent，检查并恢复到最近已知正确状态，再重新设计并发策略。

*方案 1：序列化执行涉及共享状态的 agent（首选）*

同一会话内一次只处理一个 agent 任务，不并发执行——这是刻意设计选择，不是限制。参考 OpenCode：one-message-at-a-time policy。"多 agent 并行"的价值在于任务分工，而不在于对同一共享状态的并发写入。

*方案 2：给每个 agent 分配独立工作空间*

如果必须并行，给每个 agent 分配独立的工作空间（独立目录、独立数据库 schema），消除共享可变状态。并行写不同的东西，最后合并。

*方案 3：共享只读状态可以安全并行*

只对写入操作序列化；读取操作可以并行。把操作按读写分类，精确识别需要序列化的部分。

**适用与失效**
- 适用：任何多 agent 协作且涉及共享文件 / 数据库 / 外部状态的场景
- 失效：只读并行（多 agent 只读不写）没有这个问题
- 失效：子 agent 在独立沙箱里运行且主 agent 只收最终结果，也没有这个问题

**源**

`[[multi-agent]]`（OpenCode 同会话序列化策略；"当多个 agent 共享状态时，并发会导致工具冲突和状态损坏"）。验证状态：未验证（外部 claim）。

**See also**
- P-022 Agent 越界（同属可能造成不可逆影响的 agent 行为）
- `[[serialize-writes-to-shared-state]]`（DC pattern）
