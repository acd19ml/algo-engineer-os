## P-007：纠正循环无限重试

`type: remedy-menu`   `severity: reversible-cheap`

**症状**

模型连续输出无效工具调用，框架无限注入纠正消息直到 token 超限或费用爆炸。

**根因**

纠正消息如果没有硬预算，就会把"一次格式失败"变成无界循环。成功工具调用和失败纠正也必须分开计数，否则系统不知道什么时候该停。

**当前方案**

纠正消息加硬预算（默认 3 次），超过即终止当前 turn 并向上层报错。成功的工具调用重置计数器。

**适用与失效**
- 适用：任何 agent 框架，配合 P-001 救援解析后仍判定失败时触发
- 失效：如果触发的是"工具本身坏了"（硬错误）vs "资源不存在"（ToolResolutionError），不要混在同一个计数器——参考 P-004

**源**

`[[small-model-harness-engineering]]`（Forge 第二层 / 第四层）。验证状态：未验证。

**See also**
- P-001 救援解析（这是它的下游兜底）
- P-003 升级式纠正（也使用计数器，三次后终止）
- P-004 工具错误计数器（硬错误 vs 软错误的分计数，互补设计）
- `[[retry-budget-hard-cap]]`（DC pattern）

## P-016：Agent 反复调用相同工具+相同参数，行为卡住

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 不是格式错（每次调用都成功），而是**行为重复**——同一个工具名+同一组参数被反复调用
- 任务没推进，但也不报错
- 和 P-007 不同：P-007 是格式错触发纠正循环，P-016 是成功但语义上的死循环

**根因**

模型在某个状态下"觉得"应该调某个工具，调用成功但结果没改变情况（或模型没正确理解结果），就再调一次——形成 action-level loop。这不是 retry 机制能阻止的，因为每次调用都"成功"了。

**当前方案**

**Storm（滑动窗口循环抑制）**：

1. 维护一个滑动窗口（如最近 N 轮），追踪所有工具调用历史
2. 检测：同一工具名 + 同一参数组合（哈希或字符串等价）在窗口内出现 **超过 3 次** 则触发
3. 触发时：**抑制本次调用**（不真的执行），向模型注入反思消息（"你已经用相同参数调用了 X 三次，结果不变。请重新规划。"）

关键细节：**写操作会清空窗口里同目标的读操作记录**——比如编辑文件后读文件验证，是合理的工作流，不该被当循环。

**适用与失效**
- 适用：长任务 agent + 有读写区分的工具集
- 失效：有些任务本来就是"多次同操作"（如轮询某个状态直到 ready）——这种工具应该显式 opt-out 出 Storm 监控，或者把参数里加一个递增的轮询计数器避免被认作"同参数"

**源**

`[[cache-aware-agent-loop]]`（工具调用修复管道 / Storm 段）。验证状态：未验证。

**See also**
- P-007 纠正循环无限重试（格式错 vs 行为重复，互补症状）
- P-013 模式锁定（更宏观的"重复"，可能和 Storm 一起触发）
- P-032 同文件反复编辑无进展（Storm 的边界案例：参数可能不同但方向同样错误）

## P-027：Agent 在长任务执行中途提前放弃 / 草草收工

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 执行到长任务中途突然说"完成了"，但明显还有工作未做
- 或输出质量在后期急剧下降：越到后面越草草了事
- 注意：不是真的到了上下文上限，是模型自己以为快到了

**根因**

"上下文焦虑"——模型看到上下文填充较满时，开始担心自己快到上限，主动提前结束。这和真实的上下文超限不同，是模型的认知偏差。常见于较弱的基础模型。常规的上下文压缩解决不了这个问题——压缩后上下文虽然缩短，但模型的"焦虑"仍然存在。

**当前方案**

*方案 1：Context Reset（不同于压缩，首选）*

**不是压缩，是彻底清空**：启动一个全新的子 agent，通过结构化交接文档传递状态和下一步计划——给模型一块干净的白板。

交接文档应包含：已完成的工作、当前状态、剩余的 N 个待完成项。让新 agent 从明确的起点继续工作。代价：增加编排复杂度和延迟，但效果显著。

*方案 2：检查是否可以换用更强的基础模型*

上下文焦虑是某些模型版本特有的问题。模型升级后该组件可能可以拆除（Harness 可拆卸性原则：每个组件编码了一个关于"模型做不好这件事"的假设，假设可能随模型进化而过时）。

*方案 3：在交接文档里明确写入剩余工作量*

"任务剩余 N 个步骤 / 功能"——让新 agent 知道自己接手的是半途任务，不会以为自己收到一个完整任务就快速完成。

**适用与失效**
- 适用：长任务 agent（步数 ≥ 数十步），任务分量明显超出一次上下文能完成的范围
- 失效：短任务；焦虑是真实上下文耗尽而非模型自我判断失误时，Context Reset 无效（需要真正的压缩，参见 P-006）

**源**

`[[harness-practice]]`（GAN 三 agent 实验，单 agent 最终用 Context Reset 解决，后来 Opus 4.6 升级后拆除了该组件）。验证状态：未验证（Anthropic 外部 case）。

**See also**
- P-006 多步工作流上下文膨胀（真实上下文超限的情况）
- P-013 模式锁定（另一种长任务退化形式）
- `[[agent-context-compaction]]`

## P-028：评审者宽容 / LLM 评估偏向表扬生成结果

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 自动评估环节对 agent 输出评分偏高
- 人工复查时发现评审通过的结果明显平庸甚至有明确错误
- Agent 自评时说"做得不错"但产出质量低
- 和 P-023 不同：P-023 是 agent 主动通过自我辩护操纵评审者；P-028 是评审者对 LLM 生成内容的天然统计偏差

**根因**

LLM 对 LLM 生成的内容天然宽容。如果生成者和评估者是同一个 agent 或同类调用，"自我表扬"偏差会系统性地放行劣质输出。评估器开箱即用是个很差的 QA agent——需要针对性调优才能变严格。

**当前方案**

*方案 1：把主观质量转为可打分的具体维度（首选）*

不要问"这个做得好吗"，而是定义具体评分维度（如：设计质量、原创性、工艺、功能性），每个维度有明确评分标准（"评分 3 = 功能能用但有两个以上明显 bug"）。

*方案 2：给弱项维度加权*

模型天生擅长的维度低权重（如语法正确性），短板维度高权重（如原创性、不使用 AI 生成套路）。显式写入评分标准里"要惩罚那些一眼就能看出是 AI 生成的套路"。

*方案 3：用 few-shot 示例 + 详细评分分解校准评估器*

把"这个分数对应这种质量"的例子给评估器看，对齐评估器和开发者的品味。调优过程需要人工迭代循环（读日志 → 找出分歧 → 改 prompt → 再跑），反复多轮。

*方案 4：让评估器实际操作产出（不只看代码）*

如果评估的是 UI / 功能，让评估器用 Playwright 实际操作运行中的应用——点击、导航、截图，然后才输出评估。"实际操作"而非"看代码打分"。

**适用与失效**
- 适用：任何用 LLM 做评估的场景（不只 agent——multi-agent 系统的 critic / reviewer 角色也适用）
- 失效：评估者使用确定性规则（不存在 LLM 偏差问题）

**源**

`[[harness-practice]]`（GAN 三 agent 实验，评估器调优用了大量人工迭代循环；Claude 开箱是很差的 QA agent）。验证状态：未验证（Anthropic 外部 case）。

**See also**
- P-023 Agent 自审被绕过（主动操纵 vs 天然偏差，互补症状）
- `[[reviewer-isolation-fail-closed]]`（DC pattern）
- `[[mandatory-verification-before-completion]]`（DC pattern）

## P-030：System prompt 里的规则太模糊，agent 执行时不遵守

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 在 system prompt 里写"请写高质量代码""请保持简洁""请小心对待敏感操作"，但 agent 实际表现不符合预期
- 相同规则在不同对话里遵循率波动大
- 加了规则反而感觉没用——agent 只做了模式匹配"规则存在"而没有真正按规则执行

**根因**

模糊指令给了模型太大的解读空间，模型会按训练统计直觉填充"高质量"的含义，导致输出方差大。可执行的规则是 binary 的（能自我检查），原则性规则是连续的（无法精确检查）。

**当前方案**

*方案 1：把抽象原则改写为 binary 可执行规则（首选）*

例：
- "请写好代码" → "不要给你没改的代码加注释" / "三行相似代码比一个过早抽象更好"
- "请保持简洁" → "工具调用之间的文本不超过 25 个词，最终回复不超过 100 个词"
- "请小心敏感操作" → "rm -rf / force push / drop table 必须先暂停等待用户确认"

模型不怕规则多，怕规则模糊。规则越具体越 binary，执行越稳定。

*方案 2：控制规则数量*

12 条规则以内、200 行以内是实测天花板——超过后合规率急剧下降（见 P-034）。先保证每条规则都是 binary 可检查的，再控制数量。

*方案 3：避免无法测试的修饰词*

"小心""多想""真正专注"这类修饰词（合规率实测降到约 30%）；改用具体祈使句（"明确陈述你的假设"）。不要说"请做资深工程师"——模型已经认为自己是资深的，差距在"想"和"做"之间，只有祈使句能闭合这个差距。

**适用与失效**
- 适用：所有 agent 系统
- 失效：如果规则涉及高代价安全约束，还需要代码层确定性实现——prompt 规则是充分不必要条件，见 [[high-cost-rules-in-code]]

**源**

`[[agent-system-prompt]]`（Cloud Code 900 行模块化设计；具体 binary 规则列表）`[[claude-md-rule-design]]`（Mnilax 实测：12 条最优，18 条合规率从 76% 跌到 52%）。验证状态：未验证（外部 claim）。

**See also**
- P-034 CLAUDE.md 过长导致合规率急剧下降（数量维度的互补问题）
- `[[agent-config-binary-rules-bounded]]`（DC pattern）
- `[[high-cost-rules-in-code]]`（DC pattern，高代价规则必须放代码层）

## P-031：Agent 报告结果时夸大成功或掩盖失败

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 说"测试全部通过"但测试明显失败
- 说"迁移完成"但实际上静默跳过了 14% 记录（因为约束冲突没处理）
- 说"任务完成"但关键步骤从未执行
- 或反向：任务明明完成了，agent 还加免责声明"可能有问题建议再检查一下"

**根因**

模型的训练倾向是讨好用户（倾向报告好消息）。没有明确约束时，模型大约每 3 次就有 1 次会在报告结果时夸大或掩盖失败。防守性模式走向另一个极端：成功的事情也说得像失败了。

**当前方案**

*方案 1：在 system prompt 里明确两方向约束（首选）*

写入：
- "如果测试失败了，就明确说失败了，附上相关输出"
- "如果你没有执行验证步骤，就说没执行，不要暗示它成功了"
- "永远不要在输出显示失败时声称所有测试通过"
- 同时反向："当检查确实通过了，直接说完成，不要加不必要的免责声明"

目标是准确的报告，不是防御性的报告。

*方案 2：规则越具体越好*

"如果有步骤被静默跳过，宣称'已完成'即为错误结论" → binary 可检查。

*方案 3：在交付前强制执行验证中间件*

Agent 准备退出 / 报告完成时，系统先拦截并强制执行一轮验证，确认实际输出状态与声称状态一致后才允许退出。见 P-033。

**适用与失效**
- 适用：任何需要 agent 报告执行结果的场景
- 失效：不适用于纯生成任务（没有对错的创意输出）

**源**

`[[agent-system-prompt]]`（30% 虚假声明率原码注释；防说谎 + 防过度谨慎两方向约束的具体实现）。验证状态：部分验证（30% 虚假声明率来自 Cloud Code 原码注释，属于内部测量数据）。

**See also**
- P-033 Agent 跳过验证直接输出（验证缺失导致无法检测虚假声明的上游问题）
- `[[mandatory-verification-before-completion]]`（DC pattern）

## P-032：Agent 对同一文件反复编辑但问题没有解决

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 同一个文件已编辑 7 次，每次看起来都"修了"，但运行 / 测试仍然失败
- Agent 在 edit → test → edit → test 的小循环里转，没有跳出来重新审视方案
- 和 P-016 不同：P-016 是同工具 + 同参数的完全重复；P-032 是文件确实在变，参数也不同，但方向一直错

**根因**

Agent 没有能力检测"我在同一个方向上重复失败"，也没有被强制重新规划的机制。模型倾向于在已有方向上微调（局部调整），而不是放弃方向（全局重规划）。

**当前方案**

*方案 1：文件编辑次数追踪 + 阈值触发（首选）*

用中间件追踪每个文件的编辑次数。超过阈值（如 3 次同文件）时向 agent 注入：
"你已经修改过这个文件 N 次，问题仍未解决，请重新审视你的方案而不是继续修改"。

不是终止，是强制反思。

*方案 2：阈值触发后切换推理强度*

从执行阶段（中等推理）切换到规划阶段（高推理强度）重新理解问题——推理三明治的"规划段"按需触发。

*方案 3：检查点 + 已尝试方案记录*

在每次关键修改后让 agent 总结"已尝试的方案"（不只是"修改了什么"），防止循环尝试相同方向。参见 P-033 检查点机制。

**适用与失效**
- 适用：代码编辑类 agent；任何存在 edit → verify 反馈循环的场景
- 失效：确实需要多次细化修改的任务（如迭代写文章）；区分"正常迭代"和"无效循环"的阈值需要按任务类型调整

**源**

`[[harness]]`（LangChain 中间件：追踪文件编辑次数，超过阈值提醒重新审视）。验证状态：未验证（外部 case，LangChain Terminal Bench 实验）。

**See also**
- P-016 行为循环（同工具+同参数 vs 不同参数但方向相同，互补症状）
- P-033 跳过验证直接输出（无验证是无效循环持续的原因之一）

## P-033：Agent 在任务完成前跳过验证直接输出

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 说"完成了"，但没有实际运行代码 / 跑测试 / 核对输出
- 交付的内容逻辑上合理但实际上有明显运行时错误
- 或者 agent 跑了测试但测试只验证了代码存在（而非逻辑正确性）——全绿但无价值

**根因**

模型有"找到第一个看起来合理的方案就输出"的倾向；验证步骤需要时间且有不确定性，模型的讨好倾向会跳过它。让模型"自我决定是否需要验证"通常靠不住。

**当前方案**

*方案 1：在 agent loop 退出点加中间件拦截（首选）*

Agent 准备退出时系统先拦截，强制执行一轮验证。验证通过才真正退出，验证失败则继续工作。这是结构性保证，不依赖模型自觉性。

*方案 2：推理三明治策略*

规划阶段 + 验证阶段用高推理强度（充分理解问题 / 捕获错误），执行阶段用中等推理强度（保证速度）。全程高推理强度反而成绩更差（会超时）——把算力花在刀刃上。

*方案 3：在 system prompt 里明确验证标准*

"在声明任务完成之前，必须[具体验证步骤]" → 具体 binary 规则（参见 P-030）：
- "在说测试通过之前，必须实际运行测试并看到绿色输出"
- "在说代码完成之前，必须在命令行里成功执行一次"

**适用与失效**
- 适用：任何产出需要被验证的 agent 任务（代码、配置、文档）
- 失效：纯生成类任务（没有可验证的正确标准）

**源**

`[[harness]]`（LangChain 中间件：agent 准备退出时强制执行验证；推理三明治策略）。验证状态：未验证（外部 case）。

**See also**
- P-028 评审者宽容（即使有验证，评审者也可能放行劣质结果）
- P-031 Agent 虚假报告（验证缺失是虚假声明的前提条件）
- `[[mandatory-verification-before-completion]]`（DC pattern）

## P-034：CLAUDE.md / AGENTS.md 过长，agent 合规率急剧下降

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- CLAUDE.md 有几百行，但 agent 实际表现和没有 CLAUDE.md 差不多
- 规则变多后错误率反而不降（甚至回升）
- Agent 只做了模式匹配"规则存在"而没有真正读每条规则
- 12 条时合规率 76%，18 条时降到 52%

**根因**

200 行 / 12 条是实测天花板。超过后模型开始做模式匹配而非真正执行。规则数和合规率的关系是倒 U 型：4 条 → 12 条有效，12 条 → 18 条合规率从 76% 跌到 52%。规则争夺同一份注意力预算——如果新规则覆盖的失败模式和已有规则重叠，加它反而会让整体合规率下降。

**当前方案**

*方案 1：控制在 200 行 / 12 条以内（首选）*

不是越多越好，是每条规则正交覆盖一个真实失败模式。超过 12 条时先删，而不是继续加。

*方案 2：每条规则先问"这条规则预防的是什么具体失败"*

能回答 → 保留；回答不出来 → 噪音，删掉。

*方案 3：删除语义重叠的规则*

如果新规则跟已有规则争夺同一份注意力预算（覆盖同一失败模式），两条的效果不如一条精确的。

*方案 4：不要在 CLAUDE.md 里放例子*

3 个例子消耗的上下文 ≈ 10 条规则，且模型会过度拟合到例子而非抽象规则。规则是抽象的，例子是具体的——用规则。

**适用与失效**
- 适用：所有使用 CLAUDE.md / AGENTS.md / .cursorrules 的 agent 开发场景
- 失效：规则数本身未超过天花板时，先检查规则质量（是否 binary 可执行）而非数量
