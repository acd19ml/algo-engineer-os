## P-001：模型该调工具时输出了自由文本

**症状**
- 模型该发起工具调用时，输出了包含工具名和参数描述的自然语言文本
- 框架判定"模型未调用工具"，进入错误处理或终止
- 在小模型、量化模型、本地推理后端上更频繁

**根因**

模型的**意图**正确（知道该调什么、传什么），**格式生成能力**不足。常见诱因：推理模型在思考过程中"预演"工具调用，用了非标准格式；JSON 被包在 markdown 代码块里；同一权重，原生 function calling 后端能输出干净 JSON，提示注入后端容易漂移。

**候选方案**

*方案 1：救援解析（默认先上）*

在判定"未调用工具"之前，依次尝试从自由文本里提取：

1. **代码块/反引号包裹的 JSON**：正则提取 ` ```json ... ``` ` 或反引号内的内容，尝试反序列化
2. **推理预演格式**：识别 "I will call X with {...}" 这类思考链中的模板，提取参数对象
3. **Qwen 系 XML 工具调用格式**：`<tool_call>{"name": ..., "arguments": ...}</tool_call>`

三种全部失败才视为真正解析失败，进入方案 2。

*方案 2：切换推理后端*

如果是本地模型，先看推理后端是不是问题根源。原生 function calling 模式（llama-server function-calling、vLLM tool choice）vs 纯提示注入，准确率差距可能达 70+ 个百分点（来源：Forge 评测）。**换后端是最便宜的修复，应该排在加防护层之前**。

*方案 3：合成 respond 虚拟工具*

如果模型同时也频繁在"该调工具"和"该输出文本"之间选错，给所有输出统一加一层：定义一个 `respond` 工具承接文本回复，模型永远在"调工具模式"，不再做模式选择。Forge 实测：信任模型自己选两种模式，工作流完成率从 100% 降到 4%。

**适用与失效**
- 适用：≤30B 模型 / 量化 ≤ Q8 / 本地推理。方案 1 即使在云 API 上也建议默认开
- 失效：模型输出了**错误的工具名**或**幻觉参数**——这是另一类问题（→ P-002，待写），rescue 救不了。视觉 grounding 错误（坐标偏，JSON 完美）也不在此条目范围

**来源**

Forge (ACM 2025)。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/)。验证状态：未验证（外部 claim，未在本地复现）。

**关联**
- P-002 模型调用了不存在的工具（工具名幻觉，互补症状）
- P-005 怀疑本地模型工具调用能力差（P-001 方案 2 的扩展场景）
- P-007 纠正循环无限重试
- P-008 模式选择问题（P-001 方案 3 中 respond 工具的纯粹来源）
- P-018 JSON 截断（救援解析的兄弟症状：格式错 vs 被截断）
- KNOWLEDGE: [[small-model-harness-engineering]] · [[tool-call-repair-harness]] · [[structured-output]] · [[cache-aware-agent-loop]]（Scavenge 是方案 1 在 DSML/OpenAI/自由格式三种解析下的工程实现）

## P-002：模型调用了不存在的工具

**症状**
- 模型生成的工具调用里，工具名不在已注册工具列表中
- 参数结构可能完全合理，名字是模型"想象"出来的（比如把 `find_user` 调成 `search_user`）
- 小模型上更频繁；任务描述含糊或多个相似工具时也容易触发

**根因**

模型并未真正"读"工具列表，而是基于语义相似度生成它认为"应该叫这个"的名字。和 P-001 的关键区别：P-001 是格式问题（意图正确，写法不对），P-002 是工具名层的幻觉（意图本身就指向了不存在的目标）——救援解析救不回来。

**候选方案**

*方案 1：注入工具列表强制选择*

错误响应里**直接附上可用工具列表**，告诉模型"工具不存在，从 [list] 中选一个"。比"工具不存在，请重试"信息密度高得多——避免模型在错误信息里自由再生。

*方案 2：命名收敛*

如果常错的工具名有规律（总把 `find_*` 写成 `search_*`），结构性修复：要么注册别名（同名指向同一实现），要么把工具改名向模型偏好对齐。这是运行时兜底之外的一次性修复。

**适用与失效**
- 适用：工具集合静态已知的场景
- 失效：动态工具注册时"工具列表"本身是变量。方案 1 仍能用但需每次注入当前列表；方案 2 不适用

**来源**

Forge 第二层 retry prompting。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 第二层。验证状态：未验证。

**关联**
- P-001 自由文本输出（症状互补：P-001 格式错，P-002 名字错）
- P-007 纠正循环无限重试（纠正消息要计入 retry 预算）
- KNOWLEDGE: [[small-model-harness-engineering]] · [[tool-call-repair-harness]]

## P-003：模型跳过必要的中间步骤

**症状**
- 模型直接调用终端工具（`submit` / `finalize` / `apply`），跳过了必要的前置（`search_list` → `get_detail`）
- 终端工具因为缺参数 / 缺上下文而失败
- 多步工作流的小模型上几乎必现

**根因**

小模型对"工作流应该 N 步"的理解不稳定。看到一个看起来能解决问题的终端工具，倾向直接调用，而不是按必要顺序走完。

**候选方案**

*方案 1：编码工具前置依赖 + Step Enforcer*

在工具注册时声明前置依赖：`get_detail` 只有在 `search_list` 调用之后才能使用（因为需要搜索结果里的 ID）。Step Enforcer 在工具触发前检查，发现跳步时**不让工具真正执行**，直接注入纠正消息。

*方案 2：升级式纠正消息*

- 第一次跳步：语气礼貌（"你跳过了 X 步"）
- 第二次：语气直接（"必须先调 X 才能调当前工具"）
- 第三次：措辞强硬 + 终止 turn

升级比每次同样温和的纠正有效——避免模型在同一种语气下反复尝试同一个错误。

**适用与失效**
- 适用：工作流依赖关系明确、可显式编码的场景
- 失效：依赖关系本身是数据依赖（运行时才知道哪个工具先调），无法静态编码。这时只能在 system prompt 里讲清楚

**来源**

Forge 第三层 Step Enforcer。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 第三层。验证状态：未验证。

**关联**
- P-007 纠正循环无限重试（升级式纠正三次后必须终止）
- KNOWLEDGE: [[small-model-harness-engineering]]

## P-004：工具错误计数混淆，系统过早放弃

**症状**
- 模型连续几次调用工具失败，框架终止了 turn
- 但失败原因其实是"猜错参数"（查了不存在的用户 ID）而非"工具本身坏"
- 把所有失败统一计数，模型"还在正常探索"就被砍掉

**根因**

工具失败有两类语义完全不同的：
- **硬错误**：工具实现 bug / 权限不足 / 网络挂了 → 模型重试无意义，应早停
- **ToolResolutionError**：调用格式对、参数类型对，但目标资源不存在（语义类似 HTTP 404）→ 模型换个参数就行，不该早停

混在一起，模型探索三次就被终止，即使工具完全正常。Forge 节点原文："混淆两类错误会让系统过早放弃"。

**候选方案**

工具返回错误时显式区分两类，使用独立计数器：
- 硬错误：累加全局错误计数（默认 2 次终止）
- ToolResolutionError：**不累加**全局计数，只把"资源不存在"语义返回给模型，让它换参数再试

实现层面：要么工具内部抛不同异常类型，要么返回值带 `error_kind: "not_found" | "hard_error"`。

**适用与失效**
- 适用：搜索 / 查询类工具占主导的工作流（探索型任务）
- 失效：写入 / 副作用型工具——"目标不存在"可能本身就是错误前提（"删除用户 X"，X 不存在不是 404 而是请求错了）。这种工具上不要套 ToolResolutionError 语义

**来源**

Forge 第四层错误恢复。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 第四层。验证状态：未验证。

**关联**
- P-007 纠正循环无限重试（计数器机制的另一面）
- KNOWLEDGE: [[small-model-harness-engineering]]

## P-005：怀疑本地模型工具调用能力差，下结论之前

**症状**
- 本地部署的某模型在工具调用 benchmark 上得分很低（< 20%）
- 想换更大的模型 / 加防护层 / 微调，但还没排查后端

**根因**

同一套权重，不同推理后端准确率可能差 70+ 个百分点：llama.cpp 原生 function calling 模式 7%，换成 llama-file 提示注入模式跳到 83%（来源：Forge）。**"这个模型不行"的结论可能根本就是错的——只是用错了后端**。

**候选方案**

在换模型 / 加防护层 / 微调之前，先做后端 ablation：

1. 跑原生 function calling 后端（vLLM tool choice、llama-server function-calling）
2. 跑提示注入后端（llama-file、自己拼 system prompt）
3. 在小一组样本上对比，量化差距

如果后端差距明显，先换后端——这是最便宜的修复，比加防护层、换模型、做 SFT 都便宜一个数量级。

**适用与失效**
- 适用：本地推理 / 自托管模型
- 失效：闭源云 API（你无法换后端）。这时直接对比不同模型即可

**来源**

Forge benchmark：同模型两种后端 7% vs 83%。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) "如果只改模型、不加防护层"段。验证状态：未验证。

**关联**
- P-001 自由文本输出（后端切换是 P-001 的方案 2）
- KNOWLEDGE: [[small-model-harness-engineering]] · [[structured-output]] · [[model-boundary-probing]]

## P-006：多步工作流上下文膨胀，速度骤降或 OOM

**症状**
- 工作流前几步正常，5–10 步后速度从秒级降到分钟级
- 或者直接 OOM / 上下文超长报错
- 消费级 GPU 上更频繁

**根因**

多步工作流积累大量上下文：每步的工具调用、返回结果、推理过程、纠正消息。上下文过长时：

- **本地推理**：llama.cpp 显存不足会**静默回退到 CPU**，速度降 10–100 倍，用户毫无感知
- **API 推理**：token 数线性放大成本和延迟

**候选方案**

*方案 1：三阶段 Tiered Compact（确定性压缩，零 LLM 调用）*

按程度递进，只在前一阶段不够用时进入下一阶段：

1. **第一阶段**：删除所有纠正消息；旧工具返回结果截断到前 200 字符
2. **第二阶段**：删除旧工具返回结果，但**保留模型的推理过程**——推理是模型对数据的解读，比原始数据更致命，丢推理 = 丢思路
3. **第三阶段**：删除推理和失败的文本回复，只保留工具调用骨架

**最近几轮永远不压缩**，只压旧的。三阶段都是确定性文本操作，**不调用模型，零延迟**。

*方案 2：VRAM 感知预算*

启动时查询 GPU 显存（NVIDIA SMI），自动设上下文上限：
- < 24GB → 4000 token
- 24–48GB → 32000 token
- > 48GB → 26 万 token

主动设上限好于让 llama.cpp 偷偷降级——后者的失败是无感的，前者会显式 truncate / refuse。

**适用与失效**
- 适用：长工作流（≥5 步），本地推理尤其需要
- 失效：单步任务 / 上下文本身就短的场景不需要这层
- 注意：方案 1 的"保留推理"对推理模型很重要，对非推理模型可以更激进地丢

**来源**

Forge 第五层上下文压缩 + VRAM 预算。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 第五层。验证状态：未验证。

**关联**
- P-014 缓存命中率低（**方案互冲警告**：本条方案 1 的"删除/截断旧内容"会破坏 P-014 的 Append-only 不变性。关心 API 缓存成本的场景应改用 P-015 的 Cache Line Fold）
- P-015 上下文压缩破坏缓存（同上 trade-off 的另一面）
- KNOWLEDGE: [[small-model-harness-engineering]] · [[agent-context-compaction]] · [[cache-aware-agent-loop]]

## P-007：纠正循环无限重试

**症状**

模型连续输出无效工具调用，框架无限注入纠正消息直到 token 超限或费用爆炸。

**候选方案**

纠正消息加硬预算（默认 3 次），超过即终止当前 turn 并向上层报错。成功的工具调用重置计数器。

**适用与失效**
- 适用：任何 agent 框架，配合 P-001 救援解析后仍判定失败时触发
- 失效：如果触发的是"工具本身坏了"（硬错误）vs "资源不存在"（ToolResolutionError，类似 HTTP 404），不要混在同一个计数器——参考 Forge 第四层错误恢复

**来源**

Forge。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 第二层 / 第四层。验证状态：未验证。

**关联**
- P-001 救援解析（这是它的下游兜底）
- P-003 升级式纠正（也使用计数器，三次后终止）
- P-004 工具错误计数器（硬错误 vs 软错误的分计数，互补设计）
- KNOWLEDGE: [[small-model-harness-engineering]]

## P-008：模型在"调工具"和"输出文本"之间频繁选错

**症状**
- 模型该调工具时输出了文本（不是格式错，是真的没调任何工具）
- 或者该输出最终答复时调了无关工具
- 工作流完成率显著低于"格式问题"能解释的范围

**根因**

让小模型在"输出文本 vs 调工具"两种模式之间自主选择，是一个独立的认知任务。Forge 实测：信任模型自己选两种模式，工作流完成率**从 100% 降到 4%**——这不是格式问题（P-001），也不是工具名幻觉（P-002），是**模式选择本身就难**。

**候选方案**

定义一个 `respond` 虚拟工具，模型想回复文本时调用 `respond` 并把文字传入 `content` 参数。这样模型**永远处于"调工具模式"**，不需要在两种模式间选择，彻底消除歧义。

实现细节：`respond` 不真的去任何系统，它的"执行结果"就是把参数当文本返回给用户/上层。对模型来说，输出文本和调工具走同一条路径。

**适用与失效**
- 适用：小模型 / 量化模型 / 任何模式选择不稳定的场景。和 P-001 的方案 3 是同一个机制，但出发点不同：P-001 是格式问题的兜底，P-008 是模式选择问题本身
- 失效：模型对模式选择本身稳定时（前沿大模型）这层是冗余开销；并且会把简单文本输出包一层 JSON，可读性稍降

**来源**

Forge "Respond 工具"段，100% → 4% 完成率落差。详见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 倒数第二段。验证状态：未验证。

**关联**
- P-001 自由文本输出（症状重叠但根因不同——P-001 救得回意图，P-008 救不回模式选择）
- KNOWLEDGE: [[small-model-harness-engineering]] · [[structured-output]]

## P-009：JSON 输出格式不稳（多逗号 / 加前缀 / 类型错）

**症状**
- 让模型输出 JSON，大部分时候能照做，但偶尔：
  - 前面加一句 "好的，这是结果："
  - 多一个逗号 / 漏一个引号
  - 数字输出成字符串（`"5"` 而非 `5`）
  - 字段缺失
- 单轮重试就行，但**喂给下一步的 agent 工作流会因此整链崩**

**根因**

LLM 本质是 token 生成器，**天生倾向输出自由文本**。光靠 prompt 里写"请返回 JSON"是软约束，不是物理保证——模型大部分时候会服从，但出错的概率不为零，在多步工作流里被指数放大（参见 [`KNOWLEDGE/agent/small-model-harness-engineering/`](../../../KNOWLEDGE/agent/small-model-harness-engineering/) 复合概率段）。

**候选方案**

*方案 1：原生约束解码（strict / structured outputs）*

主流 API（OpenAI / Anthropic）都提供这个能力——在工具定义或响应格式上打开 `strict`。原理是每生成一个 token 时，按 JSON Schema **物理限制**模型能选择的 token 集合（比如对象第一个字符必须是 `{`，那这一步只允许生成 `{`）。

**不是劝模型听话，而是从解码层让它无法生成不合法 token**。

*方案 2：工具调用包装（模型不支持 strict 时的兜底）*

如果模型不支持原生约束解码，把结构化输出**伪装成工具调用**：

1. 定义一个虚拟工具（比如 `submit_result`）
2. 把期望的输出格式写成它的输入 schema
3. 强制模型调用这个工具

模型通过微调已经会按 schema 生成函数调用参数，这个能力可以泛化到任意结构化输出。**几乎所有主流模型都支持工具调用，开箱即用**。

*方案 3：验证 + 重试（语法层兜底）*

方案 1/2 都失败或不可用时，模型输出后用 JSON parser 验证，失败则把错误信息连同原始输出回传给模型重新生成。和 P-001 的救援解析（先尝试从自由文本提取）配合更好。

**适用与失效**
- 适用：所有需要结构化输出的 agent / pipeline 场景
- 失效：方案 1 在指令微调模型 + 生成型任务上会触发**语义质量下降**——格式 100% 对，但内容可能更差，下游别盲目开。详见 P-010
- 失效：方案 2 把简单文本响应也包了一层 JSON，可读性稍降

**来源**

[`KNOWLEDGE/agent/structured-output/`](../../../KNOWLEDGE/agent/structured-output/) Layer 1 + Layer 3。验证状态：未验证。

**关联**
- P-001 自由文本输出（agent 工具调用上下文，方案有交集）
- P-010 strict 引发的语义质量下降（方案 1 的反向陷阱）
- P-011 JSON 语法对但值不合理（语义错误的下一站）
- KNOWLEDGE: [[structured-output]] · [[tool-call-repair-harness]]

## P-010：开启 strict / 约束解码后输出语义质量下降

**症状**
- 启用约束解码（strict mode / structured outputs）后，JSON 格式 100% 合法
- 但内容明显变差：关键信息漏了、措辞变奇怪、生成型字段（如 summary / description）质量降
- 切回非 strict 模式后质量回升

**根因**

约束解码强迫模型远离其首选 token——当模型排名靠前的 token 全部被 mask 掉，**它只能从低概率的替代 token 中选择**，输出虽然语法正确，**但语义质量下降**。

研究观察：
- **基础模型**通常从约束中受益（输出更精确简洁）
- **指令微调模型** 结果参差不齐——分类任务上保持性能，**生成任务上明显退化**

**约束解码保证 100% 格式正确，但不保证内容正确**——可能输出一个结构完美但语义荒谬的 JSON。

**候选方案**

*方案 1：推理和输出分阶段*

让模型先**自由思考**（无 schema 约束），最后**只在最终输出**时施加 schema。Claude API 已内建这种模式：thinking 阶段完全不受约束，只有最终 response 被结构化——既不损害推理质量，又保证输出格式。

*方案 2：在生成型字段上不硬启 strict*

把 schema 拆分：
- 分类 / 结构性字段（label、enum、bool、id）→ strict 开
- 生成型字段（summary、explanation、reasoning_text）→ 不强约束，靠验证+重试兜底

*方案 3：评估"是否真的需要 strict"*

参考 Bitter Lesson 视角：在不同能力等级的模型上跑同一个 agent。如果旧模型上 strict 必须开，但**新模型上关掉 strict 性能反而更好**，说明约束已经成了瓶颈——该放松了。

**适用与失效**
- 适用：用 strict 的生成型任务出现质量下降时
- 失效：纯结构化任务（提取实体、分类、打标签）通常不会触发这条——是生成型字段才有

**来源**

[`KNOWLEDGE/agent/structured-output/`](../../../KNOWLEDGE/agent/structured-output/) Layer 1 反事实段 + Bitter Lesson 视角段。

注意：节点明确标注"具体退化多少、什么场景严重，没有量化数据"——这是 Open Question 之一。验证状态：未验证（来源本身就是 qualitative claim）。

**关联**
- P-009 JSON 格式不稳（方案 1 的来源，开启它就可能落入此条）
- KNOWLEDGE: [[structured-output]] · [[model-boundary-probing]]

## P-011：JSON 语法对但值不合理（语义错误）

**症状**
- JSON 解析成功，所有字段类型也对
- 但取值越界：`rating` 字段 schema 要求 1-5，模型输出 7
- 或字段间逻辑冲突：`status: "completed"` 但 `completion_time: null`
- 约束解码 / strict 模式救不了这种错

**根因**

约束解码解决的是**语法问题**——括号配对、类型匹配、字段完整。但它**解决不了语义问题**——schema 能要求 `rating` 是整数，**无法限制取值范围**（除非用 enum，但很多业务规则是字段间的，无法静态编码）。

**候选方案**

*方案 1：业务规则 schema + 验证器 + 错误反馈重试*

1. 用 **Pydantic / Zod** 定义带业务规则的 schema（字段取值范围、字段间逻辑关系）
2. 模型输出后过验证器
3. 验证失败时，**把具体错误连同原始输出一起反馈给模型**，让它重新生成

关键是错误信息要具体（"rating 必须在 1-5，你输出了 7"），不是模糊的"验证失败"。这是自愈系统的核心机制。

节点原文："**这一层在面试中经常被忽略，但它恰恰是工程落地中最实用的一环**——约束解码保语法、验证重试保语义，两层配合才是完整方案"。

**适用与失效**
- 适用：所有有业务规则的结构化输出（几乎所有真实业务场景）
- 失效：验证重试本身要计入 retry 预算（参见 P-007），不能无限循环。3 次还不过就该上报而非死磕

**来源**

[`KNOWLEDGE/agent/structured-output/`](../../../KNOWLEDGE/agent/structured-output/) Layer 2。验证状态：未验证。

**关联**
- P-007 纠正循环无限重试（重试预算约束）
- P-009 JSON 语法不稳（语法 vs 语义，两层互补）
- KNOWLEDGE: [[structured-output]]

## P-012：工具集合大，模型频繁选错；想动态增删但会废 KV cache

**症状**
- Agent 注册了几十个工具，模型在选择下一步动作时频繁选错
- 直觉解法："当前用不到的工具动态从工具列表里移除"
- 实操后发现：**KV Cache 命中率塌方，每步延迟 / 成本暴涨**

**根因**

工具定义在上下文的**前部**，是 prompt 的稳定 prefix。任何对工具列表的变更（增 / 删 / 改顺序）都会让 prefix 的某个 token 变化，**导致从该 token 之后所有内容的 KV Cache 失效**。Agent 多步循环里这个代价被反复放大。

但"工具太多模型选错"本身是真实问题——不能不解决，也不能用直觉解法。

**候选方案**

*方案 1：Logit Masking + 工具命名前缀化*

**工具定义永远不变**（KV Cache 安全），但在解码时通过 mask logits 来屏蔽当前不需要的工具。

工具命名按功能分组、前缀化：
- 所有浏览器相关工具 → `browser_*`
- 所有命令行工具 → `shell_*`
- 所有文件操作 → `fs_*`

**按前缀做 mask 非常简单**——agent 当前阶段在浏览中，就 unmask `browser_*` 前缀的 token，其他全 mask。模型每一步能选什么是被精确控制的，但工具定义本身不动。

*方案 2：如果推理后端不支持 logit mask*

退而求其次：在 system prompt 里**动态注入"当前可用工具子集"的提示**（但工具定义本身保持完整）。这不是 mask，是软引导。效果弱于方案 1，但兼容性好。

**适用与失效**
- 适用：长上下文 agent（KV Cache 价值高的场景）+ 工具数量 > 10
- 失效：工具数 ≤ 5 时直接全开，不需要 mask。这条是"工具规模触发的问题"
- 反向陷阱：**不要为了"减少模型负担"动态删除工具定义**——这是直觉但错误的解法

**来源**

[`KNOWLEDGE/agent/structured-output/`](../../../KNOWLEDGE/agent/structured-output/) Layer 4 + Manus 实践。详见 [`KNOWLEDGE/agent/context-engineering/`](../../../KNOWLEDGE/agent/context-engineering/) 的 KV Cache 段。验证状态：未验证。

**关联**
- P-006 上下文膨胀（同样和 KV Cache 相关，但角度不同：P-006 压缩历史，P-012 稳定 prefix）
- KNOWLEDGE: [[structured-output]] · [[context-engineering]] · [[cache-aware-agent-loop]]

## P-013：Agent 跑几十步后开始机械重复之前的动作

**症状**
- Agent 在长任务（几十步以上）跑着跑着进入"复读机"状态
- 不断重复之前的动作模式：比如批量处理简历时，对每份简历都重复完全相同的操作序列，即使该简历有特殊情况
- 输出 schema 没错，但**动作选择被上下文中的历史 pattern "锁定"**

**根因**

Agent 执行几十步之后，上下文里积累了大量类似的动作和观测。模型的自回归生成会被这些重复 pattern **强化**——它陷入"模式锁定"状态，机械重复之前的输出模式，而不是基于当前情况重新决策。

Manus 在批量审阅简历时观察到这个症状。

**候选方案（注意：本条方案尚不成熟）**

*方案 1：上下文压缩（实际可行）*

减少重复 pattern 在上下文中的堆积。参考 P-006 的三阶段 Tiered Compact——旧的相似动作 / 观测被截断或丢弃，减少模型可以"模仿"的 pattern 数量。

*方案 2：阶段性 Context Reset（实际可行）*

在长任务的关键节点显式重置上下文——只保留结构化的总结（参考 OpenCode 的固定格式摘要），把详细历史丢掉。模型从一个干净的 context 重新开始决策。

*方案 3：输出微变异（思路占位，未广泛验证）*

Manus 思路：输出 schema 不变，但同一个 schema 下的**序列化方式有微小变化**（字段顺序 / 空白 / 等价表示），打破模型的自回归惯性。

**节点明确标注：具体实现细节在公开资料中披露较少，目前还不算被广泛验证的标准做法**。

**适用与失效**
- 适用：长任务 agent（步数 ≥ 数十步）+ 任务本身有重复结构（批处理、多对象循环）
- 失效：短任务 / 任务步骤差异大、本来就不重复的场景不会触发
- 注意：方案 3 在生产前先在自己场景做 A/B，不要照抄

**来源**

[`KNOWLEDGE/agent/structured-output/`](../../../KNOWLEDGE/agent/structured-output/) Layer 6 + Manus 实践分享。验证状态：未验证（方案 3 在公开资料中本身也未充分验证）。

这是 structured output 领域的一个**真正开放问题**——节点 Open Questions 段也列了。

**关联**
- P-006 多步工作流上下文膨胀（共享方案 1 上下文压缩）
- KNOWLEDGE: [[structured-output]] · [[agent-context-compaction]]

## P-014：使用前缀缓存 API 但命中率低，月账单暴涨

**症状**
- 用 DeepSeek / Anthropic / OpenAI 等支持前缀缓存的 API
- 缓存命中率只有 2~30%（API 后台或 usage metric 能看到）
- 同样的 token 量，账单远高于预期（DeepSeek miss vs hit 倍率约 50×，44× 总成本差是真实数据）

**根因**

前缀缓存按**字节前缀**匹配——这次请求的前缀和上一次完全一样才命中，**差一个字节都 miss**。客户端常见的破坏方式：

1. **工具定义重新序列化**：每次请求时 JSON 字段顺序可能变，一个空格差异让整个前缀变化
2. **思维链进入历史**：推理模型的 CoT 被记录进对话历史，下一轮请求前缀包含上一轮特有的思维链
3. **上下文压缩修改历史**：标准做法是"删/替换旧消息"，改历史 = 改前缀，缓存全废
4. **工具调用失败回滚**：回滚到旧状态再重试 → 历史改动 → 前缀失效

**候选方案**

*方案 1：三区域设计（架构层修复）*

把上下文切成三个区域，每个区域有严格的不变性约束：

- **Immutable Prefix（不可变前缀）**：system prompt、工具定义、few-shot 示例。对话开始时算一次，**整个对话不变**。工具定义的顺序、序列化方式、空白字符全部固定。可以用指纹（如 hash）追踪——发现飘移立即报错
- **Append-only Log（只追加日志）**：对话历史。**只能追加，不能修改、重排、删除**。每次新请求的前缀就是上一次请求的完整内容 + 新消息，前缀永远是上一次的超集
- **Volatile Scratch（临时暂存区）**：思维链、临时计划、中间笔记。**不发送给 API**，只在本地存在，每轮清空。CoT 里产生的有用信息通过工具调用以结构化形式追加进 Append-only Log

*方案 2：上下文压缩用 Cache Line Fold 而非删/替换*

参见 P-015——压缩时是**追加摘要到前缀后面**，不是改历史。

*方案 3：工具调用失败永远不回滚，只向前修复*

不要用"回滚到上一个状态再重试"的策略。失败信息以追加形式进 Log，让模型基于完整历史决定下一步。

**适用与失效**
- 适用：高频调用同一 API + 大量重复前缀的场景（agent loop 是典型）
- 适用：DeepSeek（自动缓存，命中规则严格）尤其需要
- 失效：低频调用 / 每次请求完全不同的内容（如批量翻译不同句子）——缓存本来就没价值
- 注意：Anthropic 缓存需要**显式标记 cache_control**，不是字节自动匹配，方案 1 还要加显式标记。OpenAI 粒度不同。**单 provider 深度适配比通用框架更值**

**来源**

[`KNOWLEDGE/agent/cache-aware-agent-loop/`](../../../KNOWLEDGE/agent/cache-aware-agent-loop/)。真实数据：用户用 ReasonX 一天处理 4.35 亿 token 花 $1.38，无缓存优化估算 $61，**44× 成本差**。验证状态：未验证（外部 case study）。

**关联**
- P-001 自由文本输出（Scavenge 是救援解析的多格式版本）
- P-006 多步工作流上下文膨胀（**方案互冲警告**：P-006 的"删除/截断旧内容"会破坏 P-014 的 Append-only 不变性。本地推理选 P-006，云 API 选 P-014）
- P-015 上下文压缩破坏缓存
- P-016 行为循环（Storm 检测，配合 Append-only）
- KNOWLEDGE: [[cache-aware-agent-loop]] · [[context-engineering]] · [[agent-context-compaction]]

## P-015：做了上下文压缩反而让缓存命中率下降

**症状**
- 上下文太长，按常规做法做了压缩（删旧消息 / 用摘要替换旧消息）
- 压缩之后**缓存命中率反而暴跌**
- 上下文短了，API 费用却涨了

**根因**

前缀缓存按字节匹配前缀。如果在第 10 轮压缩了第 3 轮的消息，**从第 3 轮开始的所有字节都变了**，本来已经累积的缓存全部失效。后续每次请求都是 miss。

朴素直觉认为"压缩 = 省钱"，但在缓存感知场景下，**压缩的代价比节省的还大**——除非压缩方式不破坏前缀。

**候选方案**

**Cache Line Fold（缓存对齐折叠）**：当上下文使用率超过阈值（如 70%）时，把旧消息总结成一条摘要，**追加到前缀后面**——不是替换旧消息，是追加。

实现细节：
1. 算 tail budget（上下文窗口的 20%），从日志末尾往前数保留最近消息直到填满
2. 剩下的旧消息（前面那块）调一次便宜模型（如 V4 Flash）生成摘要
3. 把摘要追加到 Immutable Prefix 之后、Append-only Log 之前
4. 摘要调用本身也复用主对话的不可变前缀，**摘要调用也能命中缓存**

多级触发阈值：70% 折叠 / 75% 激进折叠（tail budget 缩到 10%）/ 78% 强制提前退出 turn / 80% 生成恢复性总结 / 90% turn 开始前预防性折叠。

**适用与失效**
- 适用：使用前缀缓存的云 API + 长 agent 工作流
- 失效：本地推理 / 不计较 API 成本 / 没有缓存机制的 provider——这时用 P-006 的 Tiered Compact 更直接
- 注意：tail budget 太小会让摘要要覆盖的内容太多，摘要质量可能下降——节点 Open Question 列了这个 trade-off

**来源**

[`KNOWLEDGE/agent/cache-aware-agent-loop/`](../../../KNOWLEDGE/agent/cache-aware-agent-loop/) Cache Line Fold 段。验证状态：未验证。

**关联**
- P-014 缓存命中率低（这是根症状）
- P-006 多步工作流上下文膨胀（**方案互冲**：P-006 删/截断 vs P-015 追加。按是否关心缓存选）
- KNOWLEDGE: [[cache-aware-agent-loop]] · [[agent-context-compaction]]

## P-016：Agent 反复调用相同工具+相同参数，行为卡住

**症状**
- Agent 不是格式错（每次调用都成功），而是**行为重复**——同一个工具名+同一组参数被反复调用
- 任务没推进，但也不报错
- 和 P-007 不同：P-007 是格式错触发纠正循环，P-016 是成功但语义上的死循环

**根因**

模型在某个状态下"觉得"应该调某个工具，调用成功但结果没改变情况（或模型没正确理解结果），就再调一次——形成 action-level loop。这不是 retry 机制能阻止的，因为每次调用都"成功"了。

**候选方案**

**Storm（滑动窗口循环抑制）**：

1. 维护一个滑动窗口（如最近 N 轮），追踪所有工具调用历史
2. 检测：同一工具名 + 同一参数组合（哈希或字符串等价）在窗口内出现 **超过 3 次** 则触发
3. 触发时：**抑制本次调用**（不真的执行），向模型注入反思消息（"你已经用相同参数调用了 X 三次，结果不变。请重新规划。"）

关键细节：**写操作会清空窗口里同目标的读操作记录**——比如编辑文件后读文件验证，是合理的工作流，不该被当循环（前一次读 → 写 → 后一次读 是正常的，不是循环）。

**适用与失效**
- 适用：长任务 agent + 有读写区分的工具集
- 失效：有些任务本来就是"多次同操作"（如轮询某个状态直到 ready）——这种工具应该显式 opt-out 出 Storm 监控，或者把参数里加一个递增的轮询计数器避免被认作"同参数"
- 注意：阈值 3 次是经验值；高频探索类任务可以调高

**来源**

[`KNOWLEDGE/agent/cache-aware-agent-loop/`](../../../KNOWLEDGE/agent/cache-aware-agent-loop/) 工具调用修复管道 / Storm 段。验证状态：未验证。

**关联**
- P-007 纠正循环无限重试（格式错 vs 行为重复，互补症状）
- P-013 模式锁定（更宏观的"重复"，可能和 Storm 一起触发）
- KNOWLEDGE: [[cache-aware-agent-loop]]

## P-017：工具参数 schema 嵌套深 / 叶子多，模型频繁丢参数

**症状**
- 工具的输入 schema 有嵌套对象（如 `config.database.connection.host`）或字段数 > 10
- 模型调用时漏字段、漏层级
- 即使错误重试，模型还是丢同一个深层字段

**根因**

模型对深嵌套参数 schema 的处理稳定性差——一旦层级超过两层或叶子超过 ~10 个，丢参数的概率显著上升。这是模型能力边界问题，不是格式问题。

**候选方案**

**Flatten（参数展平）**：

1. 在工具注册时**自动检测** schema 复杂度（叶子节点数、嵌套深度）
2. 超过阈值（如 > 10 叶子 或 嵌套 > 2 层）时，把深层嵌套展平为点号分隔的扁平路径：
   - 原：`{config: {database: {host: "x", port: 5432}}}`
   - 展平后：`{"config.database.host": "x", "config.database.port": 5432}`
3. 工具 Dispatch 时再还原嵌套结构调用真实实现

模型看到的是扁平 schema，调用稳定性大幅提升。还原是确定性操作，零额外 LLM 调用。

**适用与失效**
- 适用：工具参数本来就嵌套的复杂业务场景
- 失效：参数本来就扁平 / 嵌套很浅 → 不需要这层
- 注意：展平后字段名变长，会占更多 token。但对小参数的工具几乎可忽略

## P-018：模型输出 JSON 被截断，工具层静默用空对象执行

**症状**
- 模型输出到 max_tokens 被截断
- JSON 残缺（缺右括号 / 缺引号 / 字段值不完整）
- **工具层没报错**，用了一个空对象或默认值就执行了——产生静默的错误行为
- 下游表现为"工具调用了但结果不对，原因莫名"

**根因**

很多框架对 JSON parse 失败的处理是兜底返回空对象 `{}` 或 `null`，然后把它当合法参数传给工具——结果是工具用默认参数执行，**无声的语义错误**比 loud 的崩溃难调试得多。

**候选方案**

**Truncation（截断检测 + 显式失败）**：

1. 检测不平衡括号 / 引号——计算 `{` `}` `[` `]` `"` 配对情况
2. 如果可以确定性补全（比如只差一个 `}`），尝试补全后重新 parse
3. **补不了的标记为不可恢复**，让工具层返回明确的"无效 JSON 错误"给模型，**绝不静默用空对象**

关键设计原则：**fail loud, not silent**。截断本身常见（max_tokens 限制），但下游行为应该是显式错误 → 模型重试或纠正，不是默默用错参数。

**适用与失效**
- 适用：所有用 LLM 输出 JSON 喂工具的场景
- 失效：模型不会输出 JSON（如纯文本生成）—— 这条不相关
- 注意：要配合 max_tokens 上调或工具结果分块输出（参见 cache-aware-agent-loop 的 Turn End Auto Compaction）减少截断本身

[`KNOWLEDGE/agent/cache-aware-agent-loop/`](../../../KNOWLEDGE/agent/cache-aware-agent-loop/) 工具调用修复管道 / Truncation 段。验证状态：未验证。
