## P-006：多步工作流上下文膨胀，速度骤降或 OOM

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 工作流前几步正常，5–10 步后速度从秒级降到分钟级
- 或者直接 OOM / 上下文超长报错
- 消费级 GPU 上更频繁

**根因**

多步工作流会积累工具调用、返回结果、推理过程和纠正消息。本地推理时，上下文过长可能触发显存不足和 CPU 回退；API 推理时，token 数会直接放大成本和延迟。

**当前方案**

*方案 1：三阶段 Tiered Compact（确定性压缩，零 LLM 调用）*

按程度递进，只在前一阶段不够用时进入下一阶段：

1. **第一阶段**：删除所有纠正消息；旧工具返回结果截断到前 200 字符
2. **第二阶段**：删除旧工具返回结果，但**保留模型的推理过程**——推理是模型对数据的解读，比原始数据更致命，丢推理 = 丢思路
3. **第三阶段**：删除推理和失败的文本回复，只保留工具调用骨架

**最近几轮永远不压缩**，只压旧的。三阶段都是确定性文本操作，**不调用模型，零延迟**。

*方案 2：VRAM 感知预算*

启动时查询 GPU 显存（NVIDIA SMI），自动设上下文上限：
- < 24GB → 4000 token
- 24–48GB → 32000 token
- > 48GB → 26 万 token

主动设上限好于让 llama.cpp 偷偷降级——后者的失败是无感的，前者会显式 truncate / refuse。

**适用与失效**
- 适用：长工作流（≥5 步），本地推理尤其需要
- 失效：单步任务 / 上下文本身就短的场景不需要这层
- 注意：方案 1 的"保留推理"对推理模型很重要，对非推理模型可以更激进地丢

**源**

`[[small-model-harness-engineering]]`（Forge 第五层上下文压缩 + VRAM 预算）。验证状态：未验证。

- P-015 上下文压缩破坏缓存（同上 trade-off 的另一面）

## P-014：使用前缀缓存 API 但命中率低，月账单暴涨

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 用 DeepSeek / Anthropic / OpenAI 等支持前缀缓存的 API
- 缓存命中率只有 2~30%（API 后台或 usage metric 能看到）
- 同样的 token 量，账单远高于预期（DeepSeek miss vs hit 倍率约 50×，44× 总成本差是真实数据）

**根因**

前缀缓存按字节前缀匹配，差一个字节都可能 miss。工具定义重新序列化、把思维链写进历史、压缩时改旧消息、失败后回滚历史，都会破坏稳定前缀。

**当前方案**

*方案 1：三区域设计（架构层修复）*

把上下文切成三个区域，每个区域有严格的不变性约束：

- **Immutable Prefix（不可变前缀）**：system prompt、工具定义、few-shot 示例。对话开始时算一次，**整个对话不变**。工具定义的顺序、序列化方式、空白字符全部固定。可以用指纹（如 hash）追踪——发现飘移立即报错
- **Append-only Log（只追加日志）**：对话历史。**只能追加，不能修改、重排、删除**。每次新请求的前缀就是上一次请求的完整内容 + 新消息，前缀永远是上一次的超集
- **Volatile Scratch（临时暂存区）**：思维链、临时计划、中间笔记。**不发送给 API**，只在本地存在，每轮清空

*方案 2：上下文压缩用 Cache Line Fold 而非删/替换*

参见 P-015——压缩时是**追加摘要到前缀后面**，不是改历史。

*方案 3：工具调用失败永远不回滚，只向前修复*

不要用"回滚到上一个状态再重试"的策略。失败信息以追加形式进 Log，让模型基于完整历史决定下一步。

**适用与失效**
- 适用：高频调用同一 API + 大量重复前缀的场景（agent loop 是典型）
- 适用：DeepSeek（自动缓存，命中规则严格）尤其需要
- 失效：低频调用 / 每次请求完全不同的内容——缓存本来就没价值
- 注意：Anthropic 缓存需要**显式标记 cache_control**，不是字节自动匹配

**源**

`[[cache-aware-agent-loop]]`（Reasonix case：4.35 亿 token $1.38 vs 无缓存估算 $61）。验证状态：未验证（外部 case study）。

**See also**
- P-006 多步工作流上下文膨胀（**方案互冲警告**：P-006 的"删除/截断旧内容"会破坏 P-014 的 Append-only 不变性。本地推理选 P-006，云 API 选 P-014）
- P-015 上下文压缩破坏缓存

## P-015：做了上下文压缩反而让缓存命中率下降

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 上下文太长，按常规做法做了压缩（删旧消息 / 用摘要替换旧消息）
- 压缩之后**缓存命中率反而暴跌**
- 上下文短了，API 费用却涨了

**根因**

前缀缓存按字节匹配前缀；如果第 10 轮压缩并替换了第 3 轮消息，从第 3 轮开始的所有字节都会变化，已累积缓存失效。缓存感知场景下，压缩只有在不破坏前缀时才真省钱。

**当前方案**

**Cache Line Fold（缓存对齐折叠）**：当上下文使用率超过阈值（如 70%）时，把旧消息总结成一条摘要，**追加到前缀后面**——不是替换旧消息，是追加。

实现细节：
1. 算 tail budget（上下文窗口的 20%），从日志末尾往前数保留最近消息直到填满
2. 剩下的旧消息（前面那块）调一次便宜模型（如 V4 Flash）生成摘要
3. 把摘要追加到 Immutable Prefix 之后、Append-only Log 之前
4. 摘要调用本身也复用主对话的不可变前缀，**摘要调用也能命中缓存**

多级触发阈值：70% 折叠 / 75% 激进折叠（tail budget 缩到 10%）/ 78% 强制提前退出 turn / 80% 生成恢复性总结 / 90% turn 开始前预防性折叠。

**适用与失效**
- 适用：使用前缀缓存的云 API + 长 agent 工作流
- 失效：本地推理 / 不计较 API 成本 / 没有缓存机制的 provider——这时用 P-006 的 Tiered Compact 更直接
- 注意：tail budget 太小会让摘要要覆盖的内容太多，摘要质量可能下降

**源**

`[[cache-aware-agent-loop]]`（Cache Line Fold 段）。验证状态：未验证。

**See also**
- P-014 缓存命中率低（这是根症状）
- P-006 多步工作流上下文膨胀（**方案互冲**：P-006 删/截断 vs P-015 追加。按是否关心缓存选）
- `[[agent-context-compaction]]`

## P-026：上下文压缩连续失败，系统不停重试浪费 API 调用

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 上下文压缩操作反复失败，每次失败后系统再次自动触发
- 同一对话日志显示 50+ 次连续压缩失败；极端情况下单对话失败 3000+ 次
- 全局每天浪费数十万次 API 调用，但系统没有报错，只是账单暴涨 + 对话始终无法继续
- 或者：子 agent（如记忆提取 agent）触发了压缩，然后压缩又触发子 agent，形成递归死锁

**根因**

压缩子系统缺少熔断机制。压缩失败可能是结构性的（上下文本身已不可恢复，无论重试多少次都无法压缩成功）；反复重试不仅无效还持续浪费 API 调用。子 agent 触发压缩会导致递归死锁的原因：子 agent 是为了做压缩相关任务而 fork 出来的，如果它们也能触发压缩，就会无限嵌套。

**当前方案**

*方案 1：连续失败熔断器（3 行修复，首选）*

连续压缩失败超过 N 次（默认 3 次）立即停止重试，不再自动触发；以明确错误状态向上层报告。不要用指数退避——压缩失败通常是结构性的，退避后再试仍会失败。

*方案 2：子 agent 排除逻辑*

记忆提取、压缩子任务等后台子 agent 的请求**不会触发自动压缩**。实现方式：给每个请求加来源标记（`QuerySource`），子 agent 标记的请求跳过自动压缩触发逻辑。

*方案 3：压缩本身超长时的处理*

压缩请求的输入可能本身就超长（用来缩短上下文的操作，本身的输入可能就超限）。解法：按 API 轮次分组，从最早的组开始丢弃，最多重试三次。

**适用与失效**
- 适用：使用自动上下文压缩的长对话 agent 系统
- 失效：单次短对话不会触发

**源**

`[[agent-context-compaction]]`（Cloud Code 真实事故：1279 个对话 50+ 次连续压缩失败，最多 3272 次失败/对话，全局每天浪费约 25 万次 API 调用。3 行修复：连续失败超过 3 次就停止重试）。验证状态：部分验证（事故和修复来自 Cloud Code 源码注释，符合真实生产环境）。
