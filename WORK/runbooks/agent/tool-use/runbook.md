## P-001：模型该调工具时输出了自由文本

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 模型该发起工具调用时，输出了包含工具名和参数描述的自然语言文本
- 框架判定"模型未调用工具"，进入错误处理或终止
- 在小模型、量化模型、本地推理后端上更频繁

**根因**

模型的**意图**正确（知道该调什么、传什么），**格式生成能力**不足。常见诱因：推理模型在思考过程中"预演"工具调用，用了非标准格式；JSON 被包在 markdown 代码块里；同一权重，原生 function calling 后端能输出干净 JSON，提示注入后端容易漂移。

**当前方案**

*方案 1：救援解析（默认先上）*

在判定"未调用工具"之前，依次尝试从自由文本里提取：（1）代码块/反引号包裹的 JSON；（2）推理预演格式（"I will call X with {...}"）；（3）Qwen 系 XML 工具调用格式。三种全部失败才视为真正解析失败，进入方案 2。

*方案 2：切换推理后端*

原生 function calling 模式 vs 纯提示注入，准确率差距可能达 70+ 个百分点（来源：Forge 评测）。**换后端是最便宜的修复，应该排在加防护层之前**。

*方案 3：合成 respond 虚拟工具*

如果模型同时也频繁在"该调工具"和"该输出文本"之间选错，给所有输出统一加一层：定义一个 `respond` 工具承接文本回复，模型永远在"调工具模式"，不再做模式选择。Forge 实测：信任模型自己选两种模式，工作流完成率从 100% 降到 4%。

**适用与失效**
- 适用：≤30B 模型 / 量化 ≤ Q8 / 本地推理。方案 1 即使在云 API 上也建议默认开
- 失效：模型输出了**错误的工具名**或**幻觉参数**——这是另一类问题（→ P-002），rescue 救不了

**源**

`[[small-model-harness-engineering]]`（Forge ACM 2025）。验证状态：未验证。

**See also** P-002 P-005 P-007 P-008 P-018 `[[tool-call-repair-harness]]` `[[structured-output]]`

## P-002：模型调用了不存在的工具

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 模型生成的工具调用里，工具名不在已注册工具列表中
- 参数结构可能完全合理，名字是模型"想象"出来的（比如把 `find_user` 调成 `search_user`）
- 小模型上更频繁；任务描述含糊或多个相似工具时也容易触发

**根因**

模型并未真正"读"工具列表，而是基于语义相似度生成它认为"应该叫这个"的名字。和 P-001 的关键区别：P-001 是格式问题（意图正确，写法不对），P-002 是工具名层的幻觉（意图本身就指向了不存在的目标）——救援解析救不回来。

**当前方案**

*方案 1：注入工具列表强制选择*

错误响应里**直接附上可用工具列表**，告诉模型"工具不存在，从 [list] 中选一个"。比"工具不存在，请重试"信息密度高得多——避免模型在错误信息里自由再生。

*方案 2：命名收敛*

如果常错的工具名有规律（总把 `find_*` 写成 `search_*`），结构性修复：要么注册别名（同名指向同一实现），要么把工具改名向模型偏好对齐。

**适用与失效**
- 适用：工具集合静态已知的场景
- 失效：动态工具注册时，方案 1 仍能用但需每次注入当前列表；方案 2 不适用

**源**

`[[small-model-harness-engineering]]`（Forge 第二层 retry prompting）。验证状态：未验证。

**See also** P-001 P-007 `[[tool-call-repair-harness]]`

## P-003：模型跳过必要的中间步骤

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 模型直接调用终端工具（`submit` / `finalize` / `apply`），跳过了必要的前置（`search_list` → `get_detail`）
- 终端工具因为缺参数 / 缺上下文而失败
- 多步工作流的小模型上几乎必现

**根因**

小模型对"工作流应该 N 步"的理解不稳定。看到一个看起来能解决问题的终端工具，倾向直接调用，而不是按必要顺序走完。

**当前方案**

*方案 1：编码工具前置依赖 + Step Enforcer*

在工具注册时声明前置依赖。Step Enforcer 在工具触发前检查，发现跳步时**不让工具真正执行**，直接注入纠正消息。

*方案 2：升级式纠正消息*

第一次跳步：语气礼貌；第二次：语气直接；第三次：措辞强硬 + 终止 turn。升级比每次同样温和的纠正有效——避免模型在同一种语气下反复尝试同一个错误。

**适用与失效**
- 适用：工作流依赖关系明确、可显式编码的场景
- 失效：依赖关系是数据依赖（运行时才知道哪个工具先调），无法静态编码

**源**

`[[small-model-harness-engineering]]`（Forge 第三层 Step Enforcer）。验证状态：未验证。

**See also** P-007 `[[prerequisite-gate-before-terminal-tool]]`（DC pattern）

## P-004：工具错误计数混淆，系统过早放弃

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 模型连续几次调用工具失败，框架终止了 turn
- 但失败原因其实是"猜错参数"（查了不存在的用户 ID）而非"工具本身坏"
- 把所有失败统一计数，模型"还在正常探索"就被砍掉

**根因**

工具失败至少有两类：工具本身坏了的硬错误，以及格式正确但资源不存在的 ToolResolutionError。把它们混进同一个计数器，会让模型正常探索几次参数后就被系统过早终止。

**当前方案**

工具返回错误时显式区分两类，使用独立计数器：硬错误累加全局错误计数（默认 2 次终止）；ToolResolutionError **不累加**全局计数，只把"资源不存在"语义返回给模型，让它换参数再试。

**适用与失效**
- 适用：搜索 / 查询类工具占主导的工作流（探索型任务）
- 失效：写入 / 副作用型工具——"目标不存在"可能本身就是错误前提

**源**

`[[small-model-harness-engineering]]`（Forge 第四层错误恢复）。验证状态：未验证。

**See also** P-007 `[[tool-error-split-counting]]`（DC pattern）

## P-005：怀疑本地模型工具调用能力差，下结论之前

`type: diagnostic-procedure`   `severity: reversible-cheap`

**症状**
- 本地部署的某模型在工具调用 benchmark 上得分很低（< 20%）
- 想换更大的模型 / 加防护层 / 微调，但还没排查后端

**根因**

同一套权重，不同推理后端准确率可能差 70+ 个百分点：llama.cpp 原生 function calling 模式 7%，换成 llama-file 提示注入模式跳到 83%（来源：Forge）。**"这个模型不行"的结论可能根本就是错的——只是用错了后端**。

**前置条件**
- 能在至少两个推理后端上跑同一模型权重
- 有一小组固定工具调用样本（同 prompt、同工具定义、同评分口径）

**当前方案**

*关卡 1：固定模型和样本，做后端 ablation*

原生 function calling 后端（vLLM tool choice、llama-server）vs 提示注入后端（llama-file、自拼 system prompt）。差距明显 → 先换后端；差距不明显 → 进关卡 2。

*关卡 2：按失败形态转入对应条目*

输出自由文本但意图对 → P-001；工具名不存在 → P-002；跳过前置步骤 → P-003；文本回复 / 工具调用模式选错 → P-008；仍然无法归类 → 做模型边界探测。

**适用与失效**
- 适用：本地推理 / 自托管模型
- 失效：闭源云 API（你无法换后端），直接对比不同模型即可

**源**

`[[small-model-harness-engineering]]`（Forge benchmark：同模型两种后端 7% vs 83%）。验证状态：未验证。

**See also** P-001 `[[structured-output]]` · `[[model-boundary-probing]]`

## P-008：模型在"调工具"和"输出文本"之间频繁选错

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 模型该调工具时输出了文本（不是格式错，是真的没调任何工具）
- 或者该输出最终答复时调了无关工具
- 工作流完成率显著低于"格式问题"能解释的范围

**根因**

让小模型在"输出文本 vs 调工具"两种模式之间自主选择，是一个独立的认知任务。Forge 实测：信任模型自己选两种模式，工作流完成率**从 100% 降到 4%**——这不是格式问题（P-001），是**模式选择本身就难**。

**当前方案**

定义一个 `respond` 虚拟工具，模型想回复文本时调用 `respond` 并把文字传入 `content` 参数。这样模型**永远处于"调工具模式"**，不需要在两种模式间选择，彻底消除歧义。

**适用与失效**
- 适用：小模型 / 量化模型 / 任何模式选择不稳定的场景
- 失效：模型对模式选择本身稳定时（前沿大模型）这层是冗余开销

**源**

`[[small-model-harness-engineering]]`（Forge Respond 工具段，100% → 4% 完成率落差）。验证状态：未验证。

**See also** P-001 `[[respond-single-output-channel]]`（DC pattern）

## P-009：JSON 输出格式不稳（多逗号 / 加前缀 / 类型错）

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 让模型输出 JSON，大部分时候能照做，但偶尔：前面加一句话、多一个逗号 / 漏一个引号、数字输出成字符串、字段缺失
- 单轮重试就行，但**喂给下一步的 agent 工作流会因此整链崩**

**根因**

LLM 本质是 token 生成器，**天生倾向输出自由文本**。光靠 prompt 里写"请返回 JSON"是软约束，不是物理保证。

**当前方案**

*方案 1：原生约束解码（strict / structured outputs）*

主流 API 都提供这个能力。原理是每生成一个 token 时，按 JSON Schema 物理限制模型能选择的 token 集合。**不是劝模型听话，而是从解码层让它无法生成不合法 token**。

*方案 2：工具调用包装（模型不支持 strict 时的兜底）*

把结构化输出伪装成工具调用：定义一个虚拟工具，把期望的输出格式写成它的输入 schema，强制模型调用这个工具。

*方案 3：验证 + 重试（语法层兜底）*

方案 1/2 都失败或不可用时，模型输出后用 JSON parser 验证，失败则把错误信息连同原始输出回传给模型重新生成。

**适用与失效**
- 适用：所有需要结构化输出的 agent / pipeline 场景
- 失效：方案 1 在指令微调模型 + 生成型任务上会触发**语义质量下降**，详见 P-010

**源**

`[[structured-output]]`（Layer 1 + Layer 3）。验证状态：未验证。

**See also** P-001 P-010 P-011 `[[reasoning-output-decoupling]]`（DC pattern）`[[tool-call-repair-harness]]`

## P-010：开启 strict / 约束解码后输出语义质量下降

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 启用约束解码（strict mode / structured outputs）后，JSON 格式 100% 合法
- 但内容明显变差：关键信息漏了、措辞变奇怪、生成型字段（如 summary / description）质量降
- 切回非 strict 模式后质量回升

**根因**

约束解码会在每一步 mask 掉不合法 token；当模型首选 token 被限制时，它只能从较低概率候选里选。它保证格式正确，但不保证语义质量，尤其是指令微调模型的生成型字段可能退化。

**当前方案**

*方案 1：推理和输出分阶段*

让模型先**自由思考**（无 schema 约束），最后**只在最终输出**时施加 schema。Claude API 已内建这种模式：thinking 阶段完全不受约束，只有最终 response 被结构化——既不损害推理质量，又保证输出格式。

*方案 2：在生成型字段上不硬启 strict*

把 schema 拆分：分类 / 结构性字段（label、enum、bool、id）→ strict 开；生成型字段（summary、explanation、reasoning_text）→ 不强约束，靠验证+重试兜底。

*方案 3：评估"是否真的需要 strict"*

在不同能力等级的模型上跑同一个 agent。如果新模型上关掉 strict 性能反而更好，说明约束已经成了瓶颈——该放松了。

**适用与失效**
- 适用：用 strict 的生成型任务出现质量下降时
- 失效：纯结构化任务（提取实体、分类、打标签）通常不会触发这条

**源**

`[[structured-output]]`（Layer 1 反事实段 + Bitter Lesson 视角段）。验证状态：未验证。

**See also** P-009 `[[reasoning-output-decoupling]]`（DC pattern）

## P-011：JSON 语法对但值不合理（语义错误）

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- JSON 解析成功，所有字段类型也对
- 但取值越界：`rating` 字段 schema 要求 1-5，模型输出 7
- 或字段间逻辑冲突：`status: "completed"` 但 `completion_time: null`
- 约束解码 / strict 模式救不了这种错

**根因**

约束解码解决的是语法问题——括号配对、类型匹配、字段完整。但它解决不了语义问题——schema 能要求 `rating` 是整数，无法限制取值范围（除非用 enum，但很多业务规则是字段间的，无法静态编码）。

**当前方案**

*方案 1：业务规则 schema + 验证器 + 错误反馈重试*

1. 用 Pydantic / Zod 定义带业务规则的 schema；2. 模型输出后过验证器；3. 验证失败时，把具体错误连同原始输出一起反馈给模型，让它重新生成。关键是错误信息要具体（"rating 必须在 1-5，你输出了 7"），不是模糊的"验证失败"。

**适用与失效**
- 适用：所有有业务规则的结构化输出
- 失效：验证重试本身要计入 retry 预算（参见 P-007），不能无限循环

**源**

`[[structured-output]]`（Layer 2）。验证状态：未验证。

**See also** P-007 P-009 `[[business-rule-validation-layer]]`（DC pattern）

## P-012：工具集合大，模型频繁选错；想动态增删但会废 KV cache

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 注册了几十个工具，模型在选择下一步动作时频繁选错
- 直觉解法："当前用不到的工具动态从工具列表里移除"
- 实操后发现：**KV Cache 命中率塌方，每步延迟 / 成本暴涨**

**根因**

工具定义在上下文的**前部**，是 prompt 的稳定 prefix。任何对工具列表的变更（增 / 删 / 改顺序）都会让 prefix 的某个 token 变化，**导致从该 token 之后所有内容的 KV Cache 失效**。

**当前方案**

*方案 1：Logit Masking + 工具命名前缀化*

**工具定义永远不变**（KV Cache 安全），但在解码时通过 mask logits 来屏蔽当前不需要的工具。工具按功能分组、前缀化（browser_* / shell_* / fs_*），按前缀做 mask。

*方案 2：如果推理后端不支持 logit mask*

在 system prompt 里**动态注入"当前可用工具子集"的提示**（但工具定义本身保持完整）。效果弱于方案 1，但兼容性好。

*方案 3：两级加载 + tool_search（按需发现）*

核心工具集（7-8 个高频）启动时加载，延迟工具集模型只知道名字，需要时先调用 `tool_search` 获取完整定义。工具排序时把核心工具放在前面作为连续前缀，外部插件工具放后面——插件变化不破坏核心前缀缓存。

**反向陷阱：不要为了"减少模型负担"动态删除工具定义**——这是直觉但错误的解法。

**适用与失效**
- 适用：长上下文 agent（KV Cache 价值高的场景）+ 工具数量 > 10
- 失效：工具数 ≤ 5 时直接全开，不需要 mask 或两级加载

**源**

`[[structured-output]]` `[[agent-tool-design]]` `[[context-engineering]]`（综合）。验证状态：未验证。

**See also** P-006 P-020 P-021 `[[immutable-prefix-append-only-log]]`（DC pattern）`[[cache-aware-agent-loop]]`

## P-017：工具参数 schema 嵌套深 / 叶子多，模型频繁丢参数

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 工具的输入 schema 有嵌套对象（如 `config.database.connection.host`）或字段数 > 10
- 模型调用时漏字段、漏层级
- 即使错误重试，模型还是丢同一个深层字段

**根因**

模型对深嵌套参数 schema 的处理稳定性差——一旦层级超过两层或叶子超过 ~10 个，丢参数的概率显著上升。这是模型能力边界问题，不是格式问题。

**当前方案**

**Flatten（参数展平）**：

1. 在工具注册时**自动检测** schema 复杂度（叶子节点数、嵌套深度）
2. 超过阈值（如 > 10 叶子 或 嵌套 > 2 层）时，把深层嵌套展平为点号分隔的扁平路径（原：`{config: {database: {host: "x"}}}` → 展平后：`{"config.database.host": "x"}`）
3. 工具 Dispatch 时再还原嵌套结构调用真实实现

**适用与失效**
- 适用：工具参数本来就嵌套的复杂业务场景
- 失效：参数本来就扁平 / 嵌套很浅 → 不需要这层

**源**

`[[cache-aware-agent-loop]]`（工具调用修复管道 / Flatten 段）。验证状态：未验证。

**See also** P-001 P-012 `[[agent-tool-design]]`

## P-018：模型输出 JSON 被截断，工具层静默用空对象执行

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 模型输出到 max_tokens 被截断
- JSON 残缺（缺右括号 / 缺引号 / 字段值不完整）
- **工具层没报错**，用了一个空对象或默认值就执行了——产生静默的错误行为
- 下游表现为"工具调用了但结果不对，原因莫名"

**根因**

很多框架对 JSON parse 失败的处理是兜底返回空对象 `{}` 或 `null`，然后把它当合法参数传给工具——结果是工具用默认参数执行，**无声的语义错误**比 loud 的崩溃难调试得多。

**当前方案**

**Truncation（截断检测 + 显式失败）**：

1. 检测不平衡括号 / 引号
2. 如果可以确定性补全（比如只差一个 `}`），尝试补全后重新 parse
3. **补不了的标记为不可恢复**，让工具层返回明确的"无效 JSON 错误"给模型，**绝不静默用空对象**

关键设计原则：**fail loud, not silent**。截断本身常见（max_tokens 限制），但下游行为应该是显式错误 → 模型重试或纠正，不是默默用错参数。

**适用与失效**
- 适用：所有用 LLM 输出 JSON 喂工具的场景
- 失效：模型不会输出 JSON（如纯文本生成）

**源**

`[[cache-aware-agent-loop]]`（工具调用修复管道 / Truncation 段）。验证状态：未验证。

**See also** P-001 P-009 P-011 `[[fail-loud-on-bad-json]]`（DC pattern）

## P-020：给了 agent 万能工具后，专用工具被弃用

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 你给 agent 接了三个工具：bash（执行任意命令）/ file_read / search
- 模型拿到 bash 后**什么都用 bash**：读文件 `cat`、搜索 `grep`、列目录 `ls`
- 你精心设计的 `file_read` / `search` 工具基本没被调用过

**根因**

模型发现万能工具能覆盖大量任务后，会倾向用最少工具完成最多事。问题是万能工具会绕过专用工具的结构化输出、权限边界和输出预算，后续 UI、安全和上下文管理都失去控制。**专用工具的价值不在"它能做什么"，而在"它把操作限制在可控范围"**。

**当前方案**

*方案 1：在万能工具的描述里主动让位（"别用我"反直觉）*

花大量篇幅写不要用这个工具做文件读取、搜索、列目录这些操作，并列出工具偏好表。**关键原则：把约束放在最接近决策点的地方**（工具描述），不要放在远端的 system prompt 里——工具描述是和工具调用绑定的，模型每次考虑要不要调这个工具时都会看一遍。

*方案 2：在权限层硬约束 bash 的部分能力*

参见 P-022 的分层规则——比如禁止 `cat` / `grep` / `ls` 等已有专用工具的命令，强迫模型走专用工具。这是 belt-and-suspenders 兜底。

**适用与失效**
- 适用：agent 同时配了"万能工具"+"专用工具"的所有场景
- 失效：你的 agent 本来就只有一个万能工具（没有专用替代）

**源**

`[[agent-tool-design]]`（原则一，Cloud Code 设计）。验证状态：未验证（外部 case）。

**See also** P-021 P-022 `[[constraint-at-decision-point]]`（DC pattern）

## P-021：单次工具调用输出一次撑爆上下文窗口

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 某次工具调用返回了几万 / 几十万字符（数据库查询返回大量行、日志搜索、`cat` 大文件、`find` 全目录树）
- 整个上下文窗口被这一次输出占掉大部分
- 后面所有推理都受影响——模型注意力被无关数据淹没，决策质量下降

**根因**

工具输出大小不可预测：同一个查询可能返回 10 行，也可能返回 10 万行。直接截断虽然保住窗口，但可能截掉关键证据，让模型基于残缺信息继续推理。

**当前方案**

*方案 1：每个工具显式设输出预算 + 超长存外部 + 返回预览 + 指针*

大部分工具：上限 10 万字符；搜索工具：上限 2 万字符（"无界搜索可以轻松填满上下文"）。超过上限的处理：把完整输出存到磁盘文件，给模型返回预览（前几千字节）和指针（文件路径，需要完整内容时用 `read` 工具读）。

*方案 2：搜索类工具默认带条数上限（如 250 条）*

模型确实需要更多时显式传参解除限制，但默认保守。

*方案 3：read 类工具的特殊处理（防自指）*

如果文件读取工具的输出也存磁盘，模型读回来又触发存储 = 死循环。所以 `read` 用独立机制：自己的 token 预算检查，超过直接报错并告诉模型用 offset + line_count 分段读。

**适用与失效**
- 适用：任何会返回不定长输出的工具（搜索 / 查询 / 列举 / 大文件读取）
- 失效：工具输出本来就有上界且很小（如 `get_status` 返回单个状态码）

**源**

`[[agent-tool-design]]`（原则三，Cloud Code 设计）。验证状态：未验证。

**See also** P-006 P-014 P-018 P-020 `[[tool-output-budget-with-pointer]]`（DC pattern）

## P-037：校验前对工具输入预处理，静默损坏合法输入

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- 工具处理"看起来像数组的字符串"时，把它解析成了数组，但原始输入其实是合法的字符串字面量（比如写文件的内容字段要写入一个 JSON 字符串）
- 或者预处理规则修改了本来就合法的输入，导致工具用了错误的数据但没有报错
- 合法输入被"聪明的修复逻辑"静默篡改，且极难排查（表面上工具执行成功了）

**根因**

在 schema 校验之前对输入做"智能修复"——逻辑是"我猜你想要数组，我帮你转"——会误伤本来就合法的输入。这是"先修复再校验"的典型陷阱：TOCTOU 风险（扫描通过后、写入前内容可能被改）+ 误伤合法路径。

**当前方案**

*方案 1：先校验再修复（顺序必须是这个方向，首选）*

原始输入先过 schema 校验器：通过 → 直接执行，合法输入永远不被触碰；不通过 → 读取校验器给出的 issue 列表（字段路径、期望类型、实际类型）→ 只沿出错字段尝试有限修复 → 修复后再校验一次，通过才执行。

*方案 2：正常路径零接触*

合法输入绝不被触碰，只有校验失败的输入才进入修复逻辑——不要猜哪里可能坏了，让校验器指出哪里坏了。Schema 在这里不只是约束，也充当故障定位器。

*注意修复顺序*：字符串化数组解析必须在裸字符串包装之前执行，否则会把 `["a","b"]` 先包成单元素数组而不是解析成两元素。

**适用与失效**
- 适用：所有对 LLM 输出做格式修复的场景
- 失效：输入来源完全确定性（非 LLM），没有"LLM 格式漂移"问题时不需要这层

**源**

`[[tool-call-repair-harness]]`（先校验再修复 vs 先预处理再校验的对比；schema 充当故障定位器）。验证状态：未验证（外部 claim）。

**See also** P-009 P-011 P-018 `[[schema-validate-before-repair]]`（DC pattern）

## P-038：LLM 输出 Markdown 格式路径 / URL，污染工具的文件系统操作

`type: remedy-menu`   `severity: reversible-cheap`

**症状**
- Agent 写文件路径时输出了 `[config.json](./config.json)` 这样的 Markdown 链接格式
- 工具用这个路径去找文件，找不到（非法路径）
- 在聊天场景下模型被奖励过"给文件名加链接"，在工具调用场景里这变成了非法路径

**根因**

后训练的聊天分布泄露到了工具调用场景——模型没有区分"这个输出是给人看的富文本"和"这个输出要传给文件系统"。这不是普通幻觉，是训练分布边界问题。

**当前方案**

*方案 1：局部层正则处理退化情况（快速止血）*

识别 `[text](url)` 格式，提取括号内的真实路径。处理常见 Markdown 路径格式的正则。

*方案 2：结构层——schema 字段语义化（更根本）*

把工具参数 schema 里路径相关字段的类型从通用 `string` 改为语义化的 `path-string` 类型，在 description 里明确"这是文件系统路径，不要添加 Markdown 格式化"。一次改动覆盖所有路径字段——模型的分布泄露问题在 schema 层被结构性约束。

*修复顺序注意*：先校验检测到 path 字段包含 Markdown 格式 → 再触发修复，不要在校验前预处理（见 P-037）。

**适用与失效**
- 适用：任何工具参数包含文件路径 / URL 的场景
- 失效：模型已经没有这种分布泄露时（更强的模型可能不需要这层）

**源**

`[[tool-call-repair-harness]]`（Markdown 链接格式污染文件路径；schema 语义化类型作为结构层修复）。验证状态：未验证。

**See also** P-037 `[[semantic-typed-schema-fields]]`（DC pattern）
