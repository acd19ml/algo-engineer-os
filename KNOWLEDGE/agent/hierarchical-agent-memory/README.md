# Hierarchical Agent Memory: 虚拟文件系统 + 三层按需加载

## 它要解决什么问题

把记忆系统做成平面文件列表——比如 Cloud Code 的 Active Recall（见 `KNOWLEDGE/agent/agent-memory-system/`）——在编码 Agent 的典型使用规模下够用，但扩展到更复杂的应用场景时暴露五个问题：

1. **200 文件硬上限**：Active Recall 扫描上限是 200 个文件，超过的直接忽略。更大规模的知识库——项目文档、用户历史、领域知识——200 条远远不够
2. **单次判断，无探索能力**：模型看一个平面清单，做一次选择就结束。"这个方向看起来相关，让我再深入看看"这种多步探索能力完全没有
3. **所有记忆平铺在一起，没有结构**：用户偏好、项目背景、工具使用经验混在一张清单里，模型需要同时判断类型和相关性
4. **Token 粒度粗**：要么读全文，要么只看摘要，没有"先看梗概判断值不值得深入"的中间选项
5. **Agent 的执行经验不会自动沉淀**：Active Recall 记录的是用户偏好和反馈，但 Agent 自己积累的经验（哪个工具在什么场景好用、什么错误模式反复出现）没有独立的存储位置

OpenViking 针对这五个痛点，用一套以层级结构为核心的记忆架构重新设计了整件事。

## 如果把记忆系统做成向量数据库

另一个常见方案是 embedding + 向量搜索：把所有记忆 embedding，检索时找最相似的几条。这解决了上限和结构问题，但仍有一个根本限制：**平面搜索不利用上下文信息**。

向量搜索返回的是"全局里最像的几条"——但如果你知道"这类记忆通常在 agent-learning 目录下"，那先定位到这个目录，再在里面搜，才是高效的。书架检索和全馆翻书的区别。层级结构天然携带了这种上下文：一个目录的摘要高度相关，意味着目录下的所有内容都值得看一看，**即使某些条目的向量相似度不是最高的**。

## 虚拟文件系统：统一存储接口

OpenViking 不把记忆、资源、技能分开存储，而是把所有上下文映射到一个虚拟文件系统里，用统一的 URI 标识。顶层三个目录：

- **`resources/`**：外部资源。项目文档、代码仓库、网页
- **`user/`**：用户的长期记忆。按四个子类组织：
  - `profile/`：用户画像（角色、技术背景、工作习惯）
  - `preferences/`：偏好（编码风格、沟通方式）
  - `entities/`：实体记忆（公司、人物、项目的持久信息）
  - `events/`：事件记录（发生了什么、决定了什么）
- **`agent/`**：Agent 自己的学习记忆，这是 Cloud Code 范式里缺失的部分：
  - `cases/`：具体问题加解法
  - `patterns/`：可复用的流程方法
  - `tools/`：工具使用经验（什么工具在什么场景下好用、已知坑）
  - `skills/`：技能执行策略

Agent 操作记忆就像操作文件系统一样：用 `ls` 列目录、用 `find` 搜索特定内容、用 `tree` 看层级结构——这不是比喻，是真实的操作接口。

## 三层力度按需加载

这是 OpenViking 最核心的设计。每个目录节点都有三层内容：

- **L0（Abstract）**：约 100 token，一句话摘要。用来快速判断"这个目录跟我当前的问题有没有关系"——判断成本极低
- **L1（Overview）**：约 2000 token，包含核心信息和使用场景。用来做决策："我需不需要深入看这个目录的具体内容？"
- **L2（Full content）**：完整原始内容。只有 Agent 确定需要时才加载

写入时三层同时生成，读取时按需选择。

效果是什么？集成测试数据：**输入 token 成本降低 83%–96%**——因为大部分时候 Agent 只需要看 L0 和 L1 就能做出决策，不需要加载完整内容。

反事实：如果没有分层，每次检索都要加载完整内容，或者每次都只看摘要。前者 token 成本爆炸，后者在摘要写得不好的时候会漏掉相关记忆——**分层打破了这个两难**。

## 层级递归检索：把平面搜索换成分级探索

检索流程分四步：

1. **全局向量搜索**：在所有 L0 和 L1 节点上搜索，找得分最高的目录。这步找的不是最终答案，而是**哪些目录可能包含答案**
2. **在高分目录内搜索子节点**：找到的子节点如果还是目录，继续递归下转；如果是叶子节点，收集为候选结果
3. **分数传播**：子节点的最终得分 = 自身向量相似度 + 父目录得分的加权继承。高度相关的目录下的所有内容都获得位置加成，**即使某些内容的向量相似度不是最高的**
4. **收敛检测**：如果连续三轮搜索结果没有变化，停止递归，防止无限下转

## Session Commit：自动提取记忆

当一轮对话结束时，系统自动从对话中提取记忆，分类存入对应目录，每条记忆生成 L0、L1、L2 三层内容。Memory Extractor 用大模型分析对话内容提取候选记忆，分类到上面八个类别里。

对话结束自动触发，**用户无需手动说"记住这个"**。

另一个机制是 Working Memory：系统维护一个七段式结构化工作记忆文档（对话标题、当前状态、任务目标、关键决策、相关文件、错误记录、待解决问题）。每次更新时大模型对每个段落做**保持、更新或追加**的决策，而不是重写整个文档——保证结构稳定性。

## 综合效果

集成测试里 OpenViking 的表现：任务完成率比基线提升 43%–49%，同时输入 token 成本降低 83%–96%。**这两个数字同时改善**——说明不是靠塞更多信息来提升效果，而是靠更精准地找到需要的信息。

## 一个没解决的问题

有一篇叫 MEM 的论文测试了记忆系统在级联更新场景下的表现：当一个上游事实变化（比如用户搬家），依赖它的下游事实能不能自动更新（每天骑车 15 分钟通勤的记忆还成立吗）？所有被测系统准确率只有 3%。

OpenViking 能部分缓解这个问题——相关事实在同一目录下自然聚集（`user/entities/` 里住址和通勤方式在同一处），递归检索进入这个目录时 Agent 能同时看到相关事实，**有机会**推断出不一致。但这依赖 Agent 的推理能力，不是系统层面的保证。**级联更新仍然是所有记忆系统的开放问题，目前没有任何系统从架构上解决了它**。

## Open Questions

- **L0 的摘要质量是整个检索链的瓶颈**。如果 L0 写得不好，全局搜索第一步就走错方向，后续递归完全无效。Session Commit 时的 L0 生成质量稳定性如何保证？有没有类似 Cloud Code "过期警告" 的机制？
- **分数传播的权重**（父目录加成多少）对检索精度影响很大，但材料里没说这个权重是怎么设的，是固定值还是学出来的，是否对不同深度的目录有不同权重
- `agent/cases/` 和 `agent/patterns/` 在实际使用中的边界是什么？一个"在 X 场景下用 Y 工具解决了 Z 问题"既是 case 也是 pattern 的一部分——如何避免跨类别的重复存储（参见 `KNOWLEDGE/agent/agent-memory-system/` 里 Cloud Code 的"互斥机制"）？
