# Cache-Aware Agent Loop: 前缀缓存命中率作为成本杠杆

## 它要解决什么问题

DeepSeek 的 API 对缓存命中的输入 token 收费是 miss 的五十分之一。如果你每天调用量大，这个倍率意味着：**把缓存命中率从 30% 提到 99%，输入成本直接降 97%**。

真实数据：一个用户在 2026 年 5 月 1 日，用 DeepSeek ReasonX Coding Agent 处理了 4.35 亿输入 token，花了 1.38 美元。同样的工作量，如果没有缓存优化，大约需要 61 美元。**同一个 API、同一天、同一批工作量，成本差 44 倍**。

问题在于：DeepSeek 的前缀缓存本身是自动的，命中率却完全取决于客户端。**缓存是 DeepSeek 提供的，命中率是客户端决定的**。规则只有一条：如果这次请求的前缀字节跟上一次完全一样，命中；差一个字节，miss。

大多数 Agent 框架的命中率只有 2–30%，原因很简单：每次请求都重新序列化工具定义，注入时间戳，甚至随机打乱 few-shot 顺序——**每次请求的前缀都略有不同，缓存永远 miss**。

## 如果把上下文当成普通文本管理

朴素方案：用消息列表维护对话历史，每轮请求时加一条新消息，其他不动。听起来没问题，但会在三个地方悄悄破坏缓存：

1. **工具定义重新序列化**：每次请求时 JSON 序列化的字段顺序可能变，一个空格的差异就让整个前缀变了
2. **思维链进入历史**：推理模型的 CoT 内容如果被记录进对话历史，下一轮请求的前缀就包含了上一轮特有的思维链，前缀不再可复现
3. **上下文压缩修改了历史**：标准压缩做法是删除或替换旧消息，**改了历史消息 = 改了前缀，缓存全部作废**

## 三区域设计：把上下文分成不同的不变性级别

ReasonX 把整个上下文切成三个区域，每个区域有严格的不变性约束：

**区域一：Immutable Prefix（不可变前缀）**

系统提示词、工具定义列表、few-shot 示例。对话开始时计算一次，**整个对话期间不变**。代码里有指纹追踪——如果有任何路径意外修改了前缀内容，指纹飘移，系统立刻报错。

约束很硬：工具定义的顺序不能变、系统提示词不能动态注入变量、few-shot 不能随机化。任何一个变化都让前缀字节不同，缓存就 miss。

**区域二：Append-only Log（只追加日志）**

对话历史。**只能追加，不能修改，不能重排，不能删除**。每轮 assistant 回复、工具调用结果都按时间顺序追加到日志末尾。

为什么这很重要：DeepSeek 的缓存按字节前缀匹配。如果在第 10 轮修改了第 3 轮的消息，从第 3 轮开始的所有字节都变了，**缓存全部失效**。Append-only 保证：每次新请求的前缀就是上一次请求的完整内容加新消息——前缀永远是上一次的超集，缓存永远命中。

**区域三：Volatile Scratch（临时暂存区）**

思维链、临时计划状态、每轮的中间笔记。**这个区域的内容不发送给 API**——只在本地存在，每轮开始时清空。

思维链里产生的有用信息，通过工具调用修复管道提取后以结构化形式追加进 Append-only Log，**思维链本身永远不进入请求体**。这解决了"推理过程污染前缀"的问题。

## 上下文压缩必须是缓存感知的

Append-only Log 会不断增长，总得压缩。如果压缩的方式是"删旧消息"或"用摘要替换旧消息"，前缀就变了，**缓存全部作废**。

ReasonX 的做法叫 **Cache Line Fold（缓存对齐折叠）**：当上下文使用率超过 70% 时，把旧消息总结成一条摘要，**追加到前缀后面**——不是替换，是追加。前缀本身不动，所以缓存还是命中。

具体流程：计算 tail budget（上下文窗口的 20%），从日志末尾往前数保留最近消息直到填满，剩下的旧消息调一次 V4 Flash 生成摘要。摘要调用本身也附用了主对话的不可变前缀，**摘要调用也能命中缓存**。

多级防御：到 75% 触发激进折叠（tail budget 缩到 10%），到 78% 强制提前退出当前 turn，到 80% 生成恢复性总结，到 90% 在 turn 开始前就预防性折叠。

## 工具调用修复管道：保护 Append-only 不变性

工具调用失败如果用"回滚到上一个状态再重试"的策略，就破坏了 Append-only Log 的不变性，缓存就 miss 了。ReasonX 的策略是**永远不回滚，只向前修复**：

- **Scavenge（打捞）**：模型有时把工具调用的 JSON 写在思维链里但忘了放入正式字段。Scavenge 扫描思维链，用正则和 JSON 解析器把遗漏的调用捞出来，支持三种格式（DSML 标记、OpenAI 标准格式、自由格式）
- **Flatten（展平）**：工具参数 schema 超过 10 个叶子节点或嵌套超过两层时，模型容易丢参数。Flatten 在工具注册时自动检测，把深层嵌套展平为点号分隔的路径，Dispatch 时再还原
- **Truncation（补全）**：模型输出到最大 token 长度被截断，JSON 不完整。Truncation 检测不平衡括号并尝试补全；补不了的标记为不可恢复，让工具层返回明确的无效 JSON 错误，**不静默用空对象执行**
- **Storm（抑制循环）**：滑动窗口追踪工具调用历史，同一工具名 + 参数组合在窗口内出现超过 3 次则抑制调用并注入反思。**写操作会清空窗口里的读操作记录**——编辑文件后读文件验证是正常工作流，不该被当成循环

## Flash-first 成本分层

默认用 V4 Flash，不用 V4 Pro（贵 12 倍）。但模型自己知道什么时候不够用：如果判断当前任务超出 Flash 能力，在回复第一行输出 `Need Pro` 标记，系统检测到后终止当前 Flash 调用，用 Pro 重试本轮。**这是模型决定的，不是基于错误计数或启发式规则**。

Turn End Auto Compaction：每轮结束后，超过 3000 token 的工具结果自动压缩——模型在产生回复的那轮已经看过完整输出，后续轮次只需要摘要。

## 一条设计原则

ReasonX 明确声明只支持 DeepSeek，不做多模型兼容。为什么？因为每个 API 厂商的缓存机制不同——Anthropic 需要显式标记缓存断点，OpenAI 的粒度不同。**如果要兼容所有后端，就不能对任何一个做深度优化，只能取最大公约数**。

如果你的 Agent 主要用一个模型提供商，不要用通用框架，要针对那个提供商的缓存机制、定价结构、已知模型缺陷做深度适配。**缓存策略应该是架构约束，不是事后优化**。

## Open Questions

- Append-only + 永不回滚的设计，在 Storm 检测到循环注入反思 turn 后，如果反思 turn 本身也触发循环，系统能正确退出吗？材料里没有说明这个嵌套情况
- Cache Line Fold 的 tail budget 设定（20% / 10%）是如何权衡"保留足够近期上下文"和"摘要准确性"的？如果 tail budget 太小，摘要需要覆盖的内容太多，摘要质量会不会下降，从而导致后续轮次的 Agent 误判？
