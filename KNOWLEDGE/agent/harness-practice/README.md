# Harness Practice: GAN-Inspired Three-Agent System

## 它要解决什么问题

Anthropic 用 Harness Engineering 的方法让 Claude 自主写了 4 个小时代码，最终从一句话 prompt 生成了一个能用的浏览器端音乐制作软件。**不是 Demo、不是 toy project，是有编曲视图、混音器、音轨编辑、还内置 AI 助手的完整应用**。

更关键的是：**他们在这个过程中踩了大量的坑，然后把方法论完整写了出来**。这个案例的价值不在结果本身，而在它揭示了 **"Agent = Model + Harness" 这个公式的两个具体失败模式以及怎么修复**。

## 朴素直觉为什么不够：单 agent vs 多 agent harness 对比

Anthropic 用同一句 prompt（"创建一个 2D 复古游戏制作器"），分别让一个**裸奔的单 agent** 和**他们设计的多 agent harness** 来跑：

| | 时间 | 成本 | 结果 |
|---|---|---|---|
| 单 agent | 20 分钟 | $9 | 打开看着还行，**一点就碎**——核心功能是坏的 |
| Harness | 6 小时 | $200 | 核心功能能用，且**产品范围远超单 agent**：自动规划出 16 个功能、10 个迭代周期，含精灵动画、AI 辅助关卡生成器 |

成本差了 20 多倍，**但同一个模型、同一句 prompt，产出天差地别**。

差的不是智能，**是智能周围的系统**。

## 反事实：Harness 的核心设计借鉴了 GAN

Anthropic 的设计灵感来自深度学习里经典的 **GAN（生成对抗网络）**——一个生成器负责创造，一个判别器负责挑刺，**两者对抗博弈，互相逼着对方进步**。

把这个思路搬到 agent 架构上，就是三个 agent 的协作：

- **规划器**（Planner）
- **生成器**（Generator）
- **评估器**（Evaluator）

每个 agent **解决一个明确的失败模式**。

## 规划器：故意只写"做什么"，不写"怎么做"

大多数人用 agent 写代码，上来就是一句 prompt 直接开干。Anthropic 发现这样做 **agent 会严重低估项目的范围，做出来的东西功能很单薄**。

所以加了一个规划器 agent，专门负责**把你的一句话 prompt 扩展成一份完整的产品规格说明**。

但这里有个**非常反直觉的设计决策**：**规划器只写高层产品设计，故意不写技术实现细节**。

为什么？因为如果规划器在前期把某个技术细节写错了，**这个错误会一路传导到后面所有的实现环节，越错越离谱**。**不如只约束"要做什么"，让后面的 agent 自己决定"怎么做"**。

这个思路值得记住：**当你设计多 agent 系统时，上游 agent 的输出粒度要非常克制，给方向但不给路径**。

## 生成器和"上下文焦虑"失败模式

生成器就是真正写代码的 agent。

Anthropic 一开始让它按迭代周期工作——一次只做一个功能，每个迭代周期结束后自评一次，再交给评估器。**但这里揭示了第一个重大失败模式**——

### 上下文焦虑（Context Anxiety）

当 agent 持续工作、上下文窗口越填越满时，**有些模型会开始慌**。它们会觉得自己快到上下文上限了，然后**开始草草收工，提前结束任务**。

注意——**不是真的到了上限，而是模型自己以为快到了**。

这个问题用常规的上下文压缩**解决不了**。所谓压缩就是把前面的对话摘要一下，让 agent 在缩短的历史上继续工作。**但 agent 虽然拿到了更短的上下文，心理上的焦虑并没有消失**——它仍然觉得自己快到极限了。

### 上下文重置：不是压缩，是彻底清空

Anthropic 的解法叫 **Context Reset**：**不是压缩，是彻底清空**。

启动一个全新的 agent，然后通过一个**结构化的交接文档**把前一个 agent 的状态和下一步计划传递过去——给模型一块**干净的白板**。

代价是增加了编排复杂度和延迟，**但效果显著**。

补充重要的后续发现：**当 Anthropic 后来发布了更强的 Opus 模型之后，"上下文焦虑" 这个问题基本消失了**——这意味着 Context Reset 这个组件**可以被拆掉了**。这一点和 Harness 的"可拆卸性"原则直接对应（详见 `KNOWLEDGE/agent/harness/`）。

## 评估器和"自我表扬"失败模式

这是整套 Harness 里**最关键也最难做好的部分**。

### 第二个重大失败模式：自我表扬

Anthropic 发现：**agent 在评价自己的工作时倾向于自信地表扬自己**——即使产出质量明显平庸，它也会说"做得不错"。

这个问题在主观任务上尤其严重（比如前端设计），**但即使在有客观标准的编程任务上，agent 也会犯同样的错误**——它会发现一个 bug，然后说服自己"这不是大问题"，然后放行了。

### 解法：把生成和评估彻底分离

让另一个独立的 agent 来当裁判。

**但仅仅分离是不够的**——因为评估器本身也是一个 LLM，它**天然对 LLM 生成的内容宽容**。

关键洞察：**调用一个独立的评估器让它变严格，比让生成器学会自我批判要容易得多**。**一旦严格的外部反馈存在了，生成器就有了具体的改进目标**。

### 怎么让评估器真正严格起来

Anthropic 做了两件事：

**第一件，把"主观的好不好"变成可打分的具体维度**。定义了四个评分标准：

- **设计质量**：颜色、排版、布局是否形成统一的整体感
- **原创性**：有没有定制化的设计决策？还是千篇一律的模板和 AI 生成套路
- **工艺**：技术执行力——间距一致性、对比度这些基本功
- **功能性**：用户能不能顺利完成任务

**很聪明的权重分配**：工艺和功能性权重低（Claude 默认就做得不错），**设计质量和原创性权重高**（这是模型的短板）。**评分标准里明确写了"要惩罚那些一眼就能看出是 AI 生成的套路（比如紫色渐变配白色卡片）"**。

**第二件，用 few-shot 示例 + 详细的评分分解校准评估器**——确保评估器的"品味"和开发者的"品味"对齐。

### 评估器不是看截图打分

它通过 Playwright **实际操作运行中的应用**——点击、导航、截图、仔细研究，**然后才输出评估**。每一轮反馈都回传给生成器，生成器要么继续优化当前方向，要么整体推翻重来。

**惊艳的例子**：让模型生成一个荷兰艺术博物馆的网站。前 9 轮迭代模型一直在做一个深色主题的落地页（视觉精致，但在预期范围内）。**到第 10 轮，模型突然推翻了整个方案**——把网站重新想象成一个"空间体验"：**用 CSS 透视渲染的 3D 房间、棋盘格地板、画作自由悬挂在墙上、通过门廊在画廊之间导航**。这种创造性的跳跃在单次生成中从来没见过。

**这就是对抗式反馈循环的力量**：不是让模型一次做到最好，而是让它在严格的外部反馈下**一轮一轮地逼近好的方向，甚至逼出你预料之外的方向**。

## 迭代合同：在高层规格和可测试实现之间搭桥

很精妙的机制——**在每个迭代周期开始之前，生成器和评估器先协商一份"合同"**。合同明确定义这个迭代周期"做到什么程度算完成"，包括具体要实现的功能和**可测试的验收标准**。

双方反复讨论直到达成一致，**然后生成器按合同来构建，评估器按合同来验收**。

为什么要这样做？因为规划器的产品规格说明是**故意写得高层的**——不会细到告诉你某个按钮应该怎么实现。**迭代合同就是在"高层规格"和"可测试实现"之间搭了一座桥**。

## 通信方式：通过文件而不是对话

也值得一提：**agent 之间不是通过对话沟通，而是通过文件**。一个 agent 写一个文件，另一个 agent 读这个文件然后回应。**简洁、可追溯，不会污染各自的上下文**。

## 评估器的真实严格度（举两个例子）

文章给了具体例子说明评估器到了什么级别：

- **关卡编辑器的矩形填充工具**：评估器发现它**只在拖拽起点和终点放置了瓷砖，而不是填满整个矩形区域**。**精确指出了是哪个函数没有在 mouseup 事件上被正确触发**。
- **FastAPI 的路由问题**：评估器发现 `reorder` 这个路径被定义在了带参数的路由后面，导致 FastAPI 把 "reorder" 这个字符串当成整数类型的 frame_id 去解析，**返回了 422 错误**。

**这种粒度的 QA 是真正在用的层面上找问题，而不是走过场**。

## 反事实：Anthropic 也很坦诚地说

让评估器达到这个水平**花了大量功夫**。**Claude 开箱即用的时候是一个很差的 QA agent**。他们的调优方法是**人工迭代循环**：

- 读评估器的日志
- 找到它的判断和你的判断不一致的地方
- 改 prompt
- 再跑、再看日志、再改

反复多轮之后，评估器才开始输出合理的判断。

**即便如此，评估器也不是万能的**——最终的应用仍然有一些小的布局问题、不够直觉的交互、以及深层功能里没被发现的 bug。**但和单 agent 那个"核心功能直接坏掉"的结果比，提升是巨大的**。

## 最高层认知：Harness 每个组件都编码了一个假设

整篇文章里**最有战略价值的一个认知**——

当 Anthropic 发布了更强的 Opus 4.6 模型之后，他们做了一件很重要的事：**回过头去重新审视整个 Harness，看哪些组件还有存在的必要**。

他们提出了一个原则：

> **Harness 中的每一个组件本质上都编码了一个假设——就是"模型自己做不好这件事"**。这些假设需要反复被压力测试，因为它们可能一开始就是错的，**也可能随着模型进步而过时**。

具体实践：

- 一开始想大刀阔斧地简化、直接砍掉一半组件——**失败了**（产出质量明显下降，搞不清是哪个组件的缺失导致的）
- 后来换了更稳妥的方法：**每次只移除一个组件，观察对最终结果的影响**

最终成功移除了**整个迭代周期结构**——因为 Opus 4.6 的持续工作能力大幅提升，生成器可以**连续编码两个多小时而不丧失连贯性**，不再需要把工作切成小块。

**评估器也从"逐周期评审"改成了"最后统一评审一次"**，但**评估器本身没有被移除**——因为在模型能力的边界上，评估器仍能抓到生成器遗漏的问题。**它的价值变成了动态的——任务越接近模型的能力极限，评估器的价值就越大**。

## 五个核心认知

从这篇文章里可以带走的最核心几个认知：

1. **永远不要让 agent 自己评价自己的工作**——生成和评估必须分离，然后花时间把评估器调严格
2. **主观质量是可以被打分的**——定义具体的评分维度，用 few-shot 校准，按短板加权，你就有了一个可以驱动迭代的反馈信号
3. **上游规划要克制**——只给方向不给路径，避免错误级联
4. **Harness 的每个组件都应该是可拆卸的**——模型在进步，今天需要的脚手架明天可能变成瓶颈。**保持模块化，定期审视，每次只动一个变量**
5. **Harness 的有趣组合空间不会随着模型进步而缩小，它只是在移动**——模型能力的边界往外推了，Harness 的设计空间也跟着往外推

一句话总结：**Agent 的性能天花板不取决于模型能做什么，而取决于你在模型周围搭建了什么样的系统**。**模型进步一个台阶，Harness 跟着进化一个台阶，两者叠加才是真正的前沿**。

## Open Questions

- **GAN 的对抗博弈隐含一个假设**：生成器和评估器是 "**对手**" 关系。但 Anthropic 的实践里两者还有 **协商**（迭代合同）—— 这两种关系怎么协调？纯对抗的 GAN 不需要协商。
- **评估器的调优过程是高度依赖人工的**——文章承认 "**人工迭代循环、反复多轮**"。这意味着**评估器质量和开发者品味强绑定**——同一个 harness 在不同团队手里效果差距会很大。这条路怎么规模化？
- **Context Reset 在 Opus 4.6 上不再需要**——这暗示 "Harness 是临时性脚手架"。**那哪些 Harness 组件是永久的？** 文章说"评估器是永久的"——但如果模型评估能力也强大到能客观自评呢？这种"无可拆卸的核心" 究竟存不存在，本节点没回答。
