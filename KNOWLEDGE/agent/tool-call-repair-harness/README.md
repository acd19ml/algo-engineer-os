# Tool Call Repair Harness

## 它要解决什么问题

当一个模型总是调用工具失败，第一反应往往是"这个模型工具调用能力不行"。但更精确的问法应该是：它是真的不会推理，还是没有记住你这个工具 schema 的格式？

如果是推理能力不够，Harness 很难补；如果只是格式边界不稳，Harness 可以用很小的确定性修复释放模型能力。材料里的案例是开源模型两天前连文件读取都跑不通，两天后通过不到 300 行工具输入修复层，在真实编码任务内部评测里多次逼近甚至超过前沿闭源模型。

## 工具调用失败经常是有限集合，不是随机噪声

弱模型或开源模型的工具调用错误看起来杂乱，实际常常集中在少数格式错误上。

常见四类是：可选字段不省略而传 `null` / `None` 类占位；数组被写成像数组的字符串；期望数组时传了裸字符串；期望数组时传了空对象占位。

这些错误不是模型不懂任务，而是模型输出分布和工具 schema 之间有小错位。前沿闭源模型因为见过大量工具格式，能靠统计经验猜对；开源模型没见过你的特定格式，错位被放大。

所以工具调用 Harness 的第一条判断是：先把错误分类。如果 90% 失败都落在可枚举格式错误里，就不应该把它归因成模型智能不足。

## 先校验再修复，而不是先预处理

最容易犯的工程错误是先跑预处理，再交给校验器。比如看到字符串长得像数组，就先把它解析成数组。

这个方案会误伤合法输入。写文件工具的内容字段如果本来就要写入 JSON 字符串，预处理器把它解析成真正数组，文件内容就被静默篡改。冒烟测试可能看不出来，但用户数据已经坏了。

正确顺序是反过来：先校验，再修复。

第一步，原始输入直接交给 schema 校验器。通过就直接执行，合法输入永远不被触碰。

第二步，只有校验失败时，读取校验器给出的 issue 列表。每个 issue 有字段路径、期望类型、实际类型。Schema 在这里不只是约束，也充当故障定位器。

第三步，只沿着出错字段尝试有限修复。比如解析字符串化数组、裸字符串包数组、删除无效可选占位。修复后再校验一次，通过才执行，不通过就返回可读重试提示。

这个结构的本质是：不要猜哪里可能坏了，让校验器指出哪里坏了。正常路径零开销，异常路径精确修复。

## 修复顺序和语义边界同样重要

修复不是随便堆规则。比如字符串化数组必须在裸字符串包装之前执行。否则一个内容为 `["a", "b"]` 的字符串会先被包装成单元素数组，而不是解析成两个元素，修复就变成破坏。

还有一类问题不是单字段形状错误，而是字段之间的关系不变量。读文件工具如果传了行数限制，就应该同时有偏移量；只传其中一个时，每个字段单看都合法，但组合语义不完整。

这种问题不能靠输入修复解决，因为没有哪个字段违反类型。更合适的做法是扩展语义：只传行数时默认 offset = 0，只传 offset 时默认 limit = 2000，并在返回结果里明确告诉模型系统做了什么选择。注意这不是报错，而是正常执行结果附带说明，模型下一轮可以自然修正参数。

原则是：能确定修复的修复，不能确定修复的扩展语义；无论哪种，都把 Harness 的选择透明地反馈给模型。

## 分布泄露：聊天习惯会污染工具边界

材料里有一个很典型的例子：模型在写文件路径时输出 Markdown 链接格式。聊天场景里，给文件名自动加链接是奖励过的好行为；工具调用场景里，这会变成非法路径。

这不是普通幻觉，而是后训练的聊天分布泄露到了工具调用边界。模型没有区分"这个输出是给人看的富文本"和"这个输出要传给文件系统"。

修复有两层。局部层可以用正则处理退化情况；结构层更重要：把通用 string 类型换成更语义化的 path string 类型，在 schema description 里明确这是文件系统路径，不要添加格式化。一次 schema 类型改动可以保护所有路径字段。

## Harness 的能力是画错误地图

这些修复高度模型相关。一个模型会泄露 Markdown 链接，另一个模型可能数组稳定但嵌套对象不稳。你很难写一套永远适配所有模型的通用修复。

但方法论是通用的：先校验再修复；按校验路径精确触发；把修复遥测按模型和工具维度记录；用遥测画出每个模型在哪些工具上容易错。

这和模型边界探测是同一件事。Harness Engineering 不是相信某个模型永远可靠，而是快速识别它的错误分布，再在边界上做小而确定的工程补偿。

## Open Questions

- 修复层的边界在哪里？如果 Harness 过度宽容，可能掩盖真实的模型理解错误，让系统表面可用但语义变差。
- 工具 schema 的 description 应该写到什么粒度？过短不足以约束模型，过长又会增加上下文噪声并降低缓存稳定性。
