# Structured Output for Agents

## 它要解决什么问题

面试常被问："Agent 开发中怎么让模型稳定输出结构化结果？"

如果只回答 "加个 JSON Schema"，面试官大概率会追问：

- "加了 schema 模型有时候还是乱输出怎么办？"
- "Agent 跑了 50 步，怎么保证每一步的输出都符合预期格式？"

要答透，得明白 **结构化输出不是单点问题，是一个分六层的系统问题**。

## 朴素直觉为什么不够：模型天生倾向自由文本

大模型本质上是一个 token 生成器，**天生就倾向于输出自由文本**。

你要求它返回 JSON，**大部分时候能照做**。但总有那么几次：

- 它会在前面加一句 "好的，这是结果："
- 多一个逗号
- 把数字输出成字符串

**对于单轮问答，重试一下就行**。但对于 agent 来说，**这是致命的**——因为每一步的输出都要喂给下一步，**任何一环格式出错，整个循环就崩了**。

要稳定输出结构化结果，**不是在 prompt 里加一句"请返回 JSON"** 这么简单——而是要从**解码层、验证层、工具层、架构层**逐层加固。

## 第 1 层：约束解码 —— 从语法层面杜绝格式错误

最硬核的一层。

**原理**：模型每生成一个 token 时，根据预定义的 JSON Schema **限制它能选择的 token 集合**。

例：JSON 对象的第一个字符必须是 `{`——那在这一步**只允许模型生成 `{` 这个 token，其他选项全部屏蔽掉**。

**不是靠 prompt 求模型返回合法 JSON，而是在解码阶段物理上让模型无法生成不符合格式的 token**。

主流模型 API 都提供这个能力：在工具定义中打开 `strict` 模式，能确保工具调用的参数**精确匹配 schema，消除类型不匹配和字段缺失的问题**。

### 反事实：约束解码的代价

**面试中如果只说"物理上杜绝格式错误"而不提 trade-off，在懂行的面试官面前反而会减分**。

代价：**当约束强迫模型远离其首选 token 时，如果模型排名靠前的 token 全部无效，它只能从低概率的替代 token 中选择**——最终输出虽然语法正确，**但语义质量可能下降**。

研究发现：

- **基础模型**通常从约束中受益（产生更精确和简洁的输出）
- **指令微调模型**的结果参差不齐——在分类任务上保持性能，**在生成任务上出现明显的退化**

换句话说：**约束解码保证了格式 100% 正确，但不保证内容正确**——它可能输出一个**结构完美但语义荒谬的 JSON**。

这就引出了下一层。

## 第 2 层：验证 + 重试循环 —— 覆盖语义层面

约束解码解决的是**语法问题**——括号配对、类型匹配、字段完整。但它解决不了**语义问题**——schema 要求 `rating` 字段取值 1-5，约束解码能保证它是整数，**但无法阻止模型输出 7**。

所以生产级 agent 系统中，**验证 + 重试是一个不可或缺的防御层**——**不是约束解码的替代品，而是与之互补的**。

重试机制**自动将错误上下文发回给大模型，让它有机会纠正问题**——构建自愈系统。

具体做法：

1. 用 Pydantic 或 Zod 定义带有业务规则的 schema（字段取值范围、字段间逻辑关系）
2. 模型输出后先过一遍验证器
3. 如果验证失败，**把错误信息连同原始输出一起反馈给模型**，让它重新生成

**这一层在面试中经常被忽略，但它恰恰是工程落地中最实用的一环**。

约束解码保语法、验证重试保语义，**两层配合才是完整方案**。

## 第 3 层：工具调用即结构化输出

如果模型不支持原生约束解码，**前沿产品普遍的做法是把结构化输出伪装成工具调用**。

**模型通过微调已经学会了怎么按 schema 生成函数调用参数——这个能力可以被泛化**：

1. 你定义一个虚拟工具
2. 把你期望的输出格式写成它的输入 schema
3. 强制模型调用这个工具

**就能稳定拿到结构化结果**。

Claude 的 Agent SDK 直接支持这种模式：**Agent 中间怎么折腾都行，调什么工具都行，但最终交付的结果必须符合你定义的 schema**。

OpenAI 等主流 API 提供商都支持这种模式——通过在工具定义中指定 `strict` 参数，确保模型输出**精确匹配 schema**。

好处：**几乎所有主流模型都支持工具调用，不需要你自己搭建约束解码引擎，开箱即用**。

## 第 4 层：用 Logit Masking 控制动作选择

Agent 在每一步都要从一组工具中选择下一个动作——**这本身就是一种结构化输出**。

问题是：**工具太多时模型容易选错**。

自然的想法：**动态增删工具**——当前用不到的就移除。

**但 Manus 发现不能这么做**——因为工具定义在上下文的前部，**任何变更都会导致后面所有内容的 KV Cache 失效**，代价非常大（详见 `KNOWLEDGE/agent/context-engineering/` 的第 5、6 层）。

Manus 的方案：**工具定义永远不变，但在解码时通过 mask logits 来屏蔽当前不需要的工具**。

而且工具命名经过精心设计：

- 所有浏览器相关的工具都以 `browser_` 开头
- 命令行工具都以 `shell_` 开头

**按前缀做 mask 就非常简单，KV Cache 完全安全**——但模型在每一步能选什么是被精确控制的。

## 第 5 层：多 Agent 之间的结构化通信

当一个任务被拆给多个 agent 执行时，**agent 之间传递的信息也必须是结构化的**。

Manus 的做法：**把子 agent 当工具来调用，planner 定义输出 schema，子 agent 通过约束解码填充这个 schema**——形成一种类似 MapReduce 的模式。

OpenCode 的压缩机制也遵循这个原则——当上下文过长需要压缩时，**它输出固定格式的摘要**：

- **G**：用户目标
- 重要用户指令和计划
- 对话中的重要发现
- 已完成、进行中、剩余工作的区分
- 列表

**确保每次上下文重启后信息完整且格式一致**。

核心原则：**多 agent 之间的信息传递不能依赖自然语言的模糊性，必须有明确的 schema 契约**——谁生产数据？数据长什么样？谁消费数据？每个环节都要定义清楚。

## 第 6 层：对抗长步骤中的"模式锁定"

这是**最容易被忽略的一层**。

Agent 执行几十步之后，**上下文里积累了大量类似的动作和观测**。模型会陷入一种 **"模式锁定"** 状态——**机械地重复之前的输出模式**。

Manus 在批量审阅简历的时候就遇到过这个问题：**Agent 仅仅因为上下文中看到了类似的操作，就不断重复相同的动作**。

Manus 提到的应对思路：**引入"变异"**——输出的 schema 不变，但**同一个 schema 下的序列化方式有微小的变化**，以打破模型的自回归惯性。

不过需要说明：**这个方案的具体实现细节在公开资料中披露较少**，目前**还不算被广泛验证的标准做法**。在实践中，更常见的做法是结合**上下文压缩**减少重复 pattern 的堆积，以及**事实的 context reset** 来缓解这个问题。

## 两个容易遗漏的细节

### 1. Schema 字段描述是写给模型看的指令

不是写给人看的文档。

**好的字段描述能显著提升输出质量**：

- ❌ "类型为整数"
- ✅ "评分，1 表示非常差，5 表示非常好。只根据内容质量评分，不考虑格式。"

后者**远比**前者有效。

### 2. 结构化输出和深度推理要分阶段

**先让模型自由思考，最终输出时再施加 schema 约束**。

Claude 已经做到了这一点：**思考过程完全不受约束，只有最终输出被结构化**——这样**既不损害推理质量，又能保证输出格式**。

## 反事实：苦涩的教训视角下的过度结构化警告

更高维度的思考：

Manus 创始人 Peak 警告过——**Agent 的控制机制（包括我们上面讲的所有结构化约束）可能会随着模型进步而反过来限制性能**。

这正是著名的 "**苦涩的教训**" 中所指出的挑战：

> 我们添加结构是为了**在当下**改善性能，但这种结构可能会**随着模型能力增长而成为瓶颈**。

他建议：**在不同能力等级的模型上运行 agent 的评估**。如果性能没有随着更强的模型而提升，**说明你的控制机制可能在拖累 agent**。

这意味着上面讲的六层**不是一劳永逸的**——你需要定期重新评估：**哪些约束在当前模型下仍然必要，哪些已经可以放松**。

面试中提到这个视角会显示出**对工程设计的更深层理解**。

## 答题框架

面试时这样组织答案：

1. **约束解码**是语法层面的根本保障——在 token 级别杜绝格式错误，**但要坦诚说明它有质量退化的代价**
2. **验证 + 重试循环**覆盖语义层面的正确性，和约束解码互补
3. **工具调用是结构化输出的天然载体**——开箱即用
4. **Agent 特有的挑战**：用 logit masking 控制动作空间、用结构化 schema 规范多 agent 通信
5. **三个加分点**：长步骤的模式锁定问题、推理和输出分阶段处理、bitter lesson 视角下对过度结构化的警惕

## Open Questions

- **约束解码的"质量退化"**——具体退化多少、什么场景下严重？文章说"指令微调模型上参差不齐"，但**没有给具体的退化量化数据**。这条研究线本节点没展开。
- **第 6 层"模式锁定"的应对方法**——Manus 的"引入变异"思路**没有完整公开实现**，本节点也只能列为思路。这是结构化输出领域的一个**真正开放的问题**。
- **Schema 字段描述提升质量** 这条经验**没有量化对比数据**——好描述 vs 差描述具体提升多少？这个 ablation 没看到严谨论证。
- **苦涩的教训** 视角下，工程师怎么知道某个约束已经"成了瓶颈"？需要可比较的 benchmark 设计——本节点提到了"在不同模型上跑评估"，但具体怎么搭这种对比 benchmark 没展开。
