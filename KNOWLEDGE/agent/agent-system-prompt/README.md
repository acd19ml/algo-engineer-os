# Agent System Prompt: 模块化 + 具体化

## 它要解决什么问题

每个 agent 开发者都要写 system prompt，**但大部分人写得很粗糙**——开头一段"你是一个什么什么助手"，加几条规则就完事。

Cloud Code 的 system prompt 有 **900 多行**。打开圆码你会发现，**这不是一段文本，而是一个由十几个模块化段落拼接而成的工程系统**——每个段落解决一个不同问题，**有些段落对所有用户共享、有些是每个绘画动态生成、中间还有一个明确的分界标记来控制缓存策略**。

这个节点拆五个最有启发性的设计——它们都指向同一个认知：**system prompt 不是一段介绍文字，是你的 agent 的操作系统**。

## 设计 1：静态段和动态段的分离（缓存友好）

Cloud Code 的 system prompt 被一个**边界标记**分成两半：

| 段 | 内容 | 特征 |
|---|---|---|
| **前半段（静态）** | 身份介绍 / 安全指令 / 代码风格规则 / 工具使用指南 / 行动风险评估框架 | 对所有用户、所有绘画**完全一样** |
| **后半段（动态）** | 当前绘画启用了哪些工具 / 记忆系统的索引 / 工作目录和 OS / 用户语言偏好 / 连接的插件说明 | 每个绘画**都不同** |

### 反事实：为什么这样分

大模型 API 有 **prompt cache** 机制——如果两次请求的 prompt 前缀一样，服务端可以复用之前的计算结果，**省钱省时间**。

- 如果你把动态内容和静态内容**混在一起** → 每个用户的 prompt 都不一样 → **缓存命中率很低**
- 把所有用户通用内容放前半段、绘画特有内容放后半段 → **前半段可以跨用户共享缓存**

你和另一个用户用的是同一个 Cloud Code，**你们的 system prompt 前半段完全一样**——服务端只需要计算一次。

### 可迁移原则

如果你的 agent 有多个用户：**把 system prompt 里所有用户通用的规则和当前用户特有的信息分开放**——通用规则放前面、特有信息放后面。**这一个简单的调整就能显著提高缓存命中率**。

这条直接对应 `KNOWLEDGE/agent/context-engineering/` 第 6 层 Cache-Friendly Architecture 的核心原则——**上下文结构要尽量是追加式的**。

## 设计 2：代码风格指令极其具体

大部分人在 system prompt 里写代码规则是这样：

> 请写高质量的代码。

或者：

> 遵循最佳实践。

**这种指令等于没写**——模型不知道你说的"高质量"具体是什么意思。

Cloud Code 的做法完全不同——**它的代码风格指令是一条一条非常具体的规则**：

- 不要给你**没改过**的代码加注释、文档字符串或类型标注。**只在逻辑不自明的地方加注释**
- **不要为假设的未来需求做设计**。不要给一次性操作创建工具函数。**三行相似的代码比一个过早的抽象好**
- 不要加你**没被要求加的功能**。一个 bug 修复不需要顺便清理周围的代码。一个简单功能不需要额外的可配置性
- 不要加错误处理、降级方案或校验逻辑来应对**不可能发生的场景**。信任内部代码和框架的保证，**只在系统边界做校验**

每一条都是**具体的、可执行的、有明确边界的**。模型看到这些规则，**知道该做什么、不该做什么**。

### 反事实：为什么具体规则比抽象原则更有效

模型不怕规则多，**怕规则模糊**。

- "请简洁" → 模型不知道具体意味着什么 → 输出可能仍然啰嗦
- "工具调用之间的文本不超过 25 个词，最终回复不超过 100 个词" → 模型有明确锚点 → **可以自我检查**

- "请写好代码" → 模型只能调用它对"好代码"的统计直觉 → **不同模型、不同上下文输出可能差很多**
- "不要给你没改的代码加注释" → 这条规则是 binary 的 → **稳定可执行**

> **你的 system prompt 里的规则越具体越好**。

## 设计 3：行动风险评估框架（教模型怎么判断）

Cloud Code 的 system prompt 里有一整段**专门讲什么操作需要确认、什么操作可以直接做**——但它**不是简单列一个"危险操作清单"**。

它给的是一个**判断框架**：**可逆性 × 影响范围**。

| 类型 | 判断 |
|---|---|
| **本地的、可逆的操作**（编辑文件、跑测试） | 可以自由执行 |
| **不可逆的、影响共享系统的操作**（强制推送代码、删除分支、发送消息、修改基础设施权限） | **必须先确认** |

然后列了一系列具体例子：

- 删除文件和分支
- 删除数据库表
- 强制推送
- 修改已发布的提交
- 发送消息到外部服务
- 上传内容到第三方工具

### 更有意思的：教模型怎么处理意外状态

不止"已知危险操作"——**它还教模型怎么处理"没见过的东西"**：

> 如果发现了**不认识**的文件、分支或配置，**先调查再行动**。不要直接删除或覆盖——**那可能是用户正在进行的工作**。如果遇到锁文件，**先查是什么进程持有的**，不要直接删掉。

**这个框架的价值在于：它不是死板的规则列表，而是教模型一种思维方式**。**模型遇到新的、没见过的操作时，可以用这个框架自己判断该不该确认**。

和 `KNOWLEDGE/agent/agent-permission-system/` 互补——那里讲的是**硬约束**（编码进 runtime 的权限检查），这里讲的是**软约束**（在 prompt 里教模型怎么思考）。**硬约束兜底，软约束补缝**。

## 设计 4：安全指令由专门团队维护

原码里有一段叫 **Cyber Risk Instruction** 的安全指令——它定义了模型在安全相关请求上的行为边界：

- **可以协助**：授权的安全测试、防御性安全、CTF 挑战、教育场景
- **必须拒绝**：破坏性技术、拒绝服务攻击、大规模目标攻击、供应链攻击、恶意目的的检测规避

这段指令本身不长，**但圆码注释非常引人注目**——

> 注释里写了这段指令的负责人名字，**明确要求任何修改必须经过安全团队审批和评估**——因为对这段文本的修改会对模型处理安全请求的方式产生重大影响。

### 这个设计的启发不在技术层，而在工程管理层

**System prompt 不是一个随便谁都能改的文本文件**——它是你的 agent 的行为规范。**某些段落的修改可能带来严重的安全后果**。

在团队协作中，**关键的提示词段落应该有明确的负责人和审批流程**——**就像关键代码需要 code review 一样**。

这条经常被忽略——很多团队把 prompt 当配置文件管理（人人可改），**但 prompt 实际上是和代码同等重要的"行为定义"**。

## 设计 5：防止模型说谎（也防过度谨慎）

这是 system prompt 里**最务实的一段**。它直接告诉模型：**如实报告结果**。

具体怎么写：

- 如果测试失败了，**就说失败了**，附上相关输出
- 如果你没有执行验证步骤，**就说你没执行**，**不要暗示它成功了**
- **永远不要**在输出显示失败时声称所有测试通过
- **永远不要**压缩或简化失败的检查来制造一个绿色的结果
- **永远不要**把未完成或有问题的工作说成已完成

### 不只是防说谎——也防过度谨慎

后面紧跟着一段反向约束：

- 同样地，**当检查确实通过了，或者任务确实完成了，就直接说**
- **不要**对已确认的结果加不必要的免责声明
- **不要**把已完成的工作降级成部分完成
- **不要**重新验证你已经检查过的东西

> **目标是准确的报告，不是防御性的报告**。

### 反事实：为什么需要两个方向都约束

**模型的默认倾向是讨好用户**——倾向于报告好消息。所以要约束它不能把失败说成成功。

**但反过来如果只防说谎不防过度谨慎**，模型会走向另一个极端——**给所有结果加免责声明**："测试看起来通过了，但我建议您再次验证..."、"任务可能完成了，但..."。**这种过度防御性的报告也会降低 agent 的可用性**。

### 一个惊人的数字

圆码注释说：

> 这段是针对特定模型版本**接近 30% 的虚假声明率**加的。

也就是没有这段提示词时，**模型大约每三次就有一次会在报告结果时说谎或夸大**。

### 可迁移原则

如果你的 agent 需要报告执行结果：

1. 在 system prompt 里**明确告诉它如实报告**
2. **从两个方向约束**——既不能把失败说成成功，也不能把成功说得像失败
3. 模型的默认倾向是讨好用户——你需要**用明确指令对抗这个倾向**

## 整体认知：system prompt 是 agent 的操作系统

900 多行、十几个模块，**每个段落解决一个具体问题**：

| 段落 | 解决什么 |
|---|---|
| 身份介绍 | 告诉模型它是谁 |
| 安全指令 | 画出不可逾越的底线 |
| 代码风格规则 | **用具体条目替代模糊的"写好代码"** |
| 行动风险框架 | **教模型怎么判断该不该确认** |
| 防说谎指令 | **对抗模型讨好用户的默认倾向** |

**这些段落不是一次写好的**——圆码注释里到处都是"针对某个模型版本加的"、"某个 PR 引入的"、"某次评估发现的问题"。

**System prompt 是一个持续演进的工程产物**——**每一条规则背后都有一个真实的生产问题**。

## 三条可迁移原则

如果你在给自己的 agent 写 system prompt：

1. **模块化** —— 不要写一大段文本。**把不同职责的内容拆成独立段落**（身份 / 规则 / 工具指南 / 环境信息）。可以独立修改每个段落，也可以根据场景动态组合
2. **具体化** —— 每一条规则都要**具体到模型能直接执行**。"请写好代码"不是规则，"不要给你没改的代码加注释"才是规则
3. **分离缓存** —— **把所有用户通用的内容放前面，绘画特有的内容放后面**——这一个调整就能让你的 API 成本降下来

## Open Questions

- **System prompt 写到 900 行——这本身有没有问题**？过长的 prompt 会不会**反向消耗注意力**，让模型"读了开头忘了结尾"？Cloud Code 是怎么验证它的 900 行确实都被模型遵守的？本节点没碰到。
- **"模块化段落 + 边界标记"**——但模块之间会不会冲突？比如"安全指令"说必须拒绝某类请求、"行动风险框架"说可以协助某类授权安全测试，模型遇到模糊场景时听谁的？冲突解决规则是显式写出来的还是靠模型自己权衡？
- **"30% 虚假声明率"**——这个测量是怎么做的？需要一套"诚实性 eval"。这种 eval 怎么搭？本节点没展开。
- **代码风格指令的"不要做不必要的错误处理"和软件工程里的"防御性编程"是不是直接对立**？两者各自适用什么场景？这条边界 Cloud Code 给的指令是 binary 的，但实操中可能有灰色地带。
