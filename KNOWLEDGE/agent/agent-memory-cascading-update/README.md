# Agent Memory 的级联更新问题：当一个事实变了，依赖它的事实怎么办

## 它要解决什么问题

你告诉 AI 助手 "我住在北京望京"。一个月后你说 "搬去上海了"。**大部分 AI 记忆系统能正确更新地址**——这一步几乎所有人都做得到。

但同一个 AI **同时还记得**：
- 你每天骑车 15 分钟到公司
- 你交的是北京社保
- 你的健身房在望京

**搬走之后**——这些事实哪些还成立？哪些需要被更新？哪些应该变成"不确定了"而不是"肯定是假的"？

MeMe 论文专门测了这个场景。结果是所有主流 agent 记忆系统在级联更新任务上准确率只有 **3%**；反事实推理（一个相关实体被移除后，依赖它的事实还算不算数）准确率只有 **1%**。

**不是某个系统差，是所有系统都差**。研究者试了更强的模型 / 更好的 prompt / 更深的检索 / 更少的噪音——全部失败。

唯一部分有效的配置是 **把所有记忆放进一个文件，让最强的模型一次性读完**——但代价是成本增加 **70 倍**。

这意味着：**当前所有 agent 记忆系统的底层架构有一个根本性缺陷**——它们是孤立的点存储，不是图结构。每个事实被独立记住，但系统不知道这个事实**依赖于谁、它的成立需要什么前提条件**。

## 三层更新难度

记忆系统需要处理的更新有三个层次：

| 层 | 类型 | 例子 | 当前系统能做到吗 |
|---|---|---|---|
| 1 | **直接替换** | "住在北京" → "住在上海" | ✅ 大部分系统都能。本质是检测到同一属性的新值，覆盖旧值 |
| 2 | **级联失效** | 搬上海后，"每天骑车 15 分钟到公司" 还成立吗？理论上也可能骑车 15 分钟，但**成立前提变了**，需要被重新确认 | ❌ 3% 准确率 |
| 3 | **反事实推理** | "我和张三是同事" → 张三离职了。这事实**现在是假的**？还是应该变成 "我和张三曾经是同事"？需要区分 *事实被否定* 和 *事实变成历史* | ❌ 1% 准确率 |

**第 2、3 层的共同难点**：系统需要理解事实之间的依赖关系。而当前的记忆系统——无论是向量数据库、文件存储、还是结构化数据库——**都不显式存储这种依赖**。

## 这个问题是 bi-temporal 不变量的"未完成续集"

[[memory-architecture-thesis]] 提了一个关键不变量：**UPDATE/DELETE 不应该覆盖旧记录，而是"追加一个更正事件"**，让 recall 在 query_time 下选择生效版本。这条不变量解决了 **"什么时刻什么是真的"** 的记账问题——对应本节点的**第 1 层（直接替换）**：

```
t1: "我住北京"  → ledger: location=北京, valid_from=t1
t2: "搬上海"    → ledger 追加: location=上海, valid_from=t2
query at t3 "我住哪？" → recall 按 query_time 选 valid_from=t2 的版本 → 上海 ✅
```

但**追加更正事件只发生在被显式 update 的 fact_key 上**。如果 t1 还存了 `commute=bike-15min` / `gym=望京邮乐` / `social-security=北京`——`location` 改了之后，**这些事实的 ledger 里不会自动追加任何更正事件**。recall 仍会按它们各自的 `valid_from=t1` 取出，当作"现在仍为真"。

也就是说：

| 治理层 | bi-temporal + 追加更正事件 | 本节点讨论的级联更新 |
|---|---|---|
| 闭合的问题 | **fact-level 时序**：某个 fact_key 的时间线一致性 | **cross-fact 依赖传播**：一个 fact 变了，依赖它的 fact 真值需要重算 |
| 解决了哪一层 | 第 1 层（直接替换） | 第 2、3 层（级联失效 + 反事实推理） |
| 当前状态 | 必要条件（无此不变量无法做时序治理） | **未解决**（MeMe 论文 3% / 1% 准确率） |

**bi-temporal 是必要条件，不是充分条件**。本节点四个候选方向都可以看作在试图补这个缺口：

- **方向 1（写入时触发分析）**——在 commit 时刻就把"哪些 fact 应该收到级联更正事件"算出来，但成本随记忆规模线性膨胀
- **方向 2（显式依赖图）**——把"`commute` 依赖 `location`" 这条边记进 ledger，让 update 沿边传播。**这是把 ledger 从 fact 序列升级成事实图**——但依赖关系本身需要模型推断，多跳依赖（社保 → 在北京工作 → 住在北京）更难自动发现
- **方向 3（置信度衰减）**——不试图传播，把不确定性 surfaced 给用户做主动确认
- **方向 4（全量上下文）**——放弃增量传播，每次查询都把全局重算一遍

**所以方向 2 不是"另一套方案"，是 bi-temporal 不变量的天然延伸**：从"事件序列"升级到"事件 + 依赖图"。它失败的不是设计动机，而是**依赖关系的可推断性**——模型识别多跳依赖的能力还不够。

## 为什么现有系统做不到：缺少 "外键 + 级联删除" 机制

以最常见的 agent 记忆架构为例：

- **写入**：把用户说的话提炼成一条条独立事实，存进向量库或文件
- **读取**：根据当前对话语义检索相关事实，注入上下文

**这个架构的问题在于：写入和读取之间没有维护环节**。事实存进去后就是静态的，直到被直接覆盖或被检索出来。**没有任何机制会在一个事实变化时主动去检查其他事实是否受影响**。

类比传统数据库：**没有外键约束**。你可以删除一个用户，但所有引用这个用户的订单不会自动失效。传统 DB 用外键 + 级联删除解决这个问题。但在自然语言记忆系统里，依赖关系**不是显式定义的，需要靠推理来发现**——这是难点的根源。

## 反事实推导：为什么各种朴素方案都不够

### 方向 1：写入时触发影响分析

每次写入新记忆时不只是追加，而是触发一轮检查：用模型判断这条新信息是否与已有记忆矛盾。如果矛盾，哪些已有记忆需要被标记为待确认？

**具体做法**：新记忆进来后检索所有语义相关的已有记忆，把它们和新记忆一起交给模型，让模型判断每条旧记忆的状态——"仍然有效 / 可能失效需要确认 / 明确矛盾已失效"。

**为什么不够**：
- **成本跟已有记忆数量成正比**。1000 条记忆，每次写入都要检索和判断一大批，延迟和成本都会很高
- **模型判断本身不一定准确**——MeMe 论文已证明，即使最强模型在级联推理上也只有部分准确

### 方向 2：显式依赖图

存记忆时不只存事实本身，还存它依赖的前提条件。比如存"每天骑车 15 分钟到公司"时，同时标注 *前提：住在望京附近*。当"住在望京"被更新时，系统自动把所有依赖它的事实标记为待确认。

**本质是把记忆从平面列表变成有向图**：节点是事实，边是依赖关系。

**为什么不够**：
- 依赖关系本身需要模型来推断，**推断不一定准确**。"骑车 15 分钟到公司 → 依赖住在望京" 这个依赖对人来说很明显，模型不一定能可靠识别
- **依赖可能是隐式的、多层的**。"交北京社保 → 依赖在北京工作 → 依赖住在北京"——这种多跳依赖更难自动发现

### 方向 3：置信度衰减 + 主动确认

不试图自动解决级联更新，而是承认系统做不到完美的依赖推理。**每条记忆带"最后确认时间"和"置信度"**。当一个上游事实被更新时，所有可能相关的下游事实的置信度自动降低。回答时如果涉及低置信度记忆，系统**主动追问确认而不是直接使用**。

例：搬家后用户问"我的健身房在哪儿？"——系统不是直接说"望京邮乐"，而是"你之前提到过望京邮乐，但你最近搬家了，这个信息可能已过时，要确认一下吗？"

**为什么不够**：
- **把问题推给了用户**。如果系统频繁追问确认，用户体验会很差
- **"可能相关"的范围很难界定**。搬家之后理论上你的所有记忆都可能受影响，不能每条都问

### 方向 4：全量上下文方案（MeMe 论文唯一部分有效的方案）

**把所有记忆放在一个文件里，每次查询时让模型读完整个文件再回答**。模型在一次推理中同时看到所有相关事实，**自然地发现依赖关系**并做出正确判断。

**为什么不够**：
- **成本增加 70 倍**
- **不 scale**：记忆量大了之后文件超出上下文窗口

但这条路径**证明了一件事**：**问题不在于模型推理能力不够，而在于当前的记忆架构没有给模型足够的信息来做推理**。

## 由此引出根本矛盾

四个方向其实指向同一矛盾——**级联更新需要"模型在正确时机看到所有相关事实并做一次推理"**。但 *正确时机* 和 *所有相关事实* 这两个条件很难同时满足：

| 方向 | 时机 | 相关事实 |
|---|---|---|
| 写入时触发分析 | ✅ 对 | ❌ 不确定哪些事实相关 |
| 显式依赖图 | ✅ 对 | ⚠️ 知道哪些相关，但依赖关系本身可能不准确 |
| 置信度衰减 | ⚠️ 部分对（衰减触发了） | ❌ 范围过宽 |
| 全量上下文 | ✅ 对 | ✅ 对 | ❌ 但成本爆炸不可持续 |

## 最务实的方向：写入时粗筛 + 读取时精确验证

**两阶段方案**：
1. **写入时粗筛**（成本低但可能漏网）：粗略标记可能受影响的记忆为"待验证"
2. **读取时精确验证**（成本高但只在需要时触发）：碰到待验证的记忆，触发一轮精确推理确认它是否仍然成立
3. **后台批量扫描清理**

粗筛的成本低，但精确验证的成本高——通过 "只在需要的时候才触发精确验证" 来摊销成本。

**但据所知，目前没有哪个开源项目或商业产品完整实现了这个流程**。这仍然是一个开放问题。

## 这条研究方向对 agent 工程的意义

MeMe 论文的价值**不在于它给出了解法**，而在于**它用数据证明了一件事**：

> 我们以为 AI 记忆系统已经够用了。但实际上在稍微复杂一点的场景下，它们的表现接近于 0。

**3% 和 1% 的准确率意味着这不是偶尔出错，而是几乎从来不对**。

对做 agent 产品的人：**你不能假设记忆系统会自动处理依赖关系**。如果你的产品场景涉及用户生活状态的变化（搬家 / 换工作 / 换城市 / 换伴侣），**必须在产品层面设计降级策略**——而不是指望底层记忆系统能自动搞定。

对用 AI 的人：当你发现 AI 在跨会话后记错了或给出过时建议，问题可能不是 "他记不住"，而是 **"他的记忆缺乏结构"**。他记住了每一条事实，但不知道这些事实之间的关系。

## 一句话总结

当前所有 agent 记忆系统的根本缺陷是：**它们存的是孤立的事实，不是事实之间的关系**。一个事实变了，依赖它的其他事实不会自动更新。这个问题目前没有完善解法，但它正在成为 agent **从"能用"到"可信赖"的关键瓶颈**。

## Open Questions

- **粗筛 + 精确验证的两阶段方案能 work 到什么准确率？** 没有公开数据。MeMe 测的都是单阶段方案，两阶段是否能显著超过 3%／1% 的基线是开放问题
- **依赖关系的"可学习性"**：在 `KNOWLEDGE/agent/memory-architecture-thesis/` 三类瓶颈框架下，级联更新失败可以归到 **policy 可学习性** 这一类——UPDATE/DELETE 决策需要看到 *依赖图*，而 policy 训练数据里几乎没有这种 multi-hop dependency 的反例。这是用 GRPO 训 memory tool 时一个尚未被讨论的盲区
- **"事实被否定" vs "事实变成历史" 的区分**——这本质是 `bi-temporal`（参见 memory-architecture-thesis 的 transaction_time ≠ valid_time）的应用场景。如果 ledger 显式记录 `valid_time` 区间，第 3 层反事实推理可能从 1% 提升到多少？没人测过
- **Hermes Skills 闭环的 patch 机制**（参见 `KNOWLEDGE/agent/agent-skills-closed-loop/`）是不是一种"显式依赖图"的雏形？Skill 之间的 `related_skills` 字段如果升级成 `depends_on`，能否解决 procedural memory 层的级联更新？declarative memory 层呢？
