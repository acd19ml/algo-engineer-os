# Agent Engineer Core Abilities

## 它要解决什么问题

每次 agent 新概念出来（LangChain、AutoGen、Plan-and-Execute、Multi-Agent、Context Engineering、Harness Engineering），评论区一定有人说："学这些半年就过时"。这话**有两个根据**：

- 框架迭代快——LangChain 0.1 → 0.3 接口全改、AutoGen 多 agent 写法换好几轮
- **模型能力进化在不断吞噬工程代码**——窗口从 4K → 128K，分块 RAG 流水线很多场景不需要了；模型原生支持 tool use 后，prompt 解析器代码可以删

这两个事实指向同一个焦虑：**我现在学的东西保质期多长**？

要回答这个问题，得分清两类工程：

- **补偿性工程**（patch-on-weakness）：用外部代码弥补模型能力不足。模型能力一升，对应代码就能删——保质期短
- **系统性工程**（system-around-intelligence）：解决任何智能体在真实环境中工作都必须面对的问题——不管模型多强都不会消失

Agent 工程师的核心能力**全在第二类**——四件事：上下文管理、控制流设计、错误恢复、反馈回路。

## 朴素直觉为什么不够：四件事的"似曾相识"是陷阱

四个词每个传统软件工程师都觉得熟：

| Agent 概念 | 传统对应 |
|---|---|
| 上下文管理 | 状态管理（Session / Redis / DB） |
| 控制流设计 | 流程编排（Service 层 if-else） |
| 错误恢复 | 异常处理（try-catch、重试、熔断） |
| 反馈回路 | 监控告警（QPS / 延迟 / 错误率） |

Agent 开发**没有发明新的工程学科**——它站在传统软件工程的地基上。

但**多了一层全新挑战**：你的核心引擎不再是亲手写的确定性逻辑，而是一个**概率性黑盒模型**。这一个差异把同样的四件事变成了完全不同的工程问题。

行业现在的共识：`Agent = Model + Harness`。模型负责智能 / 推理 / 决策；Harness 负责模型之外的一切（工具、知识、上下文管理、权限边界）。**模型是引擎，Harness 是方向盘和刹车**。我们造的不是发动机（那是 Anthropic / OpenAI / Google 的事），是底盘 / 方向盘 / 刹车 / 仪表盘 / 安全气囊。

## 第 1 件事：上下文管理 —— 不是数据，是程序本身

写后端时，状态管理操心的是数据存在哪、什么时候读写、一致性怎么保证。**但你从不担心一件事**：你存进去的数据不会影响业务逻辑正不正确。Service 层不会因为 Redis 里多了一条脏数据就胡言乱语。**逻辑是逻辑，数据是数据，泾渭分明**。

Agent 开发里，这个前提**不成立**。

Agent 没有你手写的 if-else，**它的全部逻辑都是模型在推理时基于上下文生成的**。你给它什么信息、什么顺序、什么措辞、信息量多少——直接决定它的推理质量和行为方向。

**上下文不是数据，上下文就是程序本身**。

这导致上下文管理至少要解决三层问题：

1. **上下文隔离**：把大任务拆成子任务时，每个子任务应该有**独立上下文**。子任务的独立上下文是一道**防火墙**，防止一个子任务的噪声污染另一个。这就是 Claude Code 的 sub-agent 模式、OpenAI Codex 的沙箱执行在做的事。
2. **按需注入**：不要把所有知识一股脑塞进 system prompt。模型需要什么、在它需要的时候再给。OpenAI 把 AGENTS.md 控制在 100 行、Anthropic 做 Skills 渐进式加载——同一个道理。
3. **上下文压缩**：Agent 持续运行，上下文不断膨胀，**模型推理能力会随上下文膨胀而退化**。数据库不会因为存太多而算错 SQL，但模型会。需要压缩策略，在保留关键信息的前提下定期为上下文瘦身。

行业从 2023 年 React 时代手写 prompt 模板，到 2025 年正式给这件事命名 "Context Engineering"——**底下的事情一直是同一件**：管理的不是数据正确性，是模型在每个决策瞬间能不能看到正确的、充分的、不冗余的信息。

## 第 2 件事：控制流设计 —— 你设计棋盘，模型在棋盘内决策

传统后端的流程编排：用户下单 → 校验库存 → 扣减库存 → 创建订单 → 发送通知。**每一步做什么、下一步走哪个分支，全由你的代码决定**。

Agent 开发里，**下一步做什么是模型决定的**。所有 agent 框架的核心都是同一个循环：调模型 → 模型说要调工具就调工具 → 把结果喂回去 → 继续循环 → 模型说停就停。**循环本身没有任何业务逻辑**。

但模型自己决定 ≠ 你什么都不用设计。**恰恰相反，你需要设计更精巧的控制结构**：

- **计划机制**：没有计划的 agent 会漫无目的飘移。给模型一个 "先列计划再执行" 的工具，引导它在行动前先想清楚步骤——实测加了计划机制的 agent 任务完成率几乎翻倍
- **任务依赖图**：不写死执行顺序，但提供结构让模型自己把大目标拆成有依赖关系的小任务，按依赖逐步推进，完成一个解锁下一个
- **自主认领机制**：多 agent 协作时不指派任务，设计一个看板让 agent 自己扫描、认领、执行

看到规律了吗？**传统控制流是你写死的 A→B→C，agent 控制流是你设计一个棋盘，模型在棋盘内自主决策**。你不控制每一步怎么走，你控制它能走的范围。

这需要从命令式转向 **声明式 + 约束式** 的设计思维。Vercel 移除 80% 的工具反而提升性能也是同一个道理——**约束解空间反而提高了产出**。

## 第 3 件事：错误恢复 —— 处理两层问题

传统异常处理（try-catch、回滚、重试、降级、熔断）**有一个隐含前提**：逻辑是对的，只是执行可能失败（网络超时、DB 挂、500）。**你的代码本身不会犯错**。

Agent 开发里，这个假设也不成立。**模型会误解意图、调错工具、传错参数，甚至产生幻觉，声称已经完成了实际没完成的操作**。这不是 bug、不是异常——**这是概率系统的本性，犯错是它工作方式的一部分**。

所以错误恢复要处理两层：

- **环境故障**（传统）：API 超时、文件不存在
- **智能故障**（全新）：模型推理本身出错

智能故障的核心原则：**在架构层面隔离错误，不让它级联传播**。子任务用独立上下文执行，失败了它的错误推理轨迹不污染主任务，主任务只拿最终结果决定下一步。**这和微服务架构的故障隔离原理完全一致**。多 agent 协作时每个 agent 在独立工作目录里——**不能假设 agent 每次都改对文件**，一个 agent 的错误不能波及其他 agent 的工作空间。

OpenAI Codex 团队的精辟经验：**当 agent 出错时，解决方案几乎从来不是让它再试一次，而是去思考"模型缺了什么能力或什么信息"，然后把那个东西补上**。

为什么？传统开发里 retry 是合理策略——错误来自环境波动，重试可能就好。但 **agent 的智能故障不是随机波动，是结构性缺陷**——模型缺了关键信息，让它试一百次，它一百次都会犯同样的错。**正确的做法是找到那个缺失的信息或能力，把它补进 Harness 里**。

## 第 4 件事：反馈回路 —— 给模型看的，不是给人看的

传统监控告警的指标（QPS / 延迟 / 错误率）**是给人类工程师看的**——你看到异常后人工介入处理。

Agent 开发里，**反馈不是给人看的，反馈是给模型看的**。

例子：

- **计划跟踪**：Agent 执行过程中，系统不断把"计划列表里还有哪些没做完"注入它的上下文，防止跑偏。这不是给人看的 dashboard，**是给模型看的实时仪表盘**。
- **后台任务通知**：耗时操作放后台异步执行，执行完毕后结果通过通知机制注入 agent 的上下文，agent 据此决定下一步。
- **自验证循环中间件**：在 agent 准备退出时拦截它，**强制执行一轮验证**。
- **推理三明治策略**：规划阶段用最高推理强度理解问题，执行阶段降低强度保证速度，验证阶段再拉回最高强度捕获错误。

**反馈回路的本质是让执行结果实时回流到模型的上下文中，驱动它自我评估和修正**——不等人来看，不等人来修，系统自己形成闭环。

## 反事实：模型变强会不会让这四件事消失

最深层的质疑：你讲的这些迟早会被模型本身内化掉。

**不会**。一个类比可以说清楚：

- CPU 性能在过去几十年提升了几百万倍，但**我们并没有因此不再需要操作系统**
- 数据库引擎越来越强大，但**我们并没有因此不再需要数据库设计**
- 编译器优化越来越智能，但**我们并没有因此不再需要软件架构**

为什么？**每一次底层能力提升都会淘汰一批补偿性工程实践，同时让系统性工程实践变得更重要**。底层能力越强，能用它构建的系统就越复杂，**越复杂的系统越需要精心的工程设计**。

具体到上下文管理：就算窗口变成无限大也仍然需要——**问题从来不是装不装得下，而是该不该装进去**。窗口从 4K 变成 128K，工程问题不是消失了，是从"怎么把信息塞进去" 变成 "怎么在海量可用信息中筛选出此刻真正相关的那一小部分"。

错误恢复也一样：**模型可以聪明到不写错代码，但它调用的第三方 API 照样会返回 500**。Agent 操作的是真实环境，真实环境的不确定性不会因为模型变强而消失。故障隔离、回滚、降级——不是因为模型笨才需要的，**是因为物理世界不可控才需要的**。

## "Agent 开发就是换皮 CRUD" 是不是？

是，也不是。

**日常操作确实像 CRUD**：注册工具、读取上下文、更新状态、压缩历史。

**但本质区别**：你的核心引擎从确定性逻辑变成了概率性黑盒。表面相似，底层逻辑完全不同。

**能不能看穿这一层，决定了你是 "agent 调包侠" 还是 "Harness 工程师"**。

## Open Questions

- 四件事在不同任务复杂度下的**优先级排序**没说清。简单 agent（如客服）可能 1+4 就够了；复杂 agent（如 deep research）四件事都重。但具体边界模糊。
- "补偿性工程 vs 系统性工程" 的二分**很清晰**，但实操中两者经常缠在一起——比如 RAG 流水线既补偿了"窗口小"的弱点（补偿性），又解决了"知识管理"的根本问题（系统性）。怎么在具体决策时判断"这个组件保质期多长"，本节点没给完整方法。
- **Harness 的 ROI 怎么量化**？文章说投入是复利的（一次加 Linter，所有 session 受益），但**反例**：太早过度工程化也是真实风险。这个权衡的实操标准没展开。
