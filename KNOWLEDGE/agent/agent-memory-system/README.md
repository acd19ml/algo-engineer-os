# Agent Memory System: 存什么 / 怎么取 / 谁来写

## 它要解决什么问题

记忆系统是所有人都想做、但大部分人做错的东西。

如果你在构建 agent，一定想过——**怎么让 agent 在多次对话之间保持连续性？怎么让他记住用户的偏好、项目的背景、之前犯过的错误**？

大部分人的做法是搞一个文件，把所有需要记住的东西往里塞：用户偏好、项目规范、历史决策、代码架构、文件路径，全部堆在一起。**文件越来越大，token 成本越来越高，而且大部分内容跟当前对话根本没关系**。

这个节点拆 Cloud Code 圆码里整个记忆系统的设计，三件大事：

1. **存什么 / 不存什么**——4 种该存、5 种不该存
2. **怎么取**——Active Recall 两阶段检索
3. **谁来写**——Extract Memories 后台自动提取

这三件事看似独立，**底层共享同一个设计哲学**：记忆是代码的补给。代码能回答的问题，不要让记忆来回答。

## 第一件大事：存什么（4 种 + 5 种不该存）

整个记忆系统里**工程量最大的部分不是怎么存，也不是怎么取，而是什么该存、什么不该存**。

### 4 种该存的

**1. 用户记忆**——用户是谁。角色 / 技术背景 / 工作习惯 / 知识水平。比如"这个用户是数据科学家、目前在做日志系统调研"，或"这个用户写了 10 年后端、但第一次碰前端"。

设计意图：**让 agent 调整沟通方式**。面对资深后端工程师可以直接用术语；面对初学者要更耐心地铺垫背景。

**重要约束**：不要记录对用户的负面评价，也不要记录与工作无关的个人信息。**记忆的目的是怎么更好地帮这个人，不是给这个人画像**。

**2. 反馈记忆**——用户纠正过 agent 的什么、肯定过 agent 的什么。

这是四种里**设计最精细的一种**。原码里有三个关键要求：

**要求 1：每条反馈必须包含三个部分**——**规则是什么、为什么有这个规则、什么时候该应用**。

举个例子。用户说"测试不要 mock 数据库——上个季度我们就是因为 mock 测试通过了、但生产环境迁移失败才出的事故"。

- 如果只记"不要 mock 数据库"——agent 在所有测试里都不敢 mock，**包括那些跟数据库迁移完全无关的单元测试**
- 如果记原因是"mock 和生产环境行为不一致导致迁移失败"——agent 就能判断：**集成测试不该 mock，但纯逻辑的单元测试 mock 是没问题的**

**记原因是为了让 agent 在新场景下做判断，而不是机械执行规则**。

**要求 2：不要只记纠正信号，也要记肯定信号**。

原码注释说得很直白：**如果你只记录"用户说不要这样做"的时刻，agent 会变得越来越保守**——他只知道什么是错的，不知道什么是对的。**时间长了，他会回避一切不确定的做法，变得畏手畏脚**。

但**肯定信号比纠正信号更难捕捉**。"不要这样做"很明显；"对，就是这样"或"默默接受了一个不寻常的方案"——这些信号很安静。Agent 需要**主动注意这些肯定信号**，比如："对，这次用一个大 PR 是对的，拆开反而是无意义的工作量"。

**3. 项目记忆**——当前项目正在发生什么、谁在做什么、为什么做、截止日期是什么。比如"本周四之后冻结所有非关键合并"，或"正在重写认证中间件，原因是法务团队指出旧的 token 存储不符合合规要求，所以做决策时要优先考虑合规性而不是技术优雅"。

**关键处理规则：所有相对日期必须转换成绝对日期**。

用户说"周四冻结"，记忆里存的是**具体年月日**。因为记忆是跨绘画的——**下周再看这条记忆，就不知道是哪个周四了**。

项目记忆**衰减得很快**——一个月前的项目状态大概率已经过时。所以原码要求项目记忆**必须记录"为什么"**——**即使事实过时了，背后的动机仍然有参考价值**。

**4. 引用记忆**——外部资源在哪里。bug 在哪个系统追踪、监控面板的地址是什么、设计文档在哪个平台。

四种里最简单但**最实用**——本质是一个"**去哪里找信息**"的索引。

### 5 种绝对不该存的

**1. 代码模式 / 架构 / 文件路径 / 项目结构**——**这是最反直觉的**。

很多人做记忆系统的第一件事就是让 agent 记住"项目用了什么框架、目录怎么组织、哪个文件负责什么"。Cloud Code 说**不要存这些**——因为这些信息可以直接从代码里读出来，agent 随时可以通过读代码和搜索获取**当前**的项目结构。

把这些存进记忆有两个问题：

- **浪费空间**——每次对话都要加载一堆本来可以实时查的信息
- **过时风险**——代码改了但记忆没更新，agent 会基于**过时信息**做决策，**而且你很难发现**

**核心原则**：如果一个信息可以从当前项目状态推导出来，**就不要存进记忆**。

**2. 版本管理历史**——谁改了什么、最近的提交记录。用 git 查就行，是**实时的、权威的**，不需要记忆来存一份可能过时的副本。

**3. 调试方案 / 修复方法**——修复已经在代码里了，提交信息里有上下文。**存"怎么修的"没有意义，因为代码本身就是最好的参考**。

**4. 项目说明文件里已经写过的东西**——配置文件里已经定义了编码规范，记忆系统不需要再存一份。**重复存储不仅浪费空间，还会在两份内容不一致时制造混乱**。

**5. 临时性的任务细节**——当前正在做什么、对话里的中间状态。这些是**短期的**，属于当前绘画的上下文，**不该进入长期记忆**。

### 一条特别值得注意的规则

**即使用户明确要求"你记住某些东西"，如果它属于上面五类，也不该记**。

如果用户说"记住这周的 PR 列表"，agent 应该反问："**这些 PR 里有什么让你意外的或不明显的？那个部分才值得记**。"

> **活动日志不是记忆，从活动中提炼出的"洞察"才是**。

### 整套体系的核心哲学

**记忆是代码的补给**——代码能回答的问题，不要让记忆来回答。**记忆只负责代码回答不了的那部分**。

这是为什么这套分类框架可以直接拿来用：先问自己"**这条信息能不能从当前代码或工具里实时获取？**"——能就不存，不能再看属于哪种类型，按对应格式存。

**这样做的好处是记忆文件会非常精简，每一条都是"代码里找不到的高价值信息"**。

## 第二件大事：怎么取（Active Recall 两阶段检索）

**光存还不够**——存了 200 条记忆，用户问了一个问题，你是把 200 条全塞进上下文，还是只挑相关的几条？

- 全塞进去 → token 成本爆炸，大部分内容跟当前对话无关，**反而干扰模型推理**
- 不塞进去 → 记忆等于白存

**Cloud Code 的解法**：一个叫 Active Recall 的多阶段筛选流水线——**不是简单的向量搜索、也不是关键词匹配**，而是每一步都在控制成本和精度的平衡。

每轮对话开始时做四件事：扫描 → 摘要 → 选择 → 注入。

### Step 1：扫描（只读 30 行，无模型调用）

系统递归扫描记忆目录，找所有记忆文件，**上限 200 个**。超过的部分**直接忽略**。

**但它不读文件全文，只读每个文件的前 30 行**。

为什么 30 行？因为 Cloud Code 的记忆文件有一个固定格式——开头是一段叫 **Front Matter** 的元数据：记忆的名字、一句话描述、类型。30 行足够覆盖这段元数据。

成本极低——200 个文件 × 每个 30 行 ≈ 几千行文本，**没有任何模型调用**。

### Step 2：摘要（拼成目录）

扫描完之后，把所有记忆的元数据拼成一个清单——每条记忆一行：`类型标签 + 文件名 + 最后修改时间 + 一句话描述`。

比如：`反馈类型 | 测试规范 | 3 天前 | 集成测试必须用真实数据库不要 mock`。

**这个清单就是模型要看的"目录"**。200 条记忆的目录大概几千个 token，**比起把 200 个文件的全文都塞进去，成本降了几十倍**。

### Step 3：选择（用便宜模型 + 后效应兜底）

把用户当前输入和记忆目录一起发给模型，让它从中选出**最多 5 条**最相关的。

**关键细节 1：用便宜模型**。不是主模型，是 **Sonnet**（更便宜更快）。**选择记忆不需要最强的推理能力**，只需要理解"这条记忆跟当前问题有没有关系"——Sonnet 完全够用。

**关键细节 2：宁缺勿滥**。选择模型的 system prompt 明确要求：

> 只选你**确定有用**的，不确定就不选，**宁可少选，不要多选**。如果没有任何记忆相关，返回空列表也完全可以。

为什么这么保守？**每一条被选中的记忆都会占用主模型的上下文空间——选错一条就浪费了一个宝贵位置，而且可能误导主模型推理**。

**关键细节 3：上下文敏感**。如果用户正在使用某些工具且用得很顺利，**系统会告诉选择模型不要选这些工具的使用文档**——因为模型已经在正常使用了，再塞一份说明就是噪声。**但如果记忆里包含这些工具的"已知问题"或"注意事项"，那还是要选**——因为正在使用的时候恰恰是最需要知道这些坑的时候。

**关键细节 4：后效应兜底防幻觉**。模型返回的文件名，**系统逐一检查是否真的存在于候选集中**。因为模型可能会幻觉出一个不存在的文件名——**这一步是确定性代码兜底，防止幻觉污染检索结果**。

### Step 4：注入（截断有边界，截断给指针）

选中的记忆文件，系统读取完整内容，作为系统提醒注入到当前对话。

**但注入也不是无脑塞**——每个文件大小上限约 **4000 字节**。超过的部分会被截断，但**不是直接丢掉，而是附上一条提示**：

> "这个记忆文件被截断了。如果需要完整内容，可以用 `read` 工具去读。"

**这和 `KNOWLEDGE/agent/agent-tool-design/` 的"工具输出预算"是同一思路**——不截断信息，而是给摘要 + 指针。

**每轮最多注入 5 条 × 每条最多 4000 字节 = 单轮上限约 2 万字节**。**整个绘画的累计上限 6 万字节**——超过这个上限 Active Recall 停止工作，不再注入新记忆。

为什么要绘画级累计上限？因为长绘画里如果每轮都注入 5 条，**几十轮下来光记忆就占了几十万字节的上下文**。

**当绘画触发上下文压缩时**（见 `KNOWLEDGE/agent/agent-context-compaction/`），之前注入的记忆会随着旧消息一起被压缩掉，**累计计数器也会重置——Active Recall 又可以重新开始工作了**。

**去重机制**：同一个绘画里已经注入过的记忆文件不会被重复注入——选择模型在选择之前就已经把已注入文件从候选集里过滤掉了。

### 贯穿整个流程的关键设计：过期警告

每条被注入的记忆都会附带一个时间标签：

- 今天或昨天写的 → **不加任何警告**
- 超过两天的 → **自动附加一段提醒**

提醒的大意：

> 这条记忆是 N 天前的。**记忆是时间点快照，不是实时状态**。里面关于代码行为或文件位置的描述可能已经过时。**在基于这条记忆做决策之前，请先验证当前代码的实际状态**。

为什么要这样做？

原码注释里解释了——**真实事故**：用户反馈说旧记忆里提到了"某个函数在某个文件的某一行"，模型直接引用了这个信息。**但实际上那个函数早就被重构到别的地方了**。而且因为记忆里给出了具体的文件和行号，**模型的回答听起来特别权威，用户一开始根本没怀疑**。

**过期警告就是为了打破这种虚假的权威感**——告诉模型"这只是过去某个时间点的观察，不是当前事实"。

### 这套架构的整体经济账

把检索成本从"所有记忆的全文大小"降到"所有记忆的摘要大小 + 5 条记忆的全文大小"：

- 200 条记忆 × 平均几千字节 = **全文加载要几十万字节**
- Active Recall 方式 = **摘要几千字节 + 5 条全文 2 万字节 = 总共不到 3 万字节**

**成本差了一个数量级**——而且不需要向量数据库、不需要 embedding、不需要复杂的相似度计算。**一个结构化摘要 + 一次便宜模型调用就够了**。

## 第三件大事：谁来写（Extract Memories 后台 agent）

前两件解决了"存什么"和"怎么取"，**但记忆文件是怎么产生的**？

你可能以为是用户手动说"记住这个"——确实有这种方式，**但它不是主要的**。

**如果记忆全靠用户手动触发，大部分值得记住的东西都会被漏掉**——用户在对话时注意力在当前任务上，不会时刻想着"这条信息值得长期保存"。

### 设计：后台自动提取

Cloud Code 用一个叫 **Extract Memories** 的系统——**每次对话结束（主模型给出最终回答、不再调用工具）时，后台悄悄启动一个独立子 agent**。

这个子 agent 的唯一任务：**回顾刚才的对话，判断有没有值得长期保存的信息**。如果有就写入记忆文件。

**整个过程用户完全无感**——不弹框、不确认、不打断。你继续用 agent，后台默默完成。

设计听起来简单，**但有四个非常讲究的细节**。

### 细节 1：权限锁死（最小权限原则）

这个后台 agent **不是一个拥有全部能力的 agent**——权限被严格限制：

**能做的**：

- 读文件、搜索文件
- 只读 bash 命令（查看目录、文件内容、文件状态）
- 在记忆目录里写入和编辑文件

**不能做的**：

- 任何会产生副作用的命令
- 修改代码
- 启动子 agent
- 调用任何外部服务

bash 工具的限制**特别细**——不是简单"禁止使用 bash"，**而是只允许只读命令**。任何涉及写入、重定向、修改状态的命令都被拒绝。

**反事实：为什么要锁这么死**？

因为记忆提取是**高价值但高风险**的操作——**它在后台运行，用户看不到它在做什么**。如果它有权限修改代码或执行命令，**一个错误判断就可能在用户不知情的情况下改坏项目**。

把权限锁死在"只读 + 记忆目录写入"——**即使子 agent 犯错，最坏的结果也只是写了一条不准确的记忆**，不会影响你的代码。

这条原则对任何"后台自动化 agent 任务"都适用：**给他最小权限，不是他大概率不会做坏事，而是即使他想做坏事也做不了**。

### 细节 2：两轮对话策略（缓存命中率最大化）

子 agent 被限制在最多 5 轮对话内完成任务，**但设计意图是 2 轮就搞定**：

- **第 1 轮**：**并行读取**所有需要的文件——现有记忆列表（避免写重复）+ 最近对话内容
- **第 2 轮**：**并行写入**所有记忆文件——把提炼出来的记忆一次性写完

**为什么强调并行**？因为这个子 agent **共享主模型的 prompt cache**——它和主模型看到的 system prompt、工具定义完全一样。**对话轮次越少，缓存命中率越高，成本越低**。

原码注释记录了实际数据——**一次典型的记忆提取，缓存命中率可以达到很高的水平**。意味着大部分输入 token 是从缓存读取，几乎不产生额外计算成本。

**5 轮是安全阀**——正常 2 轮就够，但如果子 agent 因为某种原因需要多读几个文件或多改几条记忆还有余量。**超过 5 轮直接中止，防止无意义循环**。

### 细节 3：互斥机制（避免主模型 + 子 agent 重复劳动）

**Cloud Code 的主模型本身也有写记忆的能力**——如果用户说"记住我偏好用 Tabs"，主模型会直接写一条记忆，不需要等后台子 agent。

**这时如果后台子 agent 再跑一遍**，他会看到同样的对话内容，**可能写出一条措辞不同但内容相同的重复记忆**。

所以原码做了互斥检查——**检查最近对话里有没有主模型写入记忆目录的操作**。如果有，**说明主模型已经处理了记忆写入，后台子 agent 就不需要再跑**。直接把游标推进到最新位置，下次只看新对话内容。

### 细节 4：游标 + 合并 + 节流

**游标**：子 agent 不每次都回顾整段对话历史，而是**维护一个游标**记录"上次处理到哪条消息"。每次运行只看游标之后的新消息。**对话 20 轮，前 10 轮已被提取过，下次提取只看后 10 轮**——大幅降低输入量。

**失败保护**：如果提取失败，**游标不会推进**。下次提取重新处理这些消息——**确保不会因为一次失败就永久丢失值得保存的信息**。只有成功完成提取，游标才推进。

**合并**：如果上次提取还在跑、新一轮对话又结束了，**系统不会启动第二个并行提取**——而是把新上下文暂存起来，等当前提取完成后用最新上下文跑一次。**避免多个提取子 agent 同时运行互相冲突**。

**节流**：不是每轮对话结束都触发提取，**可以配置间隔**（比如每 2 轮或每 3 轮触发一次）。高频对话场景显著降低后台计算开销。

### 这套设计的整体哲学

**让记忆生产自动化，但不失控**：

- **自动化**：用户不需要手动告诉 agent 什么该记。**后台在每轮对话结束后自动分析和提取**
- **不失控**：**权限要锁死 / 轮次要封顶 / 重复要互斥 / 进度要有游标**

**自动化 ≠ 不受控**。这条对任何后台 agent 任务都适用。

## 三件大事共享的设计哲学

回过头看整个记忆系统：

| 大事 | 在解决什么 |
|---|---|
| 存什么（4 存 5 不存） | **管记忆质量**——只存代码回答不了的高价值信息 |
| 怎么取（Active Recall） | **管检索成本**——两阶段筛选，把成本从"全文加载"降到"摘要 + 5 条全文" |
| 谁来写（Extract Memories） | **管生产可控性**——后台自动 + 最小权限 + 游标保护 |

三件事共享同一个原则——**记忆是代码的补给**。你的整个记忆系统要为"代码不能回答的那部分"服务，**不要试图复制代码已经能提供的东西**。

## Open Questions

- **"反馈记忆要包含规则 + 原因 + 适用场景"——但这要求提取时模型能从对话里推断出这三层**。如果用户只说"不要 mock 数据库"没解释原因，提取出来的反馈记忆就缺一层。Extract Memories 子 agent 应该问回去？还是只写不完整的记忆？这条原码没说清。
- **Active Recall 的"宁缺勿滥"策略**——但如果选择模型每次都返回空列表，记忆就永远不被使用。怎么避免"选择模型变得过度保守"？是否需要定期校准？本节点没碰到。
- **过期警告的"两天"阈值是怎么定的**？项目记忆衰减快、用户记忆衰减慢——是否应该按记忆类型设不同阈值？还是说"两天"已经是覆盖大多数场景的合理默认？
- **"5 条上限 + 4000 字节 + 6 万累计"这一组数字是怎么定的**？是从生产数据观测出来的、还是 token 经济学算出来的、还是早期就定下来没改？这套数字在小型 agent 应用上（比如 1 万记忆条目以下）是不是不必要地保守？
