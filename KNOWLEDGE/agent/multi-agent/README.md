# Multi-Agent: When to Use, When Not to

## 它要解决什么问题

面试常被问 "multi-agent 架构什么时候该用，什么时候不该用？"

如果你回答 "任务复杂就用多 agent"，面试官会追问：

- 怎么定义复杂？
- 代价是什么？
- 有没有不该用的情况？

要回答透彻，得**先看清 multi-agent 不是免费的**——然后才能讲清三个该用和三个不该用。

## 朴素直觉为什么不够：multi-agent 的代价是巨大的

行业前沿（Anthropic、Claude Code、Manus、OpenCode、OpenClaw）的实战经验里，**Anthropic 的立场非常明确**：

> **大多数团队不需要 multi-agent 系统**。

Anthropic 发现：在单个 agent 上反复改进 prompt，效果可以匹配那些花了几个月搭建的复杂多 agent 架构——**而且代价很大**。

具体数字（Anthropic 的数据）：

- 普通 agent 消耗大约 **4 倍**于聊天的 token
- Multi-agent 系统消耗大约 **15 倍**

这些额外开销来自：跨 agent 的上下文复制、协调消息、结果汇总。

所以判断该不该用 multi-agent 的根本标准是 **"任务价值是否高到能为这些开销买单"**——而不是"任务感觉很复杂"。

## 三种该用的场景

### 1. 上下文污染

不同子任务积累的信息互相干扰，导致推理质量下降。

**用独立的子 agent 做上下文隔离**。

Manus 的核心原则：

> **通过通信来共享上下文，而不是通过共享上下文来通信**。

实操：

- 对于有明确输入输出的独立任务——**启动全新的子 agent，只传入具体指令**
- 只有当子 agent 必须理解完整轨迹时（比如需要看到之前所有错误尝试的调试 agent）才共享完整上下文

### 2. 并行探索

任务可以拆成多个独立方向同时推进时，**Multi-agent 的效果是碾压性的**。

Anthropic 的研究系统数据：**用多 agent 架构比单 agent 高出 90%**。

为什么？因为多 agent **帮助投入了足够多的 token 来解决问题**——在他们的分析中，**token 使用量本身就解释了 80% 的性能方差**。

Claude 的 Research 功能就是这么做的：**主 agent 同时生成 3 到 5 个子 agent，每个子 agent 又并行执行多个工具调用**。对于复杂查询可以**减少多达 90% 的研究时间**。

### 3. 工具过多

当 agent 需要管理 20 个以上的工具时，**选择准确率会明显下降**。

Manus 的发现：**给模型 100 个以上的工具会导致幻觉参数或调用错误**。

解决方案：**按领域拆分给拥有聚焦工具集的专业化 agent**。

不过 Claude Code 提供了一个**替代思路**：用一个工具去搜索工具，让模型按需动态发现工具，**而不是一次性加载所有定义**——这就是当前比较火的 **Skills 渐进式加载** 的思路，**可以减少多达 85% 的 token 使用量**。

## 三种不该用的场景

### 1. 大多数编码任务

**编码的子任务之间耦合度高**——真正可并行化的部分比研究类任务少得多。

涉及 1 到 3 个文件的修改：**单 agent 更快也更便宜**。

### 2. 简单查询和低价值任务

Anthropic 在研究系统的 prompt 里直接写了 **缩放规则**：

- 简单事实核查：**只用一个 agent，3 到 10 次工具调用**
- 直接对比：用 2 到 4 个子 agent
- 只有复杂研究问题：才用 10 个以上子 agent

**早期版本甚至出现过对简单查询生成 50 个子 agent 的失控情况**——这种情况下 multi-agent 不仅没价值，还烧钱。

### 3. 拟人化的角色分工

不要按组织架构图给 agent 排角色——什么"经理 / 设计师 / 程序员互相聊天"。

**Manus 团队说得很直接**：

> **把子 agent 当作工具来调用，不要当作同事来对话**。

主 agent 调用一个函数，底层启动临时子 agent 循环并返回**结构化结果**。

Anthropic 也指出：**按角色分工会产生持续的协调开销**。

## 反事实：两个关键的实战经验

### 子 agent 的任务指令必须极其具体

要包含：**目标、输出格式、工具范围、任务边界**。

Anthropic 一开始允许主 agent 给出简短指令——比如"研究半导体短缺"。**结果发现多个子 agent 在做完全相同的搜索，完全没有有效分工**。

### 并行不是万能要

OpenCode 在同一会话内**一次只处理一条消息，按会话序列化执行**。原因很明确：**当多个 agent 共享状态时，并发会导致工具冲突和状态损坏**。

**这是刻意的设计选择，不是限制**。

## 答题框架

如果面试中被问到这个问题，组织答案：

**先说原则**：从最简单方案开始，**只在有明确证据时才引入多 agent**。

**然后说三个该用的场景**：

- 上下文污染 → 用隔离
- 并行探索 → 用分发
- 工具过多 → 用专业化

**再说三个不该用的情况**：

- 编码任务耦合度高
- 低价值查询不划算
- 拟人化角色分工是反模式

**最后补两个加分点**：

- 任务委派指令要极其具体
- 共享状态时序列化比并行更安全

## Open Questions

- **"通过通信共享上下文"** 这条原则在实操中是模糊的——哪些信息属于"上下文"应该独立，哪些属于"通信内容"应该共享？比如调试 agent 需要看历史错误尝试——这是上下文还是通信？文章说"调试 agent 应该共享完整上下文"是个例外，但**例外的判定标准没说清**。
- **"15 倍 token" 的具体来源**没拆开。其中多少来自上下文复制、多少来自协调消息、多少来自结果汇总？不同来源的优化空间不一样。
- **任务委派指令"必须极其具体"** 和 **"上游规划要克制"**（详见 `KNOWLEDGE/agent/harness-practice/`）看起来矛盾——一个说要具体、一个说要克制。**两者的边界在哪**？
- "用一个工具去搜索工具" 听起来很递归——**但 metadata 的规模化管理本身可能就是个问题**。Skills 渐进式加载的实操细节本节点没碰到。
