# Agent Evaluation Harness

## 它要解决什么问题

Agent 能跑起来之后，真正难的问题不是"它这次答案对不对"，而是"它为什么对、为什么错、下一轮该优化哪里"。简单问答系统可以只看最终输出；但一个带五六个工具、跑十几步的 agent，最终答案只是轨迹的末端。中间可能调错工具、绕远路、靠偶然恢复，也可能最终错了但暴露的是某个可修复边界。

如果评估只看最终输出，就像只看程序返回码，不看日志、调用链和异常恢复。你知道成败，却不知道该修哪里。

## 为什么直接让编码助手写评估代码会失败

最朴素的做法是对编码助手说："帮我评估这个 agent。" 这通常会得到一堆看起来专业、实际没用的代码。

第一类失败是指标泛滥。没有约束时，编码助手会生成延迟百分位、token 消耗、工具调用次数、响应长度等十几个操作性指标。这些指标能描述系统运行成本，却不回答任务有没有完成、路径是否合理、错误恢复是否有效。

第二类失败是过度工程化。本来 100 行能完成的评估，被写成 5-7 个文件的小框架。代码变多不代表评估变强，反而让后续维护和对比更困难。

第三类更隐蔽：计划和实现漂移。规划阶段说要做语义相似度、工具调用序列合理性、轨迹质量评估；实现时却退化成关键词匹配，用几十个硬编码动作词去判断旅行规划 agent。表面上实现了评估，实际上把语义问题压扁成正则。

这些失败的根源不是模型不会写代码，而是它缺少"agent 评估应该长什么样"的领域知识。缺这个知识时，它会用最擅长的方式填空：加指标、加抽象、写更多代码。

## 评估也需要 Harness：过程指令、模板、实时文档

一个更稳的做法，是把"写评估代码"本身也当作 agent 任务来加 Harness。

第一层是过程性指令。规划阶段不能自由发挥，而要先分析 agent 架构和执行轨迹，再为每个要评估的行为方面定义指标。指标数量限制在 3-5 个，每个指标必须有评分标准，不加纯操作性度量。代码生成阶段也要有限制：先写最小可工作版本，使用任何库 API 前先验证它存在，严格遵循评估计划，最终只产出指标实现文件和编排器文件。

反事实地看，如果不给这些限制，"先规划再写代码"不一定更好。无约束规划会放大模型的发散倾向，计划写得很美，实现却漂移。规划本身不是价值，有约束的规划才是价值。

第二层是可复用模板。评估计划模板规定必须包含 agent 规格、指标定义、评分标准、测试场景；报告模板规定执行摘要、主指标分析、故障根因、优先建议。模板的作用是降低方差：同一个 agent 跑两次评估，产物结构应该可比较，而不是每次长成不同形状。

第三层是实时 API 文档检索。评估框架 API 变化快，模型训练数据里的调用方式可能已经过期。材料里提到的实验里，没有实时文档检索时，评估代码第一次跑通率只有约 20%；加上实时文档检索后到 65%。这说明很多失败不是评估逻辑错，而是 API 用法过时。

## 执行轨迹决定评估什么，但实现要保持简单

Agent 评估最好基于执行轨迹，而不是只看源码。源码告诉你 agent 可以做什么，轨迹告诉你它实际做了什么。基于轨迹设计指标，相关性会明显更高，因为你能看到工具调用、重试、分支、失败恢复这些最终答案看不到的信号。

但轨迹也会增加复杂度。最务实的做法是：用轨迹决定该评估什么，用简单代码实现评估。不要因为轨迹复杂，就把评估器也写成复杂框架。

如果评估代码第一次跑不通，优先怀疑 API 调用方式而不是评估思路。材料里最大单类错误是异步模式配置错误，通常改一个参数或加几行包装就能修。更好的工程做法是让生成器先试跑，失败后把错误反馈给模型修一轮，大部分问题可以在一轮内解决。

## 这件事和 Harness Engineering 是同一个原则

Agent 评估的可靠性瓶颈，不在模型能不能写代码，而在你有没有给它足够的评估结构。过程指令告诉它每阶段做什么，模板告诉它产物长什么样，实时文档告诉它当前 API 怎么用。

这和 Harness Engineering 的主张一致：不要让 agent 自由发挥，要用结构化流程和明确标准把开放任务变成可控任务。约束不是限制模型能力，而是把模型能力从无关方向上收回来。

## Open Questions

- 轨迹评估指标如何从"项目定制"走向"可复用模式"？比如工具调用合理性、错误恢复、计划遵循度，是否能抽成跨 agent 的指标库？
- LLM-as-judge 在评估 agent 轨迹时如何校准？如果 judge 偏好长解释或表面完整的轨迹，可能会奖励绕路而不是高效完成。
