# Agent Role Isolation: 按阶段拆 + 三维隔离

## 它要解决什么问题

`KNOWLEDGE/agent/multi-agent/` 给的是"该不该用 multi-agent"的判断框架——但没回答"如果要拆，按什么轴拆"。

绝大多数人做 multi-agent 系统的第一反应是**按功能分**：前端 agent、后端 agent、测试 agent。这背后的直觉是组织架构图——既然人类团队按职能分工，agent 团队也照搬。

**这条直觉错了**。Cloud Code 的圆码给出了一个完全不同的答案——**按职责阶段分**：探索、规划、实现、验证。

这个节点拆两件事：

1. 为什么"一个 agent 不够"——不是因为能力不够，而是因为**角色冲突**
2. 怎么用**工具集 / prompt / 权限**三个维度做出真正的角色隔离

## 反事实：为什么"一个万能 agent + 不同 prompt"不够

你可能会想：为什么不能就用一个 agent，遇到不同任务时切换不同的 prompt？

举几个 coding agent 的真实病症，每一个都是"一个 agent 承担多角色"的直接后果：

- 让 agent 改一个 bug，他顺手把半个文件重构了
- 让 agent 加一个功能，他给你加了三层抽象和五个你没要求的错误处理
- 让 agent 加一行注释，他把整个文件的注释都重写了

**Anthropic 的诊断**：这不是模型的问题，是架构问题。

类比人类工程师：

- 让同一个人**既写代码又做 code review**——他天然倾向于觉得自己写的没问题
- 让同一个人**既做需求分析又做实现**——他会不自觉地把需求往自己容易实现的方向解读

**这不是能力问题，是角色冲突**。角色冲突会导致**系统性偏差**——每一个回合都偏一点，累积下来产出严重偏离。

回到 agent 的具体病症：

- 改 bug 时顺手重构 → **因为他在改 bug 的同时也在探索代码**，探索过程中他看到了"可以优化的地方"，于是动手了。**探索的好奇心污染了 bug 修复的克制**
- 加功能时加三层抽象 → **因为他同时在规划和实现**，规划阶段觉得"未来可能需要扩展"，实现阶段就提前抽象了。**规划的前瞻性污染了实现的稀疏性**

**问题的根源是：一个 agent 承担太多角色，每个角色的行为边界都模糊了**。

光用 prompt 切换角色解决不了这个——因为 prompt 是软约束，模型可能听也可能不听，**当对话进行到一半、上下文里同时塞着探索和实现两个任务的痕迹时，模型自己都分不清现在该是什么角色**。

## Cloud Code 的拆法：6 个 agent，按阶段分

Cloud Code 内建了至少 6 个 agent，**各有各的 system prompt，各有各的工具集，各有各的权限边界**：

1. **通用任务 agent**：日常的编码执行
2. **Explore agent**：纯只读的代码探索专家
3. **Plan agent**：只做规划，不执行任何操作
4. **Verification agent**：对抗性验证——专门找你代码里的问题
5. **Guide agent**：使用指导
6. **Status Line Setup agent**：状态栏配置

注意这个拆分轴——**不是 "前端 / 后端 / 测试"，是 "探索 / 规划 / 实现 / 验证"**。

这两种拆法有本质区别：

| 拆分轴 | 每个 agent 内部还要做的事 | 角色冲突 |
|---|---|---|
| 按功能（前端/后端/测试） | 仍要同时做研究、规划、实现、验证 | **没解决** |
| 按阶段（探索/规划/实现/验证） | 只负责一个阶段的职责 | **解决** |

**按阶段分** 对应的是软件工程里早就成熟的概念——**你不会让开发人员自己验收自己写的代码**。不是因为不信任他的能力，**是因为角色冲突会导致系统性偏差**。Agent 也一样。

## 反事实：光拆 prompt 不够——必须三维隔离

仅仅给每个 agent 不同的 system prompt 是不够的。**软约束（prompt）永远可能被破坏**，因为它依赖模型的自觉性。Cloud Code 在三个维度上做了真正的硬隔离。

### 维度 1：工具集隔离（最硬的一道）

**Explore agent 只能用 glob、grep、file read 这些只读工具**。连 bash 命令都只允许 `ls`、`git status`、`git log` 这种不会改变任何东西的操作。

它被明确禁止：

- 创建新文件
- 修改已有文件
- 删除文件
- 用重定向写文件
- 运行任何改变系统状态的命令

为什么这么极端？**因为探索阶段如果不小心改了东西，后面实现阶段就会基于"已经被偷偷改过的代码"做决策**——错误从根上就埋下了。

工具集隔离的最大好处：**不依赖模型自觉性**。Explore agent 想写文件也写不了，因为它的工具列表里根本就没有写文件的工具。

### 维度 2：Prompt 隔离

每个 agent 有自己的 system prompt，**明确告诉他：你是谁，你能做什么，你不能做什么**。

其中 **Verification agent 的 prompt 写了 130 行，可能是整个原码里最精心设计的一段文本**。它的核心方向只有一个：**想办法搞坏它**。

什么意思？它不是温和地"检查代码看起来对不对"，**而是像一个刻意找茬的 QA 工程师**——主动构造能让代码出错的场景，**用实际运行的方式去证明代码有没有问题**。

这就是 `KNOWLEDGE/agent/harness-practice/` 里反复强调的 **"生成和评估必须分离"** 的具体落地——让评估者**主动对抗**，而不是被动确认。

### 维度 3：权限隔离

不同 agent 有不同的权限模式。**只读 agent 在 runtime 层面就被锁死了**。

**写权限不是靠 prompt 里写"你不要写文件"来约束，是工具集压根没给他**——和维度 1 重叠但更彻底。**风险操作要停下来确认不是靠模型自己判断，是 runtime 层直接拦截**——对应 `KNOWLEDGE/agent/agent-permission-system/` 的两阶段评审。

**三层隔离叠加**，才真正实现了角色分离。

## 一个意外收获：成本优化

按阶段拆 agent 还带来了一个额外好处——**不同阶段需要的推理强度不同**。

Explore agent 在外部用户场景下**默认用 Haiku 模型**——更快更便宜。**因为探索阶段不需要最强的推理能力**，速度和成本更重要。

而需要强推理的阶段（实现、验证）才用更大的模型。

这和 `KNOWLEDGE/agent/harness/` 里讲的**推理三明治策略**是同一个思路：**把算力花在刀刃上**——规划和验证用最高推理、执行用中等推理。**全程最高推理强度反而成绩更差**（会超时）。

## 硬约束兜底，软约束补缝

Cloud Code 还把行为规范写成"制度"——它的 prompt 里有一个叫 `getSimpleDoingTasksSection()` 的函数，里面列了一组规则：

- 不要加用户没要求的功能
- 不要过度抽象，三行重复代码好过一个不成熟的抽象
- 不要给你没改的代码加注释和文档字符串
- 不要做不必要的错误处理和兜底逻辑
- 不要设计面向未来的抽象
- 先读代码再改代码，不要轻易建新文件
- 方法失败了先诊断，不要盲目重试，也不要一次失败就放弃
- 结果要如实汇报，没跑过的不要说跑过了

每一条都对应一个具体的 agent 病症——**Anthropic 把"不要犯这些错误"写成制度性的约束，而不是指望模型每次都能自己想到**。

这套设计的核心哲学：

> **不信任模型的自觉性。能用硬约束的地方，绝不用软约束**。

- Explore agent 不能写文件 → 不是 prompt 里写"请你不要写"，**是工具集里压根没给他写文件的能力**
- 风险操作要停下来确认 → 不是靠模型自己判断，**是 runtime 层直接拦截**
- "不要加用户没要求的功能" → 这种没法在代码层面拦截的行为**才退而求其次写进 prompt**

**用明确的规则代替模型的临场判断，硬约束兜底，软约束补缝**。这比把所有希望都寄托在一段 prompt 上稳定得多。

## Open Questions

- **按阶段拆 vs 按功能拆——是不是有混合方案**？比如先按阶段拆出 Explore / Plan / Implement / Verify，每个内部再按功能细分？还是说这种二级拆分会再次引入角色冲突？本节点没探讨。
- **6 个 agent 里 Guide agent 和 Status Line Setup agent 是什么定位**？它们看起来既不是阶段也不是功能，更像"特殊任务的专属 agent"。这套阶段划分是不是其实有一个"杂项"槽？
- **Explore agent 用 Haiku，Verify agent 用大模型——但 Verify agent 的 130 行 prompt 在 Haiku 上是不是就完全失效**？什么样的 prompt 能跨模型规模迁移、什么样的不能？这条没说清。
