# Agent Context Compaction: 四层压缩流水线

## 它要解决什么问题

`KNOWLEDGE/agent/context-engineering/` 已经讲了 Compaction 是上下文工程的第一道防线，但停在"为什么要做"这一层。生产 agent 真到了要落地时，会发现一个矛盾：

**上下文窗口越大、对话越长，压缩本身就越成问题**——压缩请求自己可能超长，压缩会让服务端 KV Cache 失效，连续失败会反复浪费 API 调用。Cloud Code 圆码注释里记录了一条真实事故：**1279 个绘画出现 50 次以上连续自动压缩失败，最多的一个绘画失败 3272 次，全局每天浪费约 25 万次 API 调用**。修复方案最后只有三行——连续失败超过三次就停止重试。

这条事故说明一件事：**压缩不是一次设计就能 ship 的功能，是一个持续踩坑持续修补的子系统**。这个节点把 Cloud Code 的四层压缩流水线拆开看——**每一层都假设上一层可能不够用，每一层解决一个不同力度的问题**。

## 反事实：为什么不能只做一种压缩

最朴素的方案：到上下文上限就调一次模型把整段对话总结成一个摘要，丢掉旧消息，继续。

这个方案至少踩三个坑：

1. **成本爆炸**：每次都调一次模型，对于会话密集的产品代价灾难
2. **服务端缓存失效**：API 厂商缓存 prompt 前缀来省钱，你改了旧消息，前缀变了，**缓存全部作废**
3. **压缩本身可能超长**：你触发压缩是因为上下文太长，那这段超长的输入塞进压缩调用本身就可能再次超限

所以工业级压缩必须**分级**——成本最低的先上、不够用再上更重的，每一级都精确知道自己适合什么场景。Cloud Code 把这件事拆成四层：微压缩 → 绘画记忆压缩 → 完整压缩 → 自动触发熔断。

## 第一层：微压缩（不调模型，按缓存状态分两条路径）

成本最低，**不调用模型，纯规则驱动**。

核心逻辑很简单：维护一个白名单（File Read、Bash、Grep、Glob、Web Search、Web Fetch、File Edit、File Write），**保留最近 N 个工具结果，清掉更早的**。这些工具输出通常很大但时效性短——你 10 分钟前读的文件内容大概率已经不需要了。

**但清理动作要看服务端缓存状态**，分两条路径：

### 路径一：Cached Micro Compact（缓存还有效）

用户在连续对话、prompt cache 还在服务端。这时**不修改本地消息内容**——而是通过 Cache Editing API 告诉服务端 "把这几个工具结果删了，但不动 prompt 前缀"，缓存继续有效。

### 路径二：Time-Based Micro Compact（缓存已过期）

用户离开一段时间回来，服务端缓存早过期了——那就没必要小心翼翼。系统计算距离上一条助手消息的时间间隔，超过预值直接改消息内容，**把旧工具输出替换成站位符**。

### 一个反直觉的工程细节：主线程隔离

Cached Micro Compact **只对主线程生效**。为什么？

Cloud Code 会 fork 出子 agent 做绘画记忆提取、prompt 建议等后台任务。如果这些子 agent 把它们的工具结果注册到全局状态里，主线程下次尝试删除时会去删一些**在自己对话历史里根本不存在的工具**——直接报错。

所以原码用 QuerySource 做了主线程隔离：**子 agent 的工具结果不进全局清理状态**。这是一个典型的"看起来不该有的复杂度，但生产环境逼着你加"的设计。

## 第二层：绘画记忆压缩（提炼事实，不做摘要）

微压缩解决的是"工具噪声"问题。但如果对话本身就在反复推进一个长任务呢？光删工具结果不够，**你需要把对话中学到的东西沉淀下来**。

**关键设计**：这一层不是把对话做摘要，**是从对话中提取"学到了什么"**——项目结构、用户偏好、任务进度。这些结构化事实持久化到本地的记忆目录里。详见 `KNOWLEDGE/agent/agent-memory-system/`。

具体的边界处理特别精细——它有三个保留参数：最小 token 1 万、最小消息 5 条、最大 token 4 万。从最后一条已总结消息开始往前扩展，**直到同时满足最小 token 和最小消息数，或者触及最大 token 上限**。

最关键的边界约束：**扩展时不能拆开 Tool Use 和 Tool Result**。如果保留了带 Tool Result 的用户消息，必须同时保留对应 Tool Use 的助手消息——否则 API 直接报错。流式传输时同一个 Message ID 可能被拆成多条消息（thinking 一条、tool use 一条），也必须同进同退。

**这一层最大的优势是不调用模型**——直接复用已经提取好的结构化记忆，**成本远低于完整压缩**。

## 第三层：完整压缩（调模型，用 Analysis 标签做隐式 CoT）

绘画记忆不可用或不够用时，回退到完整压缩。**这一层必须调模型**。

Prompt 设计特别值得记：模型被要求按九个维度总结——用户请求和意图 / 关键技术概念 / 文件和代码片段 / 错误和修复 / 问题解决过程 / 所有用户消息 / 待办任务 / 当前工作 / 可选的下一步。

但**最精妙的是 Analysis 机制**：prompt 要求模型先在 `<Analysis>` 标签里做详细分析、然后再在 `<Summary>` 标签里输出最终摘要。**Analysis 部分在格式化时被完全剥离，不会进入压缩后的上下文**——它的作用纯粹是让模型在输出摘要之前先想清楚。

**这本质上是一个内嵌的 chain of thought**。CoT 通常用来提升推理质量，这里被用来提升摘要质量——同一个机制不同场景。

### 一个 meta 问题：压缩请求本身超长怎么办

很反直觉但真的会发生：**用来缩短上下文的压缩操作，本身的输入可能就超长了**。

原码的解法是**按 API 轮次分组，从最早的组开始丢弃，最多重试三次**。压缩完成后还要做一系列善后工作——清空文件读取缓存、重新注入最近访问的文件内容（最多 5 个文件、总预算 5 万 token）、重新注入 plan 文件、skill 内容、MCP 工具说明、agent 列表。**这些都是压缩过程中被丢掉的关键上下文，必须在压缩后立即恢复**。

## 第四层：自动触发 + 熔断器

前三层是"压缩怎么做"。这一层是"什么时候做"——以及"什么时候停下来认输"。

### 触发逻辑

每次模型调用之后检查当前 token 用量。**超过预值（上下文窗口减 13000 token 缓冲）就自动触发压缩**。先尝试绘画记忆压缩，不可用或压缩后仍超标，再回退到完整压缩。

### 熔断器

**连续失败超过三次就停止重试**。

就是文章开头那条 25 万次 API 调用的修复——三行代码。生产环境里你才会发现 "重试是默认策略" 这个直觉在 agent 场景下是致命的：因为 agent 的对话上下文可能已经不可恢复，**反复重试只是在反复浪费 API 调用**。

熔断器的设计哲学和分布式系统里的电路保护一模一样：**不是每次都试，是知道一类失败永远不会成功，提前认输**。

### 子 agent 的排除逻辑

Session Memory 和 Compact 这两个 QuerySource 的请求**不会触发自动压缩**。为什么？

因为它们本身就是 fork 出来的子 agent。**如果它们也触发压缩，就会死锁**——压缩调用又是一个 fork，又会触发压缩，无限嵌套。

排除逻辑很简单：**只要是 fork 出来做压缩相关任务的 agent，就不能再触发压缩**。

## 一个跨层的共同 motif：把"装不下"换成"该不该装"

回过头看四层，每一层在问同一个问题：**这个 token 值不值得留在上下文里**。

- 微压缩按工具类型问：旧文件读取还有价值吗？大概率没有 → 删
- 绘画记忆压缩按事实结构问：这条对话推导出了什么持久的事实？→ 提取
- 完整压缩按九个维度问：整段对话的精华是什么？→ 总结
- 自动触发按 token 预算问：我什么时候该开始动手？

这个 motif 和 `KNOWLEDGE/agent/context-engineering/` 里 Anthropic 的原话直接对应：

> 上下文工程的目标是**找到最小的高信噪比的 token 集合，最大化期望结果的概率**。

**关键不是装不装得下，而是该不该装进去**。Cloud Code 的四层压缩流水线是这条原则在工业级 agent 上的具体落地。

## Open Questions

- **微压缩的两条路径靠"服务端缓存是否还有效"切换**——但客户端怎么知道缓存还有效？是猜的（按对话时间间隔），还是有 API 能查？如果是猜，猜错了的代价是什么——会把缓存改坏吗？这条没在节点里展开。
- **熔断器三次失败就停**——但停了之后呢？用户继续输入会怎么样？Agent 是直接 OOM 还是降级到只看最后 N 轮？文章给的修复方案是"停止重试"，但 Recovery 路径没说清楚。
- **绘画记忆压缩的"提取事实"和完整压缩的"九维度总结"是不是同一类操作的两种力度**？还是说它们提取的内容质性上不同？前者更结构化（用户偏好 / 项目结构），后者更叙述化（用户请求和意图）——但边界在哪？比如"用户提出了一个新需求"算事实还是叙述？这条 Cloud Code 的设计意图本节点没碰到。
