# RAG Failure Diagnosis

## 它要解决什么问题

RAG 上线后用户反馈 "回答不准确"，最危险的反应是立刻开药方：换 embedding 模型、调 chunk size、加 reranker。

这些手段都可能有用，但如果没有先定位瓶颈，就只是把背过的优化技巧随机套到系统上。RAG pipeline 很长，答案不准确可能发生在数据源、切分、查询表达、向量模型、排序任何一环。系统化排查的关键，是从上游到下游确认：答案是否存在、是否被完整切出来、是否能被 query 表达命中、embedding 是否能区分、排序是否把它放到可用位置。

## 第一步：答案到底在不在知识库里

很多人一看到 bad case 就怀疑检索不准，但最上游的问题可能是：知识库根本没有答案。

比如 PDF 表格解析失败，扫描件 OCR 没跑出来，某批文档没有导入，或者知识库没有及时更新。这时调 chunk、换 embedding、加 reranker 都不会救回来，因为系统没有可检索的真相。

排查方法很朴素：抽 bad case，人工到知识库里找答案。如果人都找不到，就不要把问题归因给检索模型。这一步解决的是数据源完整性。

反事实地看，如果跳过这一步，你会把一个 ingestion 问题伪装成 retrieval 问题，后续所有优化都会变成噪声。

## 第二步：chunking 有没有把答案切碎或稀释

确认答案存在之后，下一步看它在索引里变成了什么样的 chunk。

一个典型例子：用户问产品 A 的退货政策，原文里退货政策是一段完整说明，但 chunk size 设得太小，刚好把条件和流程切成两半。每个半截 chunk 语义都不完整，和 query 的相关性都不高，最后双双排到后面。

反过来，chunk 太大也会坏。一个 chunk 里混了多个话题，真正相关的几句话被大量无关内容稀释，embedding 看到的是混合语义。

所以排查 chunking 不是看配置，而是看失败样本对应的实际 chunk 长什么样：答案是否被切断，是否被淹没，是否缺标题或上下文。如果这里出问题，解法才是调整 chunk size、overlap、按标题层级切分，或者保留表格/段落结构。

## 第三步：query 和 document 之间有没有语义鸿沟

如果答案存在，chunk 也完整，仍然检索不上来，就要看用户语言和文档语言是不是错位。

用户问 "手机发烫怎么办"，文档写的是 "设备温度异常处理方案"。语义上是同一件事，但词面完全不同。向量模型有时能跨过去，有时跨不过去，尤其在业务黑话、缩写、口语表达很多的场景里。

这时问题不一定是 embedding 模型差，而是 query 侧需要帮助。Query rewrite、query expansion、HyDE、混合检索都属于这一层：它们不是改变知识库，而是让用户问题更容易命中文档表达。

混合检索的价值也在这里：向量检索擅长语义相似，关键词检索擅长精确词命中。业务系统里两者经常互补，而不是二选一。

## 第四步：embedding 模型是否适合领域

前三步都排除后，才轮到 embedding 模型本身。

通用 embedding 在医疗、法律、金融、内部运维这类垂直领域可能水土不服。它能理解常见语义，但不一定理解专业术语之间的关系。比如两个术语在人类专家看来高度相关，但通用模型可能只看到表面词差异。

排查方法不能靠感觉，要做一个小评测集：每个 query 标出应该召回的文档，看 recall@K、MRR 或命中率。如果评测里大量领域术语 case 都低，再考虑换模型、领域微调、构造正负样本。

这里的顺序很重要。只有确认不是数据源、chunk、query 表达的问题，embedding 评测才有解释力。

## 第五步：不是没召回，而是排序不对

最后一种常见情况是：相关文档其实被召回了，但排在 TopK 外面。比如系统只取 Top5，正确 chunk 排第 8。用户看到的是错答案，但根因不是完全没召回，而是排序阶段把证据截掉了。

这时 reranker 才是对症药。初始检索可以放宽到 Top20 或 Top50，再用 cross-encoder reranker 做更精细的相关性判断，最后截 Top5 给生成模型。

但 reranker 不能无限上提。Cross-encoder 精度高，是因为它联合看 query 和文档，计算量也大，所以适合小候选集精排，不适合全库扫描。

## 排查链路比优化清单更重要

完整链路是：数据源 -> chunking -> query 理解 -> embedding -> 排序。

这条链路的价值在于，每一步都有明确的诊断问题和对应证据，而不是凭经验调参。面试官真正想听到的也不是你知道多少 RAG trick，而是你拿到 bad case 后能不能把问题定位到 pipeline 的具体环节。

用一句话概括：RAG 准确率问题先诊断，再优化；否则所有优化都可能只是随机试药。

## Open Questions

- 这条排查链路需要一套自动化 eval harness 支撑。人工 bad case 能定位方向，但上线系统要持续发现 "数据源缺失 / chunk 断裂 / query 语义鸿沟 / rerank 错序" 分别占多少。
- 多轮 Agentic RAG 里，失败诊断会更复杂：一次回答可能有多次检索和多次判断。传统的 recall@K 只能评估单次检索，不能直接解释 "中间哪一步查错了"。
