# Small Model Harness Engineering: 复合概率问题与五层防护

## 它要解决什么问题

把一个 8B 本地模型投入多步 Agent 任务，它会快速失效——不是因为它不懂任务，而是因为**多步工作流里的可靠性是个复合概率问题**。

单步准确率 90% 的模型，5 步全对的概率是 0.9⁵ ≈ 59%，10 步全对只剩 0.9¹⁰ ≈ 35%。**听起来合格的模型，在 10 步工作流里通过率不到四成**。大模型看起来没这个问题，是因为前沿闭源模型单步准确率接近 99%——0.99¹⁰ 还有 90%，门槛差了一个数量级，跨越的方式不是模型变聪明，是**每步的误差都小得多**。

所以 8B 模型的真实问题不是"太笨"，而是**每步误差大了一点，经过 10 步连乘指数级放大**。Forge 项目（ACM 已接收）的回答是：不换模型，在模型外围加五层确定性防护，把每步的有效准确率提上去。结果是：Ministro 3 8B Instruct Q8 跑在 llama-file 上得分 86.5%，而同场无防护层的 Claude Sonnet 得分 87.2%。**80 亿参数本地模型通过外部防护层逼近前沿 API 模型**。

## 如果只改模型、不加防护层

朴素方案：换一个更强的模型，期望单步准确率提升。但这有三个问题：

1. **本地部署有资源上限**。消费级 GPU 装不下 700B 的模型
2. **单步准确率的提升是渐进的**，而复合误差的放大是指数的——你需要的是精确消除每一类错误，而不是全局性地稍微好一点点
3. **推理后端的选择可能比模型本身更关键**——同一套权重，在 llama.cpp 原生函数调用模式下准确率只有 7%，换成 llama-file 的提示注入模式直接跳到 83%，**75 个百分点的差距纯粹来自推理基础设施的选择**。"这个模型工具调用不行"的结论可能根本就是错的——你只是用错了后端

## 五层防护的逻辑

Forge 把 agent 的每步失败模式拆成五类，每层防护针对一类，**下一层的存在假设上一层可能失败**。

### 第一层：救援解析

最常见的失败模式是**模型知道该调什么工具、传什么参数，但输出格式不对**——把工具调用写成了自由文本而不是结构化 JSON。Forge 不急着惩罚重来，而是先尝试从自由文本里抢救出有效的工具调用。支持三种格式：JSON 被代码块包裹时用正则提取、推理模型思维链里用特殊格式预演的 "排练语法"、以及 Qwen 系列模型的 XML 格式。

关键判断：**意图对了，表达形式不对——与其让模型重来，不如把正确意图直接提取出来**。

### 第二层：重试提示

如果救援解析也失败了，Forge 注入纠正消息告知模型，最多重试三次。超过三次就终止，**不允许无限循环**。每次成功的工具调用会重置重试计数器。

调用了不存在的工具时，Forge 直接告诉模型工具列表，要求从中选一个，而不是让模型自由猜测。

### 第三层：步骤强制执行

小模型经常"跳步"——直接调用终端工具，跳过中间必要步骤。Step Enforcer 检查模型是否在前置依赖完成之前就触发终端工具。发现跳步时，注入的纠正消息是**升级式的**：第一次语气礼貌，第二次语气直接，第三次措辞强硬，三次后直接报错终止。

工具前置条件是这层的另一个机制：`get_detail` 只有在 `search_list` 调用之后才能使用（因为需要搜索结果里的 ID）。Forge 在工具注册时就编码了这种依赖关系。

### 第四层：错误恢复

工具执行失败后，Forge 区分两种错误而不是统一计入失败计数：

- **硬错误**：工具本身坏了（代码有 bug、权限不足）→ 计入连续错误计数器，超过两次终止
- **ToolResolutionError**：工具调用格式正确、参数类型也对，但资源不存在（比如查了一个不存在的用户 ID）→ **不计入连续错误计数器**，语义类似 HTTP 404，只是告诉模型换个参数再试

如果不区分，模型猜错三次参数就会触发终止——即使工具本身完全正常。**混淆两类错误会让系统过早放弃**。

### 第五层：上下文压缩

多步工作流积累大量上下文：每步的工具调用、返回结果、推理过程、纠正消息。在消费级 GPU 上上下文过长会把模型从 GPU 挤到 CPU，速度降低 10 到 100 倍。

Forge 用三阶段确定性压缩（Tiered Compact）处理：

1. **第一阶段**：删除所有纠正消息；旧工具返回结果截断到前 200 个字符
2. **第二阶段**（第一阶段不够用时）：删除旧工具返回结果，但**保留模型的推理过程**——推理是模型对数据的解读，比原始数据更致命，丢了推理等于丢了思路
3. **第三阶段**（还不够时）：删除推理和失败的文本回复，只保留工具调用的骨架

**最近几轮对话永远不压缩**，只有旧内容才被处理。三个阶段全是确定性文本操作，**不调用模型，零延迟**。

VRAM 感知的预算机制：Forge 启动时查询 NVIDIA SMI 获取 GPU 显存，自动设定上下文上限——< 24GB 配 4000 token，24–48GB 配 32000 token，> 48GB 配 26 万 token。这防止了一个隐蔽问题：llama.cpp 在显存不足时会静默回退到 CPU，速度降 10–100 倍，用户毫无感知。

## 消融实验：五层缺一不可

Forge 做了系统性消融——单独移除每一层都导致得分下降，移除上下文压缩导致长工作流得分直接崩溃。五层不是冗余堆砌，**每层针对的失败模式都真实存在且对最终通过率有独立贡献**。

## Respond 工具：彻底消除模式选择问题

五层防护之外，Forge 还加了一个设计：**合成 `respond` 工具**。问题是小模型经常搞不清楚什么时候该调工具、什么时候该直接输出文本。实测数据：**信任模型自己选择两种模式，工作流完成率从 100% 降到 4%**。

解法：给模型加一个叫 `respond` 的虚拟工具。模型想回复文本时，调用 `respond` 工具并传入文字内容。这样模型**永远处于工具调用模式**，不需要在"输出文本"和"调工具"之间做选择——彻底消除了这个歧义。

## Open Questions

- 复合概率模型假设每步错误是独立的——但实际上一次跳步或格式错误可能会扰乱后续步骤的上下文，导致后续步骤的错误率也变高（正相关）。如果引入步骤间错误相关性，所需防护强度会有多大变化？
- 五层防护的开销没有在材料里量化——比如第三层 Step Enforcer 需要追踪工具前置依赖，这在动态工具注册场景下是否能稳定维护？Forge 的 26 个评测场景里，工具集合是静态的还是运行时可变的？
