# Agentic RAG vs Long Context

## 它要解决什么问题

面试里问 "现在模型都有超长上下文窗口了，RAG 还有什么价值"，表面是在问技术取舍，实际是在问你有没有把 RAG 看成一个会演进的架构，而不是一个 "上下文塞不下时才用的补丁"。

如果只回答 "RAG 成本低、知识更新方便"，这些点没错，但防守意味太重。更好的切入是先拆层：长上下文解决的是**能不能放进去**，RAG 解决的是**该不该放进去、怎么找、找完之后够不够**。这两个问题有重叠，但不是同一个问题。

## 第一代 RAG 和长上下文确实有替代关系

传统 RAG 的直线流程是：用户提问，检索一次，把 TopK 文档塞进 prompt，再生成答案。它最早解决的一个核心限制，就是模型上下文窗口太短，不能把大量文档全放进去。

如果模型只能看几千 token，那 RAG 几乎是必需的；如果模型能看十几万甚至百万 token，"放不下"这个物理限制就被削弱了。所以面试官的问题不是空穴来风：**对一次性检索 + 拼接的第一代 RAG 来说，长上下文确实吃掉了一部分价值**。

但反过来，如果因此把所有文档都塞进去，会立刻遇到三个问题。

第一是噪声。100 篇文档都放进上下文，模型仍然要在里面找真正相关的几段。长窗口不是无限注意力，相关信息落在中间位置时尤其容易被忽略，这和 Lost in the Middle 是同一类问题。

第二是成本。长上下文输入不是免费容量，token 越多，推理成本和延迟越高。一次轻量检索如果能把 100 篇压到 5 段，本质上是在用一个便宜的外部筛选器换掉昂贵的模型注意力消耗。

第三是更新。知识库变了，RAG 更新索引就能生效；如果把知识当作 prompt 或模型参数的一部分，更新路径会重很多。

所以，长上下文可以削弱 "因为塞不下所以必须检索" 的理由，但不能消灭 "因为不该全塞所以需要筛选" 的理由。

## Agentic RAG 把检索从一次动作变成迭代过程

真正的分水岭在这里：传统 RAG 假设一次检索就能拿到回答所需的全部信息，而复杂问题经常不满足这个假设。

比如用户问："我们公司去年华东区的销售业绩为什么下滑？" 这个问题不是找一句话。它至少需要去年华东区销售数据、前年数据、下滑幅度、市场报告、竞品动作。一次检索可能召回几份看似相关的报告，但缺了对比数据，模型仍然只能编一个依据不足的解释。

Agentic RAG 的核心变化是：模型不再被动接收一次检索结果，而是自己规划、检索、判断、再检索。

它可能先拆出子问题：去年是多少、前年是多少、下滑多少、可能原因是什么。然后先查去年数据，发现还需要前年数据，再查前年的；拿到下滑 15% 之后，再查市场分析报告；报告里提到竞品补贴，于是继续查竞品动态。到信息足够时，它才整合答案。

这个过程里，检索不是一个前置步骤，而是 agent 执行链的一部分。它解决的是**多跳信息获取和证据补齐**，不是单纯的上下文容量。

## 长上下文在 Agentic RAG 里变成工作空间

这也是最容易答反的地方：在 Agentic RAG 里，长上下文不但没有替代 RAG，反而变成了 RAG 的工作空间。

因为多轮检索会产生中间状态：已经知道什么、还缺什么、哪些证据互相支持、下一步要查什么。这些状态和多次检索结果需要被模型持续维护。如果上下文太短，agent 还没完成证据闭环，中间过程就被挤掉了。

所以长上下文的角色从 "把所有知识塞进去" 变成 "承载多轮检索和推理过程"。前者是数据库替代品，后者是工作记忆。

这就是回答的主线：第一代 RAG 和长上下文有部分替代；Agentic RAG 和长上下文是互补关系。

## 选型不是信仰，而是问题形态

如果问题简单、一次检索基本能回答、业务对成本敏感，传统 RAG 足够。它的优势是便宜、快、可控。

如果知识量小、更新频率低、延迟不敏感，直接长上下文也合理。比如几十页固定合同，让模型一次性看完，可能比搭检索链路更简单。

如果问题复杂，需要多跳推理、跨多个信息源整合、边查边判断，那才是 Agentic RAG 的空间。它需要检索能力，也需要长上下文能力，还需要模型有规划和自主决策能力。

反事实地看，如果简单问题也强上 Agentic RAG，每一步都是一次模型调用，成本和延迟会很快变成负资产；如果复杂问题仍然用一次检索，系统看起来快，但答案经常缺证据。

所以这道面试题的好答案不是 "RAG 没过时"，而是："RAG 的价值从容量补丁，演进成了复杂问题的信息获取架构；长上下文削弱了第一代 RAG 的一部分价值，但增强了 Agentic RAG 的工作空间。"

## Open Questions

- Agentic RAG 的边界需要更可观测的指标：什么样的 query 可以在路由阶段判定为 "一次检索足够"，什么样的 query 必须进入多轮检索？如果只靠模型自判，可能会把简单问题复杂化。
- 长上下文作为工作空间时，哪些中间状态应该保留在上下文，哪些应该外化成结构化 scratchpad 或检索轨迹？这和 context-engineering 里的 Just-in-time Context 是同一族问题。
