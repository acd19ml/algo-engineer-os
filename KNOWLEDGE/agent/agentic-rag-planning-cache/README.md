# Agentic RAG Planning Cache

## 它要解决什么问题

Agentic RAG 准确率上去了，但响应时间从 3 秒变成 14 秒，这不是小瑕疵，而是架构层面的代价。

常见优化是限制最大迭代次数，或者把中间判断换成小模型。它们能省时间，但会伤准确率：该继续查的时候提前停，不该查的时候继续查，或者小模型误判信息是否足够。

Planning cache 的切入点不同。它不压缩执行过程，而是问：很多同质任务的规划逻辑明明一样，为什么每次都让大模型从头想一遍？

## Agentic RAG 的瓶颈常常不在检索，而在规划

传统 RAG 通常是一次检索 + 一次生成。Agentic RAG 多了分析问题、决定查什么、判断信息是否足够、再决定下一步这些环节。每个环节都可能调用大模型。

以 ReAct 形态看，模型要先输出思考：用户想知道苹果公司 2024 年营收，需要查财报；再输出动作：调用检索工具，参数是公司、年份、财报关键词；结果回来后再观察是否足够，最后生成答案。

这里最费时间的不是检索本身，而是反复生成规划和判断 token。思维链、工具参数构造、观察分析的 token 消耗，往往比最终回答还多。

但很多问题是同一类任务。查北京天气和查上海天气，实体不同，处理逻辑一样。计算苹果销量和计算橙子销量，表面词不同，本质都是 "查询数据 -> 提取指标 -> 计算"。

如果每次都从零规划，就像老员工每次处理同类工单都重新写 SOP。规划缓存要缓存的不是答案，而是这种可复用的 SOP。

## 缓存答案会过时，缓存规划模板不会直接污染事实

普通缓存的风险是把旧答案当新答案。用户问今年营收，如果命中了去年答案，系统会快但错。

Planning cache 缓存的是 "怎么做"：先查某类数据，再抽取某类指标，再做某种计算或对比。命中后，具体检索仍然发生在当前知识库和当前上下文里，最终答案仍然是实时生成。

所以它省掉的是重复规划，不是实时证据。这个差别解释了为什么它可能降低延迟而不明显牺牲准确率。

## 三步实现：意图标签、模板填空、失败积累

第一步是给问题打高层意图标签。用户问 "帮我算算这个表格里销售额的平均值"，标签不是 "销售额" 或 "表格"，而是 "均值计算"。

这里用高层意图而不是普通语义相似度，是因为语义相似度容易被表面实体干扰。"计算苹果销量" 和 "计算橙子销量" 词面不近，但任务结构相同；"苹果公司 2024 营收" 和 "苹果新品发布会" 词面近，但处理逻辑未必相同。

第二步是命中模板后填空。模板可能是：查询 {entity} 的 {metric} 数据 -> 校验时间范围 {time_range} -> 提取数值 -> 返回带来源的答案。小模型只需要把占位符换成当前实体、指标和时间，直接进入执行阶段。

第三步是没命中就积累。缓存里没有模板时，正常走完整 Agentic RAG 流程。执行成功后，把这次规划抽象出来，抹掉实体名、文件名、具体数值，只保留通用步骤框架，存为新模板。下次同类任务就能命中。

冷启动阶段缓存为空，可以先预置高频模板：查询类、对比类、汇总类、计算类。运行一两周后，模板库会覆盖越来越多重复任务。

## 为什么它比限制迭代次数更像根因优化

限制迭代次数是在执行时踩刹车。它没有减少任务本身的思考需求，只是强制 agent 少想几步。复杂问题可能因此丢证据。

Planning cache 是在重复任务上跳过已知规划。它的前提是任务类型有重复率。如果一个系统的问题高度异质、每次都要新推理，规划缓存命中率低，收益就有限。

所以面试里可以先反问一句："我们的任务类型重复率高不高？" 如果大部分请求落在少数固定模式，规划缓存是根因优化；如果请求极其开放，先做缓存可能只是增加复杂度。

可观测指标也应该围绕这个前提：模板命中率、命中后的成功率、平均规划 token 节省、延迟下降、误命中导致的失败率。

## Open Questions

- 高层意图标签怎么学得稳定？如果标签太细，命中率低；太粗，模板误用风险高。这需要真实流量上的聚类和失败分析。
- 模板抽取后的质量控制怎么做？一次成功轨迹不一定代表可复用 SOP，可能只是偶然成功。是否需要多次成功、人工审核或离线 replay 才能入库？
- Planning cache 和 procedural memory / skill 系统边界很近：缓存模板什么时候应该升级成显式 skill，什么时候只是临时性能优化？
