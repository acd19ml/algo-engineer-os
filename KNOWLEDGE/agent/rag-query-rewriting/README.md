# RAG Query Rewriting

## 它要解决什么问题

Query 改写看起来像一个简单增强：用户给一句话，让大模型改得更完整、更清晰，再拿去检索。

但很多 RAG 系统加了 query rewrite 之后效果反而下降。原因是改写不是 "让句子更漂亮"，而是让 query 更容易命中当前业务知识库。通用大模型如果没有业务上下文，可能把用户原本能命中的业务词改掉，或者把缩写扩成错误含义，最后召回率下降。

所以 query rewriting 的核心不是生成能力，而是**什么时候改、改成什么、靠什么业务知识约束它**。

## 为什么无条件改写会变差

第一类问题是浪费。很多 query 不需要改写，比如 "公司地址在哪"。如果 60% 请求都是简单查询，每次都调用大模型改写，就是纯粹增加成本和延迟。

第二类问题更危险：改写让 query 远离文档语言。用户问 "怎么开白名单"，知识库文档里也写 "白名单"。大模型把它改成 "如何将特定用户添加到访问许可列表中"，看起来更正式，但业务文档不这么写，embedding 相似度反而下降。

第三类问题是业务歧义。用户输入 "HP"，通用模型不知道是 Hewlett-Packard、首页、还是业务里的 Hearing Preparation。用户输入 "Motion"，法律语境里可能是动议，通用模型却可能理解成运动或移动。

反事实地看，如果让通用模型凭空改写业务 query，它会把业务语言拉回通用语言；而 RAG 检索恰恰需要贴近知识库语言。

## 五类通用改写策略解决不同问题

基础改写适合短 query 补全和缩写展开。用户输入 "预审批贷款案例"，可以补成 "请提供关于预审批贷款的案例分析"。它解决的是表达不完整，不解决业务语义偏差。

Multi-query 的思路是从多个角度生成 2-3 个 query 并行检索，再合并结果。比如 "气候变化对环境有什么影响" 可以拆出对生物多样性、海洋、天气模式的不同问法。它提升覆盖面，但每多一个 query 就多一次检索成本。

Subquery decomposition 适合复合问题。用户问 "比较马斯克和贝佐斯的教育背景"，单个 query 很难同时召回两个人的完整材料；拆成两个子问题分别查，证据更稳。

Step-back prompting 是把过于具体的问题退到更抽象的背景问题。比如问某电视剧里某明星穿什么衣服，知识库不一定有直接答案，但可能有该剧服装设计风格的资料。先召回背景，再回答细节。

HyDE 的假设是：知识库存的是答案，不是问题。它让模型先生成一个假设性答案，再用这个答案去检索答案形态的文档，从 "问题匹配文档" 变成 "文档匹配文档"。但如果模型对问题本身不了解，假设答案可能错得很远，检索也会被带偏。

这些策略没有银弹。它们分别解决短 query、覆盖面、复合问题、背景抽象、问答形态不一致。选错策略，改写就会变成噪声。

## 真正的下限来自业务知识注入

业务词典是第一层。缩写、同义词、产品名、内部黑话、专有术语都应该被显式维护。比如公司内部产品叫 Xflake，用户输入 Xflake 时，不应该让大模型改成普通英文，而应该保留原词或映射到业务定义。

上下文注入是第二层。在 prompt 里告诉模型当前业务背景，比如 "我们是一家法律科技公司，Motion 在这里指动议"，能显著减少跑偏。但业务知识太多，全部塞进 prompt 会爆 token，塞少了又覆盖不足。

所以更稳的做法是动态注入：先用用户 query 去一个小型业务术语库里检索相关解释，再把命中的几条注入改写 prompt。用户问 "HP 流程怎么走"，系统先命中 "HP = Hearing Preparation，指庭审准备阶段"，再让模型改写成 "庭审准备阶段的流程是怎样的"。

还可以进一步做实体识别：先识别 query 里的产品名、功能名、缩写，再针对实体查解释。这样比纯检索更精准，因为它先锁定了需要解释的对象。

用户画像也会改变语义。销售问 pipeline，多半是销售漏斗；工程师问 pipeline，可能是数据流水线。如果系统知道用户部门，就可以注入对应语境。

最后是动态 few-shot：维护历史改写成功案例库，新 query 来时检索相似案例，把 2-3 个原 query -> 改写 query 的例子放进 prompt。模型不只是拿到术语，还拿到业务偏好的改写风格。

## 落地原则：先分类，再改写

Query 改写的第一步应该是路由，而不是生成。

简单查询直接检索。复杂查询用 subquery 或 multi-query。专业术语查询先走词典映射或业务知识注入。开放模糊问题再交给大模型改写。

改写数量也要克制。Multi-query 生成 4-5 个可能覆盖更全，但线上常见取舍是 2-3 个，因为每个 query 都会增加检索和合并成本。

最重要的是评估。没有改写前后的 recall 对比，就不知道这套改写是在帮忙还是添乱。至少要有一组标注 query-doc 的评估集，能按 query 类型看召回率变化。

一句话概括：通用策略打底，业务知识兜底。大模型负责处理模糊开放问题，业务规则负责保护明确专有问题；两者缺一个，query rewrite 都容易变成不稳定的装饰。

## Open Questions

- Query 路由器应该用规则、小模型分类，还是直接用主模型自判？不同方案在成本、可解释性、误路由风险上差异很大。
- 业务术语库本身怎么维护更新？如果词典过时，动态注入会把模型稳定地带向错误解释，这和 agent memory 的过期问题是同类风险。
