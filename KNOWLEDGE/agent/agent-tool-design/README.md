# Agent Tool Design: 三原则

## 它要解决什么问题

工具设计听起来没什么——不就是定义一个函数、写个描述、让模型调用吗？

如果你这么想，说明还没踩过坑。一个具体场景：

你给 agent 接了三个工具——命令行（可以执行任意命令）、文件读取、搜索。你以为分工很清楚：读文件用 file_read，搜索用 search，跑命令用 bash。

**结果模型拿到命令行工具之后，什么都用它来做**——读文件用 `cat`、搜索用 `grep`、列目录用 `ls`。**你精心设计的专用工具，模型根本不用**。

这不是模型笨，**恰恰相反，是模型太聪明了**——它发现命令行什么都能做，那为什么还要学其他工具的参数格式呢？**用一个万能工具解决所有问题对模型来说是最省力的策略**。

但对你来说这是灾难：

- 命令行的输出是无结构纯文本——你没法做格式化展示
- 命令行的权限是全开——你没法做细粒度安全控制
- 命令行的输出大小不可预测——可能一次就把上下文打爆

这个节点拆 Cloud Code 的三个工具设计原则——**它们都指向同一个核心认知：工具设计本质上也是上下文工程**。

## 原则一：万能工具要主动让位（"别用我"反直觉）

最反直觉的设计：**Cloud Code 的 Bash 工具使用说明里花大量篇幅告诉模型"别用我去用别的工具"**。

具体写法：

> 不要用这个工具来执行读文件、搜索、列目录这些操作。除非你已经确认专用工具无法完成任务。

然后列了一个工具偏好表：

- 文件搜索 → `glob`
- 内容搜索 → `grep`
- 读文件 → `read`
- 编辑文件 → `edit`
- 写文件 → `write`

为什么要这样设计？**因为万能工具是安全和可控性的天敌**。

你给模型一个能做一切的工具，**它就会用这个工具做一切**：

- 你的权限系统被绕过 → 所有操作都走同一个工具入口
- 你的输出管理被绕过 → 命令行输出是无结构的
- 你的 UI 展示被绕过 → 没法区分这是读操作还是写操作

**专用工具的价值不在于"它能做什么"，而在于"它把操作限制在一个可控范围内"**：

- `read` 工具只能读文件 → 可以精确控制输出格式，做文件缓存去重，做 token 预算检查
- `edit` 工具只能编辑文件 → 可以做差异展示，做路径保护，做权限细分

**但模型不会自动理解这些**。它的默认策略是用最少的工具完成最多的事。**所以你必须在万能工具的说明里显式引导它**——"这些事情有更好的工具去做"。

### 反事实：如果只在 system prompt 里写"优先用专用工具"

为什么这条规则要写在**工具自己的描述**里，而不是写在 system prompt 里？

因为 system prompt 是**前缀**——模型在长对话里注意力会被新的信息冲刷掉。但**工具描述是和工具调用绑定的**——模型每次考虑要不要调用这个工具时都会看一遍。

> **把约束放在最接近决策点的地方，而不是放在远端的系统提示词里**。

这条对任何 agent 设计都适用——**如果你想引导模型在某个决策点做出某种选择，把引导文字放在那个决策点能看到的位置，而不是放在通用的 system prompt 里**。

## 原则二：工具不要一股脑全给（按需发现）

Cloud Code 内置了 **40+ 个工具**。如果把它们全塞进 system prompt，光工具定义就要占掉大量上下文空间，**而且模型面对太多工具时选择准确率会明显下降**。

Manus 团队的数据：**给模型 100 个以上的工具会导致幻觉参数或调用错误**。

### 解法：两级加载

工具分两集：

- **第一集（核心工具）**：启动时就加载，模型立刻可用。约 7-8 个高频工具——bash、file read、file edit、file write、grep、glob、agent
- **第二集（延迟工具）**：模型只知道它们的名字，**不知道参数格式，所以没法直接调用**。包括 web search、web fetch、notebook edit、task management、team collaboration

需要延迟工具时，模型先调用一个 **`tool_search`** —— **就像工具的搜索引擎**。模型传入关键词，`tool_search` 返回匹配工具的**完整定义**（包括参数格式）。拿到定义之后才能正常调用。

### tool_search 的搜索算法：search hint 字段

每个工具除了名字和完整说明，**还有一个叫 `search_hint` 的字段**——3-10 个词的能力短语：

- `read` 工具的 search_hint：`读取文件、图片、PDF、笔记本`
- `grep` 工具的 search_hint：`用正则搜索文件内容`

这些短语**专门为关键词匹配优化**——用的词尽量不和工具名重复，覆盖模型可能搜索的各种表述。

搜索权重：

1. 工具名精确匹配 → 最高
2. search_hint 匹配 → 次之
3. 完整使用说明里的关键词匹配 → 最低

这样模型用简短关键词就能快速找到需要的工具。

### 一个被忽视的好处：缓存稳定性

原码里的工具排序逻辑**把内置工具排在前面作为连续前缀，外部插件工具排在后面**。

这样**即使插件工具变化了，内置工具的前缀不变，服务端的提示词缓存不会失效**。

如果把所有工具混在一起排序，每次插件变化都会打乱前缀，**缓存就废了**——这条直接对应 `KNOWLEDGE/agent/context-engineering/` 第 6 层 Cache-Friendly Architecture 的原则。

### 反事实：如果你的 agent 工具超过 10 个

最简单的退化版实现（不用做完整 tool_search）：

> **在 system prompt 里列出所有工具的名字和一句话描述。然后告诉模型：如果你需要用某个工具，先告诉我，我会给你它的完整定义**。

比把所有工具定义一股脑塞进去**效果好得多**——因为完整工具定义可能很长，而名字 + 一句话描述只是几个 token。

## 原则三：工具输出要有预算（不截断，存外部）

这条直接呼应 `KNOWLEDGE/agent/context-engineering/`——**上一节讲对话历史怎么压缩，这一节讲工具输出怎么控制**。

Cloud Code 里**每个工具都有输出大小上限**：

- 大部分工具：**10 万字符**
- 搜索工具：**2 万字符**（更严格）
- 文件读取工具：**无穷大**（特殊原因，下面说）

### 反事实：为什么不直接截断

直觉做法：超过上限就截断。

**Cloud Code 的做法不是截断**——截断意味着**信息丢失**，模型可能因为看不到关键信息而做出错误判断。

它的做法是：**把完整输出存到磁盘文件，然后给模型一个预览 + 一个文件路径**。

- **预览**：输出的前几千字节，让模型大致了解内容
- **指针**：如果模型需要完整内容，可以用 `read` 工具去读那个文件

**这和 `KNOWLEDGE/agent/context-engineering/` 第 3 层 Just-in-time Context 是同一思路**——上下文里放的不是信息本身，而是获取信息的能力。

### 为什么 read 工具的上限是无穷大

如果文件读取工具的输出也被存到磁盘，**模型读回来的时候又会触发存储，形成死循环**。

所以 `read` 工具用另一种方式控制输出——**它有自己的 token 预算检查**：超过预算直接报错，**告诉模型用偏移和行数参数来分段读取**。

这是一个典型的"特殊工具特殊处理"——大多数工具靠"超长就外存"，**但负责把外存读回来的工具必须用别的机制**，否则会自指。

### 搜索工具的额外保护：默认 250 条上限

原码注释说得很直白：

> **无界搜索可以轻松填满上下文**。

250 条对于探索性搜索已经足够了。如果模型确实需要更多，**可以显式传入参数解除限制**——但默认是保守的。

### 反事实：如果不控制工具输出大小

你的每一个工具都应该有输出预算。**不要假设工具输出是可控的**：

- 一个数据库查询可能返回几万行
- 一个日志搜索可能返回几十万字符

**一次工具调用就能把整个上下文窗口撑爆，后面所有推理都会受影响**。

最简单的做法：设字符数上限，超过就截断 + 附提示"输出已截断，共 N 条结果"。**更好的做法**：像 Cloud Code 一样存外部 + 给摘要 + 给指针。

## 三个原则指向同一个核心认知

回过头看，三个原则都在做一件事——**管理上下文**：

| 原则 | 它在管什么 |
|---|---|
| 原则一（万能工具让位） | **管模型的注意力**——引导模型走受控路径，而不是让它自由发挥 |
| 原则二（按需发现） | **管上下文空间**——不把不需要的工具定义塞进上下文，把空间留给真正有用的信息 |
| 原则三（输出预算） | **管上下文质量**——防止一次工具调用的输出淹没其他所有信息 |

这就是 `KNOWLEDGE/agent/context-engineering/` 里那句话在工具层面的延续：

> **上下文窗口是稀缺资源。关键不是装不装得下，而是该不该装进去**。

工具设计是这个原则的另一个战场——**你的工具定义占多少上下文 / 你的工具输出占多少上下文 / 你的工具引导模型走什么路径**——这些问题和"对话历史怎么压缩、记忆怎么提取"是同一类问题。

**好的工具设计不是给模型更多能力，是给模型更清晰的边界**。

## Open Questions

- **`tool_search` 的"两阶段加载"在开源 agent 框架里怎么落地**？大部分框架（LangChain / AutoGen / CrewAI）目前还是一次性把所有工具定义都塞进去。如果只用标准 API，做不到 "工具定义动态加载到上下文"，这套方案怎么近似？
- **search_hint 的"3-10 词"是怎么调出来的**？为什么不是 1 词或 50 词？这个数字背后是什么实验？本节点没碰到。
- **"工具输出存外部 + 给摘要 + 给指针" 的摘要怎么生成**？是模型生成的还是规则生成的？如果是模型，那这是一次额外调用——成本不低；如果是规则（比如取前 N 字节），那对于结构化输出可能效果很差。Cloud Code 怎么处理这个 trade-off？
