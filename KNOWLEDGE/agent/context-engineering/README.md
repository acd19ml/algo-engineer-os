# Context Engineering

## 它要解决什么问题

面试常被问 "agent 开发里大家都在说的上下文工程到底是什么？"

如果只回答 "把相关的信息塞进 prompt 里"，面试官会追问 "那这和 prompt engineering 有什么区别？任何调用大模型的应用不都在组装 prompt 吗？为什么 agent 要单独拎出来叫一个工程？"

要讲透，得先看清两件事的根本差别——以及为什么 **agent 特别需要上下文工程**。

## 朴素直觉为什么不够：上下文 ≠ prompt

大模型推理时能用到的信息只来自两个地方：

- 参数里的知识（训练阶段学到的）
- 上下文窗口里当前放着的内容

参数里的知识在推理阶段改不了，**所以你唯一能控制的就是上下文窗口里放什么**。

**上下文工程本质上是在构建大模型的工作记忆**——你决定它在做决策的那一刻能看到什么，你就决定了它的行为质量。

Anthropic 的官方定义：

> 构建 LLM 应用，已经从 "怎么找到正确的措辞" 变成回答一个更宏观的问题——**在模型生成的那一刻，什么样的上下文配置最可能产生期望的行为**？

和 prompt engineering 的区别：

| | 关注什么 |
|---|---|
| **Prompt Engineering** | 怎么说——措辞、格式、few-shot 例子怎么写 |
| **Context Engineering** | 看到什么——每一次推理时上下文窗口里应该有 / 不该有 / 什么结构的信息 |

**对单轮问答应用，两者差别不大**。但对 agent 来说，差别巨大。

## 反事实：为什么 agent 特别需要上下文工程

Agent 不是单轮的。一个 agent 执行任务可能经过十几轮甚至几十轮推理循环——Manus 团队的数据是**一个典型任务平均要调用 50 次工具**。每一轮产生的工具调用参数、返回结果、模型中间推理都会**累积**在上下文里。

现在的前沿模型上下文窗口已到百万 token，**但窗口大不等于问题解决了**。Manus 的经验：随着上下文变长，**模型性能会明显下降**——不是因为放不下，而是因为**噪声太多，关键信息被淹没了**。

Anthropic 给的精炼指导原则：

> 上下文工程的目标是**找到最小的高信噪比的 token 集合，最大化期望结果的概率**。

注意——**不是尽可能多塞信息，而是找到最小的高质量集合**。这是反直觉的，但正是 "工程" 这个词的核心含义：精确的设计和权衡。

## 六层工程在做什么

具体到工程实践，分六层。

### 第 1 层：Compaction（压缩重启）

**Anthropic 推荐的第一号技术**，最基础的一道防线。

做法：当对话接近上下文窗口上限时，用模型自己对当前上下文做一次摘要总结，然后用这份摘要重新初始化一个新的上下文窗口——**相当于模型写了一页笔记，然后翻开新的一页继续工作**。

关键在选择保留什么、丢弃什么：

- **先最大化召回**——确保每一条重要信息被捕获
- **再迭代提升精确度**——一个容易入手的优化是把已经完成的工具调用和原始返回结果清理掉，**只保留结论**

Claude Code 在长任务执行过程中会自动触发 Compaction，把之前的对话压缩成摘要，然后在新上下文里继续，**用户甚至感知不到这个过程**。

### 第 2 层：Structured Note Taking（外化记忆）

Compaction 解决的是"上下文太长怎么办"。但还有一类信息——它不属于当前这一轮对话，但 agent 在后续步骤甚至后续会话中还需要用到。**这类信息不能只靠压缩保留在上下文里，需要写到外部**。

做法：让 agent 在执行过程中**把关键信息主动写到文件系统里**：

- Claude Code 创建 To-Do List 追踪任务进度
- Manus 维护 Notes 类文件记录中间结论

Anthropic 的生动例子：**Claude 玩宝可梦时要在数千个游戏步骤中保持连贯策略**——靠的就是在外部文件里维护精确的技能 / 训练进度 / 战斗策略 / 已探索区域地图，**不塞在上下文窗口里，需要时从文件读取**。

更深层的洞察（Manus）：**把文件系统当作"终极的无限上下文"**——代码和数据保存在文件里、原始搜索结果存入文件而不是留在对话中。**活跃的上下文里只保留结论和下一步操作**。文件系统不是辅助工具，**是上下文架构的核心组件**。

### 第 3 层：Just-in-time Context（即时加载）

传统 RAG 的思路：用户提问之后、模型回答之前，**一次性**把所有可能相关的信息检索出来塞进上下文。

但在 agent 场景里，**你没办法提前知道 agent 在第 10 步会需要什么信息**。

Anthropic 推荐的做法：agent **不预加载**所有数据，而是**维护轻量级的标识符**——文件路径、数据库查询、网页链接。需要时 agent 用工具动态加载到上下文，用完就在下一次 Compaction 时清理。

Claude Code 是典型：分析大型代码库时**不会把所有代码一次性塞进上下文**，而是用 grep 和 glob 命令即时搜索、按需读取。CLAUDE.md 在启动时注入上下文提供全局指引，**但具体的代码文件是 agent 在推理过程中自己决定什么时候去看的**。

本质：**上下文里放的不是信息本身，而是获取信息的能力**。

### 第 4 层：Multi-Agent Architecture（上下文隔离）

当一个任务足够复杂时，即使做了压缩、外化、即时加载，**单个上下文窗口也可能不够用**——或更准确地说，**不是放不下，而是不同子任务的上下文会互相干扰**。

解决方案：多 agent 架构。**主 agent 负责规划和分配任务，每个子 agent 拥有自己独立的、干净的上下文窗口**。

Manus 的架构：

- Planner 拆分任务
- Knowledge Manager 审查对话，决定什么该保存到文件系统
- Executors 负责执行具体任务

每个子 agent 的上下文里**只有它需要关注的信息**，不会被其他子任务的细节污染。

Claude Code 也从单 agent 演进到多 agent——它会启动子 agent 处理独立子任务，子 agent 完成后**只把结果返回给主 agent，中间过程不会污染主 agent 的上下文**。

可以理解为 **分治法在上下文层面的应用**。

### 第 5 层：Tool Design（工具设计）

这一层容易被忽略，但 Anthropic 明确指出：**工具设计是上下文工程的核心组成部分**。

为什么？

- 工具的定义本身就**占据上下文空间**
- 工具的返回结果是**上下文膨胀的主要来源**
- **工具集大小直接影响模型的决策质量**

Manus 的反直觉发现：**更多的工具可能让你的 agent 变蠢**。随着可选工具增多，模型更可能选择次优操作或走低效路径。Anthropic 也说："**如果一个人类工程师都无法确定在特定情况下该使用哪个工具，AI agent 也不会做得更好**。"

但这里有个**工程上的微妙之处**：你可能觉得"那就动态增删工具呗，当前不需要的就移除"。

**Manus 发现不能这么做**——因为大多数 LLM 实现里**工具定义位于上下文的前部**。如果中途增删工具，工具定义变了，**后面所有内容的 KV Cache 全部失效**，需要重新计算，导致巨大延迟和成本损失。

Manus 的解决方案：**工具定义保持不变，但通过操纵模型的 logits 来 mask 当前不需要的工具**——模型在解码时根本选不到被 mask 的选项，**但工具定义本身没动，KV Cache 保持完整**。

### 第 6 层：Cache-Friendly Architecture（缓存友好架构）

把上下文工程从"能跑"推向"能上生产"的关键一步。Manus 团队的话：

> 如果只能选一个指标来衡量生产级 AI Agent 的质量，**那就是 KV Cache 命中率**。它同时决定了延迟和成本。

经济账（以 Claude Sonnet 为例）：

- 不带缓存的输入：每百万 token $3
- 启用 Prompt Caching 后：$0.30
- **省 90%**

但比省钱更重要的是——**围绕缓存来设计架构会从根本上改变你思考上下文的方式**。

核心原则：**上下文的结构要尽量是追加式的**。

- 稳定内容（系统指令、工具定义）放在上下文最前面，形成**固定前缀**
- 每一轮新产生的内容**追加在后面**

每次推理时**前缀部分的 KV Cache 完全复用**，只需计算新追加的部分。

反过来，如果在中途修改了前面的内容（动态增删工具、重写系统指令），前缀变了，**后面所有内容的缓存全部作废**——代价灾难性。

这就是为什么前面说工具设计要用 **masking 而不是增删**——它不只是工程技巧，背后是整个缓存架构的约束。

## 两个容易被忽略的加分点

### 保留错误信息

直觉上你会觉得 agent 执行过程中犯了错应该把错误步骤清理掉给模型一个干净的上下文。

**但 Manus 的经验相反：不要擦除失败**。**因为失败就是证据**——模型看到之前的错误才能避免重复同样的错误。**清理掉错误等于移除了学习信号**。

### Todo 复述对抗 Lost in the Middle

Manus 发现长任务执行过程中 agent 容易**忘记自己的总体目标**——不是因为信息丢了，**而是因为目标被埋在了上下文的中间，模型的注意力没有对准它**。这就是 "Lost in the Middle" 现象。

解决方案很简单：**每一步结束后让 agent 重写它的 to-do list，把当前目标和剩余任务刷新到上下文末尾**。这利用了模型对末尾内容注意力更强的特性，**用自然语言偏置模型的注意力焦点**——不需要任何架构改动。

## Open Questions

- 六层之间的**优先级排序**：在固定预算下应该先做哪一层？文章按"基础 → 进阶"列了顺序，但实操中决策依赖很多变量（任务类型、模型 / 框架选型、上下文长度上限），没有给统一指南。
- **Cache 命中率的具体测量方法**：怎么在实际系统里量化、监控？文章说"如果只能选一个指标"，但**测量方法**没展开。
- 第 5 层用 logit masking 替代工具增删——这要求**对 inference 接口有底层控制权**。如果只用 OpenAI / Anthropic 的标准 API（没有 logit_bias 的细粒度控制），这套方案怎么落地？这条线本节点没碰到。
- "保留错误信息"和"Compaction" 看起来矛盾——Compaction 要清理冗余，错误信息也是冗余但要保留。**两者的边界在哪**？没说清。
