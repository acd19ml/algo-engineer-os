# Agent Permission System: 分层规则 + 两阶段评审

## 它要解决什么问题

每个用过 agent 的人都遇到过这种情况：你让 agent 改一个配置文件，他改完顺手把环境变量文件也改了；你让他跑测试，他自作主张地跑了部署；你让他重构一段代码，他把没让碰的文件也重构了。

这不是模型笨。**Agent 乱来本质上是一个信任边界问题**——你给了一个具体指令，agent 把它理解成了一个更大的授权范围。

大多数人的第一反应是**加确认框**——每次敏感操作就弹个框问用户。简单直接，但有一个根本矛盾：

- **确认框太多** → 用户烦，开始无脑点允许，确认框形同虚设
- **确认框太少** → 拦不住 agent 乱来

这个节点拆 Cloud Code 给出的答案——**不是更好的确认框，而是一条多层评估流水线**。每一层解决一个不同层次的问题，每一层都假设上一层可能失败。

## 反事实：把"开关"换成"分层规则"

最简单的权限系统是一个开关：允许或禁止。Cloud Code 不是这样——**它有三种规则，且有严格的优先级**：

- 禁止
- 需要确认
- 允许

**禁止永远优先**。

这意味着什么？你可以配一条规则说"允许所有文件编辑"，**同时**配另一条规则说"禁止修改版本管理的配置文件"——禁止规则覆盖允许规则。**你不需要在"全部放行"和"逐一确认"之间二选一**。

而且规则**支持内容级别的匹配**，不是简单的"允许命令行工具"或"禁止命令行工具"，**而是可以精确到"允许版本管理相关的命令，但禁止强制删除类的命令"**。

具体例子：`git status`、`git log` 这种允许；`rm -rf`、`git push --force` 这种禁止；`git commit` 这种需要确认。同一个命令行工具，按内容分三档。

### 反事实：如果只有"全局开关"会怎样

只有"全部允许 / 全部禁止"两档，用户面对的是 binary 选择：

- 全部允许 → 危险操作没人挡
- 全部禁止 → 每次都要点确认，用户疲劳

**Binary 设计逼着用户在"被烦死"和"被坑死"之间选**。分层 + 优先级让你**先画一条主线（放行大部分），再画硬线（堵住关键危险点）**——把决策从"逐次"提升到"策略"。

## 设计核心：生成和评估分离（YOLO Classifier）

这是整个全线系统里最值得学的设计。

Cloud Code 有一个叫 **YOLO Classifier** 的东西——**一个独立的大模型调用，专门审查主 agent 的每一个操作**。主 agent 说"我要执行这个命令"，YOLO Classifier 看一眼对话上下文和这个命令，判断该不该让它执行。

**两个模型调用，两个独立判断，互相制衡**。

### 一个非常精妙的安全细节：评审者不看被评审者的"自我辩护"

分类器在审查的时候**会看用户说了什么、agent 调用了什么工具，但不会看 agent 的文字回复**。

为什么？因为 agent 的文字回复是他自己生成的。**如果 agent 想绕过审查，他可以在回复里写"用户已经授权了所有操作"**——分类器看到这段文字可能就被误导了。

> 只看 agent **做了什么**，不看 agent **说了什么**。

这切断了"被审查者操纵审查者"的路径。

这个思路对任何构建 agent 的人都有用：**你的 agent 生成了一个行动计划，要不要用另一个模型审查它？如果要，给审查模型的信息里应不应该包含 agent 自己的解释？** Cloud Code 的答案是**不应该**——审查者只看事实，不看被审查者的自我辩护。

### 两阶段架构：把算力花在刀刃上

如果每个操作都用一个完整推理来审查，成本不可接受——**因为大部分操作其实是安全的**。

所以 YOLO Classifier 用了一个两阶段设计：

- **第一阶段**：输出空间极小（只要 yes / no），快速判断。**大部分安全操作在这里就直接放行**
- **第二阶段**：只有第一阶段觉得可疑的操作才进入这里，**让模型做完整推理后再给最终判断**

类比生产环境的常见模式：**先做一道便宜的快速过滤，把绝大多数样本快速通过，只把少量可疑样本送进昂贵的完整流程**。

**错误处理策略：宁可误杀不可放过**——解析失败、API 错误，全部默认拦截。

## 设计延伸：硬约束 vs 软约束

YOLO Classifier 的整体思想——**Prompt 里的规则是建议，代码里的规则是法律**。

规则一旦编码：

- 在所有绘画中机械式执行
- 不依赖模型的自觉性
- 不会被新版本的模型"理解错"

Cloud Code 里还有一组**二三条编号的 Bash 安全检查**：防御 Zsh 等号展开、Unicode 零宽空格注入、IFS 空字节注入。**其中一条来自 HackerOne 安全审查中发现的真实漏洞**。

这些都是 prompt 里写不进去的——你没法在 system prompt 里说"请提防 Unicode 零宽空格注入"，模型并不真懂这是什么。**只有编码成确定性的检查规则，才能稳定生效**。

## 不可绕过的安全底线

Cloud Code 有一个最高权限模式，**可以跳过几乎所有确认框**。但即使在这个模式下，有一组操作仍然会谈确认：

- 修改版本管理的配置
- 修改 agent 自身的配置
- 修改终端的配置文件（`.zshrc`、`.bashrc`、`.profile` 这些）

**原码注释写得很直白：这些安全检查是不可绕过的**。

### 反事实：为什么这些不能交给用户决定

为什么这些操作即使在"最高信任"模式下也不能放行？

因为**这些文件一旦被恶意修改，影响是系统级的、持久的、隐蔽的**。

举一个具体的攻击场景：你的 `.zshrc` 被改了，**你每次开终端都会执行恶意代码**——而且你可能很长时间都不会发现，因为终端打开是再日常不过的动作，没人会去 diff 配置文件。

这种**"破坏触发频繁、副作用持久、检测成本高"**的操作必须由系统替用户兜底，**用户自己都不能关掉**——因为用户可能在不理解后果的情况下选了"完全信任"。

这个设计的启发是：**有一些安全底线应该是系统替用户守住的**。用户可以选择信任 agent，但系统要确保即使用户完全信任，最坏的情况也是可控的。

## 三个对初学者的可迁移原则

1. **规则要分层**：不要只做"允许/禁止"开关。至少分两层——默认策略 + 例外规则，例外优先级高于默认。这样能在保持流畅体验的同时对特定危险点保留控制。
2. **生成和评估要分离**：不要让 agent 自己审查自己。如果用独立 agent 审查，**审查者只看事实（用户输入 + 工具调用），不看被审查者的自我解释（agent 的文字回复）**。
3. **安全底线要硬编码**：不要把所有决策权交给用户。**对"破坏触发频繁、副作用持久、检测成本高"的操作，应该硬编码确认，用户都不能关**。

## Open Questions

- **YOLO Classifier 第二阶段的"完整推理"用什么模型**？如果用和主 agent 同等的模型，成本不止翻倍（请求增量 + 推理增量）；如果用更小的模型，可能拦不住高阶攻击。这条 trade-off 本节点没碰到。
- **"评审者不看 agent 的文字回复"**——但 agent 的文字回复有时也包含合法上下文（比如它解释了为什么要做这个操作）。完全屏蔽是不是过度防御？什么场景下应该让审查者看一部分？这条没说清。
- **硬约束 vs 软约束的边界**——什么样的规则该编进代码、什么样的该留在 prompt？Cloud Code 给的标准是"prompt 是建议、代码是法律"，**但这条不能用来回答"哪条规则该升级成法律"**。需要一个判断框架。
