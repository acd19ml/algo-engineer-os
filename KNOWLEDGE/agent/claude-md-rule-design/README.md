# CLAUDE.md 规则设计：用户侧行为合约，不是许愿清单

## 它要解决什么问题

CLAUDE.md 是 AI 编程栈里最被低估的文件。大多数开发者的 CLAUDE.md 处于三种状态之一：

- 什么都往里塞，膨胀到 4000+ token，**遵循率掉到 30%**
- 完全跳过，每次手动 prompt——**5 倍 token 浪费**，会话之间零一致性
- 复制一个模板就忘了，用两周还行，**代码库一变就静默失效**

Anthropic 官方文档说得很清楚：**CLAUDE.md 是建议性的，Claude 大约遵循 80%**。超过 200 行，遵循率急剧下降——因为重要规则被埋在了噪音里。

这个节点拆 Mnilax 在 Karpathy 4 条规则基础上 **6 周 × 30 个代码库 × 50 个标准任务** 实测出的 8 条新规则——把错误率从 **41% 压到 3%**。但更重要的是它揭示了**写 CLAUDE.md 是一种特殊的工程实践**：你不是在写文档，你是在和模型签一份**行为合约**——每条规则都必须能回答 "这条规则预防的是什么具体失败？"

## 朴素直觉为什么不够

最直觉的做法是 "想到啥写啥"：编码规范、目录约定、技术栈说明、注释要求、依赖列表……结果文件膨胀到 4000+ token，**Claude 只做模式匹配"规则存在"，不真正去读**。

Mnilax 实测了规则数量 vs 遵循率：

| 规则数 | 合规率 | 错误率 |
|---:|---:|---:|
| 4 条（Karpathy 原版） | 78% | 11% |
| 12 条（Mnilax 扩展） | 76% | **3%** |
| 18 条（继续加） | **52%** | 回升 |

**12 条是 sweet spot**。从 4 → 12 几乎没增加合规执行开销（78% → 76%），但**错误率从 11% 降到 3%**——因为新加的规则**覆盖了原始 4 条没触及的失败模式**，不在争夺同一份注意力预算，而是形成互补增强。

200 行 / 12 条是真实天花板，过了之后**模式匹配替代真正阅读**。

## Karpathy 原版 4 条（底线，不是天花板）

2026 年 1 月底 Andrej Karpathy 吐槽 Claude 写代码的三个问题：**静默错误假设 / 过度复杂化 / 波及非目标代码**。Forrest Chang 把这些问题提炼成 4 条行为规则放进 CLAUDE.md。文件第一天 5582 颗星，两周 6 万书签，到 5 月 12 万星——**2026 年增长最快的单文件仓库**。

四条规则覆盖了无监督 Claude Code 会话中 **~40% 的失败模式**：

1. **先想后写**——杜绝隐蔽假设。明确声明前提，阐明权衡。有疑问先确认，切忌盲目猜测。若存在更简单的实现路径，应果断提出异议
2. **简单至上**——仅以最少代码解决问题。不添加 "以防万一" 的冗余功能。不为仅用一次的代码强行设计抽象层。资深工程师会觉得过度复杂就立即简化
3. **外科手术式修改**——仅改动绝对必要的部分。切勿"顺手优化"相邻代码、注释或格式。未出问题的代码绝不重构
4. **目标驱动**——明确定义成功标准（验收条件）。持续迭代直至验证通过。不要规定具体操作步骤，只需清晰描述"成功的最终形态"

但 5 月的 Claude Code 生态有了 **1 月不存在的新病灶**：Agent 冲突 / Hook 级联 / Skill 加载冲突 / 跨会话断裂的多步工作流。原版没覆盖这些——剩余 60% 失败模式在这里。

## 反事实：Karpathy 原版的 4 个 limitation

| 维度 | 缺什么 | 后果 |
|---|---|---|
| **长任务 Agent** | 预算约束 / 状态检查点 / "显式报错"机制 | 在多步流水线里**语义漂移**，最终偏离预定目标 |
| **多代码库一致性** | 在 monorepo 数十个微服务里没有决策优先级 | 多种代码风格冲突时**随机采样**或产出**平庸的"风格平均值"** |
| **测试质量** | 把"测试通过"和"任务成功"画等号 | 大量逻辑空洞测试用例——全绿通过但毫无价值，**虚假安全感** |
| **生产 vs 原型** | "简约至上"在原型阶段成为枷锁 | 早期探索性脚手架被**过度触发**，抑制试错速度 |

这四个 limitation 给出了新增 8 条规则的设计目标：补上长任务 / 多代码库 / 测试质量 / 阶段感知。

## 新增的 8 条（每条来自一个真实翻车场景）

### 规则 5 — 模型只做判断，不做决策

**适用**：分类、内容起草、摘要总结、非结构化文本信息提取  
**禁止**：路由分发、重试机制、状态码处理、确定性数据转换

> 翻车现场：一段调用 Claude 去"判断 503 该不该重试"的代码稳定运行两周，然后开始出问题——**模型开始把请求体内容当成决策的上下文**。重试策略变成了随机的，因为 prompt 是随机的。

本质：**确定性逻辑用代码，判断与裁量用模型**。Karpathy 4 条对此只字未提，结果模型越俎代庖去裁决本该由确定性代码处理的事——你实际上在以 token 0.003 美元/个的成本运行一套**脆弱不稳定的 if-else 逻辑**。

### 规则 6 — Token 预算硬性约束，没有例外

- 单任务预算：**4,000 token**
- 单会话预算：**30,000 token**
- 接近上限时**立即摘要 + 重置上下文**

> 翻车现场：一个 90 分钟调试会话。模型对着同一条 8KB 错误日志反复迭代，逐渐丢失了已尝试方案的上下文记忆。**对话末尾它建议我早在 40 条消息前就明确否决的方案**。有 token 预算的话 12 分钟就会被强制熔断。

反事实：**未设预算的 CLAUDE.md = 一张空白支票**。每次循环都可能失控，最终螺旋膨胀为 5 万 token 上下文灾难。模型自身**绝不会主动终止**。

### 规则 7 — 暴露冲突，不要折中

代码库中既有的两种模式相互矛盾时，**切勿强行融合**。明确选择其一（优先更新或经过更充分测试的版本），阐明选择理由，将另一种标记为待清理项。

> 翻车现场：代码库并存两种错误处理（async/await + try/catch 局部 vs Global Error Boundary 集中拦截）。Claude 新代码**将两者强行叠加**，错误处理逻辑翻倍。30 分钟才查清楚为什么异常被重复捕获并静默"吞掉"两次。

**试图同时迎合两套规则的"折中"代码，往往是最糟糕的代码**。

### 规则 8 — 先读后写

追加代码前**必须通读该文件的导出接口、直接调用方、公共工具函数**。无法理解现有代码为何采用当前结构，**必须先提问确认**。

> 翻车现场：Claude 在一个已有相同功能的函数旁边加了一个新函数——**它没读到那个已有函数**。两个函数做一样的事。新函数因为导入顺序而优先执行。旧的那个已经作为唯一真相来源运行了 6 个月。

> "在我看来这部分跟现有代码互不干涉（Looks orthogonal to me）"——**本代码库中最危险的一句话**。

### 规则 9 — 测试验证意图，而非仅验证行为

每个测试**必须明确体现该行为为何重要（WHY）**，而不仅断言它做了什么（WHAT）。

```js
// ❌ 没用的测试
expect(getUserName()).toBe('John')   // 函数内部硬编码返回 'John' 也能过

// 真测试要求：函数内业务逻辑变更时测试必须失败
```

> 翻车现场：Claude 为某认证函数编写 12 个测试，全部通过。但生产环境的认证功能**彻底失效**。测试仅验证"函数是否返回了数据"，而非"是否返回了正确数据"——全通过纯粹是因为函数内部直接硬编码返回了一个固定常量。

### 规则 10 — 长任务必须设检查点

多步任务每完成一个关键节点后**必须执行状态同步**：明确总结已实施操作、已验证结果、剩余待办事项。无法清晰描述当前上下文与执行状态，**绝不可继续推进**。

> 翻车现场：一个 6 步重构在第 4 步出了问题。等我发现时 Claude 已经在错误状态上继续完成了第 5、6 步。**拆解比整个重做还花时间**。检查点机制会在第 4 步就抓住问题。

### 规则 11 — 约定胜于新奇

在已有规范的代码库中，Claude 总是倾向另起炉灶。即便其方案在局部技术上"更优"，**两种范式共存所带来的维护成本与认知割裂，远大于固守其中任何一种**。

- 项目用 snake_case 而你偏好 camelCase → 用 snake_case
- 项目用 class component 而你倾向 hooks → 沿用 class component
- 技术分歧另开议题讨论，**代码库内部规范一致性 > 个人技术偏好**

> 翻车现场：Claude 在一个 class 组件代码库里引入了 React hooks。能跑。但搞坏了**测试模式假定的 `componentDidMount`**。花了半天删除和重写。

### 规则 12 — 显式失败，不要静默

> 最贵的 Claude 失败是那些**看起来像成功的**。一个函数 "能跑" 但返回了错误数据。一次迁移 "完成" 但跳过了 30 条记录。一个测试 "通过" 但只是因为断言本身就写错了。

约束：
- 无法确信操作正确执行 → 必须明确声明
- 30 条记录被静默跳过 → 宣称"迁移完成"便是**错误结论**
- 任意测试被跳过 → 宣称"测试通过"便是**错误结论**
- 未验证明确要求的边界情况 → 宣称"功能正常"便是**错误结论**

默认原则：**主动暴露不确定性，绝不掩盖**。

> 翻车现场：Claude 说一次数据库迁移"成功完成"。它**悄悄跳过了 14% 因为约束冲突没处理的记录**。跳过虽然记了日志但没被暴露。**11 天后报告数据开始对不上时才发现**。

## 试过但放弃的 6 条路径（反事实证据）

Mnilax 在最终确定 12 条之前试过的失败方向：

| 失败方向 | 为什么不行 |
|---|---|
| 从 Reddit/X 看来的"网友规则" | 大多是 Karpathy 4 条的翻版，或不可泛化的领域特化规则（如 "始终用 Tailwind class"）—— 删 |
| 超过 12 条 | 测到 18 条时合规率从 76% 掉到 52%。**200 行天花板是真的**——过了之后 Claude 只做模式匹配"规则存在"，不真正去读 |
| 依赖不一定存在的工具 | "始终用 eslint"——eslint 没装的时候规则静默失效。**替换为跟能力无关的措辞**："匹配代码库强制执行的风格" |
| 在 CLAUDE.md 里放例子 | 例子比规则重。**3 个例子消耗的上下文 ≈ 10 条规则**，且 Claude 会过度拟合。规则是抽象的，例子是具体的——**用规则** |
| 用 "小心" / "多想" / "真正专注" 这类指令 | **纯噪音**。合规率掉到约 30%——因为它们**不可测试**。替换为具体祈使句（"明确陈述你的假设"） |
| 让 Claude "做高级工程师" | 没用。**Claude 已经觉得自己是高级的**。差距在"想"和"做"之间——祈使句规则能闭合差距，身份提示不能 |

**这六条比正面 12 条还有价值**——它们是 prompt 设计领域少见的、由系统化测试得到的负面证据。

## 41% → 3% 错误率的真正含义

| 规则数 | 合规执行率 | 失败率 |
|---:|---:|---:|
| 0 条（裸用 Claude Code） | n/a | 41% |
| 4 条（Karpathy 原版） | 78% | 11% |
| 12 条（Mnilax 扩展） | 76% | **3%** |

数据真正值得关注的**不是头条降幅**，而是：

**规则从 4 扩展到 12，几乎未增加合规执行开销（78% → 76%），却将错误率进一步降低了 8 个百分点**。新增规则覆盖了原始 4 条没触及的失败模式——它们**并非在争夺同一份注意力预算，而是形成了互补增强**。

这给出了 CLAUDE.md 设计的一条核心原则：**新规则的边际价值看的是"覆盖的失败模式 是否与已有规则正交"**，而不是规则总数。如果一条新规则跟已有规则争夺同一份注意力预算，加它反而会让整体合规率下降。

## 12 条规则完整模板（可粘贴）

```markdown
# CLAUDE.md — 12 条规则模板
除非显式覆盖，否则本规则适用于本项目中的所有任务。
核心倾向：非琐碎工作，谨慎优先于速度；琐碎任务可自主判断处理。

## 规则一：先思后码（Think Before Coding）
明确声明前提假设。遇不确定处，先提问而非盲目猜测。
存在歧义时，列出多种可能的理解路径。
若存在更简方案，应果断提出异议。
陷入困惑时立即暂停，并明确指出模糊之处。

## 规则二：简单至上（Simplicity First）
仅用最少代码解决问题。杜绝任何"以防万一"的猜测性实现。
不实现需求之外的功能。不为仅用一次的代码强行设计抽象。
自检：资深工程师是否会认为此实现过度复杂？若是，立即简化。

## 规则三：外科手术式修改（Surgical Changes）
仅改动绝对必要的部分。仅清理自身引入的冗余或错误。
切勿"顺手优化"相邻代码、注释或排版格式。
未出问题的代码绝不重构。严格贴合项目既有风格。

## 规则四：目标驱动执行（Goal-Driven Execution）
明确定义成功标准（验收条件）。持续迭代直至验证通过。
不要死板遵循步骤。定义成功形态并自主迭代。
清晰的成功标准赋予你独立闭环执行的能力。

## 规则五：仅将模型用于判断与裁量场景（Use the model only for judgment calls）
适用于我：分类、起草、摘要总结、信息提取。
切勿用于：路由分发、重试机制、确定性数据转换。
若常规代码能给出答案，就由代码处理。

## 规则六：Token 预算绝非软性建议（Token budgets are not advisory）
单任务上限：4,000 Token。单会话上限：30,000 Token。
接近预算上限时，执行上下文摘要并重置状态。
主动暴露超支。切勿静默越界消耗。

## 规则七：显式暴露冲突，拒绝折中调和（Surface conflicts, don't average them）
若两种模式相互矛盾，明确择一（优先更新或更经测试的版本）。
阐明选择理由。将另一处标记为待清理项。
切勿强行融合冲突范式。

## 规则八：落笔前先阅读（Read before you write）
添加代码前，通读该文件的导出接口、直接调用方及公共工具函数。
"看似互不干涉"是最危险的判断。若不理解现有代码为何如此设计，先提问。

## 规则九：测试验证意图，而非仅验证行为（Tests verify intent, not just behavior）
测试必须体现该行为*为何重要*（WHY），而非仅断言它*做了什么*（WHAT）。
若业务逻辑变更时测试仍不报错，则该测试设计错误。

## 规则十：关键步骤后强制设立检查点（Checkpoint after every significant step）
总结已完成事项、已验证结果及剩余待办。
若无法向我清晰描述当前状态，绝不可继续推进。
若丢失上下文或逻辑偏离，立即暂停并重新声明当前状态。

## 规则十一：严格遵从代码库既有规范，即便持保留意见（Match the codebase's conventions, even if you disagree）
在代码库内部：规范一致性 > 个人技术偏好。
若确信某规范存在实质危害，请显式提出。切勿暗中另起范式。

## 规则十二：显式失败（Fail loud）
若有步骤被静默跳过，宣称"已完成"即为错误。
若有测试被跳过，宣称"测试通过"即为错误。
默认原则：主动暴露不确定性，绝不掩盖。
```

项目专属规则（技术栈 / 测试命令 / 错误模式）另外追加，**总行数控制在 200 以内**。

## 两条心智模型

1. **CLAUDE.md 不是许愿清单，是行为合约**。每条规则都应该能回答一个问题：**这条规则预防的是什么错误**？没法回答这个问题的规则就是噪音
2. **6 条针对你真踩过的坑的规则 > 12 条里有 6 条你永远用不上的**。Karpathy 4 条是地基——别跳过。补齐的 8 条选你**实际需要的**：
   - 如果你不跑多步流水线，规则 10 不关你的事
   - 如果你的代码库有 linting 强制执行的统一风格，规则 11 就是冗余
   - 读完全部 12 条，**保留映射到你真实翻车现场的规则**，其余可以不要

## Open Questions

- **遵循率天花板**：78% / 76% 的合规率是 Claude Code 当前版本特有的，还是大模型 RLHF 的某种结构性天花板？跨模型实测过吗？
- **CLAUDE.md vs 项目根目录的 AGENTS.md / .cursorrules**：同样的 12 条规则在 Cursor / OpenAI Codex CLI 上表现如何？规则的"可迁移性"靠什么保证？
- **多代码库一致性的"自适应规则"**：如果一个 monorepo 有 20 个微服务、每个服务有不同的风格——CLAUDE.md 是否需要按子目录分裂为多份？怎么避免规则冲突？规则 11 "约定胜于新奇" 在这种场景下怎么具体落地？
- **跟 `KNOWLEDGE/agent/agent-system-prompt/` 的关系**：那个节点讲的是 Claude Code **内部**的 system prompt 设计（900 行模块化、静态/动态分离）。本节点讲的是 **用户侧** 给 Claude Code 的行为合约。**两层是怎么交互的**？用户的 CLAUDE.md 加载顺序如何影响内部 prompt 的处理？
