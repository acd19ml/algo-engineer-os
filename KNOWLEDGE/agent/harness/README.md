# Harness Engineering

## 它要解决什么问题

最近 agent 开发领域最火的概念叫 **Harness Engineering**。大部分人对它的理解停留在"给 agent 加约束"这个层面——**这远远不够**。

要看清这件事的份量，先看一个反直觉的事实：**LangChain 的 Coding Agent 在 Terminal Bench 排行榜上从 30 名开外冲到前 5——整个过程底层模型一行没换，始终是同一个模型**。他们只动了三个东西：系统提示、工具配置、中间件钩子。

这个结果直接挑战了 AI 开发中一个根深蒂固的假设——**更好的性能需要更大或更新的模型**。LangChain 用实际数据证明：**模型不变的前提下，光靠优化模型周围的系统就能带来数量级的提升**。

他们用的方法论叫 **Harness Engineering**。

## Harness 的命名隐喻：缰绳和马鞍

"Harness" 来自马具——缰绳、马鞍这些用来引导马匹朝正确方向走的装备。这个比喻是**刻意的**：

- 马 = AI 模型（强大、快速，但自己不知道该往哪走）
- 骑手 = 人类工程师（提供方向，而不是亲自跑）
- **Harness = 骑手和马之间的那套控制系统**

最早的简洁定义：

> 每当你发现 agent 犯了一个错误，就花时间设计一个解决方案，让 agent 永远不会再犯同样的错误。

LangChain 给了一个更精炼的公式：

> **Agent = Model + Harness**。模型包含智能，Harness 让这个智能变得有用。

**模型是引擎，agent 是整辆车，harness 是方向盘和刹车**——最好的引擎没有方向盘和刹车，**去不了任何有用的地方**。

## 反事实：Harness 和 Context Engineering 是什么关系

容易混的概念：Context Engineering（上下文工程）。

**一句话讲清**：

- **Context Engineering 问的是**："我们给 agent 看什么？"
- **Harness Engineering 问的是**："系统预防了什么、测量了什么、修复了什么？"

更准确地说：

- 上下文工程**主要关注怎么管理 agent 的上下文窗口**——给它看什么、不看什么、什么时候看
- Harness Engineering 范围**更广**——还包括架构约束、自验证循环、长治理和系统的可演进性

两者**互相包含**。详见 `KNOWLEDGE/agent/context-engineering/`——上下文工程是 Harness 的一个核心子集。

## Harness 包含的六个关键词

### 1. 上下文架构

前沿团队一致发现：**给 agent 塞太多信息反而有害**。研究表明 agent 性能在上下文利用率超过约 40% 之后开始下降。

关键不是给 agent 一本百科全书，**而是给它一张地图，让它按需查找**。

- OpenAI：把 AGENTS.md 控制在大约 100 行，**只充当目录，指向更深层的文档**——agent 需要什么自己去查
- Anthropic：Skills 渐进式加载理念

### 2. 架构约束

大多数人靠 prompt 来约束 agent 行为——写 "请遵循以下规则"。**但 prompt 里的规则本质上是建议，模型可以听也可以不听**。

前沿团队的做法是**用确定性的工具来机械式执行约束**：

- 自定义 Linter
- 结构化测试
- Schema 验证

**规则一旦编码，就在所有 agent 会话中同时生效，不依赖模型的自觉性**。

反直觉的发现：**Vercel 一开始给 agent 提供大量工具（什么都能用），结果 agent 反而变得困惑、做冗余调用**。后来移除 80% 的工具、只留最必要的，**agent 反而更快更可靠**——**约束解空间反而提高了产出**。

### 3. 自验证循环

Agent 有两个常见失败模式：

- **陷入死循环**：对同一个文件反复编辑十几次，但问题始终没解决
- **交付时跳过验证**：第一个看起来合理的方案就直接输出

LangChain 的方案是用**中间件钩子**：

- 一个中间件**跟踪每个文件的编辑次数**，超过阈值就提醒 agent 重新审视方案
- 另一个中间件在 **agent 准备退出时拦截它，强制执行一轮完整验证**

非常有价值的策略：**推理三明治**：

- **规划阶段**：最高推理强度，充分理解问题
- **执行阶段**：降到高等推理强度，保证速度
- **验证阶段**：再拉回最高推理强度捕获错误

全程最高推理强度反而成绩更差——**会超时**。**把算力花在刀刃上，效果最好**。

### 4. 上下文隔离

任务复杂到需要多 agent 协作时，**关键不是按角色分工**（什么"前端 agent / 后端 agent"），而是**把子 agent 当作上下文防火墙**。

副 agent 只看到给子 agent 的指令和子 agent 的最终结果——**中间所有的工具调用和中间产物都被隔离掉了**。每个执行单元的上下文都保持干净，不被无关信息污染。

### 5. 长治理

Agent 持续运行的时间越长，系统的混乱度就越高：文档过时、架构漂移、知识库和代码不一致。

OpenAI 的方案：**引入一个后台运行的"文档梳理 agent"**——定期扫描过时的文档并自动提交修复。**为 agent 服务的文档由 agent 来维护，形成自维护的闭环**。

### 6. 可拆卸性

这是最高维度的一层：**更好的模型会让某些 Harness 组件变成瓶颈**。

- 2024 年需要复杂流水线的任务
- 2026 年可能一个 prompt 就搞定了

所以 Harness **必须是模块化的、可拆卸的**。LangChain 的中间件架构是目前最好的参考——**每个中间件独立添加特定能力，不需要的时候直接移除，不影响其他部分**。

## ROI 的关键认知：复利投入 vs 过度工程化

**Harness 的投入是以复利形式生效的**：

- 今天加一条 Linter 规则 → 之后所有会话中这个错误都被预防
- 今天加一条验证中间件 → 之后所有任务的交付质量都提升

这意味着 Harness Engineering **投入越早，累积收益越大**。

但**同时要警惕过度工程化**。一个很务实的原则：**只在 agent 确实犯过的错误上投入 Harness**——不要为想象中的错误预先建造防御层，那是浪费工程预算。

## Open Questions

- **"40% 上下文利用率开始下降"**这个数字——具体测量方法是什么？不同模型、不同任务的曲线一样吗？这个具体阈值在不同场景下有多稳定，本节点没展开。
- **"推理三明治"** 在闭源 API 上需要 API 支持设置 reasoning effort（OpenAI 的 `reasoning_effort` 或 Anthropic 的 `thinking` 配置）。**没有这种 API 支持的模型上**怎么落地？
- 第 5 层"长治理 agent"听起来像 "用 agent 治理 agent" 的递归——**这个治理 agent 自己会不会出错、谁来治理它**？这条递归的终点在哪？
- 第 6 层 "可拆卸性" 需要持续的"压力测试 / 移除组件 / 看效果"流程。但**生产环境的 agent 通常不能轻易做这种 ablation**——怎么把这种实验文化落到生产工程？
