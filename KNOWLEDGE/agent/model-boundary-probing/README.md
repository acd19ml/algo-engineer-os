# Model Boundary Probing

## 它要解决什么问题

Agent 开发里最容易过时的是具体技巧：某个模型不会用工具，于是写格式适配；某个模型上下文太短，于是做复杂 RAG；某个模型不稳定，于是加很多提示词。模型一升级，这些补丁可能就失效。

但有一类能力不容易过时：拿到一个新模型后，快速判断它在你的场景里能稳定完成什么、会在哪些条件下失控，然后围绕边界设计 Harness。

这叫模型边界探测。它不是问 benchmark 分数，而是问模型在真实 agent 任务中的行为曲线。

## 为什么模型越强，边界问题反而越重要

2026 年模型层的进化开始明显服务长程 agent：百万 token 上下文、更低 KV cache 成本、更快推理、更适合 agent 延迟敏感场景的混合架构。直觉上，模型更强应该让工程更简单。

反事实地看，模型更强确实会淘汰一批补偿性工程。但它也让 agent 能跑更长任务。任务越长，错误越不是单步智力问题，而是连续执行几十步后系统还能不能保持轨道。

典型失控包括：上下文漂移，跑着跑着忘了初始目标；缺乏自验证，犯错后不知道自己错了；没有经验积累，同一个错误反复犯；记忆管理失败，压缩或长上下文导致关键信息丢失；死循环，对同一文件反复编辑但问题没解决。

这些问题的共同点是：单步看模型可能很聪明，长程看系统仍然会失控。

## 前沿 Agent 系统都在补同一类边界

很多看似朴素的机制，其实都在对抗长程边界。

SpecLock 每生成一页 PPT 前重新读取设计规范，是在对抗上下文漂移。Trading Agents 让多空 agent 互相反驳，是在对抗单一视角偏见。Cloud Code 连续失败触发熔断，是在对抗失败循环。Hermes 把有效方案提炼成 Skill，是在对抗经验不积累。压缩前先写记忆，是在对抗信息丢失。

单独看，这些都不像高深算法。但真正的技术含量不在"知道加熔断器"，而在理解 agent 为什么会在长程任务里这样失败：早期目标在长上下文中注意力衰减，压缩丢细节，缺少"我已经试过"的状态记忆，单模型决策容易强化已有证据。

理解失败机制，才知道熔断阈值、锚定力度、辩论介入点、记忆写入时机该怎么选。

## Boundary map 比通用 benchmark 更有用

真实模型边界至少有四维：单步推理准确率、有效上下文利用率、工具调用可靠性、长程任务行为稳定性。

Benchmark 很难告诉你这些。比如一个模型在 25 万 token 下能稳定做信息定位，在 100 万 token 下准确率明显衰减。这条曲线比"上下文窗口标称 100 万"更重要，因为它告诉你什么时候需要外部锚定或检索。

模型边界还会快速变化。上个月需要 Harness 补偿的能力，下个月可能被模型内化；上个月稳定的工具调用格式，新模型可能反而退化。靠背"某模型擅长什么"不够，真正不过时的是快速重测边界的方法。

## 怎么培养边界探测能力

第一，理解底层机制。你不一定要训练模型，但要理解 attention 长序列衰减、KV cache 成本、上下文压缩丢失什么、工具调用协议怎么传递。这样才能从原理预判风险，而不是只靠线上事故。

第二，用不同模型跑真实任务。不是只跑 benchmark，而是跑你的 coding、research、RAG、tool-use 场景，观察它第几轮开始偏、工具调用在哪类字段上错、长上下文下漏掉什么。

第三，读前沿 agent 系统源码。每个中间件、熔断器、记忆机制、工具校验器背后，都是一个已经被验证过的失败模式。读源码不是为了抄代码，而是积累错误地图。

第四，建立自己的模型体检清单。把典型任务压缩成几分钟可跑的探针，每次新模型出来就跑一遍。时间久了，你会有自己的能力地图，而不是依赖厂商宣传。

## 什么不会过时

容易过时的是补某个模型短板的局部技巧。不容易过时的是两件事：对 agent 系统行为的深层理解，以及快速探测模型边界的方法论。

还有一类更稳定：组织内部隐性知识。团队代码审查规范、行业合规要求、项目架构约定，这些不在互联网公开语料里，模型训练不到，始终需要通过 Skills、规范文档、Harness 注入 agent。

一句话概括：模型会换，边界会变，但画边界地图的能力不会过时。

## Open Questions

- 模型体检清单如何避免变成另一个过拟合 benchmark？如果探针太固定，模型或工程师都可能对探针优化，而不是对真实任务优化。
- 边界探测结果如何转化为 Harness 投入优先级？知道模型在第 20 轮漂移是一回事，决定是加锚定、压缩、记忆还是任务拆分是另一回事。
