# AI 产品决策四问: 面试官真正在追问什么

## 它要解决什么问题

AI 方向到今天已经不缺"做过 AI 项目"的人——大家都在卷履历、卷项目数量。**项目大不大、平台响不响，对面试不太重要**。真正决定胜负的是面试官追问"**你当时怎么想的**"时能不能讲扎实。

这套四问框架是某 AI 产品方向团队 leader 给出的内部评估候选人标准。它不是"AI 产品经理才用"的——对任何 AI 工程师讲自己实习项目时同样适用。**核心价值**：把"我做了什么"的项目描述升级为"我每一步决策都能讲清楚"的决策叙事。

## 朴素叙事为什么不够

最容易的项目讲法是 "**我们做了什么**" ——大厂 / 大项目 / 大团队的人一聊就发现全程都在描述功能、技术栈、规模。这种叙事的根本问题是：**它无法区分"参与过"和"理解过"**。

**反事实推导**：如果"我做了什么"是唯一叙事，面试官问"你当时为什么选这个方案不选那个"时**只能打磕巴**——因为这套叙事路径里根本没有"为什么"的存在位置。每个决策点都被压缩成了"我们决定了 X"，背后的权衡和反对意见全部丢失。

反过来：一个候选人就做了一个小小的客服场景，但每一步决策都能讲扎实——**为什么用 AI 不用规则、幻觉怎么控、置信度阈值怎么定、什么条件下必须转人工**——这种人无论项目规模多小都是稀缺的。

所以叙事不能停留在"做了什么"，要进入决策层。**这套四问就是把决策层显性化的工具**。

## 四问框架

### 第一问: 场景判断

> 你拿到一个需求，凭什么觉得这个地方该上 AI？是拍脑袋还是有依据？你有没有见过不该用 AI 但硬上的反面案例？

**核心**：检验你**对 AI 适用边界的判断力**——什么场景上 AI 是杠杆，什么场景上 AI 反而是负担。

**好答案的结构**：
- 描述需求特征（确定性 vs 模糊性、单步 vs 多步、有标注数据 vs 无标注、可容错 vs 不可容错）
- 给出"该上 AI"的具体依据（规则覆盖率上限、人力成本计算、对比传统方法的具体收益）
- **反面案例**：有没有见过"该用规则却硬上 LLM"的场景？讲清楚为什么是反面

**反事实推导**：如果答不出反面案例，说明你**没真正思考过 AI 边界**——只是知道"AI 好用"。这条问句的本质是排除"伪 AI 工程师"。

### 第二问: 风险意识

> AI 不是万能的，尤其大模型会胡说八道这件事，你有没有真正处理过？你是靠提示词压一压就算了，还是在产品层面做了兜底设计？

**核心**：检验你**对幻觉风险的工程化处理能力**——产品层面有没有兜底，不是 prompt 层面假装处理。

**好答案的结构**：
- 具体的幻觉案例（你项目里出现过的、不是抽象的）
- prompt 层做了什么、效果上限在哪
- 产品层做了什么兜底（置信度阈值 / 人工审核环节 / 多源交叉验证 / 结构化输出强制 schema / 失败时的降级路径）

**反事实推导**：如果只说"我们调了 prompt 让它不要胡说"，面试官立刻知道你**没经历过 AI 上线**。线上场景调 prompt 永远无法保证 100%，必须有产品层兜底。

### 第三问: 标准感

> 什么叫"AI 做得好"？你拿什么指标衡量？这个标准是上线前就定好了，还是上线后拍脑袋补的？

**核心**：检验你**事前定义成功标准的能力**——事后拍脑袋补指标几乎等于没指标。

**好答案的结构**：
- 上线前定的具体指标（accuracy / precision / recall / 用户满意度 / 转化率 / 业务指标，看场景）
- 为什么选这个指标（跟业务价值的关系）
- 上线后的实际数字 vs 预期，差异在哪、为什么

**反事实推导**：如果上线后才补指标，**整个迭代周期已经走完**——优化方向只能靠拍脑袋。事前定标准的本质是"用未来的失败定义现在的成功"。

### 第四问: 边界感

> AI 处理不了的问题，你怎么办？转人工的触发条件是什么？这条线是怎么画的、有没有调过？

**核心**：检验你**对系统边界的工程意识**——AI 不是闭环，是有显式 fallback 路径的子系统。

**好答案的结构**：
- 转人工 / 转规则 / 降级响应的**具体触发条件**（置信度低于 X / 输出违反 schema / 调用工具超过 N 次 / 关键字段缺失）
- 这条线最初怎么画的（基于什么数据 / 假设）
- 上线后调过吗（往严格还是宽松调，为什么）

**反事实推导**：如果"边界没画 / 没调过"，要么你做的是 demo（线上不可能没碰到边界）、要么你**没意识到边界是 AI 系统的一等公民**。边界感是产品上线和 demo 跑通的根本分水岭。

## 四问之间的因果链

四问不是孤立的，**形成一个完整的决策叙事链**：

```
场景判断（该上 AI 吗？）  →  风险意识（上了之后幻觉怎么处理？）
                                    ↓
       边界感（处理不了的怎么 fallback？）  ←  标准感（什么叫处理好？）
```

一个完整的 AI 项目叙事**必须四问都能答**。任何一问答不出，对应一个具体的工程盲点：

| 答不出 | 暴露的盲点 |
|---|---|
| 场景判断 | 你没思考过"该不该上 AI"，只是被指派上 |
| 风险意识 | 你没经历过线上幻觉，所有处理停留在 prompt 层 |
| 标准感 | 你没事前定义成功，只在事后补指标 |
| 边界感 | 你做的是 demo，没碰过 production 边界 |

## 如何用四问框架反向重塑你的项目叙事

四问的真正价值是**反向重塑**——拿任何一个旧项目，**用四问框架重新写一遍叙事**，原本干瘪的"做了什么"立刻变成有决策深度的故事。

具体操作：

1. **挑一个你参与最深、踩坑最多的场景**——不是最大、最响亮的，是最深的
2. **把四问写下来**——逐条回答
3. **答不出的部分诚实标注**——"这一问我当时没想清"比假装答出来强
4. **把答不出的部分作为面试时的成长叙事**——"我当时没意识到 X，后来在 Y 时才理解"

这正是 `KNOWLEDGE/methodology/three-tier-decision-docs/` 三类决策文档的"实现设计"层落地——四问对应的是**实现设计 + 反思**的混合叙事。

## 与"挑参与最深的项目"的关系（最易被忽视的核心建议）

原文有一句金句：**"别急着堆项目，挑一个你参与最深、踩坑最多的场景，把决策过程从头到尾理一遍"**。

**反事实推导**：如果按"项目数量"思路准备面试，你会铺很广、每个项目都讲得浅。面试官随便挑一个深问就破防。
反过来如果只准备 1-2 个最深的项目，每一问都答得扎实，**面试官追问什么你都站得住**——这种候选人到哪儿都稀缺。

所以四问框架不是"对所有项目都问一遍"，是**给一个最深的项目做四问压力测试**。

## Open Questions

- **四问之外是否还有第五问值得加**——比如"你的 AI 系统怎么演进 / 留下了什么不该留的技术债务"。原文给的是四问，但 AI 产品在迭代周期里如何**演进 / 退役 / 重构**也是高频追问点。是否应该作为四问的扩展
- **不同岗位（AI 工程师 vs AI 产品经理 vs AI 训练师）四问的侧重点是否不同**——原文是 AI 产品方向的标准，工程岗位可能更侧重 2 + 4（风险 + 边界），产品岗位更侧重 1 + 3（场景 + 标准）。这跟你的目标岗位画像应该对齐
