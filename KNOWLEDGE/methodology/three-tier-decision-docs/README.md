# 技术提案 / 架构设计 / 实现设计: 三类决策层级文档

## 它要解决什么问题

很多团队把 PEP、RFC、架构图、API 文档、协议说明、状态机、表结构都笼统叫"设计文档"。这会带来一个实际问题：**大家在不同阶段讨论不同粒度的事情，却以为自己在讨论同一件事**。

更糟的是当 AI 也能"生成设计文档"后，**文档归属哪一类型变得更模糊**——AI 给的可能是技术提案、架构设计、实现设计的混合体，但你团队需要的是某一个特定层级的产出。

这个节点的核心观点：**这三类不是"写得粗还是细"的区别，而是决策层级的区别**。一旦看清这点，就能解决三类项目难题：

- 团队在某阶段争论错粒度的问题（前置决策缺失或后置决策已定）
- AI 生成的文档怎么定位 / 怎么校验
- 实习 / 面试时讲架构经验讲不清自己做的是哪一类

## 朴素方案为什么不够（"详略论"是最常见的误解）

最容易的混淆是：**技术提案 = 简略架构设计 = 详略不同**。这条路径错在两个层面：

**反事实推导（层面 1）**：如果三类文档只是详略差异，那"详细的技术提案"就等于"架构设计 + 实现设计"——但事实不是。技术提案的核心是**论证 + 共识**，它可能只有 1-2 页但回答了"该不该做"；架构设计可能 30 页都在画系统结构但完全不论证"该不该做"。**两者的核心问题不同**，详略只是表象。

**反事实推导（层面 2）**：如果以为详细就是好，技术提案阶段团队会陷入争论字段名（实现设计层级问题）→ 决策方向被字段名绑架；实现设计阶段还在争论"是否要做这个方向"（技术提案层级问题）→ 编码无从开始。**层级搞错的代价是讨论效率降到零**。

所以三类文档必须按**决策层级**分清楚，不是按详略分。

## 三类的核心定义

| 文档类型 | 核心问题 | 典型产物 | 决策层级 |
|---|---|---|---|
| **技术提案**（Tech Proposal / RFC） | **该不该做、为什么做这个不做那个** | PEP / RFC / IETF Internet-Draft / KEP / Go proposal / Rust RFC | **方向** |
| **架构设计**（Architecture Design） | **系统由什么组成 / 如何协作 / 部署拓扑** | 系统架构图 / 组件图 / 部署图 / 数据流图 / 控制平面说明 | **结构** |
| **实现设计**（Implementation Design） | **API 怎么定义 / 字段含义 / 状态流转 / 协议格式 / 数据库表 / 错误处理** | API reference / protocol spec / schema / state machine / sequence diagram | **落地** |

注意"实现设计"比传统说法"详细设计"更准——"详细设计"容易被误解成"写得很细"，但本质是面向编码落地。

### 三类决策层级形成因果链

```
技术提案（方向）→ 设定目标和约束 → 架构设计（结构）→ 设定接口契约 → 实现设计（落地）
```

每一层的产物为下一层提供**前提**：
- 技术提案给出"为什么做"——架构设计才知道往哪个目标设计
- 架构设计给出"组件 + 边界"——实现设计才知道某个模块的接口契约和职责范围

## 开发流程中的顺序

**重大变化**：

```
技术提案 → 架构设计 → 实现设计 → 编码实现 → 测试验证 → 发布反馈
```

**普通功能**：

```
架构设计 → 实现设计 → 编码实现
```

**小功能 / bugfix**：

```
实现设计 → 编码实现
```

**这不是严格瀑布**——开发过程中会反向修正：
- 实现太复杂 → 回头调实现设计
- 模块边界不合理 → 回头调架构设计
- 架构假设不成立 → 回头修改或否决技术提案

**关键判断**：

- 团队在**技术提案阶段**争论字段名 → 讨论过细，跳层级了
- 团队在**实现设计阶段**还在争论"是否要做这个方向" → 前置决策缺失
- 编码时发现**模块边界不清** → 架构设计不足

## 经典案例: Kubernetes 三类同时存在

Kubernetes 是同时含三类文档的典型开源项目：

- **技术提案**：[Kubernetes Enhancements / KEPs](https://github.com/kubernetes/enhancements)
- **架构设计**：[Kubernetes Cluster Architecture](https://kubernetes.io/docs/concepts/architecture)——讲 control plane / worker node / kube-apiserver / etcd / kube-scheduler / kube-controller-manager / kubelet / kube-proxy / Pods 的关系
- **实现设计**：[Kubernetes API Conventions](https://github.com/kubernetes/community/blob/main/contributors/devel/sig-architecture/api-conventions.md)——讨论 kind / apiVersion / metadata / spec / status / REST verbs / defaulting / concurrency control 等具体规则

这组对比的关键观察：

- **架构设计层级问题**："集群有哪些组件，它们如何协作？"
- **实现设计层级问题**："一个 API 对象有哪些字段，这些字段什么语义？"

同一个项目里，**架构层级讲"组件协作"，实现层级讲"对象表达"**——决策层级不同。

类似的对比也适用于 PostgreSQL（架构 = client/server 模型 + parser/planner/executor / 实现 = Frontend/Backend Protocol 字节格式）、GitLab（架构 = NGINX/Workhorse/Puma/Sidekiq/Redis/PostgreSQL/Gitaly 协作 / 实现 = Merge Requests API 字段定义）、opencode（架构 = Architecture Overview / 实现 = Session Management + Tool System + Provider Architecture）、OpenClaw（架构 = Gateway 为中心 / 实现 = Agent Loop + Gateway Protocol 字节级）。

## 用关键词判断文档归属

| 关键词出现频次高 | 文档类型 |
|---|---|
| motivation / proposal / alternatives / drawbacks / compatibility / consensus | **技术提案** |
| architecture / component / control plane / data flow / deployment / service / node | **架构设计** |
| API / schema / field / message format / state / protocol / endpoint / error code | **实现设计** |

更简单的判断公式：

> "**该不该做**" → 技术提案
> "**整体怎么组织**" → 架构设计
> "**具体怎么实现**" → 实现设计

## 三类文档在团队开发中的实际作用

**技术提案**：
- 避免未经论证的大改动
- 记录背景 / 权衡 / 反对意见
- 帮助团队或社区达成共识
- 给后续架构设计设定目标和约束

**架构设计**：
- 统一系统理解
- 明确模块边界
- 降低跨团队协作成本
- 提前暴露扩展性 / 安全性 / 部署 / 性能风险

**实现设计**：
- 把架构拆成可开发任务
- 降低接口歧义
- 指导开发和测试
- 减少返工
- 让代码评审有依据

## 常见误区

- **误区一**：技术提案 = 简略架构设计 → **错**。技术提案可包含架构设计片段，但核心是论证和共识
- **误区二**：架构设计越详细越好 → **错**。架构设计明确边界 + 关键决策，不应替代实现设计
- **误区三**：实现设计 = 代码翻译 → **错**。不是把代码翻译成文字，是把接口 / 状态 / 异常 / 协议 / 数据结构 / 测试约束说清楚
- **误区四**：所有功能都需要三份文档 → **错**。小变更只要实现设计或直接编码；重大变更才需要技术提案

## 与"架构设计 6 步法"的联系

这个节点跟 `KNOWLEDGE/methodology/architecture-design-six-steps/` **互补不重叠**：

- 6 步法说的是**架构设计这一层内部的执行顺序**（定位→原型→API→数据表→模块→详细）
- 三类决策层级说的是**架构设计层在整个流程中的位置和外延**

七牛云 ZeroOps 项目失败教训正好印证：**前置的"技术提案"层缺位**——三次方向变更没有 1-2 页论证文档让团队达成共识就跳到架构设计层反复返工。**6 步法走得稳，但底层方向没共识**。CEO 后期拍板的本质是替团队补了缺位的"技术提案 + 共识"环节。

## Open Questions

- **AI 生成"混合层级文档"的检测和分层切割**——当前 AI 给的设计文档常常是技术提案 + 架构设计 + 实现设计的混合体。能不能用关键词分布做自动分层？这是个工程工具化方向
- **三类文档的最小必要内容**——技术提案最少要包含哪几项？1 页能写完吗？这跟项目规模 + 团队规模都有关，缺乏统一答案。值得收集开源项目的最短有效 RFC 做横向对比
