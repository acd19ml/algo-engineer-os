# 架构设计 6 步法: 避免推倒重来的顺序

## 它要解决什么问题

接到一个新产品，团队通常被要求"先做架构设计"——但**从哪里入手**？是先画架构图、还是先定 API、还是先做数据表？拍脑袋选起点的后果是**几周后推倒重来**：API 定好了发现界面用不上、数据表设好了发现 API 调不通、模块拆完发现耦合度极高。

这个节点回答的不是"架构设计有哪些组成部分"（那是百度搜得到的清单），而是**为什么必须按一个特定顺序走，跳序会产生什么具体损失**。这套顺序来自七牛云 1024 实训营的实训项目实战经验（mentor 王昶敏总结），是经过踩坑验证的工业 SOP。

## 朴素方案为什么不够（"从架构图直接开画"是最常见的坑）

最容易的起点是"先画一张大架构图"——把组件框框画出来，箭头连一连。看起来快，但有个根本问题：**架构图本身没有依据**。组件凭什么这样划分？API 凭什么这样设计？数据表凭什么这样建？

**反事实推导**：如果跳过"产品定位 + 原型图"直接画架构图，会发生什么？

1. **组件划分依赖工程师拍脑袋**——没有"产品要做成什么样"这个外部锚点，每个工程师按自己理解划组件，团队共识为零
2. **API 形状跟随组件画法**——画了组件再补 API，API 服从于组件而不是服从于功能需求
3. **数据表跟随 API**——但 API 是错的，数据表也跟着错
4. **几周后做完联调发现产品形态对不上需求**——回头改原型图就要全链路推倒

这正是七牛云 ZeroOps 项目踩过的坑——产品形态三次重大调整，**每次调整都触发原型图重画 → API 重设 → 数据表重新设计的连锁返工**。所以顺序不是 "建议性"，是 "强约束性"。

## 正确顺序: 六步法

```
明确产品定位 → 定义产品原型与原型图 → 定义功能 API → 完成数据表设计 → 模块拆分 → 模块详细设计
```

每一步建立在前一步的基础上，**每一步都在解决上一步留下的、推进不下去的问题**。

### 一、明确产品定位

回答几个具体问题：

- 我们是要做一个**工具**，还是**全托管系统**？（ZeroOps 实训营当时争论的就是这个——做"运维工具"还是"全托管运维系统"，方向不同后续一切都不同）
- 用户群体是 X 还是 Y？
- 解决的核心用户故事是什么？哪些是必须优先实现、哪些可留到后续迭代？

这步属于产品设计范畴，但**没共识就启动后续所有步骤都是浪费**。

### 二、定义产品原型与原型图

把模糊的业务需求转化为**清晰、可验证、无歧义**的技术需求——这是后续设计开发的唯一依据。

具体做法：
- 用交互界面方式收集需求，每个界面的功能列表 / 功能划分 / 页面跳转关系
- 不涉及界面的功能（如 IDE 扩展）用**功能流程图**表达
- 提示：在原型设计前先画**需求草图**明确不变的核心要素，让后续原型设计更高效

**避免两类陷阱**：
- **过度设计**：设计了实际用不到的功能
- **伪需求**：某些"功能"更适合通过技术特性实现，而不该以产品功能形态存在

**MVP 思想**也是关键——先聚焦最核心、不可或缺的功能。

### 三、定义功能 API

原型图确定后，每个页面的功能列表、要展示的信息都已明确，进入 API 设计。**两个经验**：

**经验一：API 设计要结合界面，但更要考虑逻辑关联性**
- 例：界面需要"某服务的发布任务"，可以设计为 `/v1/services/:service/deployments`
- 但如果系统要支持多种关联查询，更合理的是 `/v1/deployments?service=serviceName`
- **不一定按界面层级硬翻 API**

**经验二：避免过度通用的 payload**
- 不要把各种数据结构都往一个复杂 payload 里塞——解释成本高、依赖方难理解
- 在定义 API 时只包含**明确且稳定**的数据结构

⚠️ 重要：**API 初版不是最终定稿**——等模块拆解完成后，要根据模块边界和耦合度做微调。

### 四、数据表定义

API 定义完成后，**不直接拆模块，先定义数据表结构（ER 图）**。

**为什么数据表在模块拆分之前**：
- 数据表本身是模块划分的重要参考依据——表的边界决定模块边界
- 不同 API 操作哪些表，也反映 API 与数据表的耦合关系

**避免**：
- 不必为了兼容多种数据库而抽象表结构——这种"通用设计"在现实中通常不会用到
- 数据库选型一旦确定通常不会轻易改

**建议**：注意数据库特性。比如 Postgres 支持 ARRAY 类型，在某些场景下可以减少拆表、简化 ER 图——七牛云 ZeroOps 项目就因此把告警 labels / metrics 等结构化数据存在 ARRAY 里减少了拆表。

### 五、模块拆分

完成 API 和数据表设计后，进入模块拆分。**三个判断标准**：

1. **表的边界决定模块边界**——通常不会有多个模块同时操作同一张表
2. **代码量考虑**——代码量大的功能值得独立成模块，反之合并
3. **耦合度**——如果一个模块大量依赖另一个模块的**私有 API**，说明拆分不合理；如果只依赖**公共 API** 就能完成任务，则可以独立

模块确定后还要理清依赖关系，**合理使用依赖倒置原则**减少耦合、提高复用性。

### 六、模块详细设计

把每个模块展开到**可指导开发的程度**，包括：

- 输入输出定义
- 接口与数据结构
- 内部逻辑分层
- 关键流程图、类图

## 顺序为什么不能反过来

每一步对下一步的约束是**强且单向的**：

```
产品定位 ──约束──→ 原型图能定的功能集
原型图 ──约束──→ API 该提供的端点和参数
API ──约束──→ 数据表要支持哪些查询
数据表 ──约束──→ 模块边界
模块边界 ──约束──→ 模块详细设计
```

**反事实推导**：跳过"数据表"直接做模块拆分会怎样？模块边界没有"表边界"做参考，工程师会按"功能直觉"拆。当后续做数据表时发现两个所谓"独立模块"实际上反复读写同一张表——拆分不合理，要重做。

## 团队共识比文档质量更重要（也是最易被忽视的点）

mentor 在原文里强调：**架构设计讨论，团队参与的人越多越好**。因为它是一项非常重要的共识工程。

但这条**有前提**——参与的人需要在相关领域有 context。七牛云 ZeroOps 项目实操踩过坑：让算法同学参与后端架构设计**既帮不上忙也不一定能理解，还浪费了他调优算法的时间**。

更可行的做法是**按子系统的领域专家小组讨论**：Agent 设计由算法 + 一名后端，发布系统由后端 + 运维 mentor，前端由前端独立设计后跟其它模块对齐接口。**6 人全员讨论是低效形态**。

## 何时不需要走完整 6 步法

不是所有变化都需要 6 步。判断标准：

- **重大方向变更**（项目级、架构级）→ **走完整 6 步法 + 前面还要补技术提案**（见 `KNOWLEDGE/methodology/three-tier-decision-docs/`）
- **普通功能新增** → 走架构设计（步骤一可跳，从原型图开始）→ 实现设计
- **小变更 / bugfix** → 实现设计 → 编码

强行让小变更也走 6 步是过度工程。

## Open Questions

- **每一步的产物如何审核才算"达标"**——原文给的是"做什么"，但具体到验收标准（如原型图到什么粒度算够细、API 到什么程度算稳定）需要每个团队自己摸索，缺乏统一答案
- **方向变更时回滚的最小代价路径**——理论上每一步约束下一步、所以方向一变全套返工。但实战中能不能用"接口稳定 + 实现替换"的方式让上层不重做？这跟 `KNOWLEDGE/methodology/three-tier-decision-docs/` 的"技术提案先行"是同一个问题在不同维度
