# KV Cache

## 它要解决什么问题

面试常被问 "KV Cache 是什么？为什么需要它？" 大多数人会答："自回归生成的时候把之前算过的 key 和 value 缓存下来，生成下一个 token 就不用重复计算了，让推理更快。"

这个答案只能拿 60 分。因为面试官真正想考的不是定义，而是**三个核心问题**：

1. 为什么缓存的是 K 和 V，**不是 Q**？
2. KV Cache 的**代价**是什么？
3. 为什么说 KV Cache 是当前大模型推理的**核心瓶颈**？

要把这三件事讲清，得先看清自回归生成每一步在做什么。

## 自回归生成的不对称性

每生成一个新 token，模型需要：

- 用**这个新 token 的 query** 去和**所有历史 token 的 key** 做点积，算出注意力权重
- 再用这些权重对**所有历史 token 的 value** 做加权求和

注意这里有一个非常关键的**不对称性**：

- **Query 永远只有一个**——就是当前这个新 token 的
- **Key 和 Value 是所有历史 token 的**——它们会随着生成不断累积

如果不做缓存，每生成一个新 token，你都得把之前所有 token 重新过一遍 K 和 V 的投影：

- 生成第 100 个 token 时，前面 99 个 token 的 K 和 V 都要重算一遍
- 生成第 101 个 token 时，前面 100 个又全部重算一遍
- ……

**大量的计算是重复的**。

## KV Cache 的做法

每一步只算**当前新 token** 的 Q、K、V，然后把新的 K 和 V **追加到缓存里**。下一步直接从缓存读取，不再重复计算。

那 Q 为什么不用缓存？因为**每一步的 Q 就是当前这一个 token 的向量，用完就不需要了**——下一步不会再用到它。所以缓存 K 和 V，不缓存 Q——这是不对称性决定的，不是任意选择。

## KV Cache 的代价：用显存换算力

KV Cache 省下来的是**计算量**，付出的是**显存**——这不是免费的加速。

具体多大？算一笔账。假设模型 40 层、32 个注意力头、头维度 128、上下文长度 4096、用 FP16 存储。那一个请求的 KV Cache 大小：

```
40 (层) × 32 (头) × 4096 (位置) × 128 (维度) × 2 (K+V) × 2 (字节, FP16) ≈ 2.5 GB
```

**这只是一个请求**。如果同时服务 64 个用户，就是 **160 多 GB**——已经超过一张 A100 80GB 的显存。

所以 KV Cache 的大小**直接决定了你能同时服务多少个用户**。

## 为什么说 KV Cache 是推理的核心瓶颈

大模型推理分两个阶段：

- **Prefill**（预填充）：用户的输入一次性并行处理，生成初始的 KV Cache。这个阶段**算力是瓶颈**——GPU 越强越快。
- **Decode**（解码）：每生成一个 token，模型要从显存里把整个 KV Cache 读出来，和当前的 query 做一次注意力计算。

Decode 阶段的关键观察：**计算量很小**——就一个向量和一堆向量做点积。但**要搬运的数据量很大**——整个 KV Cache 都得读出来。

GPU 的算力根本吃不满，**大部分时间都在等数据从显存搬到计算单元**。这就是为什么 Decode 阶段是典型的 **memory-bound（内存带宽瓶颈）**，而不是 compute-bound（算力瓶颈）。

理解这一点，才能理解业界为什么搞出那么多 KV Cache 优化方案：

- **MQA**（Multi-Query Attention）：让所有注意力头**共享一套 K/V**——KV Cache 大小直接除以头数
- **GQA**（Grouped-Query Attention）：让几个头一组共享 K/V——MQA 的折中版本
- **量化 KV Cache**：用 INT8 甚至 INT4 来存储 K/V——空间减半甚至减到四分之一
- **PagedAttention**：用**分页管理**减少显存碎片

这些手段的核心目标都一样：**把 KV Cache 搞小**。因为 KV Cache 越小，显存占用越低，能同时服务的请求就越多，系统的吞吐量就越高。

## 表面上是缓存优化，本质上是资源瓶颈

KV Cache 这个东西的迷惑性在于——**名字叫"缓存"听起来像锦上添花的优化技巧**，但它实际上是大模型推理系统里**最核心的资源约束**。

讨论推理系统时，能不能把这件事讲清楚——
- 不对称性（为什么是 K/V 不是 Q）
- 显存代价（具体数字 GB 量级）
- Memory-bound 而非 compute-bound
- 优化方向都在压缩 KV Cache

——直接决定了面试官觉得你是**背过答案还是真正理解推理系统**。

## Open Questions

- **MQA / GQA 通过共享 K/V 来压缩 cache，但理论上应该损害表达能力**（多头的"多种关系图"现在被压缩到 1 套或几套 K/V）。实际效果损失却很小。这暗示什么？是不是 K/V 的分化对模型表达力贡献远小于 Q 的分化？这条线和 `multi-head-attention` 的 Open Question 相关，可能要单独节点 `KNOWLEDGE/transformer/mqa-gqa/`。
- KV Cache 的量化（INT8 / INT4）显然会损失精度，但实际生成质量影响很小。**为什么 K/V 这么"耐量化"**？可能和 attention 输出在 softmax 后被归一化，对绝对值不敏感有关——但这是猜测，没有看到严谨论证。
- **PagedAttention 的分页机制借鉴了操作系统虚拟内存**——这个设计具体怎么处理跨页的注意力计算的？我还没读 vLLM 源码，需要单独节点处理。
