# Multi-Head Attention

## 它要解决什么问题

面试常被问："为什么 attention 要用多个头而不是一个头就够了？"

很多人的第一反应是 "多个头可以让模型关注不同子空间的信息"。这个回答方向对，**但太抽象了**——面试官追问 "什么叫不同子空间" 就接不住。

要讲透这道题，得反过来：**先想清楚如果只用一个头会出什么问题**。把这个问题想透了，多头的动机自然就出来。

## 单头注意力的根本瓶颈：一个权重分布承载不了多种关系

设输入序列有 N 个 token，每个 token 维度 `d_model`（比如 512）。单头注意力就是用**一组** Q/K/V 投影矩阵，在这个完整的 512 维空间里计算注意力。每个 token 生成一个 512 维 query，去和所有 token 的 512 维 key 做点积，得到一个注意力分数过 softmax，再对 value 做加权求和。

结果是什么？**每个 token 最终得到一个注意力分布——一组 softmax 权重，一种加权方式**。

意味着：对每一个 token，它**只能用一种方式**去理解自己和其他 token 的关系。

但语言中 token 之间的关系**从来不是一维的**。

## 反事实：把"多种关系"摊开看

读这句话："小明把妈妈昨天刚买的那个又大又红的苹果吃了"。

"吃了" 这个词需要同时关注几个不同层面的信息：

- **句法层面**：`吃了` 的主语是谁？是 `小明`。需要跨越一长串修饰语，找到最前面的小明。这是一种**长距离的结构性依赖**。
- **语义层面**：`吃了` 的宾语是什么？是 `苹果`。需要理解 `又大又红` 都是修饰苹果的，真正被吃的对象是 `苹果`。
- **局部搭配层面**：`吃了` 和紧挨着它的 `苹果` 之间存在直接的动宾关系。这种**短距离搭配**本身也是一个需要捕捉的模式。

三种关系，**三种完全不同的关注模式**。有的需要看远处，有的需要看近处，有的关注句法角色，有的关注语义内容。

但单头注意力只有**一组** softmax 权重——它被迫把所有这些不同层面的关系压缩进同一组权重里。结果是**妥协**：为了同时照顾长距离句法依赖和短距离语义搭配，注意力权重会变成一个**折中分布**——哪个都想关注，哪个都关注不充分。

打个比方：你只有一双眼睛，但需要同时盯着黑板上的公式、手里的笔记和旁边同学的表情。你只能快速轮流扫视，或者干脆只盯其中一个，其他的就丢了。

**这就是单头的根本问题**：不是不能工作，而是**表达能力被严重限制**——一个注意力分布承载不了多种独立的关系模式。

## 多头的设计：把高维空间拆成多个低维子空间

既然一个注意力分布不够用，那就多来几个。多头注意力的核心思路：

- 把那个 512 维的空间**拆成 8 个 64 维的子空间**
- 每个子空间有自己**独立的一组 Q/K/V** 投影矩阵
- 每个子空间**独立计算**注意力，生成各自的注意力分布
- 最后把 8 个头的输出**拼接起来**，通过一个线性变换融合

这样一来，每个头**只需要负责捕捉一种关系模式**：

- 第 1 个头可以专注于学习局部的词序关系——它的 Q 和 K 投影矩阵可能会特别强调位置信息
- 第 2 个头可以专注于长距离的指代关系——投影矩阵会把语义角色特征放大
- 第 3 个头可以学习句法结构——主语、谓语、宾语的依赖
- 第 4 个头可以捕捉共现模式

**各学各的，互不干扰，最后拼在一起信息互补**。

这不是猜测——实验确实验证了这种功能分化。有研究系统性分析了 BERT 的注意力头：有的头几乎只关注相邻位置，有的头专门追踪指代链（"他" 指向 "猫"），有的头呈现清晰的句法结构（主语盯着谓语，谓语盯着宾语）。如果这些功能全部压缩到一个头里，模型不可能同时保持这么多种清晰的模式。

## 反事实：每个头维度变小会不会损失表达能力？

会损失一点，但**换来的远大于失去的**。算一笔账：

- 单头：在 512 维空间里计算注意力，计算量正比于 `N² · 512`
- 8 头：每头在 64 维里计算，总计算量正比于 `8 · N² · 64 = N² · 512`

**总计算量基本没变**。但信息容量变了：

- 单头只能产生**一个** N×N 的注意力矩阵
- 8 头能产生 **8 个不同**的 N×N 注意力矩阵

模型可以同时维护**八种不同的 token 间关系图**——这是质的变化。

每个头从 512 维降到 64 维，单个头的表达能力确实不如原来 512 维强。但换个角度：一个 512 维的头试图在**一个**注意力分布里塞下**所有类型**的关系，其中很多维度其实在做无谓的折中；而 64 维的头只需要在**一个清晰的子任务**上做好就行——**有效利用率反而更高**。

类比：全科医生 vs 8 个专科医生。全科医生一个人看所有病，每种病的诊断深度有限；8 个专科医生每人看一类病，合在一起的诊疗能力远超一个全科医生。虽然每个专科医生不懂其他科，但**那不重要——他们各自负责的领域互不重叠**。

## 为什么是在子空间上拆分，而不是用 8 个完整的 512 维头？

如果面试官追问这个，答案是两方面：

**第一，参数量和计算量**。如果 8 个头各自都在 512 维空间上操作，参数量和计算量直接翻 8 倍，工程上不可接受。

**第二（更重要）**：在子空间上拆分隐含了一个**归纳偏置**——它告诉模型："你不需要在全维度空间里搜索关系模式，你只需要在低维子空间里找就行"。这是一种**正则化**，防止每个头在高维空间里过拟合，强迫它们只学习**紧凑、本质的**关系模式。

这个逻辑和 QKV 三组矩阵分解的低秩论证**一脉相承**（参见 `KNOWLEDGE/transformer/qkv-three-matrix-design/`）：Q 和 K 各自从 d_model 投影到 d_k，乘积构成一个 d_k 秩的相关性矩阵。多头进一步推广了这个思想——不是在一个 d_k 子空间里算一种相关性，而是在**多个 d_k 子空间里同时算多种相关性**。

低秩保证了每种关系的学习是正则化的，多头保证了多种关系可以并行捕捉——两个设计在数学上嵌套在一起，共同服务于同一目标。

## 实验上的最优平衡点

有人尝试过把头的数量调到非常多、每个头的维度调到很小，也尝试过很少的头、每个头维度很大。结果发现存在一个**最优平衡点**：

- 头太少 → 功能分化不够
- 头太多 → 每个头的维度太小，子空间里装不下有意义的关系

Transformer 原论文用的 512 维 / 8 头 / 每头 64 维，就是在这个权衡中找到的一个好工程选择。

## Open Questions

- 头之间到底是不是真的"功能分化得很干净"？BERT 的 attention 头可视化工作显示功能分化存在，但很多头的注意力模式接近**冗余**——有研究表明可以剪枝大量头而性能不显著下降。这说明 8 头的"信息容量翻 8 倍"是上界，实际利用率远低。如果是这样，**为什么训练不出更高效的稀疏多头结构**？这一点我没想清。
- MQA / GQA 这类工程优化把多头的 K 和 V 进一步**共享**，按本节点的逻辑似乎应该明显损害表达能力，但实际效果损失很小。这是不是说明多头的真正价值更多在 **Q 投影的分化**而非 K/V 的分化？这条线值得单独节点处理（`KNOWLEDGE/transformer/mqa-gqa/`，待建）。
