# SFT Data Quality

## 它要解决什么问题

说"训练完效果好，所以数据好"是事后判断。真正的数据工程要在训练前和训练中判断：几万条候选数据哪些该留、哪些该扔、哪些要修；训练效果不对时，是数据问题还是超参数问题。

SFT 数据质量不是单条样本漂亮就够，而是整个数据集在格式一致性、分布合理性、多样性和纯净度上健康。

## 训练前：先筛出可学习区间

第一种方法是困惑度过滤。用目标模型或同系列基座模型计算回复部分 perplexity。困惑度特别低，说明模型本来就会，训练价值有限；困惑度特别高，说明回复要么质量有问题，要么远超模型能力边界，硬学可能是噪声。

最有价值的是中间区域：模型能理解但还没掌握。一个经验起点是去掉困惑度最低和最高的各 10%，再根据小实验校准阈值。

第二种方法是奖励模型打分。奖励模型比困惑度更贴近人类质量判断，可以评估有用性、准确性、安全性、格式规范。但奖励模型有自己的训练偏见，适合辅助筛选，不适合当唯一标准。

第三种是 LLM-as-judge。它灵活，可以按场景定制维度，比如特别强调格式一致性或简洁性。但它常偏好更长、更详细的回复，所以 prompt 里要明确长度不是质量指标。

第四种是交叉一致性检验。如果同一问题有多个标注者或多个生成模型答案，一致性高说明可信度高；差异大说明问题歧义或来源质量不稳。它本质上是在训练前量化标注者一致性。

## 训练中：用症状反推数据问题

训练 loss 下降、验证 loss 上升，是经典过拟合信号。在 SFT 中，这往往不只是 epoch 太多，也可能说明数据多样性不足：模型记住了窄模式，而不是学到泛化行为。

如果某类评估任务异常差，优先回到训练数据里看对应类型样本数量和质量。把评估集按任务类型分组，是定位数据缺口的实用方法。

如果模型输出格式或风格突然变化，比如开始套用固定句式、过度冗长、或偏离目标格式，也常常是数据分布太窄或格式不一致。

如果目标任务变好但通用能力退化，是灾难性遗忘信号。根因通常是训练数据分布太窄，需要混入通用数据，而不是只调学习率。

## 清洗第一步：格式一致性

格式问题是多源数据里最高频的问题。System prompt 是否存在且一致，角色标记是否正确，特殊 token 是否完整，回复是否有结束标记，这些都应该用脚本自动检查。

这一步听起来低级，但回报很高。格式错的数据会把 loss 信号导向错误位置，甚至让模型学习错误对话边界。

## 去重：精确去重和模糊去重都要做

重复样本会让模型记忆化，降低训练效率，也损害泛化。精确去重用哈希即可；模糊去重要用 embedding 相似度或编辑距离。

阈值没有固定答案，但可以先画相似度分布。相似度超过 0.95 的样本通常可以直接审查或去掉，超过 0.9 的样本至少值得检查。

反事实地看，如果不去重，训练 loss 可能降得很好，但模型只是在重复样本上变熟。

## 异常值、安全过滤和分布审计

异常样本包括回复极短、极长、语言不匹配、答非所问。这些不一定全部删除，但要标出并审查。一个 "好的" 两字回复可能适合判断题，但也可能是坏标注。

如果数据来自网络或用户交互，还必须做毒性、偏见、隐私过滤。这不只是安全问题，也会影响模型行为；有害样本进入 SFT 后，后续对齐不一定能完全消除影响。

清洗后还要做分布审计：任务类型比例、回复长度、难度梯度、语言和格式覆盖是否合理。单条数据再好，整体分布偏了，模型也会学偏。

OpenAI 那个常见例子可以说明：训练数据里 60% 是拒答，但生产环境只有 5% 需要拒答，模型就会过度拒答。问题不是单条拒答数据错，而是分布错。

## 数据质量是持续工程

训练前用困惑度、奖励模型、LLM judge、交叉一致性筛选；训练中用 loss 走势、分类评估、输出模式和能力退化诊断；全程做格式清洗、去重、异常检测、安全过滤和分布审计。

一句话概括：SFT 数据质量不是"每条数据都完美"，而是数据集作为系统是否健康。

## Open Questions

- 困惑度过滤会不会误删真正有价值的高难样本？对小模型是噪声的样本，对大模型可能正是有效挑战。
- LLM-as-judge 与奖励模型如何校准到特定产品目标？通用 helpfulness 分数不一定等于业务质量。
