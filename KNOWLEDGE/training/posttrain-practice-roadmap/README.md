# Pretraining Practice Roadmap

## 它要解决什么问题

面试大模型岗位时有个很苛刻的问题：**"你从零训练过模型吗？"**

行业现状的真实分布：

- **会做 AI 应用的人**满地都是
- **跑过推理的人**遍地都是
- **错过微调的人**也不稀缺
- **真正从零训练过模型、踩过坑、能把预训练讲明白的人——凤毛麟角**

**看过数十篇论文不如自己训过一次模型**。

但大家都觉得预训练成本太高（几百万算力预算），普通人摸不到——于是陷入死亡循环：**不进大厂接触不到预训练，没有实战经验又进不了大厂**。

要回答 "怎么打破这个循环"，得先打消一个误解，然后看清成本，最后给出一份可执行的 4 周计划。

## 朴素直觉为什么不够：不是参数量大才有经验

**预训练的大部分坑，1B 的模型会遇到，100B 的模型也会遇到**。

当然，规模大了会有一些新问题：

- 通信瓶颈更严重
- 显存管理更复杂
- 需要更激进的并行策略
- 多机多卡的容灾、checkpoint 恢复、网络抖动同步

**但这些是"更难"，不是"不同"——核心的工程方法论是相通的**。

类比：你在驾校学会了开车，上路遇到的交通规则、突发情况处理、路况判断和老司机一样——**区别只是老司机开的是复杂的拉力赛道，你开的是城市公路。但"开车"这件事，你已经会了**。

要向面试官证明的不是 "**我烧过多少钱**"，而是 "**我知道这条路上有哪些坑、我踩过、我知道怎么爬出来**"。

## 反事实：成本到底多少？1B-7B 是甜蜜点

算一笔账，让你看到自己的能力边界。

**云算力平台已经成熟**：80GB 显存的 A100/A800，租用价格大概 5-10 块钱/小时。按 6 块钱算，一天跑满也就 144 块——**一个月不到 5000 块**。

如果想踩多机多卡分布式训练的坑，**可以租更便宜的单卡组成集群**。**几千块换一个极其宝贵的预训练经验，投入产出比极高**。

那单卡 A100 80GB 到底能训多大的模型？粗略算：

- 模型参数本身：1B 参数用 BF16 存储约 2GB
- **训练不只是存参数**：优化器状态、梯度、激活值才是大头
- 用 AdamW 优化器：1B 模型训练大概需要 **20GB 左右显存**

所以 80GB 的 A100：

- 不做任何优化直接跑：能训 **3B-4B** 模型
- 加上梯度检查点 + 混合精度 + 优化器状态 offload：上限推到 **7B**

**7B 是什么概念**？

- 私有化部署最常用的尺寸
- 开源社区最活跃的模型规模
- 已经能涌现出相当不错的能力

足够你把预训练的核心流程**全部走一遍**。

## 反事实：怎么才算"真正攒到经验"

**千万别觉得到 GitHub 上随便拉一个训练框架，把代码一跑、等着 loss 下降就行了**——那叫**烧电费**，不叫攒经验。

真正的经验是你能回答这些问题：

- **学习率怎么调？warmup 要多少步？为什么？**
- **训练吞吐量是多少？token/sec？MFU 大概多少？合理吗？有没有优化空间？**
- **数据配比对模型能力有什么影响？怎么评估清洗后数据的质量？**
- **什么时候该存 checkpoint？存多频繁？**
- 训练中后期 loss 下降变慢——**怎么判断是正常的收敛减速，还是陷入了局部最优？还是学习率该 decay 了？**
- **BF16 和 FP16 混合精度训练各有什么坑？什么情况下会出现数值不稳定？**

**论文里不会细讲、博客里一笔带过**——但面试官一定会问。**这才是区分"真干过"和"只看过"的分水岭**。

## 一个月的可执行计划

### 第 1 周：数据工程（先不上 GPU）

**别急着采购 GPU**——先下载几个开源的预训练数据集。

经验公式：**训练所需的 token 数 ≈ 模型参数量的 20 倍**。比如训练 1B 的模型需要 20B token。开源数据集很多都远超这个规模——比如 SkyPile 有 150B token。

**反常识认知**：大厂里做预训练的人，占大头的工作**并不是看论文设计新算法**——而是**清洗数据、维护一整套数据处理流水线**。

**数据是预训练的地基，地基不牢，上层建筑必然塌方**。

这一周要亲手做的：

- **去重**：大模型有很强的记忆能力——一段话重复很多次它就记住了开始复读，**泛化能力直接归零**。亲眼看到那些重复内容如何污染训练集，然后写脚本用 MinHash 或 SimHash 把它们剔除——**会让你对"数据质量"四个字有切肤之痛的理解**
- **分词**：分词的核心是**信息压缩率**。试着训练一个自己的词表，对比不同词表大小的压缩率差异。**当词表压缩率提升 10%，意味着同样的文本能节省 10% 的训练算力**——这笔账算清楚就理解了为什么各家都在词表上下功夫
- **数据配比**：代码数据放多少、维基百科放多少、网页文本放多少？代码多了逻辑推理强但文采下降；小说多了说话流畅但开始胡说八道。**故意弄几个极端配比的数据集**，留着后面做对比实验

### 第 2 周：跑通流程，制造事故

这周开始真金白银烧算力。

**先别急着训大的**——找一个成熟的训练框架，用小参数量把完整流程跑通。

**跑通之后，最有价值的事来了——故意制造事故**：

- 把学习率调大 10 倍，看 loss 怎样
- 关掉 warmup，观察训练初期的震荡
- 把 batch size 调得很小，感受梯度噪声

**亲眼见证传说中的 "Loss Spike"** 长什么样——跑着跑着 loss 突然像心电图一样窜上去，然后可能就下不来了。

这时候你要做的**不是慌，而是分析**：

- 是哪个 batch 触发的？
- 那个时间点的梯度范数是什么样的？
- 是数据问题还是超参问题？
- 然后思考怎么设计一套监控机制来捕捉这些异常
- Checkpoint 存多频繁才能在出事时快速回滚

这些问题想清楚，**就理解了为什么大厂训练团队要花大量精力在训练稳定性上**。

> Google 训练 PaLM 的时候整个过程出现了**超过 20 次 Loss Spike**——每次都要回滚 checkpoint、跳过问题 batch 才能恢复。**这不是段子，是真实的工程挑战**。

### 第 3 周：正式训练 + 深度调优

前两周铺垫做完，这周开始认真训一个模型。**建议从 1B 参数起步**——这个规模在单卡 A100 上跑得动，训练周期不会太长，方便快速迭代。

重点关注：

- **学习率调度 + warmup 步数**：试不同配置，观察收敛曲线差异
- **梯度裁剪**：用来防止梯度爆炸——**试着关掉它，看会发生什么**
- **Checkpoint 策略**：存太频繁 IO 会把存储打爆、拖慢训练；存太稀疏出事了要回滚到很久之前算力白烧。**1 步 checkpoint 可不可以试试加上去**？
- **监控指标**：除了 loss，还要盯**梯度范数、学习率曲线、吞吐量**——这些指标能帮你在问题恶化之前发现苗头

这周结束应该有一个训练到一定程度的模型 + **一份详细的训练日志**。

### 第 4 周：评估与复盘

模型训出来了，怎么知道它好不好？这周建立**自己的评估体系**：

- 找评测集、下游任务来测：文本生成、简单问答、代码补全
- **不用追求分数多高**——关键是建立 **"改了什么、效果变了多少"** 的对应关系

然后做对比实验：

- 把第 1 周准备的不同配比数据集拿出来，分别训练对比效果差异
- 试不同的学习率、batch size，记录对收敛速度和最终效果的影响

**这些对比实验的结论，就是你面试时最硬的谈资**。

最后写一份复盘文档——把整个过程中遇到的问题、尝试的解决方案、最终的结论都记下来。**这份文档就是你的预训练经验证明**。

如果预算允许，第 3 周可以尝试**双卡多卡训练**——体验数据并行的配置、梯度同步开销、多卡通信带来的效率损失。

## 一个深层认知：预训练是系统工程

大模型训练**本质上是一个系统工程**——不是单纯的算法问题，也不是单纯的代码问题，**而是对算力、数据、算法、工程基建的综合考验**：

- **算力**不够：练不动
- **数据**不好：练傻了
- **工程**不稳：练不成
- **评估**不准：不知道练了个啥

**每一个环节都不能有短板**。

但**也正因为它难，所以壁垒高**。所以值得花时间、钱去认真踩一遍坑。

当你走完这些坑，面试官再问 **"你从零训练过模型吗？"**，**你可以正眼看着他说："训过，踩过坑。我来给你讲讲。"**——这句话的底气，**可不止值几千块钱**。

## Open Questions

- **多机多卡分布式训练**的核心坑（通信瓶颈、容灾、节点故障 checkpoint 恢复）和单机训练**有质的差异**——这条路自己复现成本高、收益相对单机训练边际更小。怎么用最低成本踩这部分的坑？
- **MFU 的合理范围**因模型架构、硬件配置、并行策略而异——什么是 "合理的 MFU"？这条线本节点没给具体数字。
- 预训练后**对齐流程**（SFT + RLHF + DPO 等）是否也属于"必踩的坑"？还是预训练 + SFT 就够了？这条线和 SFT/RL 节点联通，但本节点专注预训练范围。
