# SFT Data Sourcing

## 它要解决什么问题

问 "SFT 数据从哪来"，表面答案是人工标注或大模型生成。真正的问题是：不同来源的数据，质量上限、成本结构、迭代速度、法律风险和分布对齐都不同。

SFT 数据来源不是选一条路，而是设计一条数据供应链：人工定标杆，教师模型出主体，进化方法补难度，线上交互做持续闭环。

## 人工标注：质量天花板高，但不适合纯堆量

人工标注的优势是质量上限高，尤其在医疗、法律、金融这类专业领域，专家能写出模型和普通标注者都写不出的高质量答案。

但它有三个瓶颈。第一是成本，专业标注单条可能几十到上百元。第二是一致性，不同标注者的风格、判断标准、详细程度可能不一致。OpenAI 的实践强调，模型性能上限受标注者一致性限制；如果标注者只有 70% 一致率，模型很难学出超过这个上限的稳定行为。第三是速度，人越多产出越快，但一致性越难控。

所以人工标注最适合两个位置：冷启动阶段用 50-100 条专家数据定义质量标杆；质量校准阶段作为评估基准，衡量其他来源数据是否偏离。

反事实地看，如果用人工标注铺满全部数据量，预算和一致性都会很快成为瓶颈。

## 教师模型蒸馏：工业主体数据来源，但要防同质化

教师模型蒸馏用更强模型生成训练数据，再微调更小、更便宜的模型。它速度快、成本低、规模可控，是工业界常用主体路径。

但蒸馏有三个陷阱。

第一是同质化。单一教师模型会把自己的偏见、盲区、表达风格一起传给学生。教师不擅长的区域，学生也学不到；教师的错误如果没过滤，会被当成标准答案灌进去。

第二是条款风险。商业模型输出是否允许训练竞品模型，取决于具体服务条款。要商业化时，这不是技术细节，而是合规风险。

第三是能力坍缩。教师倾向生成自己最擅长的中等难度样本，缺少特别简单和特别困难两端。没有主动控制，多样性会变窄。

应对方式是多教师混合和交叉验证。多个教师对同一问题分歧大时，要人工审查或丢弃。

## Self-Instruct / Evo-Instruct：价值在扩展难度，不在凭空造能力

Self-Instruct 从少量种子指令出发，让模型生成新指令和回复，再过滤低质量样本。Evo-Instruct 进一步系统化地增加约束、推理步骤、专业术语和输出格式，让指令逐轮变复杂。

它的根本限制是：模型不能稳定生成超出自身能力边界的数据。一个不擅长数学推理的模型，自己生成的数学数据大概率也不可靠。

所以自指令生成经常仍然依赖更强生成模型，本质又回到蒸馏。但 Evo-Instruct 的进化思想很有价值：即使原始数据来自人工或蒸馏，也可以用系统化进化补足难度覆盖。

## 真实用户交互：长期最有价值，但必须筛选

线上交互数据的最大优势是分布完全对齐。它来自真实用户、真实场景、真实失败，不存在 "训练数据和生产 query 不一样" 的大偏差。

但它不能直接拿来训。第一要脱敏和合规，用户数据可能包含隐私和敏感信息。第二质量参差，用户问题和模型回复都可能很差。第三需要反馈信号，判断哪些样本值得进入训练集。

常见筛选信号包括：用户是否采纳回答，是否重复追问同一问题，对话是否自然结束，人工审核是否通过。

理想状态是数据飞轮：上线模型服务用户，收集交互，筛选高质量样本，加入训练集，微调新版本，再上线。每轮让模型更贴近真实用户需求。

## 多样性是贯穿所有来源的约束

不管数据从哪来，最终数据集必须有多样性。

任务多样性：对话、编辑、改写、摘要、推理、数学、编程各有目的，不能只堆一种任务。

难度多样性：不能全是简单题，也不能全是高难题。小模型尤其需要去掉超出能力边界的噪声，同时保留可学习挑战。

表达多样性：同一意思要有不同问法、长度、语气，否则模型只学会某种句式。

语言和格式多样性：中英文混合、Markdown、JSON、代码等输出格式如果生产环境会出现，训练里就要覆盖。

## 一个实用配方

一个高效 SFT 数据供应链可以是：人工专家数据占 5%-10% 用来定标杆；教师模型蒸馏占 60%-70% 做主体；Evo-Instruct 或难度扩展占 10%-20% 补高难样本；真实用户交互筛选占 10%-15% 做持续更新。

比例不是固定答案，需要消融实验。但主线是稳定的：人工定标，蒸馏扩量，进化补难度，线上闭环校准分布。

## Open Questions

- 多教师蒸馏中，教师分歧到底是噪声还是有价值的多样性？全部丢弃会提高一致性，但可能损失开放问题的合理多解。
- 线上交互数据进入训练集的反馈信号如何设计？用户采纳不一定代表答案正确，用户放弃也不一定代表模型失败。
