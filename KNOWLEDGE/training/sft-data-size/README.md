# SFT Data Size: How Much is Enough

## 它要解决什么问题

面试官问 "SFT 微调的数据量怎么选择？"

如果你回答 "数据的质量比数量重要、选择少而精的数据"——**这句话早就已经是业界共识了**。

**但当你真正坐下来准备做 SFT 数据集的时候，这句话帮不了你**。**真正卡住人的是这些问题**：

- **标多少条？500 条够不够？5000 条会不会浪费**？
- **手上的数据，哪些该留哪些该扔**？
- **"高质量" 的可操作标准到底是什么**？

要回答这些，得对照 OpenAI、Anthropic、Google、Hugging Face 四家的官方最佳实践。

## 朴素直觉为什么不够：从多少条开始

四家大厂在**起步数量**上的建议高度一致——但**也有差异**：

| 厂商 | 起步建议 |
|---|---|
| OpenAI | **50 条**精心制作的示范数据 |
| Anthropic | **50-100 条**高质量数据 |
| Google | **100 条**起步，可以扩展到数千 |
| Hugging Face | **1000 条**起步，推荐 1 万条以上 |

注意**差异 ≠ 矛盾**——**它们面向的场景不同**：

- **OpenAI / Anthropic** 的 API 微调面向**定义清晰的商业任务**（格式控制、分类、信息抽取）——**这类窄任务确实几十条就能见效**
- **Hugging Face** 做的是**开源模型的通用指令微调**——**需要覆盖的能力维度更多**，自然需要更大的基数

**关键要点是"思路而非数字"**：

> **先用最小可行数据量验证方向对不对，再决定要不要加码**。

## 四家共识：怎么定义"高质量"

### OpenAI：格式一致性 + 分布对齐

**最强调的一点，也是最多人忽视的一点**：**所有训练样本的格式必须一致，并且和推理时的格式完全一样**。

**听起来像废话——但实操中大量人会犯这个错**：

- 训练数据里有的加了系统提示词、有的没加
- 有的用 Markdown 格式、有的用纯文本
- 有的回复 300 字、有的回复 3000 字

**这些不一致会直接拉低模型的学习效率**。

**OpenAI 还指出一个很容易被忽视的问题**：

> **如果多个人参与创建训练数据，模型的性能上限会受限于"标注者之间的一致性水平"**。比如在文本抽取任务中，如果标注者之间只有 70% 的一致率——**模型也不太可能超过这个水平**。

意味着——**在你开始收集大量数据之前，先花时间统一标注规范**——确保少数几个核心标注者之间的一致性。**10 条高度一致的数据比 50 条标准混乱的数据有用得多**。

### 数据分布对齐

**OpenAI 举了一个很直观的例子**：

> 如果你训练数据里 60% 的助手回复是 "我无法回答这个问题"——**但实际推理时只有 5% 的情况需要这样回复——你的模型就会产生过多的拒答**。

**训练数据的分布必须跟你实际生产环境中的输入分布对齐**——如果业务里 8 成是短问答、2 成是长文生成，**训练数据也应该大致遵循这个比例**。

### OpenAI 关于系统提示词的实用建议

**把微调前效果最好的指令和提示词，原封不动地包含在每一条训练样本中**——尤其在训练数据不到 100 条的时候，**这能帮你获得最好、最通用的效果**。

如果你为了省成本去缩短每条样本里重复出现的指令——**可能反而需要更多的训练数据才能达到同样效果**。

**这条信息量很大**：SFT 数据里**不光是输入输出对本身重要，系统提示词的存在和一致性同样重要**——每条数据里都带上完整的系统提示词，看起来冗余，**但实际上是在帮模型建立稳定的行为框架**。

## Anthropic 的实战数据点

Anthropic 在 Claude Haiku 微调中的硬数据：

**用微调后的 Haiku 做论坛内容审核**：

- 分类准确率从 **81.5% 提升到 99.6%**
- 同时将每次查询的 **token 消耗减少了 85%**

**81% → 99% 这个提升幅度不是一般的 prompt engineering 能做到的**。**且 token 消耗减少 85% 意味着微调后你可以用更短的提示词达到更好的效果**——**直接说明了微调的投入回报比**。

### 超参数层面的发现

**当数据量较大（1K-10K 条）时**：**学习率对性能的影响更显著**。

**当数据量较小（32-100 条）时**：**Batch Size 的大小反而成为主导因素**。

**实操指导意义**：

- **几十条数据**：不要只调学习率——**优先调 Batch Size**
- **几千条数据**：学习率的调优优先级更高

**大多数教程只会告诉你"调学习率"——但在小数据量场景下，这可能不是最关键的地方**。

### Anthropic 独特的格式建议

强烈推荐：

- **在训练样本中包含系统提示词**——明确定义模型的角色和任务
- **在 XML 标签中包含推理过程**

**这个细节很多人不会注意到**：**在训练数据中嵌入推理链，而不只是给出最终答案——可以帮助模型学到更稳健的推理模式**。

## Google 的关键洞察

### 去重是预处理中最关键的步骤

**Google 明确指出**——**预处理中最关键的步骤之一是去重**。

**训练数据中的重复样本会**：

- **导致记忆化**——损害模型的泛化能力
- **低效训练**——模型在相似的样本上做冗余的学习
- **训练集和验证集之间的重复或近似重复样本**会造成**数据泄漏**——**人为拉高评估成绩**

**很多人在构建 SFT 数据集时会从多个来源拼凑数据**——**不同来源之间很容易出现重复或近似重复**——尤其是当多个数据集都用同一个教师模型生成的时候。

### 极其实用的数据验证策略

> **微调一个大模型之前，先在一个更小、更便宜的模型上测试你的微调数据**。

**逻辑**：

- 如果数据能有效提升小模型的准确率——**说明你的数据质量是好的，微调大模型时大概率也会有效**
- 如果小模型上效果不好——**那问题大概率出在数据上，而不是模型上**

**在小模型上跑一轮微调的成本可能只有大模型的 1/10 甚至更低**。**如果小模型学不到东西，换个更大的模型也不会有本质改善**。

## Hugging Face 的开源工程方法论

**SmallLM 系列的实践**：

- 观察到用公开 SFT 数据集微调的模型表现不如使用专有指令数据集的模型
- 创建合成数据集来弥补差距
- 在 17 亿参数规模上进行**一系列数据消融实验**确定最优数据混合配比

**注意关键词**：**"一系列数据消融实验" + "大量精选和过滤"**——**不是随便拿 40 万条数据就用，而是生成大量候选数据之后通过系统化的消融测试来决定留什么、去什么、按什么比例混合**。

### 具体的混合方式

每个数据子集**都有明确的目的**：

- **10 万条 OpenHermes** → 保持和提升通用基准测试成绩
- **5 万条 MetaMath QA** → 提升数学和推理能力
- **NuminaMath** → 提升数学难题的表现
- **编程数据集** → 提升代码能力
- **3 万条 SystemChats** → 让模型支持多种系统提示词格式

**这不是"什么都加一点"——而是每个组件都经过消融测试，确认了它对哪些能力维度有贡献，贡献多大**。

### 不同大小模型需要不同数据

**对更小的模型（135M / 360M）**：

- 用了一个**专门的子集**——**只保留对话更简洁的样本**
- **去掉了高难度的数学数据集**

**这恰好呼应了学术界的一个重要发现**：

> **最好的数据不是"客观上最优质"的数据——而是"跟你的目标模型最匹配"的数据**。

**一条对于大模型来说很有价值的高难度数学推理，对于一个一亿多参数的小模型来说可能完全学不到——反而是噪声**。

### 一条容易误解的核心原则

**Hugging Face 强调**：

> **SFT 不会教给模型"新的事实"——它教的是"新的行为模式"**。

**模型在预训练阶段已经有了关于世界的知识——SFT 教的是"如何利用这些知识做一个有用的助手"**。

**这条原则直接影响你怎么设计数据**：

- **如果你的目的是让模型掌握"新的领域知识"——SFT 可能不是最佳路径**
- **检索增强生成 或 继续预训练 可能更合适**
- **SFT 最擅长的是改变模型的输出格式和风格，让模型学会遵循特定的工作流程，教模型在什么情况下该怎么回应**

详见 `KNOWLEDGE/training/continue-pretrain-vs-finetune/`。

## 综合：6 条可直接落地的操作要点

1. **从 50-100 条高质量数据开始**——先跑一版微调，用明确的评估指标验证效果。**不要一上来就规划几万条的标注预算**。如果在小数据量上就看不到方向性的改善——**加数据也大概率救不了**。**而且可以先在更小的模型上验证数据质量**（Google 给的聪明做法）。

2. **格式一致性是很多人不当回事但实际影响巨大的因素**。每条训练样本的格式和推理时完全一致、每条都带上完整的系统提示词。**如果有多人标注**——先统一标准，**测量标注者间一致性**——**这个一致性就是模型的性能上限**。

3. **训练数据的分布必须和生产环境的输入分布对齐**。如果你的真实场景中只有 5% 的问题需要拒答——**那训练数据中拒答样本的比例也不要超过 5-10%**。**数据分布偏差是微调效果不好最常见也最隐蔽的原因**。

4. **做去重**。用精确匹配和模糊匹配至少过一遍，移除重复和近似重复样本。**特别是从多个来源拼凑数据集的时候——这一步不做，你可能在不知不觉中给模型引入了严重的分布偏差**。

5. **不同模型需要不同的数据**。Hugging Face 给小模型用了专门的精简版数据集——**这不是偷懒，而是工程上的最优选择**。如果你在微调一个小模型——**不要直接套用为大模型设计的数据集**。可以**用目标模型跑一遍困惑度，优先保留困惑度适中的样本**。

6. **把数据集当成一个"有明确目的的配方"——而不是直接堆一坨数据**。Hugging Face 的 SmallTalk 不是随便凑的 100 万条——**每个子集都有明确的目标**：这 5 万条提升数学、那 10 万条保持通用能力、这 3 万条增强系统提示词支持。**每一组数据都经过消融测试确认了贡献**。如果你有这个条件——**也应该对你的数据集做类似的结构化拆解和验证**。

## 一句话总结

**SFT 数据量的问题——不是一个"多少条"的技术问题，而是一个"需要对照评估反复调校"的工程问题**。

四家大厂的共识：**从少量高质量数据起步，用评估驱动迭代，在格式一致性、分布对齐、去重和模型匹配这些细节上下功夫**。**做好这些，你的数据效率会远超盲目堆量**。

## Open Questions

- **"标注者一致性即性能上限"** 这个论点——**有没有反例**？某些任务可能存在"一致性低但模型能学到深层结构"的场景吗？
- **小模型上验证数据质量** 这个 trick 在跨架构时是否还成立？比如用 Llama 验证后给 Qwen 微调——验证结论是否还可迁移？
- **6 条要点之间的优先级** 如果只能做 3 条，哪 3 条最重要？文章列了 6 条但**没排优先级**。
- **困惑度筛选样本** 是 Hugging Face 提到的方法——**具体困惑度阈值怎么定**？太低（模型已经会了）和太高（模型完全不懂）都不好——中间区间怎么找？
