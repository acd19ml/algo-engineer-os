# Q: 讲一次团队卡在共识失败、最后是怎么破局的

> 也适用于："讲一次你做组长的决策瞬间"、"讲一次项目方向需要重大调整的经历"、"讲一次失败的项目复盘"

## STAR

### Situation

七牛云 1024 实训营 ZeroOps 智能运维项目，2025 年 7-10 月。CEO 许式伟亲自立项，没有 PRD，团队 6 人从产品形态 0→1。

项目方向经历了**三次重大调整**——最初想做对话入口辅助 SRE 排障；然后调整为完全替代 SRE 的无人化系统；最后收敛到"变更场景的多版本灰度发布管理 + AI 根因下钻 + 简单自愈"。

每一次方向变更都触发原型图重画 / API 重设 / 数据表重新设计的连锁返工，**团队几周陷入"设计停摆"**——每个人都积极完成自己的工作，但产品完整形态没有共识。

### Task

我作为项目组长 + Agent 设计负责人，需要让这个 6 人团队对 MVP 形态达成一致，**同时还要兼顾 Agent 设计和每日协调推进**。

第一次 milestone 路演当场失败——5 位导师集中批评：

- 产品形态前端还是后端不清晰
- 选 Agent 框架没有调研过程
- 路演堆砌专业名词、听众听不懂
- 没思考为什么联调失败
- 演示文档准备不足、缺整体流程演示

我需要带团队从路演失败中恢复，并真正解决 MVP 形态共识问题。

### Action

1. **路演当场记录所有导师批评 + 团队复盘**——产出一份完整的复盘文档，列出 5 类核心问题 + 5 类经验教训
2. **推动 MVP 共识清单**——明确"先思考 MVP 形态再做模块、把功能串起来再细化"。具体是收敛到只做变更场景（不做线上场景）、故障治愈只做重启 + 回滚（不做复杂治愈）、时序检测只做 STL + 百分位数（不做复杂算法）
3. **milestone 路演时直接请 CEO 介入裁决**——CEO 在汇报后基于他的独到见解**拍板**了产品功能边界 + 产品形态 + 前端操作逻辑，同时辅助验收了数据表 / API / 架构设计
4. **CEO 拍板后立即推动新版完整设计文档**——重新走完 6 步法（产品定位 → 原型图 → API → 数据表 → 模块拆分 → 模块详细设计），让团队在新形态下重建共识
5. **复盘文档同步给 mentor 和 CEO**，记录失败原因 + 经验教训 + 下一步行动清单

### Result

- 团队在 CEO 拍板后才真正达成 MVP 形态共识，**重新走通了完整的 6 步法**
- 项目完整设计文档产出（原型图 + 5 大类 API + 数据表 + 6 层架构 + 模块详细设计）
- 最终 milestone 路演演示（详见 `qiniu-roadshow-emergency-rescue.md`）超出所有人预期
- **项目获七牛云第三届 1024 创作节优胜奖**
- ⚠️ **坦诚的局限**：项目停留在 MVP / Mock 阶段未上线，Agent 模块独立跑通但端到端主链路未完整接入。这件事教会我的是：上线状态不是组长成功的唯一标准——**让团队走出共识失败的方式本身是更重要的能力信号**

## 可强调的能力信号

- **组长决策能力**——当团队卡死时主动升级到 CEO 这层求助而不是自己硬扛
- **复盘能力**——路演当场失败立刻产出结构化复盘文档，把负面事件转化为团队共识工具
- **诚实**——能区分"我做对了什么"和"我作为组长应该做但没做的事"
- **学习能力**——从这件事提炼出"技术提案缺位"的根本原因（详见下方）

## 我从这件事学到的最具体的一件事（最有杀伤力的面试金句）

> 我们项目最大的教训是**技术提案缺位**——三个月里产品形态变了三次，每次方向变更其实应该先写 1-2 页技术提案让团队论证"为什么变 / 不变会怎样 / 替代方案是什么"再动。我们直接跳到架构设计层反复返工，每次都要重新画原型图、改 API、调数据表，**6 步法是走完的、走得也是稳的，但底层方向没共识**。CEO 拍板的本质是替我们补了那个缺位的"技术提案 + 共识"环节。
>
> 如果重做，组长（我）应该在每次重大方向变更前主导一份**技术提案**（问题 / 替代方案 / 选择 / 共识范围），让团队在方向层达成一致再动架构设计——而不是默认"方向是上面给的、我们只管实现"。

## 时代背景补充（防止"项目过时"的面试 framing）

> 项目时间是 2025 年 7 月到 10 月。**当时市面上还没有开源的 agent loop 系统**，上下文工程刚兴起，最火的开源 agent 项目是基于 DAG 的 DeepResearch。Coding Agent / Harness / Skills 这些概念都是项目结束后涌现的。所以**评判我们的产品选型必须放在 2025 年 7 月的技术坐标里看**——CEO 拍板的"灰度发布管理 + 多智能体根因下钻"在当时是合理的 MVP 选择。

## 最常见的深挖问题（提前准备）

1. **"CEO 拍板后你具体改了什么"** → 答：重新走 6 步法 / 重做原型图 / 改 API / 调数据表 / 同步复盘文档
2. **"团队为什么卡这么久"** → 答：6 人对产品边界没共识 + 没有"一锤定音的人" + 我作为组长没意识到该升级到上层求助
3. **"如果再来一次你会怎么做"** → 答：(a) 早一周升级到 CEO / mentor 求助；(b) 每次重大变更前写技术提案；(c) 让每个组员每天都有产出不要卡死在一个点
4. **"最后上线了吗 / 效果怎么样"** → 答：**MVP 停在 Mock 阶段未上线，Agent 模块独立跑通但端到端主链路未接入**。项目获七牛云第三届 1024 创作节优胜奖
