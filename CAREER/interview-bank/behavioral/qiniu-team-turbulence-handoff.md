# Q: 讲一次你顶住人员流失 / 项目压力的经历

> 也适用于："讲一次你的角色被迫扩大的经历"、"讲一次你做出技术取舍的决策"、"讲一次失败 + 反思"

## STAR

### Situation

七牛云 ZeroOps 项目中后期。这时项目已经经历了第一次路演失败 + CEO 拍板 MVP 形态（详见 `qiniu-ceo-pivot-decision.md`），团队正在按新形态重新推进。

**问题陆续叠加**：

1. **有人离职**——团队减员
2. **团队整体信心不足**——产品形态反复调整后，部分组员对项目能否成功产生怀疑
3. **路演压力同时压上来**——下一次 milestone 路演 deadline 临近
4. **项目分工本来就模糊**——后端同学干了全栈，我原本干算法和后端，**没有明确的前后端分界**

### Task

我作为组长 + Agent 设计 + 后端，需要在人员减少 + 团队信心不足的情况下**不让进度断档**，同时确保下一次路演能拿出可演示的成果。

**具体的角色扩大**：

- 接手所有前后端联调工作（接口对齐 / 数据格式对齐 / mock 数据格式对齐）
- 继续推进 Agent 设计 + 团队协调 + 每日晨会 + 每日领导汇报 + 公司周报 + milestone 路演准备

### Action

#### 取舍：先放弃算法效果调优

我没有去做"全部接住"——这做不到。**主动取舍把算法效果调优搁置**，只保证算法 MVP 状态"能跑"（STL 分解 + 百分位数检测），把节省下来的时间挪到联调和团队协调上。

#### 具体动作

1. **接手前后端联调**——每天与后端联调接口、与前端联调展示、与算法联调 mock 数据格式
2. **每日晨会保持节奏**——确保每个组员每天都有明确的小产出，避免团队再次卡死
3. **每日领导汇报 + 周报**——把进度透明化给 mentor 和领导，**主动 expose 问题而不是藏着**
4. **路演前加班补救**——详见 `qiniu-roadshow-emergency-rescue.md`

### Result

- **保住了 milestone 路演**——加班补救让最终路演呈现超出团队所有人的预期
- **算法效果调优被搁置**——我接受了这个代价，**算法停在 MVP 状态没做调优**。事后看，这是对的取舍——路演成败的关键是端到端能跑 + 演示效果，不是算法精度
- **团队没有再次崩溃**——每日晨会保持了节奏，组员们都有产出
- ⚠️ **坦诚的局限**：Agent 模块独立跑通了但端到端主链路因为联调持续变更最终没有完整接入。这是 MVP / Mock 系统始终未能突破的边界

## 反事实复盘（如果重做我会怎么改）

这是这段经历最值钱的部分——三件事：

1. **让每个同学每天都有产出**——不能让团队卡在一个点不动。组长的核心职责之一是"保证每个人都在前进"，而不是"自己冲在前面替组员扛任务"
2. **尽早找领导帮助**——可以多寻求帮助，不要试图一个人扛。组长不是"什么都要自己解决"，是"知道什么时候该把事情升级"
3. **意见不统一时立刻开会沟通达成一致**——不要拖。拖的代价比开会的代价高得多

## 隐藏的取舍逻辑（让面试官看到你的工程判断力）

> 当时还有一个更深的取舍是 **agent 范式选型**——项目初期我试过编排 agent loop / 类 coding agent 式，但调试困难。在团队信心已经不足 + 非算法成员要能接手的约束下，**主动从 agent loop 退到 Dify 工作流 + ReAct 模式**。这是用工作流可调试性换 agent loop 灵活性的合理 ROI 取舍——**今天 coding agent 生态成熟了我可能选不同，但当时 2025 年 7 月那个技术坐标下这是对的**。
>
> 这跟"放弃算法效果调优"是同一类决策——**在压力下识别什么是必须 ship 的、什么是可以延后的**，是组长的核心能力。

## 可强调的能力信号

- **取舍能力**——主动放弃算法调优、把时间投到联调，是有意识的决策不是被动反应
- **责任心 / Ownership**——人员流失 + 信心不足时主动接手额外职责
- **透明沟通**——主动 expose 问题给 mentor 和领导而不是藏着
- **学到了"组长该做什么"的具体边界**——不是冲在前面扛、是保证团队前进

## 最常见的深挖问题（提前准备）

1. **"你为什么放弃算法效果调优"** → 答：路演成败的关键是端到端能跑 + 演示效果，不是算法精度。MVP 阶段算法只要"能跑"就够，调优要等产品形态稳定后再做
2. **"如果你不接手联调，会有什么后果"** → 答：路演大概率失败 / 项目可能停摆 / 团队信心进一步崩
3. **"接手联调时最难的是什么"** → 答：接口对齐——前端期望的数据结构、后端实际返回的、Mock 系统能模拟的、Agent 工具调用需要的，四方对齐成本极高。如果重做我会在更早期统一接口契约
4. **"组长 + 算法双角色你怎么平衡"** → 答：实际比例算法 30% / 协调 70%，**这不是个好状态**。如果重做我会主动把联调任务在团队内重新分发，不是自己扛
5. **"你具体写代码 vs 协调的时间占比"** → 答：30% / 70%。后期甚至更倾向协调。**这意味着我作为'AI 算法工程师'的算法产出可能比简历看起来少**——这是诚实的局限
